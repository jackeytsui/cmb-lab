### Pitfall 1: Sentry Leaks Student PII Through URLs, Breadcrumbs, and Replay

**What goes wrong:**
Sentry captures student data by default: full request URLs (e.g., `/students/clerk_abc123/grades`), query strings (`?email=student@example.com`), console log breadcrumbs containing student submissions, and Session Replay network bodies with grading responses. Chinese text submissions containing student names appear in error reports accessible to anyone with Sentry dashboard access.

**Why it happens:**
Sentry's default behavior captures request URLs, query parameters, and console logs as breadcrumbs. The existing LMS has 50 API routes with student IDs in paths. Console logs during development often include `console.log('grading response:', response)` with full student data. Developers enable `sendDefaultPii: true` for debugging without understanding the scope. Session Replay captures XHR/fetch request and response bodies if `networkDetailAllowUrls` is configured.

**How to avoid:**
```typescript
// sentry.client.config.ts
Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  sendDefaultPii: false, // NEVER enable in production

  beforeSend(event) {
    // Scrub student IDs from URLs
    if (event.request?.url) {
      event.request.url = event.request.url
        .replace(/\/students\/[^/]+/g, '/students/[REDACTED]')
        .replace(/\/users\/[^/]+/g, '/users/[REDACTED]');
    }
    // Scrub query params
    if (event.request?.query_string) {
      event.request.query_string = '[REDACTED]';
    }
    // Remove breadcrumbs containing Chinese text (likely student submissions)
    if (event.breadcrumbs) {
      event.breadcrumbs = event.breadcrumbs.filter(b =>
        !/[\u4e00-\u9fff]/.test(JSON.stringify(b.data || ''))
      );
    }
    return event;
  },

  // Disable console breadcrumbs entirely
  integrations: [
    Sentry.breadcrumbsIntegration({ console: false }),
  ],

  // Low sample rates for a small user base
  tracesSampleRate: 0.2,
  replaysSessionSampleRate: 0,    // No passive recording
  replaysOnErrorSampleRate: 0.5,  // Only on errors
});
```

**Warning signs:**
- Student names or Chinese text visible in Sentry event data
- Clerk user IDs appearing in Sentry breadcrumbs or URLs
- Console breadcrumbs containing grading responses or chat messages
- Session Replay showing student dashboard content

**Phase to address:** Error Monitoring (Sentry) phase -- configure scrubbing before any events are sent

**Sources:**
- [Sentry Data Collected for Next.js](https://docs.sentry.io/platforms/javascript/guides/nextjs/data-management/data-collected/) - HIGH confidence
- [Sentry Sensitive Data Scrubbing](https://docs.sentry.io/platforms/javascript/data-management/sensitive-data/) - HIGH confidence

### Pitfall 2: Sentry Over-Alerting Buries Real Errors in Noise

**What goes wrong:**
First week after Sentry integration, team receives 500+ alerts per day. Most are network timeouts from students on mobile, expected 401s from expired Clerk sessions, and browser extension errors. Team starts ignoring all Sentry notifications. A real grading API failure goes unnoticed for 3 days because it was buried in noise.

**Why it happens:**
Default Sentry setup captures everything -- including expected errors, client-side network failures, third-party script errors, and browser-specific quirks. With 50 API routes and no existing error handling strategy, the signal-to-noise ratio is abysmal. Small teams (2-3 coaches) cannot triage hundreds of alerts.

**How to avoid:**
```typescript
// 1. Filter known noise in beforeSend
beforeSend(event, hint) {
  const error = hint.originalException;

  // Ignore network errors (students on bad connections)
  if (error?.message?.match(/fetch failed|network error|timeout/i)) {
    return null;
  }

  // Ignore expected auth errors
  if (error?.message?.match(/UNAUTHENTICATED|session expired/i)) {
    return null;
  }

  // Ignore browser extension errors
  if (event.exception?.values?.[0]?.stacktrace?.frames?.some(
    f => f.filename?.includes('extension://')
  )) {
    return null;
  }

  return event;
},

// 2. Set up Sentry alert rules (not just defaults)
// - Only alert on NEW issues (not every occurrence)
// - Only alert if issue occurs > 5 times in 1 hour
// - Critical alert only for API routes matching /api/grade/* or /api/chat/*
// - Weekly digest for everything else

// 3. Start with error tracking only, add performance later
tracesSampleRate: 0,  // Start at 0, enable after baseline
```

**Warning signs:**
- More than 20 unique Sentry issues in first week
- Team members muting Sentry notification channels
- Alert fatigue -- "I don't check Sentry anymore"
- Real production bugs discovered by users, not monitoring

**Phase to address:** Error Monitoring (Sentry) phase -- configure filtering and alert rules before going live

### Pitfall 3: E2E Tests Become Permanently Flaky Due to Clerk Auth and Mux Video

**What goes wrong:**
E2E test for "student completes lesson" flow: sign in via Clerk, navigate to lesson, wait for Mux video to load, interact at checkpoint, submit Chinese text for grading. This test passes locally, fails in CI 40% of the time. Mux video takes variable time to buffer. Clerk session sometimes expires mid-test. AI grading webhook returns different results for same input. Test suite becomes "red by default" and developers stop trusting it.

**Why it happens:**
The CantoMando LMS combines multiple external service dependencies in a single user flow: Clerk auth, Mux video streaming, n8n webhook AI grading. Each adds latency variance. Video elements have non-deterministic load times. Clerk's testing mode requires specific setup. Chinese text input may trigger IME-related issues in headless browsers.

**How to avoid:**
```typescript
// 1. Use Clerk's testing helpers -- authenticate once, reuse session
// playwright.config.ts
import { clerkSetup } from '@clerk/testing/playwright';

export default defineConfig({
  globalSetup: async () => {
    await clerkSetup(); // Sets up test user session
  },
  use: {
    storageState: '.clerk-auth-state.json', // Reuse across tests
  },
});

// 2. Mock external services in E2E tests
test('student completes interaction', async ({ page }) => {
  // Mock Mux video -- don't depend on actual video loading
  await page.route('**/stream.mux.com/**', route =>
    route.fulfill({ status: 200, body: '' })
  );

  // Mock n8n grading webhook -- deterministic response
  await page.route('**/webhook/**grade**', route =>
    route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify({ correct: true, feedback: 'Good job!' }),
    })
  );

  // Use test-specific video player state instead of real video
  await page.evaluate(() => {
    window.__TEST_SKIP_VIDEO__ = true;
  });
});

// 3. Separate test categories
// - Unit: grading logic, progress calculations (fast, reliable)
// - Integration: API routes with mocked DB (medium speed)
// - E2E: Only critical happy paths with mocked externals (slow, few)
// - Smoke: Real services, runs nightly, allowed to be flaky
```

**Warning signs:**
- Same test passes locally but fails in CI more than 10% of the time
- Test times exceed 30 seconds per test
- Tests depend on real external service responses
- Chinese text input tests fail only in headless mode
- Team starts adding `test.skip()` or `test.fixme()` annotations

**Phase to address:** E2E Testing phase -- establish mocking strategy and test categories before writing first test

**Sources:**
- [Clerk Testing with Playwright](https://clerk.com/docs/guides/development/testing/playwright/overview) - HIGH confidence
- [Clerk Playwright Example Repo](https://github.com/clerk/clerk-playwright-nextjs) - HIGH confidence

### Pitfall 4: PWA Service Worker Caches Stale LMS Content After Updates

**What goes wrong:**
Coach updates a lesson's interaction timestamps or corrects a grading rubric. Students on PWA still see the old version because the service worker serves cached pages. Student submits answer at wrong checkpoint, gets marked incorrect. Clearing cache requires students to manually uninstall the PWA -- which most won't know how to do.

**Why it happens:**
Service workers intercept all network requests and can serve cached responses. The `stale-while-revalidate` strategy (default in most PWA libraries) serves the cached version immediately and updates in the background. But if the student completes the lesson before the background update finishes, they experience the old version. Next.js App Router's dynamic routes and server components make caching even more complex -- the service worker doesn't know which routes are static vs. dynamic.

**How to avoid:**
```typescript
// 1. NEVER cache lesson content or API routes
// service-worker.ts
const CACHE_STRATEGIES = {
  // Cache static assets aggressively
  static: /\.(js|css|png|jpg|svg|woff2?)$/,

  // NEVER cache these -- always network-first
  neverCache: [
    /\/api\//,           // All API routes
    /\/lessons\//,       // Lesson content (changes frequently)
    /\/courses\//,       // Course structure
    /\/dashboard/,       // Student progress
  ],

  // Cache shell only (navigation)
  networkFirst: [
    /\/$/,               // Home page
    /\/settings/,        // Settings page
  ],
};

// 2. Implement update notification
// When new SW is available, prompt user to refresh
self.addEventListener('install', () => self.skipWaiting());
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then(keys =>
      Promise.all(keys.map(key => caches.delete(key)))
    )
  );
  self.clients.claim();
});

// 3. Version your SW and force update checks
// In your app layout:
useEffect(() => {
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.ready.then(reg => {
      // Check for updates every 5 minutes
      setInterval(() => reg.update(), 5 * 60 * 1000);
    });
  }
}, []);
```

**Warning signs:**
- Students report seeing old lesson content after coach updates
- "I answered correctly but it said wrong" complaints
- Cache storage growing unbounded on student devices
- Different students see different versions of the same lesson

**Phase to address:** PWA phase -- define caching strategy before writing any service worker code

**Sources:**
- [Next.js PWA Stale Data Discussion](https://github.com/vercel/next.js/discussions/52024) - HIGH confidence
- [next-pwa-pack for App Router](https://hackernoon.com/how-i-made-nextjs-and-pwa-finally-get-along-with-next-pwa-pack) - MEDIUM confidence
- [Service Worker Unregistration Issues](https://github.com/vercel/next.js/discussions/32402) - HIGH confidence

### Pitfall 5: In-Memory Rate Limiting Fails Silently in Serverless

**What goes wrong:**
Rate limiting is implemented using an in-memory Map (or `express-rate-limit` with default store). Works perfectly in development. In production, each serverless function invocation gets a fresh memory space. A bot can hit the grading API 1000 times because each request goes to a different function instance, and no instance knows about the others. The n8n webhook gets overwhelmed, grading breaks for all students.

**Why it happens:**
Serverless functions are stateless -- when the function runs, the rate limit counter starts at zero. With Vercel's auto-scaling, there could be 50 instances running simultaneously, each with its own empty rate limit map. This is the #1 rate limiting mistake in serverless architectures.

**How to avoid:**
```typescript
// Use Upstash Redis for distributed rate limiting
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const redis = new Redis({
  url: process.env.UPSTASH_REDIS_REST_URL!,
  token: process.env.UPSTASH_REDIS_REST_TOKEN!,
});

// Different limits for different endpoints
const gradingLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(10, '1m'), // 10 per minute
  prefix: 'ratelimit:grade',
});

const chatLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(30, '1m'), // 30 per minute
  prefix: 'ratelimit:chat',
});

// Apply in middleware or API route
export async function POST(req: Request) {
  const userId = auth().userId; // Clerk user ID
  const { success, limit, remaining, reset } = await gradingLimiter.limit(userId);

  if (!success) {
    return Response.json(
      { error: 'Too many requests. Please wait before submitting again.' },
      {
        status: 429,
        headers: {
          'X-RateLimit-Limit': limit.toString(),
          'X-RateLimit-Remaining': remaining.toString(),
          'X-RateLimit-Reset': reset.toString(),
          'Retry-After': Math.ceil((reset - Date.now()) / 1000).toString(),
        },
      }
    );
  }
  // ... handle request
}
```

**Warning signs:**
- Rate limit tests pass locally but bots still get through in production
- Using `new Map()` or `express-rate-limit` without external store
- n8n webhook receives bursts of identical requests
- No `429` responses in production logs despite known abuse

**Phase to address:** Rate Limiting phase -- must use external store from the start, no "upgrade later" path

**Sources:**
- [Upstash Rate Limiting for Next.js](https://upstash.com/blog/nextjs-ratelimiting) - HIGH confidence
- [Next.js Rate Limiting Discussion](https://github.com/vercel/next.js/discussions/12134) - HIGH confidence
- [API Rate Limiting Best Practices 2025](https://dev.to/zuplo/10-best-practices-for-api-rate-limiting-in-2025-358n) - MEDIUM confidence

### Pitfall 6: CI/CD Database Migrations Corrupt Production Data

**What goes wrong:**
Developer creates a Drizzle migration that renames a column. CI/CD pipeline runs `drizzle-kit push` against production Neon database during deploy. The running application still references the old column name. All API routes that query that column start returning 500 errors. With Neon's serverless auto-suspend, there's no easy "rollback the last 5 minutes" option.

**Why it happens:**
Teams couple database migrations to application deployment. The migration runs before the new code is live, creating a window where old code hits new schema. Or worse, `drizzle-kit push` is used in production instead of versioned migrations -- `push` makes destructive changes without confirmation. Neon's branch-based architecture could help but is often not utilized.

**How to avoid:**
```bash
# 1. NEVER use drizzle-kit push in production
# Development only: drizzle-kit push
# Production: drizzle-kit generate + drizzle-kit migrate

# 2. Run migrations as separate CI step BEFORE deploy
# .github/workflows/deploy.yml
jobs:
  migrate:
    runs-on: ubuntu-latest
    steps:
      - name: Run migrations
        run: npx drizzle-kit migrate
        env:
          DATABASE_URL: ${{ secrets.NEON_DATABASE_URL }}

  deploy:
    needs: migrate  # Only deploy after migration succeeds
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Vercel
        run: vercel deploy --prod

# 3. Use Neon branching for preview deployments
# Each PR gets its own database branch
# .github/workflows/preview.yml
- name: Create Neon branch
  uses: neondatabase/create-branch-action@v5
  with:
    project_id: ${{ secrets.NEON_PROJECT_ID }}
    branch_name: preview/${{ github.event.pull_request.number }}
    api_key: ${{ secrets.NEON_API_KEY }}

# 4. Make migrations backward-compatible
# BAD: Rename column (breaks old code)
# GOOD: Add new column -> deploy new code -> backfill -> drop old column
```

**Warning signs:**
- Using `drizzle-kit push` in any environment other than local development
- Migrations and deploys in a single step with no separation
- No preview/staging database for testing migrations
- Column renames or drops without a multi-step migration plan
- No Neon branching strategy for feature branches

**Phase to address:** CI/CD phase -- establish migration workflow before first automated deploy

**Sources:**
- [Database Migrations in CI/CD](https://stonetusker.com/automating-database-migrations-in-your-ci-cd-pipeline/) - MEDIUM confidence
- [Database DevOps Git Strategy](https://www.harness.io/blog/how-git-strategy-can-break-your-database-pipeline) - MEDIUM confidence
- [Neon Database Branching](https://neon.com/docs/introduction/branching) - HIGH confidence

### Pitfall 7: PDF Certificate Generation Crashes Serverless Functions

**What goes wrong:**
Student completes a course and requests a certificate. The API route uses Puppeteer to render an HTML template to PDF. The serverless function either times out (default 10s on Vercel) or exceeds the 250MB bundle size limit because Puppeteer bundles a full Chromium binary. Certificate generation fails silently, student sees a loading spinner forever.

**Why it happens:**
Puppeteer + headless Chrome is the most common PDF generation approach but is fundamentally incompatible with serverless constraints. The Chromium binary alone is ~130MB compressed. Cold starts add 3-5 seconds before any rendering begins. Complex certificates with custom Chinese fonts, images, and layout can take 5-10 seconds to render.

**How to avoid:**
```typescript
// Option A: Use lightweight PDF library (no browser needed)
// @react-pdf/renderer generates PDFs without Puppeteer
import { renderToBuffer } from '@react-pdf/renderer';
import { CertificateTemplate } from './CertificateTemplate';

export async function POST(req: Request) {
  const { studentName, courseName, completionDate } = await req.json();

  const buffer = await renderToBuffer(
    <CertificateTemplate
      studentName={studentName}
      courseName={courseName}
      date={completionDate}
    />
  );

  return new Response(buffer, {
    headers: {
      'Content-Type': 'application/pdf',
      'Content-Disposition': `attachment; filename="certificate-${courseName}.pdf"`,
    },
  });
}

// Option B: Use @sparticuz/chromium for serverless Puppeteer
// (if you need pixel-perfect HTML rendering)
import chromium from '@sparticuz/chromium';
import puppeteer from 'puppeteer-core';

// Increase function timeout
export const maxDuration = 30; // seconds (Vercel Pro)

// Option C: Offload to background job
// Generate async, email/notify when ready
import { inngest } from '@/lib/inngest';

export async function POST(req: Request) {
  await inngest.send({
    name: 'certificate/generate',
    data: { studentId, courseId },
  });
  return Response.json({ status: 'generating', message: 'Certificate will be ready shortly.' });
}
```

**Warning signs:**
- `puppeteer` or `chrome-aws-lambda` in dependencies
- Serverless function bundle size approaching 100MB+
- Certificate generation taking more than 5 seconds
- Timeout errors only in production, works locally
- Custom Chinese fonts not rendering in generated PDFs

**Phase to address:** Certificates phase -- choose generation approach before implementation

**Sources:**
- [Vercel 250MB Function Limit](https://vercel.com/kb/guide/troubleshooting-function-250mb-limit) - HIGH confidence
- [Solving Next.js Timeouts](https://www.inngest.com/blog/how-to-solve-nextjs-timeouts) - HIGH confidence
- [PDF Generation Styling in Serverless](https://github.com/vercel/next.js/discussions/61243) - MEDIUM confidence

### Pitfall 8: Notification Fatigue Causes Students to Disable All Notifications

**What goes wrong:**
In-app notification system sends alerts for: coach feedback received, new lesson available, grading complete, chatbot message, course announcement, system maintenance. Students get 10-15 notifications per study session. They mute the notification bell or ignore it entirely. When a critical notification arrives (e.g., "Coach reviewed your audio submission"), they miss it.

**Why it happens:**
Every feature team adds "and send a notification" without a holistic notification strategy. There's no distinction between urgent, informational, and ambient notifications. No frequency caps. No user preference controls. The notification system becomes a dumping ground.

**How to avoid:**
```typescript
// 1. Define notification tiers with clear rules
const NOTIFICATION_TIERS = {
  urgent: {
    // Student action required, time-sensitive
    examples: ['coach_feedback_received', 'submission_rejected'],
    delivery: ['in_app', 'email'],
    frequency: 'immediate',
    canMute: false,
  },
  informational: {
    // Useful context, not time-sensitive
    examples: ['lesson_unlocked', 'grading_complete', 'course_announcement'],
    delivery: ['in_app'],
    frequency: 'batched_hourly',
    canMute: true,
  },
  ambient: {
    // Nice to know, easily ignored
    examples: ['weekly_progress_summary', 'new_knowledge_base_entry'],
    delivery: ['in_app'],
    frequency: 'batched_daily',
    canMute: true,
  },
};

// 2. Implement frequency caps
const MAX_NOTIFICATIONS_PER_HOUR = 5;
const MAX_NOTIFICATIONS_PER_DAY = 20;

// 3. User preferences from day one
// Settings > Notifications
// - Coach feedback: Always / In-app only / Off
// - Lesson updates: Always / In-app only / Off
// - Progress summaries: Weekly / Monthly / Off

// 4. Batch non-urgent notifications
// Instead of 5 individual "grading complete" notifications:
// "3 interactions graded in Lesson 5 -- see results"
```

**Warning signs:**
- Notification count badge regularly shows 20+ unread
- Students report "I didn't see the coach's feedback"
- No notification preferences in user settings
- Every feature adds notifications without coordination
- Engagement with notifications drops below 10% click rate

**Phase to address:** In-App Notifications phase -- define notification tiers and frequency caps before building UI

### Pitfall 9: Full-Text Search Returns Irrelevant Results for Chinese Content

**What goes wrong:**
Student searches for "tone change rules" and gets results about "changing course settings." Or searches in Chinese characters but gets no results because the search index only tokenized English text. Pinyin searches like "shengdiao" return nothing because content was indexed as Chinese characters only.

**Why it happens:**
PostgreSQL's built-in `tsvector` full-text search uses language-specific dictionaries. The default English dictionary can't segment Chinese text (no spaces between words). Mixed Chinese/English content creates poor search quality. Pinyin romanization is a completely separate text representation that needs explicit handling.

**How to avoid:**
```sql
-- 1. Use pg_search (BM25) on Neon instead of tsvector
-- Neon now includes pg_search extension (ParadeDB)
CREATE INDEX idx_lessons_search ON lessons
USING bm25 (title, description, content)
WITH (text_fields = '{
  "title": {"tokenizer": {"type": "chinese_compatible"}},
  "description": {"tokenizer": {"type": "chinese_compatible"}},
  "content": {"tokenizer": {"type": "chinese_compatible"}}
}');

-- 2. If using tsvector, create separate indexes
-- English index for English text
-- Unaccented/simple index for Chinese (character-level matching)
CREATE INDEX idx_lessons_en ON lessons USING gin(
  to_tsvector('english', title || ' ' || description)
);
-- Simple config treats each character as a token (works for Chinese)
CREATE INDEX idx_lessons_zh ON lessons USING gin(
  to_tsvector('simple', title || ' ' || description)
);
```

```typescript
// 3. Store Pinyin alongside Chinese for searchability
const lessonSearch = pgTable('lesson_search', {
  lessonId: uuid('lesson_id').references(() => lessons.id),
  titleEn: text('title_en'),           // "Tone Change Rules"
  titleZh: text('title_zh'),           // "声调变化规则"
  titlePinyin: text('title_pinyin'),   // "shengdiao bianhua guize"
  titleJyutping: text('title_jyutping'), // "sing1diu6 bin3faa3 kwai1zak1"
});

// 4. Search across all representations
const searchResults = await db.execute(sql`
  SELECT * FROM lessons
  WHERE title ILIKE ${`%${query}%`}
     OR title_pinyin ILIKE ${`%${query}%`}
     OR title_jyutping ILIKE ${`%${query}%`}
  ORDER BY
    CASE WHEN title ILIKE ${`%${query}%`} THEN 1
         WHEN title_pinyin ILIKE ${`%${query}%`} THEN 2
         ELSE 3
    END
  LIMIT 20
`);
```

**Warning signs:**
- Chinese character searches return zero results
- Pinyin/Jyutping searches return nothing
- Search returns irrelevant English matches for Chinese queries
- Search works perfectly in English but fails for Chinese

**Phase to address:** Search phase -- validate Chinese search tokenization early in implementation

**Sources:**
- [Neon pg_search Extension](https://neon.com/docs/extensions/pg_search) - HIGH confidence
- [PostgreSQL Full-Text Search](https://www.postgresql.org/docs/current/textsearch.html) - HIGH confidence

### Pitfall 10: Analytics Dashboard Tracks Vanity Metrics That Don't Help Coaches

**What goes wrong:**
Analytics dashboard shows: total logins (432), page views (12,847), average session duration (14 min). Coaches look at it and ask: "Which students are stuck? Which lessons have the hardest interactions? Are students actually improving at Chinese?" The dashboard can't answer any of these questions. It becomes a feature nobody uses.

**Why it happens:**
LMS analytics that track consumption metrics (logins, page views, time spent) instead of learning metrics (completion rates per interaction, retry counts, time-to-mastery) provide no actionable insight. In education, "95% completion rate" can mean students clicked through without learning anything. The metrics that matter for language learning are fundamentally different from generic web analytics.

**How to avoid:**
```typescript
// Track learning-specific metrics, not web metrics
const MEANINGFUL_LMS_METRICS = {
  // Per-student
  interactionPassRate: 'What % of interactions does student pass on first try?',
  averageRetries: 'How many attempts before passing each interaction?',
  stuckPoints: 'Which interactions took > 5 attempts?',
  progressVelocity: 'Lessons completed per week (trending up/down)?',
  lastActive: 'When did student last complete an interaction?',

  // Per-lesson (for coaches)
  lessonDropOff: 'What % of students start but don\'t finish this lesson?',
  hardestInteraction: 'Which interaction has the lowest first-try pass rate?',
  averageCompletionTime: 'How long does this lesson take to complete?',
  coachReviewBacklog: 'How many audio submissions are waiting for review?',

  // System-wide
  activeStudentsThisWeek: 'Students who completed >= 1 interaction',
  atRiskStudents: 'Students with no activity in 7+ days',
  coachResponseTime: 'Average time from audio submission to coach review',
};

// DO NOT track:
// - Page views (tells you nothing about learning)
// - Session duration (longer != better in education)
// - Raw login count (logging in != learning)
// - Total enrolled (vanity metric for coaches)
```

**Warning signs:**
- Dashboard shows numbers but coaches can't answer "who needs help?"
- Metrics go up but student outcomes don't improve
- Coaches never visit the analytics page after launch week
- No per-lesson or per-interaction breakdowns
- Dashboard requires expensive aggregation queries that slow down

**Phase to address:** Analytics phase -- define metrics with coaches before building any UI

**Sources:**
- [Why LMS Metrics Are Vanity Metrics](https://albimarketing.com/blog/the-half-life-of-skills-is-now-2-5-years-why-your-lms-metrics-are-a-vanity-metric-and-how-to-measure-real-competence/) - HIGH confidence
- [7 LMS Metrics You Should Track](https://www.docebo.com/learning-network/blog/lms-metrics/) - MEDIUM confidence

### Pitfall 11: Neon Cold Starts Make Search and Analytics Queries Feel Broken

**What goes wrong:**
Student opens search, types a query, waits 3-5 seconds for results. First analytics dashboard load takes 8 seconds. Users think the feature is broken. This only happens after periods of inactivity because Neon's compute auto-suspends after 5 minutes of no queries.

**Why it happens:**
Neon's serverless architecture auto-suspends compute to save costs. When a query arrives after suspension, the compute must wake up (500ms-3s). Search and analytics queries are often the first queries after idle periods because they're used less frequently than core lesson flows. Combined with complex aggregation queries, the total response time can exceed 5 seconds.

**How to avoid:**
```typescript
// 1. Keep compute warm for critical paths
// Use Neon's auto-suspend timeout configuration
// Set to 300s (5 min) minimum, or longer for active hours

// 2. Add loading states that feel intentional
const SearchResults = () => {
  const [results, setResults] = useState(null);
  const [isSearching, setIsSearching] = useState(false);

  // Show skeleton immediately, not a spinner
  if (isSearching) return <SearchSkeleton />;
  // Never show a blank page while waiting
};

// 3. Pre-warm with lightweight query on page load
// When student navigates to search page, fire a cheap query
useEffect(() => {
  // Warm the connection before user types
  fetch('/api/search/warm', { method: 'HEAD' });
}, []);

// 4. Cache analytics results (they don't need real-time)
// Refresh every 15 minutes, not on every page load
const ANALYTICS_CACHE_TTL = 15 * 60; // 15 minutes
```

**Warning signs:**
- First query after idle period takes 3-5x longer than subsequent queries
- Users report "search is slow" but only sometimes
- Analytics dashboard has inconsistent load times
- Neon dashboard shows frequent compute wake-ups

**Phase to address:** Search and Analytics phases -- implement connection warming and caching strategies

**Sources:**
- [Neon Cold Start Benchmarks](https://neon.com/docs/guides/benchmarking-latency) - HIGH confidence
- [Neon Auto-Suspend Configuration](https://neon.com/docs/introduction/auto-suspend) - HIGH confidence

### Pitfall 12: SSE/WebSocket Notifications Don't Work on Vercel

**What goes wrong:**
In-app notification system uses Server-Sent Events (SSE) for real-time updates. Works locally. In production on Vercel, SSE connections drop after 10-30 seconds because serverless functions have execution time limits. Notifications arrive late or not at all. Switching to WebSockets fails because Vercel doesn't support persistent WebSocket connections.

**Why it happens:**
Vercel's serverless architecture is fundamentally incompatible with persistent connections. Each function invocation has a maximum duration (10s free, 60s Pro with Fluid Compute). SSE and WebSockets require long-lived connections that exceed these limits.

**How to avoid:**
```typescript
// Option A: Polling with smart intervals (simplest, most reliable)
const useNotifications = () => {
  const [notifications, setNotifications] = useState([]);

  useEffect(() => {
    const poll = async () => {
      const res = await fetch('/api/notifications?since=' + lastChecked);
      const data = await res.json();
      if (data.length > 0) setNotifications(prev => [...data, ...prev]);
    };

    // Poll every 30 seconds when tab is active
    const interval = setInterval(poll, 30_000);
    // Also poll on tab focus (user comes back)
    const onFocus = () => poll();
    window.addEventListener('focus', onFocus);

    return () => {
      clearInterval(interval);
      window.removeEventListener('focus', onFocus);
    };
  }, []);
};

// Option B: Upstash Redis + polling (if you need < 10s latency)
// Store notifications in Redis, poll frequently
// Much cheaper than real-time infrastructure for small user base

// Option C: Third-party real-time service (only if truly needed)
// Pusher, Ably, or Supabase Realtime
// Adds cost and complexity -- not justified for 2-3 coaches + small student base
```

**Warning signs:**
- SSE or WebSocket code in API routes
- "Connection reset" errors in production but not locally
- Notifications work locally but are delayed/missing in production
- Investigating Pusher/Ably/Socket.io before trying polling

**Phase to address:** In-App Notifications phase -- use polling for initial implementation, upgrade only if needed

**Sources:**
- [Vercel Serverless Limitations for Real-Time](https://northflank.com/blog/vercel-backend-limitations) - HIGH confidence
- [Upstash Real-Time Notifications](https://upstash.com/blog/realtime-notifications) - MEDIUM confidence

### Pitfall 13: Sentry Bundle Size Bloats Client-Side JavaScript

**What goes wrong:**
Adding `@sentry/nextjs` increases the client bundle by 30-50KB gzipped. For students on mobile with slow connections (common for language learners in various countries), this adds noticeable load time.

**How to avoid:**
- Set `__SENTRY_DEBUG__` and `__SENTRY_TRACING__` to false via webpack DefinePlugin
- Set `disableLogger: true` in Sentry's Next.js config
- Start with error monitoring only (no performance tracing, no replay)
- Lazy-load replay integration only when needed
- Use `@next/bundle-analyzer` to verify impact

**Phase to address:** Error Monitoring (Sentry) phase

**Sources:**
- [Sentry Bundle Size Reduction Blog](https://blog.sentry.io/javascript-sdk-package-reduced/) - HIGH confidence
- [Sentry Build Options for Next.js](https://docs.sentry.io/platforms/javascript/guides/nextjs/configuration/build/) - HIGH confidence

### Pitfall 14: E2E Tests Don't Cover Chinese Text Input Edge Cases

**What goes wrong:**
Tests use English text for all inputs. In production, students type Chinese characters which trigger IME (Input Method Editor) composition events. The interaction input field doesn't handle `compositionstart`/`compositionend` events properly, causing premature submission or garbled text.

**How to avoid:**
- Include Chinese text input in at least one E2E test: `await page.type('#answer', '我很好')`
- Test IME composition events: `await page.keyboard.insertText('你好')`
- Test Pinyin input sequences that produce Chinese output
- Test mixed Chinese/English input

**Phase to address:** E2E Testing phase

---

### Pitfall 15: CI/CD Secrets Leak Through Preview Deployment Logs

**What goes wrong:**
GitHub Actions workflow logs expose environment variables. A developer adds `echo $DATABASE_URL` for debugging and commits it. Preview deployment logs show the Neon connection string. Or Clerk API keys appear in build output.

**How to avoid:**
- Use GitHub Actions secrets, never hardcode in workflow files
- Add `::add-mask::` for any dynamically generated secrets
- Review workflow files in PRs before merge
- Use separate API keys for preview vs. production environments
- Neon branch credentials should be scoped to that branch only

**Phase to address:** CI/CD phase

## Integration Gotchas

Common mistakes when adding v2.0 features to the existing CantoMando system.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Sentry + Clerk | Clerk user IDs logged as PII in Sentry events | Use Sentry's `setUser({ id: hashedId })` with a non-reversible hash, never raw Clerk ID |
| Sentry + Mux Player | Mux player errors flood Sentry (buffering, codec issues) | Filter Mux-related errors in `beforeSend`, or use Mux's own monitoring |
| E2E Tests + n8n webhooks | Tests depend on live n8n webhook for grading | Mock webhook responses in Playwright, test n8n integration separately |
| PWA + Clerk auth | Service worker caches auth-gated pages, serves to logged-out users | Never cache routes behind Clerk middleware, use network-first for auth routes |
| Notifications + existing email | Duplicate notifications: in-app AND email for same event | Single notification dispatch that routes to correct channels based on preferences |
| Search + Chinese fonts | Search results don't render Pinyin/Jyutping annotations | Apply same custom font rendering pipeline to search result snippets |
| Analytics + Neon connection pooling | Analytics queries hold connections too long, starving lesson queries | Use separate connection pool for analytics, or run analytics on read replica |
| Rate limiting + Clerk webhook | Rate limiter blocks Clerk's enrollment webhook | Whitelist webhook endpoints from rate limiting using shared secret verification |
| Certificates + custom fonts | Chinese characters render as boxes in generated PDF | Bundle font files with PDF generation, don't rely on system fonts |
| CI/CD + Drizzle migrations | `drizzle-kit push` used in CI instead of `drizzle-kit migrate` | Use versioned migrations in CI, reserve `push` for local development only |

## Performance Traps

Patterns that work during development but fail at scale.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Analytics queries scan full tables | Dashboard takes 10s+ to load | Pre-aggregate metrics on write, or use materialized views | >1000 students or >50k interactions |
| Notification polling every 5 seconds | Database overwhelmed by polling queries | Poll every 30s, use tab visibility API to pause when hidden | >100 concurrent students |
| Search without database indexes | Search takes 2-3 seconds per query | Add GIN/BM25 indexes on searchable columns | >500 lessons/entries |
| Certificate PDF generation in request/response cycle | Timeout errors, poor UX | Generate async, notify when ready | >10 concurrent certificate requests |
| Sentry capturing all transactions | Sentry bill spikes, bundle bloat | Start with `tracesSampleRate: 0`, increase gradually | Any production traffic |
| Full student progress recalculation on every page load | Slow dashboard for students with many courses | Cache progress, invalidate on interaction completion | >20 lessons per student |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Sentry PII exposure | Student personal data in third-party service | Configure `beforeSend` scrubbing, keep `sendDefaultPii: false` |
| Rate limiter identifies by IP only | Students behind shared WiFi (classroom) all get blocked | Identify by Clerk user ID + IP combo, with generous per-IP limits |
| Analytics endpoint exposes all student data | Coach dashboard leaks student PII | Aggregate-only endpoints for dashboards, individual data requires auth + role check |
| CI/CD secrets in workflow logs | Database credentials exposed in GitHub | Use `::add-mask::`, separate preview/prod credentials, never echo secrets |
| Certificate tampering | Students modify PDF certificates | Include verification URL/QR code linking to server-verified completion record |
| Preview deployments with prod data | Neon branch copied from prod contains real student data | Use anonymized seed data for preview branches, never fork prod |

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Sentry PII leak | HIGH | Audit all existing events, delete events with PII, update scrubbing rules, notify affected users if required by privacy policy |
| Alert fatigue | LOW | Configure alert rules in Sentry UI (no code change), set up issue grouping and deduplication |
| Flaky E2E tests | MEDIUM | Add mocking layer, reduce external dependencies, may need to rewrite tests |
| Stale PWA cache | HIGH | Deploy SW update that clears all caches, but must wait for users to reconnect; may need to add banner telling users to "refresh" |
| In-memory rate limiting | LOW | Replace Map with Upstash Redis, deploy new version (quick code change) |
| Migration corrupted prod | HIGH | Restore from Neon point-in-time recovery, but may lose recent data; create forward migration to fix |
| PDF generation timeout | LOW | Switch to lighter library or async generation; no data loss |
| Notification fatigue | MEDIUM | Add frequency caps and preferences UI; retroactively mark old notifications as read |
| Poor search quality | MEDIUM | Add proper indexes and tokenization; may need to rebuild search index |
| Vanity metrics dashboard | MEDIUM | Rebuild queries around learning metrics; existing tracking data may not have the right events |
| Neon cold start | LOW | Increase auto-suspend timeout, add connection warming; configuration change only |
