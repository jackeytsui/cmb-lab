## 1. Chart/Visualization Library Recommendation

### Verdict: Use shadcn/ui Charts (built on Recharts)

**Confidence: HIGH** -- Verified via official shadcn/ui documentation.

shadcn/ui has an official Chart component that wraps Recharts with shadcn/ui-native theming. This is the clear winner for this project because:

1. **Already in the stack.** The project uses shadcn/ui for all UI components. Charts integrate seamlessly with existing Card, Tooltip, and Dialog components.
2. **Dark mode out of the box.** All 53+ chart variants use CSS variables that automatically adapt to dark/light mode. The project's zinc-900 dark aesthetic will carry into charts without custom theming work.
3. **Composition-based, not abstracted.** shadcn/ui does NOT wrap Recharts -- it provides composition helpers (ChartContainer, ChartTooltip, ChartTooltipContent, ChartLegend, ChartLegendContent) that you use alongside raw Recharts components. No lock-in.
4. **Copy-paste model.** Like all shadcn/ui components, charts are copied into your project, not imported from a package. Full control over styling.
5. **Recharts v3 path.** shadcn/ui is actively working on Recharts v3 support. When it lands, the migration path is direct since there's no abstraction layer.

### Installation

```bash
pnpm dlx shadcn@latest add chart
```

This adds:
- `components/ui/chart.tsx` with ChartContainer, ChartTooltip, ChartTooltipContent, ChartLegend, ChartLegendContent, ChartStyle
- Recharts as a dependency (currently v2.x, ~50KB gzipped with tree shaking)

### Chart Types Available via Recharts

| Chart Type | Component | Use Case in This Project |
|-----------|-----------|-------------------------|
| Area Chart | `<AreaChart>` | XP timeline (daily/weekly/monthly) |
| Line Chart | `<LineChart>` | Score trends over time |
| Bar Chart | `<BarChart>` | Weekly XP comparison bars |
| Radar Chart | `<RadarChart>` | Skill mastery spider web |
| Pie Chart | `<PieChart>` | Course completion breakdown |
| Radial Bar | `<RadialBarChart>` | Activity ring alternative |

### Theming Pattern

```tsx
// Chart config defines colors per data series using CSS variables
const chartConfig = {
  xp: {
    label: "XP Earned",
    color: "var(--chart-1)", // Maps to CSS variable in globals.css
  },
  streak: {
    label: "Streak Days",
    color: "var(--chart-2)",
  },
} satisfies ChartConfig;
```

Colors are defined in `globals.css`:
```css
:root {
  --chart-1: oklch(0.646 0.222 41.116);
  --chart-2: oklch(0.6 0.118 184.704);
}
.dark {
  --chart-1: oklch(0.488 0.243 264.376);
  --chart-2: oklch(0.696 0.17 162.48);
}
```

### Alternatives Considered

| Library | Why Not |
|---------|---------|
| **Tremor** | Built on Recharts + Tailwind + Radix (same stack!) but adds another dependency layer. Since shadcn/ui has native chart support now, Tremor would be redundant. |
| **Nivo** | Excellent D3-based library with SSR support and calendar heatmaps. However, it has a separate theming system that does not use CSS variables, making dark mode integration with shadcn/ui painful. Also much heavier (~100KB+ for multiple chart packages). |
| **Chart.js / react-chartjs-2** | Canvas-based, not SVG. Loses composability with React components. Harder to style with Tailwind. |
| **Victory** | Good library but less community momentum than Recharts. No shadcn/ui integration. |

### Bundle Size Notes

- **Recharts v2.x (current):** ~50KB gzipped with tree shaking via proper imports
- **Recharts v3.x (latest):** v3.7.0 is current; shadcn/ui is testing v3 support. V3 adds z-index support, 3500 new unit tests, and performance improvements. When shadcn/ui officially supports v3, upgrade.
- **Impact on this project:** Recharts is already pulled in by the `chart` shadcn component. No additional weight for chart features.

## 2. XP Timeline Visualization

### Recommended: Area Chart with Period Selector

**Confidence: HIGH**

Use a shadcn/ui `<AreaChart>` with a gradient fill showing XP earned over time. This is the hero visualization of the progress dashboard -- the first thing students see.

### Design Pattern

```
[Daily] [Weekly] [Monthly]    <-- Period toggle tabs
 ___________________________________
|                          ____      |
|                     ___/    \___   |
|                ___/             \  |
|           ___/                   | |
|      ___/                        | |
| ___/                             | |
|/___________________________________|
 Jan    Feb    Mar    Apr    May
```

### Implementation Details

**Data shape:**
```typescript
interface XpDataPoint {
  date: string;        // ISO date
  xpEarned: number;    // XP earned on this date/period
  cumulativeXp: number; // Running total
}
```

**Chart configuration:**
- **Daily view:** Last 30 days, one point per day
- **Weekly view:** Last 12 weeks, one point per week (sum of daily XP)
- **Monthly view:** Last 12 months, one point per month (sum of daily XP)
- **Gradient fill:** Emerald-500 at top fading to transparent at bottom (matches project accent color)
- **Hover tooltip:** Shows exact XP, date, and what earned it (lessons, practice, conversations)
- **Cumulative line overlay:** Optional toggle to show total XP growth curve behind the daily bars

**Visual details for premium feel:**
- Smooth curve interpolation (`type="monotone"` on Recharts Area)
- Subtle grid lines at low opacity (zinc-800)
- Animated entrance on page load (Recharts supports `isAnimationActive`)
- Current day/week/month highlighted with a dot + label
- "Personal best" indicator line showing highest daily XP achieved

### XP Sources to Track

| Activity | XP Value | Rationale |
|---------|---------|-----------|
| Complete a lesson | 50 XP | Core engagement loop |
| Complete a practice set | 30 XP | Reinforcement |
| Pass pronunciation exercise | 20 XP | High-effort activity |
| Chatbot conversation (5+ turns) | 10 XP | Encourages practice |
| Voice AI conversation (3+ min) | 15 XP | Speaking practice |
| Daily login | 5 XP | Baseline engagement |
| Perfect score on practice set | 20 XP bonus | Rewards mastery |

These values should be stored in a config table or constants file, not hardcoded across the app, so coaches can adjust without code changes.

## 4. Mastery Map Design

### Recommended: Linear Course Progress with Module Segments

**Confidence: HIGH** -- Based on project's existing linear progression model.

### Why NOT a Skill Tree

The project enforces **linear progression** (lesson N+1 requires lesson N complete). A branching skill tree implies choice and non-linear paths. Using a skill tree visualization for linear content would be misleading and confusing.

Instead, use a **segmented progress bar per course** that breaks down into modules, with each module breaking into lessons.

### Mastery Map Visual Pattern

```
Course: Beginner Cantonese
[===Module 1====][===Module 2====][===Module 3====][  Module 4  ]
 L1  L2  L3  L4  L5  L6  L7  L8  L9  L10 L11 L12  L13 L14 L15

Legend: [filled] = completed, [half] = in progress, [empty] = locked
```

### Implementation: Segmented Progress Component

```tsx
// MasteryMap component
// For each enrolled course:
//   - Horizontal segmented bar
//   - Each segment = 1 module
//   - Within each segment, subdivisions = lessons
//   - Color coding:
//     - Completed lesson: emerald-500
//     - In-progress lesson: amber-500 (pulsing)
//     - Locked lesson: zinc-700
//     - Current lesson: emerald-400 with glow ring
```

### Chinese-Specific: Character Mastery Tracking

Since the LMS teaches Mandarin and Cantonese, a valuable differentiator is showing characters mastered:

**Character Mastery Card:**
```
Characters Practiced
[你] [好] [我] [是] [很] [的] [也] [不] [吗] [他]
  A    A    B    A    C    B    -    -    -    -

A = Mastered (green)  B = Practiced (amber)  C = Attempted (red)  - = Not yet
```

This could pull from:
- Practice set results (exercises involving specific characters)
- Pronunciation scores per character (from Azure Speech data already stored)
- Chatbot conversation analysis (characters used)

**Note:** This is a differentiator feature, not table stakes. Defer to post-MVP if scope is tight. It requires a character-tracking table or aggregation from existing practice/pronunciation data.

### Radar/Spider Chart for Skill Dimensions

For a high-level mastery overview, use a Recharts `<RadarChart>` showing skill dimensions:

```
        Listening
           /\
          /  \
Reading --    -- Speaking
          \  /
           \/
        Writing
```

Each axis represents a skill area:
- **Listening:** Video completion rate
- **Speaking:** Pronunciation scores + voice AI time
- **Reading:** Practice set scores (text-based exercises)
- **Writing:** Free text exercise scores + chatbot conversations

This provides a motivating "shape of your skills" visualization.

## 5. Milestone & Badge System

### Design Philosophy: Premium, Not Childish

**Confidence: HIGH** -- Based on gamification research for adult learners.

The PROJECT.md explicitly states: "Premium adult learners prefer personal progress over competition." Badges must feel like **achievements**, not stickers. Think LinkedIn skill endorsements, not Duolingo gems.

### Milestone Categories

#### Learning Progress Milestones
| Milestone | Trigger | Badge Name | Icon Suggestion |
|-----------|---------|-----------|-----------------|
| First lesson completed | Complete any lesson | "First Step" | Footprint |
| First course completed | Complete all lessons in a course | "Course Scholar" | Graduation cap |
| All courses completed | Complete every enrolled course | "Master Learner" | Crown |
| 10 lessons completed | 10 lesson completions | "Dedicated" | Star |
| 50 lessons completed | 50 lesson completions | "Committed" | Shield |

#### Practice & Assessment Milestones
| Milestone | Trigger | Badge Name | Icon Suggestion |
|-----------|---------|-----------|-----------------|
| First practice set | Complete any practice set | "Practice Begins" | Target |
| Perfect score | 100% on any practice set | "Perfect" | Diamond |
| 10 perfect scores | 10x 100% practice scores | "Perfectionist" | Double diamond |
| Pronunciation Pro | 90%+ pronunciation score | "Clear Voice" | Microphone |

#### Engagement Milestones
| Milestone | Trigger | Badge Name | Icon Suggestion |
|-----------|---------|-----------|-----------------|
| 7-day streak | 7 consecutive active days | "Week Warrior" | Flame |
| 30-day streak | 30 consecutive active days | "Monthly Master" | Calendar check |
| 100-day streak | 100 consecutive active days | "Century" | Trophy |
| 100 XP in one day | Earn 100+ XP in a single day | "Power Day" | Lightning |
| 1,000 total XP | Accumulate 1,000 XP | "Rising Star" | Rocket |
| 10,000 total XP | Accumulate 10,000 XP | "Language Legend" | Mountain peak |

#### Conversation Milestones
| Milestone | Trigger | Badge Name | Icon Suggestion |
|-----------|---------|-----------|-----------------|
| First AI conversation | Complete any chatbot or voice AI session | "Conversationalist" | Speech bubble |
| 10 AI conversations | 10 conversations | "Dialogue Pro" | Two speech bubbles |
| 30-min voice practice | Cumulative 30 min with voice AI | "Vocal" | Waveform |

### Visual Design Pattern

**Earned badges:** Full color, subtle shadow/glow, lucide icon at center
**Locked badges:** Zinc-700 monochrome, lock overlay, progress indicator ("3/7 days")
**Recently earned:** Emerald glow pulse animation (framer-motion)

```tsx
// Badge card layout
<div className="flex flex-col items-center gap-2 p-4">
  {/* Circle with icon */}
  <div className={cn(
    "h-16 w-16 rounded-full flex items-center justify-center",
    earned ? "bg-emerald-500/20 border-2 border-emerald-500" : "bg-zinc-800 border-2 border-zinc-700"
  )}>
    <Icon className={cn("h-8 w-8", earned ? "text-emerald-400" : "text-zinc-600")} />
  </div>
  {/* Badge name */}
  <span className={cn("text-sm font-medium", earned ? "text-white" : "text-zinc-500")}>
    {badge.name}
  </span>
  {/* Progress or date */}
  <span className="text-xs text-zinc-500">
    {earned ? format(earnedAt, 'MMM d') : `${progress}/${target}`}
  </span>
</div>
```

### Storage

Milestones should be stored in a `milestones` table:

```sql
milestones (
  id UUID PK,
  user_id UUID FK -> users,
  milestone_key TEXT NOT NULL,  -- e.g., 'streak_7', 'xp_1000'
  earned_at TIMESTAMP NOT NULL,
  metadata JSONB               -- e.g., { streakLength: 7, totalXp: 1042 }
)
```

Use a `milestone_definitions` constant (not a DB table) for the badge metadata (name, description, icon, threshold). Only earned milestones get stored.

### Anti-Patterns to Avoid

- **Too many badges too fast.** Space them out. First badge at lesson 1, next at lesson 5, not 1-2-3-4-5.
- **Childish names/icons.** No "Baby Steps" or cartoon animals. Use professional iconography.
- **Notification spam.** Show badge earned in-context (progress dashboard), not as a popup interruption during lessons.
- **Removing earned badges.** Never un-earn a badge. If a streak breaks, the "7-day streak" badge stays earned.

### Recommended: In-App Summary Card (not email)

**Confidence: HIGH**

An in-app weekly summary card is more impactful than an email digest for this audience. Email is noisy; an in-app card appears when they're already engaged.

### What to Show

| Stat | Source | Visual |
|------|--------|--------|
| XP earned this week | Sum of xp_events this week | Large number with animated counter |
| vs. last week | Compare to previous 7 days | Green up arrow / red down arrow with percentage |
| Lessons completed | Count of lesson completions | Number with progress ring |
| Practice sets completed | Count of practice attempts | Number with progress ring |
| Current streak | Consecutive days with activity | Flame icon + number |
| Longest streak | Historical max streak | Small text below current |
| Daily goal hit rate | Days meeting XP goal / 7 | "5/7 days" with mini bar chart |
| Time spent | Sum of session durations | "2h 34m" |
| Top skill area | Highest radar chart dimension | "Speaking improved 15%" |

### Layout Pattern

```
+--------------------------------------------------+
| This Week                           Jan 27 - Feb 2 |
|                                                    |
| [   245 XP   ]    [  3 Lessons  ]  [  5 Practice  ]|
|   +23% vs last       completed       sets done     |
|                                                    |
| Streak: 12 days     Daily Goal: 5/7 days          |

|  M   T   W   T   F   S   S                        |

+--------------------------------------------------+
```

### Implementation Notes

- **Animated counters:** Use framer-motion `useSpring` + `useTransform` for counting up from 0 to value on mount. Already installed in project (`framer-motion: ^12.29.2`).

- **Mini bar chart:** 7 small bars (one per day) using inline divs with height proportional to XP. No Recharts needed for this -- pure CSS.
- **Card wrapper:** shadcn `Card` component with zinc-800 background.
- **Calculation:** Server-side aggregation query. Cache the weekly summary for the user and recalculate daily at midnight or on-demand.

For v5.0, build the in-app card first. Email digest can be a future phase using:

- Same data as the in-app card, rendered as HTML email

## 8. Activity Scoreboard

### Design Philosophy: Personal Bests, Optional Cohort

**Confidence: HIGH** -- Research strongly supports opt-in rankings for adult learners.

### Key Research Finding

Poorly designed leaderboards **demotivate** lower-ranked learners. For high-ticket adult learners, public mandatory rankings are an anti-pattern. Instead:

1. **Default view: Personal bests only** -- no other students visible
2. **Opt-in cohort view** -- student explicitly enables "see how I compare"
3. **Relative positioning, not absolute** -- "You're in the top 30%" not "#47 of 156"
4. **Multiple dimensions** -- different "leaderboards" for different skills so everyone can find their strength

### Personal Bests Section

```
Your Personal Records

[Longest Streak]     [Highest Daily XP]     [Best Practice Score]
   34 days              245 XP                  100%
   Current: 12          Today: 65               Last: 85%

[Most Lessons/Week]  [Best Pronunciation]    [Fastest Practice]
   8 lessons           95% accuracy            4m 12s
   This week: 3        Last: 88%               Last: 7m 30s
```

### Optional Cohort Rankings (Opt-In)

```
See how you compare? [Enable] [Disable]

This Week's Activity         Your Position
Total XP earned              Top 25% (8 of 32 students)
Streak length                Top 10% (3 of 32 students)
Practice sets completed      Top 50% (16 of 32 students)

Note: Rankings are anonymous. Other students cannot see your position.
```

### Implementation Details

- **Opt-in toggle:** Store in user settings (`show_cohort_rankings: boolean`)
- **Percentile calculation:** Server-side query: `SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY xp_total)` to find where the user falls
- **Anonymous by default:** Even when opted in, students see "Top 25%" not individual names
- **Multiple categories:** XP, streak, practice scores, pronunciation scores, lessons completed
- **Refresh:** Recalculate daily, not real-time (prevents obsessive checking)

### Anti-Patterns to Avoid

- **Showing names of other students** -- Privacy concern and discouraging for lower-ranked
- **Mandatory visibility** -- Must be opt-in, default OFF
- **Single ranking metric** -- Use multiple dimensions so everyone can excel somewhere
- **Real-time updates** -- Creates anxiety. Daily recalculation is sufficient.
- **Showing exact rank** -- "#47 of 156" is demoralizing. "Top 30%" is encouraging.

### Sources

- [Leaderboard Design Principles (PMC)](https://pmc.ncbi.nlm.nih.gov/articles/PMC8097522/)
- [Psychology of Leaderboards in Learning](https://cluelabs.com/blog/understanding-the-psychology-behind-leaderboard-design/)
- [Level Up: Leaderboard Opt-Out](https://docs.levelup.plus/xp/docs/leaderboard-opt-out)

## 9. Activity Rings (Daily Goals)

### Recommended: Custom SVG Component (no library needed)

**Confidence: HIGH**

The PROJECT.md explicitly calls for "Apple Watch-style activity rings" that "feel premium; Duolingo flames feel juvenile for this audience." Build a custom SVG ring component rather than using a library.

### Why Custom Over Library

| Option | Verdict |
|--------|---------|
| **Custom SVG + framer-motion** | RECOMMENDED. Full control over styling to match zinc/emerald palette. 50-80 lines of code. Uses existing framer-motion for animation. |
| **react-activity-rings** (JonasDoesThings) | Functional but limited. Fixed styling, no Tailwind integration. 12 GitHub stars. |
| **Recharts RadialBarChart** | Could work via shadcn/ui charts. But the visual aesthetic is "chart" not "ring." |

### Ring Design for Learning

Three concentric rings representing daily goals:

```
Ring 1 (outer, emerald-500): Lessons progress -- "Learn"
Ring 2 (middle, amber-500):  Practice progress -- "Practice"
Ring 3 (inner, blue-500):    Conversation progress -- "Speak"
```

### Technical Implementation

```tsx
// ActivityRing uses SVG circles with stroke-dasharray/dashoffset
// Each ring: background circle (low opacity) + progress circle (animated)
// framer-motion animates dashoffset from full to target on mount

interface RingData {
  label: string;
  progress: number; // 0-1
  color: string;    // Tailwind color
  target: string;   // "2 lessons", "1 practice set", "5 min conversation"
}
```

The SVG technique:
1. Set `stroke-dasharray` to full circumference
2. Set `stroke-dashoffset` to `circumference * (1 - progress)`
3. Animate `stroke-dashoffset` with framer-motion `motion.circle`
4. Round linecaps for Apple Watch feel

### Daily Goal Configuration

Students set their own daily goal tier:

| Tier | Lessons | Practice | Conversation | Daily XP Target |
|------|---------|----------|-------------|----------------|
| Casual | 1 | 0 | 0 | 50 XP |
| Regular | 1 | 1 | 5 min | 100 XP |
| Serious | 2 | 1 | 10 min | 150 XP |
| Intense | 3 | 2 | 15 min | 250 XP |

Store the selected tier in user preferences. The activity rings show progress toward the selected tier's targets.

## 10. Celebration Animations

### Recommended: canvas-confetti + framer-motion

**Confidence: HIGH**

The project already has framer-motion installed. For confetti specifically, use `canvas-confetti`:

```bash
npm install canvas-confetti
# Types: npm install -D @types/canvas-confetti
```

**Bundle size:** ~6KB gzipped. Lightweight.

**Key features:**
- Canvas-based (no DOM manipulation overhead)
- `disableForReducedMotion` option for accessibility
- Works in web workers for off-main-thread rendering
- Configurable: particle count, spread angle, velocity, colors

### When to Trigger Celebrations

| Event | Animation | Intensity |
|-------|-----------|-----------|
| Lesson completed | Confetti burst + score card slide-in | Medium |
| Practice set 100% | Full-screen confetti rain + "Perfect!" text | High |
| Practice set passed | Gentle confetti + "Well done!" | Low |
| Badge earned | Ring glow animation + badge slide-in | Medium |
| Streak milestone (7, 30, 100) | Full confetti + milestone card | High |
| Course completed | Extended celebration + certificate CTA | Highest |

### Implementation Pattern

```tsx
import confetti from 'canvas-confetti';

function celebrateLessonComplete(score: number) {
  if (score === 100) {
    // Full celebration
    confetti({
      particleCount: 100,
      spread: 70,
      origin: { y: 0.6 },
      colors: ['#10b981', '#34d399', '#6ee7b7'], // emerald palette
    });
  } else {
    // Gentle celebration
    confetti({
      particleCount: 30,
      spread: 50,
      origin: { y: 0.7 },
    });
  }
}
```

Combine with framer-motion for score reveals:
```tsx
<motion.div
  initial={{ scale: 0, opacity: 0 }}
  animate={{ scale: 1, opacity: 1 }}
  transition={{ type: "spring", bounce: 0.5 }}
>
  <h2 className="text-4xl font-bold text-emerald-400">92%</h2>
</motion.div>
```

## 11. Database Schema Implications

### New Tables Needed

```sql
-- XP tracking
xp_events (
  id UUID PK,
  user_id UUID FK -> users,
  amount INTEGER NOT NULL,        -- XP earned
  source TEXT NOT NULL,            -- 'lesson', 'practice', 'conversation', 'login', 'bonus'
  source_id UUID,                 -- lesson_id, practice_set_id, etc.

-- Index on (user_id, created_at) for timeline queries

user_streaks (
  user_id UUID PK FK -> users,

  last_active_date DATE NOT NULL,
  streak_freeze_available INTEGER NOT NULL DEFAULT 0,
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
)

-- Milestones
milestones (
  id UUID PK,
  user_id UUID FK -> users,
  milestone_key TEXT NOT NULL,     -- 'streak_7', 'xp_1000', etc.

  metadata JSONB,
  UNIQUE(user_id, milestone_key)
)

-- Daily goal settings (extend users or separate)
user_goals (
  user_id UUID PK FK -> users,
  daily_xp_target INTEGER NOT NULL DEFAULT 50,
  goal_tier TEXT NOT NULL DEFAULT 'casual',  -- casual/regular/serious/intense
  show_cohort_rankings BOOLEAN NOT NULL DEFAULT false,
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
)
```

### Existing Tables Used (No Changes Needed)

- `lesson_progress` -- Lesson completion counts, video progress
- `practice_attempts` -- Practice scores, per-exercise results
- `conversations` -- Voice AI conversation data
- `chat_conversations` -- Chatbot conversation data

### Aggregation Queries

For the weekly summary and heatmap, these queries run against `xp_events`:

```sql
-- Daily XP for heatmap (last 365 days)
SELECT DATE(created_at) as day, SUM(amount) as xp
FROM xp_events
WHERE user_id = $1 AND created_at > NOW() - INTERVAL '365 days'
GROUP BY DATE(created_at);

-- Weekly summary
SELECT SUM(amount) as weekly_xp
FROM xp_events
WHERE user_id = $1 AND created_at > NOW() - INTERVAL '7 days';
```

Add indexes:
```sql
CREATE INDEX idx_xp_events_user_date ON xp_events(user_id, created_at);
CREATE INDEX idx_milestones_user ON milestones(user_id);
```

## 12. Pitfalls & Anti-Patterns

### Critical Pitfalls

#### Pitfall 1: Recharts in Server Components
**What goes wrong:** Recharts components require browser APIs (SVG rendering, event handlers). They cannot render in Next.js Server Components.
**Prevention:** Always use `"use client"` directive on any component that contains Recharts/chart elements. Fetch data in a Server Component parent, pass as props to a Client Component chart wrapper.
**Detection:** Build error: "window is not defined" or hydration mismatch.

#### Pitfall 2: XP Inflation / Deflation
**What goes wrong:** If XP values are too easy to earn, they become meaningless. If too hard, students feel no progress.
**Prevention:** Start conservative. Set XP values in a config constant, not scattered across code. Track average daily XP per user in the first week and adjust values based on real data.
**Detection:** If 80%+ of students hit daily goal every day, it's too easy. If less than 30% hit it, it's too hard.

#### Pitfall 3: Streak Anxiety for Adult Learners
**What goes wrong:** Adult learners have jobs, families, travel. Losing a 50-day streak to a sick day is demoralizing, not motivating.
**Prevention:** Implement streak freezes (Duolingo data shows this reduces churn by 21%). Allow 1-2 free freezes per week. Never show "streak lost" in a punishing way -- frame it as "new streak started."
**Detection:** User complaints, high dropout after streak loss.

#### Pitfall 4: N+1 Queries on Dashboard
**What goes wrong:** Dashboard shows XP timeline + heatmap + badges + weekly summary + activity rings. Each widget makes separate DB queries. With 5-10 widgets, that's 5-10 round trips.
**Prevention:** Create a single server action or API endpoint that returns all dashboard data in one query batch. Use Drizzle's query builder to join efficiently.
**Detection:** Dashboard load time > 2 seconds. Check Network tab.

#### Pitfall 5: Heatmap Data Volume
**What goes wrong:** Loading 365 days of daily activity data for every dashboard visit is expensive.
**Prevention:** Cache the heatmap data. It only changes once per day. Use a server-side cache (or edge cache with 24-hour TTL). Only fetch today's data fresh and merge with cache.

### Moderate Pitfalls

#### Pitfall 6: Badge Notification Fatigue
**What goes wrong:** Early users earn 5-10 badges in their first session (first login, first lesson, first practice, etc.), creating popup spam.
**Prevention:** Queue badge notifications. Show max 1-2 per session. Batch the rest into a "You earned 4 badges today!" summary on the dashboard.

#### Pitfall 7: Chart Animations on Mobile
**What goes wrong:** Recharts animations + confetti + framer-motion all running on a low-end phone = janky.
**Prevention:** Reduce animation on mobile. Check `window.matchMedia('(prefers-reduced-motion: reduce)')`. Use `isAnimationActive={false}` on Recharts for mobile viewports.

#### Pitfall 8: Timezone Issues with Streaks
**What goes wrong:** Server calculates "day" in UTC. Student in PST does their lesson at 10pm. Server thinks it's the next day. Streak logic breaks.
**Prevention:** Store the user's timezone in settings (or detect from browser). Calculate streak days in the user's local timezone, not UTC.

### Minor Pitfalls

#### Pitfall 9: Chart Responsiveness
**What goes wrong:** Recharts charts have fixed dimensions that break on small screens.
**Prevention:** Use `<ResponsiveContainer>` wrapper (from Recharts) inside shadcn `ChartContainer`. Set min-height, let width be 100%.

#### Pitfall 10: Stale Weekly Summary
**What goes wrong:** Student completes a lesson, goes to dashboard, sees stale weekly summary without the lesson they just finished.
**Prevention:** Invalidate or revalidate the weekly summary data when XP is earned. Use `revalidatePath('/dashboard')` in the XP-granting server action.

## 13. Recommended Phase Structure

Based on this research, the dashboard features should be structured into phases as follows:

### Phase A: XP & Streak Foundation (Build First)

**Rationale:** Everything else depends on XP data existing. Without the XP events table and streak tracking, there's nothing to visualize.

Build:
- `xp_events` table + migration
- `user_streaks` table + migration
- `user_goals` table + migration
- XP awarding logic (hook into lesson completion, practice completion, conversation end, login)
- Streak calculation logic (daily cron or on-login check)
- Daily goal tier settings UI

### Phase B: Progress Dashboard (Core Visualizations)

**Rationale:** This is the student-facing value. Show them their data.

Build:
- Install shadcn chart component (`pnpm dlx shadcn@latest add chart`)
- XP timeline area chart (daily/weekly/monthly toggle)
- Activity heatmap (GitHub-style calendar)
- Activity rings (daily goal progress)
- Weekly summary card (stats + comparison to last week)
- Animated counter for stat values (framer-motion)

### Phase C: Milestones & Celebrations

**Rationale:** Depends on XP data and streak data being tracked (Phase A). Celebrations enhance existing lesson/practice completion flows.

Build:
- `milestones` table + migration
- Milestone detection logic (check after XP award)
- Badge display grid (earned vs locked)
- `canvas-confetti` integration
- Lesson completion celebration screen
- Practice set completion celebration screen
- Badge earned animation

### Phase D: Coach Practice Analytics

**Rationale:** Depends on practice data already being collected (v4.0 shipped). Can be built in parallel with student dashboard.

Build:
- Practice results page (`/coach/practice-results`)
- Per-student attempt detail view
- Aggregate analytics (avg score, completion rate, difficulty ranking)
- Filtering by student/practice set/date range
- Score trend line chart per student

### Phase E: Activity Scoreboard

**Rationale:** Build last. Depends on XP, streaks, and milestones all existing. Opt-in design means it's additive, not blocking.

Build:
- Personal bests section
- Opt-in cohort toggle
- Percentile calculation API
- Multi-dimension rankings
- Settings toggle for visibility

### Mastery Map (Integrate Into Phase B)

The mastery map (segmented course progress) is a visual component that uses existing `lesson_progress` data. It doesn't need new tables. Build it as part of the progress dashboard phase.
