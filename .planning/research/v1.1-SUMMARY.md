# v1.1 Research Summary

**Project:** CantoMando Blueprint LMS - v1.1 Coach Tools & Student AI Assistant
**Domain:** Language Learning Platform - Bulk Content Management, RAG Chatbot, AI Prompts
**Researched:** 2026-01-28
**Confidence:** HIGH

## Executive Summary

The v1.1 milestone introduces four interconnected features to support coaches and students: bulk video upload, AI-powered student chatbot, knowledge base management, and an AI prompts dashboard. Research reveals that these features follow well-established patterns from platforms like Intercom (chatbot), Notion (bulk content management), and enterprise video CMS systems, but require specialized adaptations for Chinese language output with Pinyin/Jyutping annotations.

The recommended architecture maximizes integration with the existing Next.js 15 + Neon Postgres + Drizzle stack. The critical decision is using pgvector within Neon (not external vector databases like Pinecone) for RAG functionality—this eliminates data sync complexity while maintaining performance for the expected scale (<100K documents). Mux Direct Uploads will power bulk video upload, with rate-limiting required to avoid API throttling. The chatbot will use Vercel AI SDK with assistant-ui components for production-ready UI, and all AI prompts will be database-stored for coach editability.

Key risks center around three areas: (1) Mux rate limiting during bulk uploads (requires queuing), (2) RAG hallucinations from poor knowledge curation (requires systematic evaluation), and (3) Chinese text chunking destroying semantic meaning (requires language-aware splitting). All are mitigatable with the right patterns from day one.

## Key Findings

### Recommended Stack

The v1.1 features require four new technology integrations, all chosen to extend rather than replace the existing architecture. The core principle is minimizing operational complexity by using Neon Postgres for vector storage instead of external services.

**Core technologies:**
- **pgvector in Neon**: Vector storage for RAG — eliminates external vector DB, maintains transactional consistency, handles scale (<100K vectors easily)
- **Vercel AI SDK v6 + assistant-ui**: Chatbot framework — production-ready components, shadcn/ui compatible, streaming support built-in
- **OpenAI text-embedding-3-small**: Embeddings model — 5x cheaper than ada-002 ($0.02/1M tokens), better performance, 1536 dimensions
- **@langchain/textsplitters**: Text chunking — battle-tested RecursiveCharacterTextSplitter, handles Chinese sentence boundaries correctly
- **unpdf**: PDF text extraction — serverless-compatible, zero dependencies, works in Next.js Edge Runtime
- **Mux Direct Uploads API**: Bulk upload — signed URLs for client-side upload, webhooks for status, rate limit aware (1 RPS)

**Critical version requirements:**
- ai package 6.0.57+ (Vercel AI SDK with tool support)
- unpdf 1.4.0+ (PDF.js based, TypeScript-first)
- @mux/mux-node latest (Direct Upload support)

**What NOT to add:**
- Pinecone/Weaviate/Qdrant (external vector DBs add sync complexity)
- LangChain full framework (heavy; only need text splitters)
- pdf-parse (abandoned, maintenance issues)
- Custom chat UI from scratch (2-4 weeks work; assistant-ui handles it)

### Expected Features

Research across enterprise video CMS platforms, education chatbots, and knowledge base tools reveals clear patterns for what users expect versus what differentiates.

**Must have (table stakes):**
- **Bulk Upload**: Drag-and-drop multi-file upload with per-file progress — users expect this from every file manager since 2015
- **Batch Editing**: Multi-select + bulk metadata edit — Notion/Airtable trained users to expect this
- **Drag-to-Reorder**: Visual reordering of lessons/modules — universal pattern from Trello, Notion, every todo app
- **Floating Chat Widget**: Bottom-right positioned, minimizable — Intercom/Drift established this as the standard placement
- **Searchable Knowledge Base**: Type question, find answer — Zendesk/Intercom trained users for this
- **AI Prompts Editor**: View/edit prompts with descriptions — coaches need visibility into AI behavior

**Should have (competitive advantage):**
- **Chinese Language Output**: Chatbot outputs Simplified/Traditional Chinese with Pinyin/Jyutping annotations — core differentiator for language learning
- **RAG-Powered Chatbot**: Knowledge base powers responses — single source of truth, fewer hallucinations
- **Language Preference Awareness**: Respects user's Cantonese/Mandarin setting from v1.0 — personalized experience
- **Prompt Versioning**: Rollback capability for AI prompts — essential for production prompt management
- **PDF Upload for Knowledge Base**: Upload docs, auto-extract text — time-saver for coaches
- **Smart Video Assignment**: Suggest lesson assignment based on filename patterns — reduces manual work

**Defer (v2+):**
- Video editing in browser (massive scope; users have preferred tools)
- Automatic captioning (Mux paid add-on; don't replicate)
- Multi-language UI translation (English UI with Chinese content is correct)
- Prompt A/B testing (requires analytics infrastructure)
- Saved chat history across sessions (storage/privacy concerns)

### Architecture Approach

The v1.1 features integrate cleanly by extending existing patterns: Mux video infrastructure (add uploads), Drizzle schema (add tables), API routes (add endpoints). The architecture maintains separation of concerns with clear component boundaries.

**Major components:**

1. **Bulk Upload Pipeline** — Three-stage flow: (1) Create Direct Upload URLs server-side via `/api/mux/upload`, (2) Client uploads directly to Mux with XHR progress tracking, (3) Webhooks update database status as Mux processes. Database tracks upload batches and individual items with status (pending → uploading → processing → ready). Rate limiting via queue prevents 429 errors.

2. **RAG Knowledge System** — Four-table schema: categories, entries, embeddings (chunked), chat sessions. Enable pgvector extension, store 1536-dim vectors with HNSW index. Embedding generation via OpenAI API, semantic chunking with language-aware separators for Chinese (。！？；，), similarity search with cosine distance (threshold >0.5). Integration point: Vercel AI SDK tool calls `searchKnowledge()` during chat.

3. **Chatbot Interface** — FloatingChat component in root layout (SignedIn wrapper), powered by Vercel AI SDK's useChat hook. System prompt loaded from database via `getPrompt()`, includes user language preferences (Simplified/Traditional, Pinyin/Jyutping). Chat API route at `/api/chat` with `streamText()`, tool definition for knowledge retrieval. Context management via conversation summarization when approaching token limits.

4. **AI Prompts Management** — Two-table schema: prompts (active content) and versions (audit trail). Registry at `lib/prompts.ts` with fallback constants. Admin UI for CRUD, Monaco/textarea for editing. Integration: existing AI routes (`/api/grade`, `/api/realtime/token`) call `getPrompt(key)` instead of hardcoded strings. Changes apply on next API call (no hot reload needed).

**Key data flows:**
- Upload: File → CreateURL → Mux Direct → Webhook → DB Status → Assignment UI → Lesson
- Chat: Message → API → LoadPrompt → SearchKnowledge → StreamResponse → FloatingChat
- Prompt Edit: Editor → SaveVersion → UpdateActive → NextAPICall uses new prompt

### Critical Pitfalls

Research identified 13 pitfalls across critical, moderate, and minor severity. Top 5 require architectural decisions before implementation begins.

1. **Mux Rate Limiting (Critical)** — Bulk upload rapidly creates Direct Upload URLs, hitting Mux's 1 RPS API limit. Silent failures occur if not queued. Prevention: Implement rate-limited queue (p-queue or similar) with 1 request/second, show UI feedback as each completes. Detection: HTTP 429 responses, inconsistent upload counts. Address in: Bulk Upload Phase from day one.

2. **Large File Memory Exhaustion (Critical)** — Multi-gigabyte video uploads crash Next.js serverless functions if body parser loads files into memory. Prevention: Use Mux Direct Uploads (client uploads directly to Mux, bypasses server), disable Next.js body parser. Detection: "Body exceeded 1mb limit" errors, 504 timeouts, memory warnings. Address in: Bulk Upload architecture decision.

3. **RAG Hallucinations (Critical)** — 80% of RAG projects fail before scaling due to poor document curation and lack of evaluation. Chatbot confidently states wrong information if knowledge base is uncurated. Prevention: Curate aggressively (10 quality docs > 1000 random), implement relevance scoring, add explicit "I don't know" handling in system prompt. Detection: Students report incorrect information, support tickets increase. Address in: Knowledge Base Phase with evaluation metrics.

4. **Chinese Text Chunking (Critical)** — Fixed-size chunking splits Chinese sentences mid-character, destroying semantic meaning. Standard tokenizers break on spaces (Chinese has none). Prevention: Use RecursiveCharacterTextSplitter with Chinese punctuation separators (。！？；), 20% overlap, keep Pinyin/Jyutping as metadata (not in chunk text). Consider BGE-M3 embeddings for better Chinese support. Address in: Knowledge Base Phase, choose strategy early.

5. **Context Window Overflow (Critical)** — Long chatbot conversations exceed model token limits, causing "memory loss" mid-conversation. Prevention: Implement conversation summarization when approaching 80% of token limit, keep recent 10 messages + summary of older ones. Set MAX_CONVERSATION_MESSAGES = 50, notify user when approaching limit. Address in: Chatbot Phase, design context management from start.

**Additional notable pitfalls:**
- Content reordering creates database inconsistencies (use gap-based indexing like Figma)
- Chatbot outputs wrong character variant (enforce preferences in system prompt)
- PDF scanned documents fail text extraction (detect PDF type, flag for OCR/review)
- AI prompt edits break output format (require versioning, test before activation)
- Bulk operations lack per-item error feedback (return succeeded/failed arrays with retry)

## Implications for Roadmap

Based on dependencies and integration complexity, four phases are recommended in this specific order:

**Delivers:**
- Prompts management UI at `/admin/prompts`
- Database tables (ai_prompts, ai_prompt_versions)
- Prompt registry at `lib/prompts.ts` with getPrompt() function
- Versioning and rollback capability
- Integration with existing AI routes (grading, voice)

**Addresses (from FEATURES.md):**

**Avoids (from PITFALLS.md):**

**Research flag:** Standard CRUD pattern, skip `/gsd:research-phase`

### Phase Ordering Rationale

**Why this sequence:**
1. **Prompts first** because it's standalone and establishes the dynamic prompts pattern that chatbot needs
2. **Bulk upload second** because it's isolated from chatbot/knowledge systems and unblocks content creation
3. **Knowledge base third** to validate RAG patterns before exposing chatbot to students
4. **Chatbot last** as final integration of all v1.1 components

**Dependency chain:**
```
Phase 1 (Prompts) ──┐
                    ├──> Phase 4 (Chatbot)
Phase 3 (KB+RAG) ───┘

Phase 2 (Upload) ────> (independent)
```

**How this avoids pitfalls:**
- Testing RAG quality (Phase 3) before student access (Phase 4) prevents hallucination exposure
- Rate limiting patterns proven in Phase 2 before applying to other bulk operations
- Prompt versioning (Phase 1) established before chatbot makes breaking changes more costly
- Chinese chunking validated on knowledge base before chatbot depends on it

### Gaps to Address

**During planning:**
- **Chinese text chunking validation**: Phase 3 planning should include small-scale testing with actual Chinese help content to validate RecursiveCharacterTextSplitter separator strategy. May need to adjust chunk size (currently 500 chars) and overlap (20%) based on content.
- **Embedding model comparison**: text-embedding-3-small is cost-effective and documented, but BGE-M3 may perform better for Chinese. Consider benchmarking during Phase 3 if retrieval quality is poor.
- **Mux webhook reliability**: Local development requires ngrok tunneling for webhooks. Production deployment needs Vercel URL stability. Document setup process clearly.

**During implementation:**
- **RAG evaluation metrics**: Define what "good retrieval" means before building chatbot. Suggested: relevance score >0.5, at least 2 chunks retrieved, manual review of 50 test queries.
- **Chinese font consistency**: Ensure FloatingChat applies same font classes as existing lesson content (v1.0 has custom Pinyin/Jyutping fonts). Test with actual Chinese speakers.
- **Prompt testing infrastructure**: Phase 1 should include sample inputs for testing prompts (grading examples, voice scenarios, chatbot queries). Manual testing acceptable for v1.1, automate in v2+.

**Open questions (low priority, defer if time-constrained):**
- Whether to support scanned PDFs with OCR (adds complexity, unpdf doesn't handle this)
- Whether to persist chat history across sessions (storage/privacy considerations)
- Whether to implement A/B testing for prompts (requires analytics infrastructure)
