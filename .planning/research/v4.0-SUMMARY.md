# v4.0 Practice & Homework Platform Research Summary

**Project:** CantoMando Blueprint LMS
**Domain:** Language Learning Platform - Practice System Integration
**Researched:** 2026-02-06
**Confidence:** HIGH

## Executive Summary

v4.0 transforms CantoMando Blueprint from a video-only LMS into a full practice platform with homework builder, pronunciation scoring, and enhanced chatbot. The research reveals a critical architectural principle: **practice sets must be a parallel system, not an extension of the existing video interactions**. The existing interaction system is tightly coupled to video timestamps and lesson progression. Practice sets are standalone homework units assignable to lessons, modules, courses, or directly to students.

The technical path is clear and requires minimal new dependencies. Only 3 packages are needed: `@dnd-kit/react` (drag-and-drop builder), `@dnd-kit/helpers` (sorting utilities), and `microsoft-cognitiveservices-speech-sdk` (pronunciation assessment). Custom fonts require no new packages (built-in `next/font/local`), exercise rendering uses existing shadcn/ui components, and the enhanced chatbot extends the current AI SDK integration. The n8n grading webhook pipeline handles free-text and audio exercises without modification.

The highest-risk area is pronunciation scoring for Cantonese. Research literature shows tone recognition accuracy tops out at ~71.5% for speaker-independent systems, and Azure Speech's prosody assessment (needed for tone-specific scoring) is currently English-only. The workaround is post-processing: compare Azure's transcription against expected pinyin/jyutping character-by-character using the already-installed `pinyin-pro` and `to-jyutping` libraries. For Mandarin, Azure Speech provides phoneme-level accuracy suitable for production. SpeechSuper offers superior Mandarin tone scoring but lacks Cantonese entirely, making Azure the pragmatic choice for dual-language support.

## Key Findings

### Recommended Stack (from STACK.md)

The existing Next.js 16 + React 19 + Neon Postgres + Drizzle ORM + Clerk stack handles 90% of v4.0 requirements. Only 3 production dependencies needed:

**New packages:**
- **@dnd-kit/react 0.2.1** — Drag-and-drop for homework builder canvas (sortable lists, palette-to-canvas, grid support). The new rewrite (not legacy `@dnd-kit/core`) with simplified API and React 19 support. Chosen over `@hello-pangea/dnd` (list-only) and `react-beautiful-dnd` (archived Aug 2025).
- **@dnd-kit/helpers 0.2.1** — Array manipulation utilities for sortable lists (companion to @dnd-kit/react).
- **microsoft-cognitiveservices-speech-sdk 1.47.0** — Pronunciation assessment for zh-CN (Mandarin) and zh-HK (Cantonese). Supports both languages with phoneme-level accuracy scores. Free tier: 5 hours/month STT. ~$0.003 per 8-second pronunciation check at standard tier.

**No new packages for:**
- Custom fonts — Use built-in `next/font/local` from Next.js 16
- Exercise rendering — Build with existing shadcn/ui + framer-motion + react-hook-form
- Enhanced chatbot — Extend existing AI SDK `useChat` hook and ChatWidget component

**External service required:** Azure Speech service (free tier sufficient for development/early users).

**Critical version note:** Use `@dnd-kit/react` (new rewrite), NOT the legacy `@dnd-kit/core` + `@dnd-kit/sortable` packages. The legacy packages are being replaced.

**Tone scoring limitation:** Azure Speech provides phoneme accuracy for zh-CN/zh-HK but prosody assessment (tone-specific scoring) is currently English-only. Workaround: post-process transcription to compare recognized characters against expected tones using pinyin-pro/to-jyutping.

### Expected Features (from FEATURES.md)

v4.0 adds 5 major capabilities spanning different maturity levels in the language learning ecosystem.

**Must have (table stakes):**
- **Multiple choice, fill-in-blank, matching, audio recording, free text exercises** — Universal question types in every quiz builder (H5P, Articulate Rise, Duolingo, HelloChinese all have these)
- **Drag-and-drop question arrangement** — Coaches expect visual reordering (H5P, Typeform, Articulate, Google Forms all support this)
- **Sequential question flow with immediate feedback** — Students answer one-at-a-time with instant correct/incorrect display (Duolingo pattern)
- **Overall score and summary** — Post-completion screen with score, time, areas of weakness (universal in assessment tools)
- **Progress persistence** — Resume mid-practice if student leaves (expected in any serious learning platform)
- **Practice set assignment to lessons/modules/courses/students** — Flexible homework system (Moodle, Canvas, Google Classroom pattern)
- **Overall pronunciation score + per-character highlighting** — Color-coded feedback (green/yellow/red) after speaking (ELSA Speak, HelloChinese standard)
- **iMessage-style chat bubbles** — Modern chat UI (rounded bubbles, left/right alignment, timestamps)
- **Lesson-specific conversation starters** — Suggested practice prompts based on lesson context (Speak app pattern)

**Should have (differentiators):**
- **Canto-to-Mando comparison exercises** — "Which Mandarin tone matches this Cantonese word?" — NO competitor does this (unique to CantoMando pedagogy)
- **Dual-language annotation toggle** — Switch between pinyin, jyutping, or both on any exercise (leverages existing language preference system)
- **Pronunciation scoring with Canto-Mando comparison** — Cross-language insight: "Your Mandarin 3rd tone is correct! In Cantonese, this word uses tone 5" (no competitor does this)
- **Video + Practice bundling** — Practice sets attached to video lessons: "After watching Lesson 3 video, complete these exercises before proceeding" (combines FluentU video + Duolingo exercises)
- **Coach review of practice attempts** — Coaches see not just scores but attempt details with coaching annotations (most LMS tools show only scores)

**Defer (v2+ / anti-features):**
- **Gamification (points, streaks, badges)** — Already in PROJECT.md "Out of Scope." Adult learners find this condescending. Motivation comes from progress and coach feedback, not artificial rewards.
- **Spaced repetition vocabulary system** — Already in PROJECT.md "Out of Scope" for v4.0. SRS is a deep product (Anki has decades of refinement). Provide vocabulary exports instead.
- **Character stroke order / handwriting input** — HelloChinese and Pleco do this excellently. Support typed Chinese input (IME already built). Handwriting is a different product.
- **AI-generated exercises** — Quality control for Chinese exercises is critical. Let coaches author manually, AI can suggest improvements but coaches must approve.
- **Real-time multiplayer / competitive practice** — Requires WebSocket infrastructure for small user base. Focus on single-player mastery and coach-student interaction.

**Exercise type roadmap:**
- **Phase 1 (MVP):** MCQ, fill-in-blank, matching, free text, audio (5 types)
- **Phase 2 (post-MVP):** Word order, tone selection, listening comprehension (3 additional types)

### Architecture Approach (from ARCHITECTURE.md)

The v4.0 integration is **additive with surgical modifications**. New tables, new components, new API routes. Nothing in the existing video interaction pipeline changes.

**Critical architectural decision:** Practice exercises are a SEPARATE entity from video interactions. The existing `interactions` table is timestamp-bound (video cue points), lesson-required (NOT NULL FK), limited to 2 types (text/audio), consumed by XState video player machine, tied to `lessonProgress.interactionsCompleted` counter. Practice sets are standalone homework units assignable to any content level or directly to students. Overloading the interactions table would break the CuePoint type system, video player state machine, progress calculation, and InteractionForm UI.

**New database tables (4):**
1. **practice_sets** — Container for homework (title, description, language, status, creator, sortOrder)
2. **practice_exercises** — Individual exercises with JSONB definitions (type enum: mcq, matching, ordering, fill_blank, free_text, audio; prompt, definition as JSON string validated by Zod)
3. **practice_set_assignments** — Polymorphic assignment model (assignmentType enum: lesson, module, course, student, tag; targetId references appropriate table based on type)
4. **practice_attempts** — Student attempt records (userId, practiceSetId, startedAt, completedAt, score, totalPoints, earnedPoints, answers as JSON)

**Major components:**
1. **PracticeSetBuilder (new)** — Coach drag-and-drop canvas with dnd-kit, polymorphic ExerciseBlockEditor per type, BuilderPreview for live rendering, useReducer state management (NOT XState — simpler CRUD on ordered list)
2. **PracticePlayer (new)** — Student-facing sequential flow, polymorphic ExerciseRenderer per type, client-side grading for deterministic types (MCQ/matching/ordering/fill-blank), server-side AI grading via n8n webhooks for free_text/audio (reuses existing pipeline)
3. **ChatPanel v2 (modified)** — Add lessonId context, practice mode toggle, generateExercise tool in /api/chat, inline exercise rendering in ChatMessage
4. **VoiceTutor components (modified)** — Pronunciation scoring display (overall + per-character), connect to Azure Speech SDK via new /api/pronunciation/assess route

**Integration points — what changes in existing files:**
- `src/db/schema/index.ts` — Add `export * from "./practice"` (new practice.ts schema file)
- `src/app/api/chat/route.ts` — Add `generateExercise` tool to streamText config
- `src/app/api/realtime/token/route.ts` — Pass languagePreference param from request body
- `src/components/chat/ChatPanel.tsx` — Add lessonId prop, practice mode toggle
- `src/components/chat/ChatMessage.tsx` — Add inline exercise detection + rendering
- `src/hooks/useChatbot.ts` — Add lessonId + practiceMode to transport body
- `src/lib/lesson-context.ts` — Add practice set context to voice tutor instructions
- `src/app/layout.tsx` — Add custom font loading via next/font/local (2 fonts: Hanzi-Pinyin, Cantonese Visual)

**Data model decision — JSONB for exercise definitions:** Store exercise-specific data (MCQ options, matching pairs, fill-blank templates) as JSON text in `practice_exercises.definition` column. Validate with Zod discriminated unions in application layer. This matches existing patterns (`kbChunks.metadata`, prompt template variables) and avoids Drizzle JSONB type casting complexity. Trade-off: cannot query inside JSON at DB level without GIN indexes, but practice sets are retrieved whole (not queried by option content).

**Pattern reuse:**
- Free-text and audio exercises flow through same n8n webhooks as existing video interactions (N8N_GRADING_WEBHOOK_URL, N8N_AUDIO_GRADING_WEBHOOK_URL)
- MCQ, matching, ordering, fill-blank graded client-side (deterministic, instant feedback, zero server load)
- FreeTextRenderer and AudioRenderer wrap existing TextInteraction and AudioInteraction components (adapt props, reuse UI)
- Enhanced chatbot extends AI SDK useChat with new tool, follows existing searchKnowledgeBase pattern

### Critical Pitfalls (from PITFALLS.md)

14 pitfalls identified, top 5 critical:

1. **New exercise types break existing progress system** — Adding question types to practice sets either makes them invisible to progress OR conflates practice progress with video interaction progress OR breaks the `interactionTypeEnum` (only "text"/"audio"). **Prevention:** Build practice as SEPARATE system with own tables (practice_sets, practice_exercises, practice_attempts, practice_progress.ts). Then create unified progress aggregation view for dashboard. Do NOT alter `interaction_type` enum or modify `interactions` table.

2. **Polymorphic assignment table destroys foreign key integrity** — Single `practice_assignments` table with `assignable_type` string and `assignable_id` UUID means PostgreSQL cannot enforce FK constraints. Orphaned assignments pile up when lessons/modules deleted. **Prevention:** Use separate join tables per assignment target (practice_set_lesson_assignments, practice_set_module_assignments, practice_set_course_assignments, practice_set_user_assignments, practice_set_group_assignments). Each has proper FK with onDelete cascade.

3. **Drag-and-drop builder state becomes unrecoverable** — Coach builds 15-question homework, accidentally drags question group into itself (cycle), state corrupts during reorder, no undo. Entire homework lost. **Prevention:** Implement undo/redo from day one using command pattern with inverse operations. Generate unique IDs when dragging from palette. Flatten nested structures for dnd-kit. Memoize components aggressively. Auto-save to localStorage every 30 seconds.

4. **Pronunciation scoring gives wildly inaccurate tone scores for Cantonese** — Research literature shows tone recognition accuracy tops out at ~71.5% speaker-independent. Azure Speech provides phoneme accuracy but prosody (tone-specific) is English-only. Students get "wrong" scores, lose trust. **Prevention:** Set honest expectations ("pronunciation practice" not "scoring"). Display tone scores as ranges ("Good"/"Needs Practice"/"Try Again") not exact percentages. Allow self-assessment with audio playback. Build "Was this accurate?" feedback mechanism. Test with actual Cantonese/Mandarin speakers before shipping.

5. **Custom CJK font with ruby annotations causes layout disasters** — CJK fonts are 5-50MB+ (Noto Sans CJK is ~15MB per weight). Pinyin annotations overflow adjacent characters. Line heights inconsistent. Next.js `next/font` cannot preload CJK fonts (too many glyphs, split into 100+ files). **Prevention:** Use next/font/local with `display: 'swap'` and `preload: false`. Subset font to only used characters (pyftsubset). Serve WOFF2 only. Use unicode-range for split loading. Use HTML `<ruby>`, `<rt>`, `<rp>` elements for annotations (NOT specialty fonts). Set explicit line-height (2x normal). Test Chrome/Safari/Firefox/mobile.

**Additional moderate pitfalls:**
- Question type schema uses JSONB for everything, losing query power (use hybrid: common columns + typed JSONB)
- Practice progress doesn't integrate with existing dashboard (design unified aggregation layer early)
- Enhanced chatbot message type explosion (use discriminated union + renderer registry)
- dnd-kit library maintenance uncertainty (abstract DnD layer, monitor GitHub activity)
- Existing `interaction_type` enum cannot be extended without migration pain (create NEW enum for practice)

## Implications for Roadmap

Based on research, suggested phase structure prioritizes dependency chains and incremental value delivery.

### Phase 1: Foundation (Fonts + Voice Fix)
**Rationale:** Quick wins with no dependencies. Fonts affect all UI and should load before building components that depend on them. Voice fix is isolated bug fix.
**Delivers:** Custom font loading in layout.tsx, voice AI language preference bug fix, font switching CSS
**Addresses:** Custom font requirement from FEATURES.md
**Avoids:** Pitfall 5 (CJK font layout disasters) by using next/font/local correctly from start
**Research flag:** LOW — next/font/local is well-documented Next.js feature

### Phase 2: Practice Data Model + Admin CRUD
**Rationale:** Database tables and Zod schemas are foundation for everything else. Get the architecture right before building UI.
**Delivers:** practice.ts schema with 4 tables + 3 enums, Drizzle migration, exercise-schemas.ts (Zod discriminated unions), API routes for CRUD, basic admin practice set list
**Addresses:** Homework builder backend from FEATURES.md
**Avoids:** Pitfall 1 (breaks existing progress), Pitfall 2 (polymorphic FK), Pitfall 6 (JSONB-only schema), Pitfall 10 (enum extension)
**Research flag:** LOW — follows established Drizzle patterns from existing codebase

### Phase 3: Exercise Renderers + Student Practice Player
**Rationale:** Renderers needed by both student player AND builder preview. Build standalone first. Client-side grading for 4/6 types reduces server load.
**Delivers:** ExerciseRenderer with 6 type renderers, practice-grading.ts (client-side for MCQ/matching/ordering/fill-blank), /api/practice/grade route (server-side for free_text/audio via n8n), PracticePlayer component, PracticeResults screen, student /practice pages, practice-progress.ts, usePracticeSet.ts
**Addresses:** Practice sets from FEATURES.md (sequential flow, immediate feedback, score summary, retry incorrect)
**Avoids:** Pitfall 7 (progress not integrated) by building practice-progress.ts parallel to existing progress.ts
**Research flag:** LOW — reuses existing interaction components, standard React patterns

### Phase 4: Drag-and-Drop Practice Builder
**Rationale:** Most complex new UI. Requires data model (Phase 2) and exercise renderers (Phase 3 for live preview).
**Delivers:** Install @dnd-kit/react + @dnd-kit/helpers, PracticeSetBuilder canvas, all 6 ExerciseBlockEditor components, BuilderPreview panel, coach /coach/practice pages, usePracticeBuilder.ts with undo/redo
**Addresses:** Homework builder from FEATURES.md (drag-and-drop arrangement, question preview, reusable question bank)
**Avoids:** Pitfall 3 (unrecoverable builder state) by implementing command pattern undo from day one, Pitfall 9 (dnd-kit maintenance) by abstracting DnD layer
**Research flag:** MEDIUM — dnd-kit 0.2.x is new rewrite, needs performance testing with 20+ questions

### Phase 5: Practice Set Assignments
**Rationale:** Connects practice sets to curriculum. Requires practice sets to exist (Phase 2) and student player to work (Phase 3).
**Delivers:** Assignment API routes, coach assignment UI (lesson/module/course/student/tag), student dashboard integration, lesson page integration, due date display/filtering
**Addresses:** Practice set assignment from FEATURES.md (flexible assignment, due dates, availability windows)
**Avoids:** Integration Pitfall B (completion breaks unlock logic) by making practice additive, not blocking
**Research flag:** LOW — standard LMS assignment patterns

### Phase 6: Enhanced Chatbot
**Rationale:** Leverages exercise renderers from Phase 3 for inline widgets. Otherwise independent of builder.
**Delivers:** generateExercise tool in /api/chat, lesson-context-aware system prompt, InlineExercise component, ChatMessage modification for exercise tool results, practice mode toggle in ChatPanel, conversation persistence
**Addresses:** iMessage-style practice chat from FEATURES.md (lesson-specific starters, student text annotation, conversation history)
**Avoids:** Pitfall 8 (message type explosion) by using discriminated union + renderer registry
**Research flag:** LOW — extends existing AI SDK useChat pattern

### Phase 7: Pronunciation Scoring
**Rationale:** Most complex integration (Azure Speech SDK + WebRTC + scoring visualization). Highest-risk feature due to Cantonese tone accuracy. Can be built independently from practice sets.
**Delivers:** Install microsoft-cognitiveservices-speech-sdk, /api/pronunciation/assess route, Azure Speech integration, post-processing for tone identification, PronunciationScore component (overall + per-character), conversation history review, suggested practice topics per lesson
**Addresses:** Pronunciation scoring from FEATURES.md (overall score, per-character accuracy, tone indicators, retry comparison)
**Avoids:** Pitfall 4 (inaccurate Cantonese tones) by using qualitative ranges not percentages, setting honest expectations, building feedback mechanism
**Research flag:** HIGH — needs dedicated research spike for Azure Speech zh-HK accuracy, tone post-processing validation with actual speakers

### Phase Ordering Rationale

**Dependency chains:**
- Fonts load first (site-wide dependency)
- Data model before UI (builder and player depend on schema)
- Renderers before builder (builder preview reuses them)
- Practice sets before assignments (must exist to assign)
- All above before pronunciation scoring (can attach to any practice context)

**Groupings based on architecture:**
- Phases 2-3 form the practice core (data + student experience)
- Phase 4 adds coach authoring (builder)
- Phase 5 connects to curriculum (assignments)
- Phase 6 extends chat (orthogonal to practice sets)
- Phase 7 enhances voice (pronunciation feedback)

**Pitfall avoidance:**
- Schema designed correctly in Phase 2 prevents Pitfalls 1, 2, 6, 10
- Undo/redo built into Phase 4 prevents Pitfall 3
- Qualitative scoring in Phase 7 mitigates Pitfall 4
- Font strategy in Phase 1 prevents Pitfall 5
- Progress integration in Phases 2-3 prevents Pitfall 7
- Message type design in Phase 6 prevents Pitfall 8

### Gaps to Address

Areas where research was inconclusive or needs validation during implementation:

- **Cantonese tone scoring accuracy:** Azure Speech supports zh-HK for pronunciation assessment, but prosody (tone-specific) scoring is English-only per docs. The post-processing workaround (comparing transcription against expected pinyin/jyutping) is an engineering approach not a verified product feature. **Validation:** Needs research spike in Phase 7 with actual Cantonese speakers to test Azure transcription accuracy + post-processing algorithm before committing to this path. Consider SpeechSuper for Mandarin-only if Azure proves insufficient (but SpeechSuper has no Cantonese).

- **@dnd-kit/react 0.2.x production stability:** Version 0.2.1 is new (Jan 2026 publish). GitHub issues show it fixes React 19 and StrictMode problems, but it is 0.x semver. **Validation:** Phase 4 should include performance testing with 20+ questions in builder canvas. Monitor dnd-kit GitHub for maintenance activity. Build abstraction layer so library can be swapped if needed (Pitfall 9 mitigation).

- **JSONB vs native columns trade-off:** Storing exercise definitions as JSON text (validated by Zod) is pragmatic but loses database-level query power. If analytics later needs "find all MCQs with exactly 4 options", this becomes complex. **Validation:** Phase 2 schema design should prototype a few reporting queries to confirm JSONB+GIN indexes are sufficient. Consider hybrid approach: common fields in columns, type-specific in JSONB.

- **Custom font file quality:** The Hanzi-Pinyin font and Cantonese Visual font will be user-provided. File size, glyph coverage, and annotation positioning quality are unknowns. **Validation:** Phase 1 should test provided fonts early with sample characters. Subset to only needed glyphs, convert to WOFF2, measure load times. If fonts are unsuitable, fall back to HTML ruby markup exclusively.

- **Practice completion vs lesson unlock:** The PROJECT.md doesn't specify whether practice set completion should gate lesson progression. Current system: video watched 95% + all interactions passed = lesson complete. **Validation:** During Phase 5 (assignments), decide and document: are practice sets optional enrichment OR required for progression? Recommendation: optional by default (Pitfall B mitigation), with per-assignment "required for unlock" flag.
