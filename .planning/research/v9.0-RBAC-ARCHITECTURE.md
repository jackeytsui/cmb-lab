# Architecture: Discord-Style RBAC for CantoMando LMS

**Domain:** Role-based access control integration into existing LMS
**Researched:** 2026-02-14
**Confidence:** HIGH (all code paths verified against codebase, patterns derived from existing schema conventions)

---

## Executive Summary

This document specifies how a Discord-style RBAC system integrates with the existing CantoMando LMS architecture. The system adds four new database tables (`roles`, `role_courses`, `role_features`, `user_roles`) alongside the existing `course_access` table. A centralized Permission Resolver replaces scattered `courseAccess` queries. The existing `hasMinimumRole()` function for coach/admin dashboard gating remains **unchanged** -- it is orthogonal to student RBAC.

The architecture follows the **additive union model**: roles stack, permissions are the union of all assigned roles plus any direct `course_access` overrides. There is no "deny" mechanic. The existing enrollment webhook gains a `roleId`/`roleName` parameter while retaining backward compatibility with `courseId`-only payloads.

## Current Architecture (Before RBAC)

### Access Control Flow

```
User Request
    |
    v
[middleware.ts] ---- Edge Runtime, claims-only ----
    | reads sessionClaims.metadata.role
    | blocks: student -> /admin, student -> /coach
    | allows: everything else through
    v
[Server Component / API Route]
    | calls hasMinimumRole("coach") or hasMinimumRole("admin")
    | resolveRole(): claims first, DB fallback
    v
[Course Access Check] ---- per-page, scattered ----
    | direct SQL: courseAccess JOIN users JOIN courses
    | WHERE users.clerkId = clerkId
    |   AND courseAccess.expiresAt IS NULL OR > now()
    v
[Render / Respond]
```

### Current Authority Boundaries

| Concern | Authority | Mechanism | Files |
|---------|-----------|-----------|-------|
| "Is this user a coach/admin?" | `users.role` enum + Clerk claims | `hasMinimumRole()` in `src/lib/auth.ts` | ~117 files |
| "Can this student see Course X?" | `course_access` table | Direct SQL JOINs, no centralized function | ~50 files |
| "Can this student use Feature Y?" | None (all features available) | No gating exists | 0 files |
| "What tier of access?" | `course_access.accessTier` | `preview` / `full` enum per row | ~5 files |

### Key Architectural Constraints

1. **Edge Runtime in middleware**: Cannot import `db` (Neon HTTP driver not available on Edge). Middleware can only read Clerk session claims.
2. **Server-only DB**: `src/db/index.ts` has `import 'server-only'` guard. All DB access is server components or API routes.
3. **Neon HTTP driver**: Uses `@neondatabase/serverless` with `neon()` HTTP connector, NOT WebSocket. Each query is an independent HTTP request. No persistent connection pool.
4. **Drizzle ORM**: Schema-first with `drizzle-kit` migrations. Schema files in `src/db/schema/`. Migrations in `src/db/migrations/`.
5. **Clerk session token**: Refreshes every 60 seconds. Custom claims via `publicMetadata`. ~1.2KB available for custom data.

## Recommended Architecture (With RBAC)

### Access Control Flow (New)

```
User Request
    |
    v
[middleware.ts] ---- UNCHANGED ----
    | Same claims-based routing for /admin, /coach
    | Does NOT check RBAC roles (Edge Runtime limitation)
    | Student RBAC is invisible to middleware
    v
[Server Component / API Route]
    |
    +--- Coach/Admin pages: hasMinimumRole() ---- UNCHANGED
    |
    +--- Student content: resolvePermissions(userId) ---- NEW
            |
            v
       [Permission Resolver] ---- src/lib/permissions.ts
            | 1. Load user_roles (filter expired)
            | 2. Load role_courses for all active roles
            | 3. Load role_features for all active roles
            | 4. Load direct course_access (legacy/overrides)
            | 5. Union everything
            | 6. Return PermissionSet
            v
       [PermissionSet]
            .canAccessCourse(courseId) -> boolean
            .canUseFeature(featureKey) -> boolean
            .getAccessTier(courseId) -> 'preview' | 'full' | null
            .getAccessibleCourseIds() -> string[]
            .getSourcesForCourse(courseId) -> { roleIds: string[], directGrant: boolean }
```

### Component Boundaries

| Component | Responsibility | Location | Communicates With |
|-----------|---------------|----------|-------------------|
| **Permission Resolver** | Computes effective permissions from roles + direct grants | `src/lib/permissions.ts` | DB (read), called by pages/routes |
| **Role Schema** | Drizzle table definitions for roles, role_courses, role_features, user_roles | `src/db/schema/roles.ts` | DB via Drizzle |
| **Role Admin API** | CRUD for roles, role-course/feature mappings | `src/app/api/admin/roles/` | Permission Resolver, DB |
| **Role Assignment API** | Assign/remove roles on users | `src/app/api/admin/students/[id]/roles/` | DB, Clerk (metadata sync) |
| **Feature Gate Component** | Server component wrapper that checks feature permission before rendering | `src/components/auth/FeatureGate.tsx` | Permission Resolver |
| **Enrollment Webhook** | Modified to accept roleId/roleName | `src/app/api/webhooks/enroll/route.ts` | DB, Clerk |
| **hasMinimumRole()** | **UNCHANGED** -- coach/admin gating | `src/lib/auth.ts` | Clerk claims, DB fallback |
| **middleware.ts** | **UNCHANGED** -- route-level auth | `src/middleware.ts` | Clerk claims only |

## Schema Design

### New Tables

#### `roles` -- Named permission packages

```typescript
// src/db/schema/roles.ts

export const roles = pgTable("roles", {
  id: uuid("id").defaultRandom().primaryKey(),
  name: text("name").notNull().unique(),      // "Bronze", "Premium", "Trial"
  description: text("description"),
  color: text("color").notNull().default("#6b7280"), // Hex for UI badges
  allCourses: boolean("all_courses").notNull().default(false), // Wildcard flag
  isDefault: boolean("is_default").notNull().default(false),   // Auto-assign on signup
  sortOrder: integer("sort_order").notNull().default(0),
  createdBy: uuid("created_by").references(() => users.id),
  createdAt: timestamp("created_at").notNull().defaultNow(),
  updatedAt: timestamp("updated_at")
    .notNull()
    .defaultNow()
    .$onUpdate(() => new Date()),
  deletedAt: timestamp("deleted_at"),       // Soft delete (codebase pattern)
});
```

**Design decisions:**
- `name` is UNIQUE: webhook lookup by name requires uniqueness. Case-insensitive lookup in query.
- `allCourses` boolean: Avoids maintaining N role_courses rows for "Premium = everything". Resolver checks this before querying role_courses.
- `isDefault`: If true, auto-assigned to new users via enrollment webhook. Only one role should have this flag.
- `deletedAt`: Soft delete follows codebase convention (`courses`, `lessons`, `notifications` all use this pattern).
- No `accessTier` on roles table itself. Tier is per role-course pair (in `role_courses`).

#### `role_courses` -- Which courses a role grants

```typescript
export const roleCourses = pgTable("role_courses", {
  id: uuid("id").defaultRandom().primaryKey(),
  roleId: uuid("role_id")
    .notNull()
    .references(() => roles.id, { onDelete: "cascade" }),
  courseId: uuid("course_id")
    .notNull()
    .references(() => courses.id, { onDelete: "cascade" }),
  accessTier: accessTierEnum("access_tier").notNull().default("full"),
  createdAt: timestamp("created_at").notNull().defaultNow(),
}, (table) => [
  uniqueIndex("role_courses_role_course_unique").on(table.roleId, table.courseId),
  index("role_courses_role_id_idx").on(table.roleId),
]);
```

**Design decisions:**
- `accessTier` per role-course: "Trial" role can grant `preview` for Course A but `full` for Course B.
- Reuses existing `accessTierEnum` from `src/db/schema/access.ts` -- no new enum needed.
- `ON DELETE CASCADE` on both FKs: Deleting a role cleans up mappings. Deleting a course cleans up mappings.
- Unique constraint on (roleId, courseId): Prevents duplicate mappings.
- Not needed when `roles.allCourses = true` -- resolver skips this table for wildcard roles.

#### `role_features` -- Which platform features a role grants

```typescript
// Feature keys -- fixed enum, defined in code
export const featureKeyEnum = pgEnum("feature_key", [
  "ai_conversation",    // Voice AI practice
  "practice_sets",      // Coach-created exercises
  "dictionary_reader",  // Text reader with dictionary lookup
  "listening_lab",      // YouTube listening exercises
  "video_threads",      // Video conversation threads
  "certificates",       // Completion certificates
  "ai_chat",            // Text-based AI assistant
]);

export const roleFeatures = pgTable("role_features", {
  id: uuid("id").defaultRandom().primaryKey(),
  roleId: uuid("role_id")
    .notNull()
    .references(() => roles.id, { onDelete: "cascade" }),
  featureKey: featureKeyEnum("feature_key").notNull(),
  createdAt: timestamp("created_at").notNull().defaultNow(),
}, (table) => [
  uniqueIndex("role_features_role_feature_unique").on(table.roleId, table.featureKey),
  index("role_features_role_id_idx").on(table.roleId),
]);
```

**Design decisions:**
- `featureKeyEnum` is a Postgres enum, NOT arbitrary text. Adding a feature requires a code change + migration, not admin configuration. This is intentional -- prevents permission sprawl (Anti-Feature AF-1 from FEATURES.md).
- Seven features map 1:1 to actual platform entry points. Verified against codebase.
- No `accessLevel` on features -- features are binary (granted or not). This avoids the complexity of "partial feature access."

#### `user_roles` -- Which roles are assigned to a user

```typescript
export const userRoles = pgTable("user_roles", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  roleId: uuid("role_id")
    .notNull()
    .references(() => roles.id, { onDelete: "cascade" }),
  assignedBy: uuid("assigned_by").references(() => users.id), // null = webhook/system
  expiresAt: timestamp("expires_at"),   // null = permanent
  createdAt: timestamp("created_at").notNull().defaultNow(),
}, (table) => [
  uniqueIndex("user_roles_user_role_unique").on(table.userId, table.roleId),
  index("user_roles_user_id_idx").on(table.userId),
  index("user_roles_role_id_idx").on(table.roleId),
]);
```

**Design decisions:**
- `expiresAt` nullable: Matches existing `course_access` pattern. Resolver filters expired at query time -- no background job needed.
- `assignedBy` nullable: Webhook/system assignments have no human actor. Manual assignments track the admin/coach who assigned.
- Unique on (userId, roleId): A user can hold each role at most once. Prevents duplicates from webhook retries.
- Multiple roles per user: This is the core Discord pattern -- roles stack additively.
- No `source` column (unlike `course_access.grantedBy`): The `assignedBy` being null implies webhook/system. Non-null implies manual. This is simpler and sufficient.

### Existing Tables -- Changes

#### `course_access` -- NO CHANGES

The `course_access` table **remains as-is**. It becomes the "direct override" layer:

- Existing rows continue working (backward compatible)
- New role-derived access is resolved via `user_roles` + `role_courses`
- If a student has BOTH a role granting Course X AND a direct `course_access` row for Course X, the resolver unions them (both count)
- When roles grant `preview` but direct grant says `full`, `full` wins (highest tier wins)
- Admin/coach can still grant direct `course_access` for one-off overrides

**Why not deprecate:** ~50 files query `course_access` directly. Deprecation requires migrating all 50 files. The resolver abstracts both sources, so existing queries can be migrated incrementally.

#### `users` -- NO CHANGES to schema

The `role` column stays. It continues to serve `hasMinimumRole()` for coach/admin gating. It is NOT involved in student RBAC at all. The two systems are orthogonal:

| System | Purpose | Column/Table | Function |
|--------|---------|-------------|----------|
| Legacy role | Coach/admin dashboard access | `users.role` | `hasMinimumRole()` |
| RBAC roles | Student content/feature access | `user_roles` + `roles` | `resolvePermissions()` |

This separation is critical. Conflating them (e.g., putting RBAC roles in `users.role`) would require migrating 117 files that call `hasMinimumRole()`.

## Permission Resolver

### Core Interface

```typescript
// src/lib/permissions.ts

export interface PermissionSet {
  /** All course IDs this user can access (from roles + direct grants) */
  courseIds: Set<string>;

  /** All feature keys this user can access (from roles) */
  features: Set<string>;

  /** Course ID -> highest access tier (full > preview) */
  accessTiers: Map<string, "preview" | "full">;

  /** Whether any role has allCourses=true */
  hasWildcardAccess: boolean;

  /** Attribution: course ID -> { roleIds, directGrant } */
  sources: Map<string, { roleIds: string[]; directGrant: boolean }>;

  // Convenience methods
  canAccessCourse(courseId: string): boolean;
  canUseFeature(featureKey: string): boolean;
  getAccessTier(courseId: string): "preview" | "full" | null;
}
```

### Resolution Algorithm

```typescript
export async function resolvePermissions(userId: string): Promise<PermissionSet> {
  // 1. Load active role assignments (filter expired)
  const activeRoles = await db
    .select({
      roleId: userRoles.roleId,
      allCourses: roles.allCourses,
    })
    .from(userRoles)
    .innerJoin(roles, eq(userRoles.roleId, roles.id))
    .where(
      and(
        eq(userRoles.userId, userId),
        isNull(roles.deletedAt),
        or(
          isNull(userRoles.expiresAt),
          gt(userRoles.expiresAt, new Date())
        )
      )
    );

  const roleIds = activeRoles.map(r => r.roleId);
  const hasWildcardAccess = activeRoles.some(r => r.allCourses);

  if (roleIds.length === 0) {
    // User has no RBAC roles -- fall through to direct grants only
    return resolveDirectGrantsOnly(userId);
  }

  // 2. Load role-course mappings (parallel with features)
  const [roleCourseRows, roleFeatureRows, directGrants] = await Promise.all([
    // Only query if no wildcard (wildcard = all courses)
    hasWildcardAccess
      ? Promise.resolve([])
      : db
          .select({
            roleId: roleCourses.roleId,
            courseId: roleCourses.courseId,
            accessTier: roleCourses.accessTier,
          })
          .from(roleCourses)
          .where(inArray(roleCourses.roleId, roleIds)),

    // Role features
    db
      .select({
        roleId: roleFeatures.roleId,
        featureKey: roleFeatures.featureKey,
      })
      .from(roleFeatures)
      .where(inArray(roleFeatures.roleId, roleIds)),

    // Direct course_access grants (legacy + overrides)
    db
      .select({
        courseId: courseAccess.courseId,
        accessTier: courseAccess.accessTier,
      })
      .from(courseAccess)
      .where(
        and(
          eq(courseAccess.userId, userId),
          or(
            isNull(courseAccess.expiresAt),
            gt(courseAccess.expiresAt, new Date())
          )
        )
      ),
  ]);

  // 3. Build permission set
  const courseIds = new Set<string>();
  const features = new Set<string>();
  const accessTiers = new Map<string, "preview" | "full">();
  const sources = new Map<string, { roleIds: string[]; directGrant: boolean }>();

  // From role-course mappings
  for (const row of roleCourseRows) {
    courseIds.add(row.courseId);
    mergeTier(accessTiers, row.courseId, row.accessTier);
    mergeSource(sources, row.courseId, row.roleId, false);
  }

  // From role features
  for (const row of roleFeatureRows) {
    features.add(row.featureKey);
  }

  // From direct grants
  for (const row of directGrants) {
    courseIds.add(row.courseId);
    mergeTier(accessTiers, row.courseId, row.accessTier);
    mergeSource(sources, row.courseId, null, true);
  }

  return {
    courseIds,
    features,
    accessTiers,
    hasWildcardAccess,
    sources,
    canAccessCourse(courseId: string): boolean {
      return hasWildcardAccess || courseIds.has(courseId);
    },
    canUseFeature(featureKey: string): boolean {
      return features.has(featureKey);
    },
    getAccessTier(courseId: string): "preview" | "full" | null {
      if (hasWildcardAccess) return "full"; // Wildcard = full access
      return accessTiers.get(courseId) ?? null;
    },
  };
}

// Tier merge: full > preview (highest wins)
function mergeTier(
  map: Map<string, "preview" | "full">,
  courseId: string,
  tier: "preview" | "full"
) {
  const current = map.get(courseId);
  if (!current || tier === "full") {
    map.set(courseId, tier);
  }
}

// Source merge: track which roles and direct grants contribute
function mergeSource(
  map: Map<string, { roleIds: string[]; directGrant: boolean }>,
  courseId: string,
  roleId: string | null,
  isDirect: boolean
) {
  const current = map.get(courseId) ?? { roleIds: [], directGrant: false };
  if (roleId) current.roleIds.push(roleId);
  if (isDirect) current.directGrant = true;
  map.set(courseId, current);
}
```

### Query Cost Analysis

The resolver executes **3 parallel queries** (or 2 if wildcard):

| Query | Expected Rows | Index Used |
|-------|--------------|------------|
| `user_roles JOIN roles WHERE userId = ?` | 1-5 rows per user | `user_roles_user_id_idx` |
| `role_courses WHERE roleId IN (?)` | 5-50 rows (roles * courses) | `role_courses_role_id_idx` |
| `role_features WHERE roleId IN (?)` | 5-35 rows (roles * features) | `role_features_role_id_idx` |
| `course_access WHERE userId = ?` | 0-10 rows per user | `course_access_user_id_idx` |

**Total:** 3 parallel queries, each returning <50 rows. With Neon HTTP driver, each query is an independent HTTP roundtrip (~50ms each). Running in parallel = ~50ms total wall time. Acceptable for per-request resolution.

### Per-Request Caching

The resolver should be called **once per request** and the result threaded through. Do NOT call it multiple times in the same render.

```typescript
// Pattern: Resolve once at the top of the server component, pass down

// app/(dashboard)/dashboard/page.tsx
export default async function DashboardPage() {
  const user = await getCurrentUser();
  if (!user) redirect("/sign-in");

  const permissions = await resolvePermissions(user.id);

  // Pass permissions to child components
  return (
    <div>
      <CourseGrid permissions={permissions} />
      {permissions.canUseFeature("ai_conversation") && <VoiceConversation />}
    </div>
  );
}
```

For API routes that need multiple permission checks, resolve once at the top:

```typescript
export async function GET(req: NextRequest) {
  const user = await getCurrentUser();
  const permissions = await resolvePermissions(user.id);

  // Use permissions.canAccessCourse() multiple times -- no extra DB queries
}
```

### No Global Cache / No Redis

The codebase does not use Redis. The Neon HTTP driver has no connection pool to manage. Adding Redis for permission caching introduces infrastructure complexity that is not justified for the current scale (<1000 users). The 3-query parallel resolver is fast enough.

If scale becomes an issue later (>10K concurrent users), add Redis with a short TTL (30s) keyed by userId, invalidated on role change webhook. But do not build this upfront.

## Feature Gating Pattern

interface FeatureGateProps {
  featureKey: string;
  permissions: PermissionSet;
  fallback?: React.ReactNode; // What to show when locked

export function FeatureGate({
  featureKey,
  permissions,
  fallback,
  children,

  if (permissions.canUseFeature(featureKey)) {
    return <>{children}</>;
  }
  return fallback ? <>{fallback}</> : null;

## Enrollment Webhook Integration

### Modified Webhook Payload

```typescript
// BACKWARD COMPATIBLE -- all new fields are optional
interface EnrollmentPayload {
  email: string;
  name?: string;

  // EXISTING (legacy) -- still works
  courseId?: string;
  accessTier?: "preview" | "full";
  expiresAt?: string;

  // NEW (RBAC) -- optional
  roleId?: string;     // UUID of a role
  roleName?: string;   // Case-insensitive name lookup
  roleExpiresAt?: string; // Expiration for this role assignment
}
```

### Webhook Handler Logic

```typescript
export async function POST(req: NextRequest) {
  // ... existing auth/rate-limit code ...

  const body = await req.json() as EnrollmentPayload;

  // 1. Find or create user (EXISTING logic, unchanged)
  const dbUser = await findOrCreateUser(body.email, body.name);

  // 2. Handle role assignment (NEW)
  if (body.roleId || body.roleName) {
    let role: Role | null = null;

    if (body.roleId) {
      role = await db.query.roles.findFirst({
        where: and(eq(roles.id, body.roleId), isNull(roles.deletedAt)),
      });
    } else if (body.roleName) {
      // Case-insensitive lookup
      role = await db.query.roles.findFirst({
        where: and(
          sql`LOWER(${roles.name}) = LOWER(${body.roleName})`,
          isNull(roles.deletedAt)
        ),
      });
    }

    if (!role) {
      // CRITICAL: Do not silently skip. Return error so webhook retries.
      return NextResponse.json(
        { error: `Role not found: ${body.roleId || body.roleName}` },
        { status: 404 }
      );
    }

    // Upsert role assignment (idempotent)
    await db
      .insert(userRoles)
      .values({
        userId: dbUser.id,
        roleId: role.id,
        expiresAt: body.roleExpiresAt ? new Date(body.roleExpiresAt) : null,
      })
      .onConflictDoUpdate({
        target: [userRoles.userId, userRoles.roleId],
        set: {
          expiresAt: body.roleExpiresAt ? new Date(body.roleExpiresAt) : null,
        },
      });
  }

  // 3. Handle direct course access (EXISTING logic, unchanged)
  if (body.courseId) {
    // ... existing courseAccess upsert logic ...
  }

  return NextResponse.json({ success: true, userId: dbUser.id });
}
```

**Key design choices:**
- `onConflictDoUpdate` on the unique (userId, roleId) constraint: Idempotent. Sending the same webhook twice updates expiresAt but does not create duplicates.
- Unknown roleId/roleName returns 404, NOT 200: External system should be alerted to misconfiguration. Returning 200 silently loses the enrollment.
- Both `roleId` and `courseId` can be present in the same payload: They are processed independently. This supports "assign role AND grant a specific course override" in a single webhook.

## Migration Strategy: courseAccess to Roles

### Phase 1: Additive Schema (No Behavior Change)

1. Add `roles`, `role_courses`, `role_features`, `user_roles` tables via Drizzle migration
2. Add `resolvePermissions()` function
3. All existing code continues using `courseAccess` directly -- no breakage

### Phase 2: Dual-Path Resolution (New Code Uses Resolver)

1. New pages/routes use `resolvePermissions()` instead of raw `courseAccess` queries
2. Existing pages/routes are unchanged (still query `courseAccess` directly)
3. Both paths produce correct results because the resolver includes `courseAccess` in its union

### Phase 3: Admin Creates Roles and Assigns

1. Admin creates roles via new admin UI
2. Admin assigns roles to students
3. Students now have access from BOTH roles (via resolver) AND direct `courseAccess` (via existing queries)
4. This dual access is correct and harmless -- it's additive

### Phase 4: Gradual Migration of Existing Queries

Migrate files that query `courseAccess` directly to use the resolver:

**Priority 1 -- Dashboard and course listing (3 files):**
- `src/app/(dashboard)/dashboard/page.tsx`
- `src/app/(dashboard)/courses/page.tsx`
- `src/app/api/courses/route.ts`

**Priority 2 -- Lesson access check (1 file):**
- `src/app/(dashboard)/lessons/[lessonId]/page.tsx`

**Priority 3 -- Assignment resolution (3 files):**
- `src/lib/assignments.ts`
- `src/lib/video-assignments.ts`
- `src/lib/thread-assignments.ts`

**Priority 4 -- Admin/coach views (remaining files):**
- Student access management routes
- Bulk operations route
- Analytics routes

Each file migration is independent and can be done in any order. The resolver always includes `courseAccess` data, so switching a file from direct query to resolver does not change behavior.

### Phase 5: Migration Tool (Optional)

After roles are configured, an admin tool can:
1. Analyze existing `course_access` patterns
2. Suggest roles that match common access patterns
3. Create roles and assign them to matching students
4. Optionally archive the now-redundant direct `course_access` rows

Direct `course_access` rows are never deleted automatically. They become "override" grants. Archiving is optional and manual.

### Backward Compatibility Guarantees

| Scenario | Before RBAC | During Migration | After Full Migration |
|----------|-------------|-----------------|---------------------|
| Student with `course_access` only | Works via direct query | Works via direct query OR resolver | Works via resolver (includes `course_access`) |
| Student with roles only | N/A | N/A | Works via resolver |
| Student with roles + `course_access` | N/A | Works (union of both) | Works (union of both) |
| `hasMinimumRole("coach")` | Works | Works (unchanged) | Works (unchanged) |
| Enrollment webhook with `courseId` | Works | Works (unchanged) | Works (unchanged) |
| Enrollment webhook with `roleId` | N/A | Works (new code) | Works |
| Bulk `assign_course` | Works | Works (unchanged) | Works (unchanged) |
| Bulk `assign_role` | N/A | Works (new operation) | Works |

**No destructive changes. No behavior changes for existing users. No migration required for existing students to retain access.**

## Patterns to Follow

### Pattern 1: Resolve Once, Use Many

**What:** Call `resolvePermissions()` once at the top of a server component or API route. Pass the result down. Never call it multiple times in the same request.

**Why:** Each call makes 3 parallel DB queries. Multiple calls = wasted queries.

```typescript
// GOOD: Resolve once
export default async function Page() {
  const permissions = await resolvePermissions(userId);
  return (
    <>
      <CourseGrid permissions={permissions} />
      <FeatureGate featureKey="ai_chat" permissions={permissions}>
        <ChatWidget />
      </FeatureGate>
    </>
  );
}

// BAD: Resolve multiple times
export default async function Page() {
  const p1 = await resolvePermissions(userId); // Query 1
  const p2 = await resolvePermissions(userId); // Query 2 (duplicate!)
}
```

### Pattern 2: Orthogonal Auth Systems

**What:** `hasMinimumRole()` and `resolvePermissions()` are completely separate. Never mix them.

**Why:** `hasMinimumRole()` controls "which dashboard" (student/coach/admin). `resolvePermissions()` controls "what content/features within the student experience."

```typescript
// GOOD: Clear separation
// Coach page: uses hasMinimumRole
export default async function CoachPage() {
  if (!(await hasMinimumRole("coach"))) redirect("/dashboard");
  // ... coach logic, no resolvePermissions needed
}

// Student page: uses resolvePermissions
export default async function StudentPage() {
  const permissions = await resolvePermissions(userId);
  if (!permissions.canAccessCourse(courseId)) redirect("/dashboard");
  // ... student logic, no hasMinimumRole needed
}

// BAD: Mixing systems
export default async function Page() {
  if (await hasMinimumRole("tier_2")) { ... } // tier_2 is not a legacy role!
}
```

### Pattern 3: Feature Gate at Entry Points Only

**What:** Check feature permissions at the page/route level, not deep inside components.

**Why:** Deep permission checks are invisible, hard to test, and create confusing UX (component renders but sub-component is missing).

```typescript
// GOOD: Gate at page level
export default async function ReaderPage() {
  const permissions = await resolvePermissions(userId);
  if (!permissions.canUseFeature("dictionary_reader")) {
    return <UpgradePrompt feature="dictionary_reader" />;
  }
  return <Reader />;
}

// BAD: Gate deep inside
function DictionaryLookup({ word }) {
  const permissions = await resolvePermissions(userId); // Extra DB hit
  if (!permissions.canUseFeature("dictionary_reader")) return null; // Silent disappearance
  return <Definition word={word} />;
}
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Storing Permissions in Clerk Claims

**What:** Putting role IDs, feature keys, or course IDs in Clerk `publicMetadata`.

**Why bad:** Clerk session tokens have ~1.2KB available for custom data. Role names are fine (~20 bytes). Course ID lists would overflow. Feature arrays would overflow. And the data would be stale for up to 60 seconds.

**Instead:** Keep `publicMetadata` minimal: `{ role: "student" | "coach" | "admin" }`. Resolve RBAC permissions from the database on each request.

### Anti-Pattern 2: Querying courseAccess in New Code

**What:** New pages or API routes that import `courseAccess` and write raw JOIN queries.

**Why bad:** Bypasses the permission resolver. Does not account for role-based access. Creates another file that must be migrated later.

**Instead:** All new code uses `resolvePermissions()`. Existing code is migrated incrementally.

### Anti-Pattern 3: Per-Student Feature Overrides

**What:** A table like `user_features` that grants features directly to individual users.

**Why bad:** Creates a shadow permission system. "Why can this student use AI chat?" becomes unsolvable when both roles AND individual overrides exist. Debugging nightmare.

**Instead:** Create a small role. "AI Chat Add-on" role with just the `ai_chat` feature. Assign it to the one student. Roles are cheap. The resolver handles the union.

### Anti-Pattern 4: Role Hierarchy / Inheritance

**What:** "Premium inherits from Standard which inherits from Basic."

**Why bad:** Editing Basic silently changes Premium. Coaches will not understand cascading effects. Every RBAC guide warns against this for small-team systems.

**Instead:** Flat roles with explicit permissions. If Premium needs everything Basic has, list the same courses/features on both. The data duplication (a few join table rows) is trivial; the clarity is worth it.

## Scalability Considerations

| Concern | At 100 users | At 1K users | At 10K users |
|---------|-------------|-------------|-------------|
| Resolver query time | ~50ms (3 parallel queries) | ~50ms (same, index-backed) | ~80ms (slightly larger result sets) |
| user_roles table size | ~200 rows | ~2K rows | ~20K rows |
| role_courses table size | ~30 rows | ~30 rows (courses, not users) | ~50 rows |
| Migration to Redis caching | Not needed | Not needed | Consider if >100ms p95 |
| Resolver calls per page | 1 | 1 | 1 (resolve once pattern) |
| Course listing query | Simple (resolver returns Set) | Simple | Consider pagination if >100 courses |

The bottleneck at scale is NOT the resolver. It is the existing raw courseAccess JOIN queries that do full table scans. The resolver with proper indexes is actually faster than the existing pattern.
