# Production Readiness Research Summary

**Project:** CantoMando Blueprint LMS - Milestone v2.0
**Domain:** Production Infrastructure & Observability for Educational SaaS
**Researched:** 2026-01-30
**Confidence:** HIGH

## Executive Summary

Version 2.0 transforms the working CantoMando LMS prototype into a production-grade platform that coaches can rely on and students can use seamlessly on mobile devices. Research across 9 production readiness features reveals a clear pattern: most features follow well-established patterns with proven library choices. The recommended approach prioritizes infrastructure protection first (error monitoring, CI/CD, rate limiting), then quality assurance (E2E testing), followed by user-facing enhancements (PWA, notifications, search), and finally insights/rewards (analytics, certificates).

The critical finding is that these features are **additive** — they wrap or extend existing code rather than requiring major refactoring. The Next.js 16 App Router architecture with Clerk auth, Drizzle ORM, and Neon Postgres provides solid foundations. Key risks center on four areas: (1) Sentry leaking student PII without proper scrubbing, (2) E2E tests becoming permanently flaky due to Clerk/Mux/n8n external dependencies, (3) PWA service workers caching stale lesson content, and (4) serverless limitations breaking SSE-based notifications. Each risk has well-documented mitigation strategies.

For a 2-3 coach team with paid students, the sweet spot is solid fundamentals without enterprise overhead. Error monitoring and CI/CD are the highest-priority items because they protect everything else. Total additional monthly cost at launch is $0 — all services stay within generous free tiers for this scale (Sentry 5K errors/month, PostHog 1M events/month, Upstash 10K commands/day, Vercel Hobby, GitHub Actions 2000 min/month).

## Key Findings

### Recommended Stack

The production stack adds 7 npm packages across 9 features. No major framework changes required — everything integrates with existing Next.js 16 architecture.

**Core technologies:**
- **@sentry/nextjs (10.37.0)** — Error monitoring, performance tracking, session replay with verified Next.js 16 compatibility
- **@playwright/test (1.57.0)** — E2E testing with cross-browser support (Chromium + WebKit for Safari) and parallel execution
- **@serwist/next (9.5.0)** — PWA service worker (maintained successor to next-pwa, works with Next.js 16 Turbopack in production)
- **@upstash/ratelimit (2.0.8)** — Serverless-friendly rate limiting with Redis (connectionless HTTP-based)
- **@react-pdf/renderer (4.3.2)** — PDF certificate generation with React components (React 19 compatible, no Puppeteer bloat)
- **posthog-js + posthog-node** — Product analytics with session replay, feature flags, and LMS-specific funnels
- **No new packages for:** Notifications (custom SSE + Postgres), Search (Postgres FTS), CI/CD (GitHub Actions YAML)

**Critical decision: Avoid over-engineering**
- No Knock/Novu for notifications (custom SSE is sufficient for low volume)
- No Meilisearch/Algolia for search initially (Postgres FTS handles < 10K documents)
- No Puppeteer for PDFs (serverless-incompatible, @react-pdf/renderer is faster and lighter)

**Version compatibility:** All packages verified compatible with Next.js 16 and React 19. Sentry test suite explicitly includes Next.js 16.0.7.

### Expected Features

Research identified clear feature tiers based on domain expectations for production LMS platforms.

**Must have (table stakes):**
- **Error monitoring** — Frontend/backend error capture, source maps, user context, environment separation (Sentry's default setup handles this)
- **E2E testing** — Critical path coverage (enrollment → lesson → interaction → grading), auth flow, mobile viewport tests (Playwright required over Cypress for Safari support)
- **Rate limiting** — Webhook protection, AI endpoint throttling, graceful 429 responses with retry headers (Upstash Redis essential for serverless)
- **PWA basics** — Manifest, service worker, install prompt, offline fallback page (Serwist handles generation)
- **In-app notifications** — Bell icon, unread count, dropdown list, mark as read, notification types (custom build, no third-party needed)
- **Basic search** — Search bar in header, courses/lessons by title, search results page (Postgres ILIKE initially)
- **CI/CD pipeline** — Automated build/lint/typecheck on PR, auto-deploy on merge to main, preview deployments (GitHub Actions + Vercel)

**Should have (competitive):**
- **Session Replay** — See exactly what student did before error occurred (Sentry built-in, just enable)
- **Performance monitoring** — Track slow pages, Web Vitals, API route latency (Sentry tracing)
- **PWA install prompt** — Banner prompting mobile users to install (beforeinstallprompt event handler)
- **Notification preferences** — Students choose which events trigger in-app vs email (settings UI + preferences table)
- **Fuzzy search** — Tolerate typos, Chinese character search, Pinyin/Jyutping search (pg_search with ICU tokenizer)
- **Analytics dashboard** — Course completion rates, lesson drop-off points, coach response time, pending review count (queries on existing tables)
- **PDF certificates** — Student name, course name, completion date, unique ID, professional design (@react-pdf/renderer)

**Defer (v2+):**
- Push notifications (PWA) — Requires Safari 16.4+, still inconsistent on iOS
- Background sync — Queue submissions when offline, send when back online
- Real-time notifications via WebSocket — Polling every 30-60s is fine for small user base
- Custom certificate designs per course — Single template sufficient initially
- Public certificate verification page — Nice to have, not launch-critical
- Lighthouse CI — Performance regression tracking (defer until baseline established)

### Architecture Approach

The existing LMS has 50 API routes, 58 components, 12 Drizzle schema tables, and clean separation between Server Components (data fetching) and Client Components (interactivity). Production features integrate at four distinct layers with minimal code changes.

**Major components:**

1. **Infrastructure layer (wraps existing code)** — Sentry wraps all layers via instrumentation hooks, CI/CD via GitHub Actions workflows, rate limiting via new middleware.ts. Zero changes to existing business logic.

2. **Data layer (extends schema)** — Three new tables: `notifications` (id, userId, type, title, body, readAt, createdAt), `certificates` (id, userId, courseId, issuedAt, certificateNumber, pdfUrl), search extensions via `tsvector` generated columns on existing courses/lessons tables with GIN indexes.

3. **Client layer (new capabilities)** — PWA adds service worker (src/app/sw.ts), manifest (src/app/manifest.ts), and InstallPrompt component. Certificates add PDF template component. NotificationBell component in header.

4. **Testing layer (observes externally)** — Playwright tests sit outside production code, use @clerk/testing for auth, mock Mux/n8n responses. No production code changes required.

**Key patterns identified:**
- Use Clerk's `@clerk/testing/playwright` with `clerkSetup()` for E2E auth (reuse session state, avoid re-login)
- Service worker caching strategy: CacheFirst for static assets, NetworkFirst for API routes, NetworkOnly for auth/chat/realtime, NEVER cache lesson content (changes frequently)
- Rate limiting in middleware with Upstash Redis (in-memory Map fails silently in serverless due to stateless functions)
- SSE for notifications will hit Vercel's serverless timeout — use polling fallback every 30s
- Sentry's `beforeSend` hook is critical for PII scrubbing (student IDs in URLs, Chinese text in breadcrumbs, console logs with grading responses)

**New files estimate:** ~52 new files total across all 9 features. ~8 modified files (next.config.ts, package.json, layout.tsx, schema/index.ts, .gitignore).

### Critical Pitfalls

Research identified 15 pitfalls across critical/moderate/minor categories. Top 5 with highest impact:

1. **Sentry leaks student PII through URLs, breadcrumbs, and replay** — Default behavior captures full request URLs (e.g., /students/clerk_abc123), query params, console logs with student data, and Session Replay network bodies with Chinese text submissions. **Prevention:** Configure `beforeSend` scrubbing, set `sendDefaultPii: false`, disable console breadcrumbs, filter Chinese text with regex `/[\u4e00-\u9fff]/`, scrub student IDs from URLs. Set Session Replay sample rates to 0 for passive recording, 0.5 only on errors.

2. **E2E tests become permanently flaky due to Clerk auth and Mux video** — Tests combining Clerk auth, Mux video streaming, and n8n webhook AI grading have variable latency. Video load times are non-deterministic. Chinese text input triggers IME composition events that headless browsers handle poorly. **Prevention:** Use Clerk's testing helpers to authenticate once and reuse session state. Mock Mux video and n8n webhook responses — test interface behavior, not external service reliability. Separate test categories: unit (fast), integration (medium), E2E (slow, critical paths only with mocked externals), smoke (real services, nightly, allowed to be flaky).

3. **PWA service worker caches stale LMS content after updates** — Coach updates lesson interaction timestamps or grading rubric. Students on PWA still see old version because service worker serves cached pages. Clearing cache requires manual PWA uninstall. **Prevention:** NEVER cache lesson content or API routes — only static assets (JS/CSS/fonts/icons) and app shell. Implement update notification when new SW is available. Version service worker and force update checks every 5 minutes. Use `stale-while-revalidate` only for navigation, `NetworkFirst` for all dynamic content.

4. **In-memory rate limiting fails silently in serverless** — Rate limiting with Map or express-rate-limit works locally but fails in production because each serverless function invocation gets fresh memory. Bot hits grading API 1000 times because each request goes to different function instance. **Prevention:** Use Upstash Redis from the start with `@upstash/ratelimit` package. Different limits for different endpoints (grading: 10/min, chat: 20/min, default: 60/min). Identify by Clerk userId, fallback to IP for anonymous. Return proper 429 responses with X-RateLimit-* headers.

5. **CI/CD database migrations corrupt production data** — Developer creates Drizzle migration that renames a column. CI/CD runs `drizzle-kit push` against production. Running app still references old column name, all API routes return 500 errors. **Prevention:** NEVER use `drizzle-kit push` in production (development only). Use versioned migrations: `drizzle-kit generate` then `drizzle-kit migrate`. Run migrations as separate CI step BEFORE deploy. Use Neon branching for preview deployments (each PR gets its own database branch). Make migrations backward-compatible (add new column → deploy new code → backfill → drop old column).

**Other notable pitfalls:**
- Sentry over-alerting buries real errors in noise (filter network errors, expected 401s, browser extension errors in `beforeSend`)
- PDF certificate generation crashes serverless functions (use @react-pdf/renderer, not Puppeteer)
- Notification fatigue causes students to disable all notifications (implement notification tiers, frequency caps, user preferences)
- Full-text search returns irrelevant results for Chinese content (use pg_search with ICU tokenizer for CJK, or store Pinyin/Jyutping in separate indexed columns)
- SSE/WebSocket notifications don't work on Vercel (serverless functions have 10-60s timeout, use polling with smart intervals)

## Implications for Roadmap

Based on dependency analysis and risk mitigation priorities, research suggests a 4-phase structure grouping features by integration layer and risk level.

### Phase 1: Infrastructure Protection (Build First)

**Rationale:** Error monitoring protects all subsequent work by providing visibility into production issues. CI/CD automates quality gates. Rate limiting protects AI endpoints from cost exposure. These three features are fully independent and have zero user-facing components, making them low-risk foundation work.

**Delivers:**
- Sentry error monitoring with PII scrubbing, alert rules, source maps
- GitHub Actions CI pipeline (lint, typecheck, build on every PR)
- Upstash rate limiting in middleware (protects /api/grade, /api/chat, /api/webhooks)

**Addresses:**
- Error monitoring (table stakes from FEATURES.md)
- CI/CD pipeline (table stakes)
- Rate limiting (table stakes)

**Avoids:**
- Pitfall #1: Sentry PII leak (address during setup with beforeSend config)
- Pitfall #4: In-memory rate limiting failure (use Upstash from start)
- Pitfall #6: Database migration corruption (CI/CD migration workflow established)

**Stack elements:** @sentry/nextjs, @upstash/ratelimit, @upstash/redis, GitHub Actions workflows

**Architecture:** Infrastructure layer — wraps existing code, no business logic changes

**Research flag:** Standard patterns, skip deeper research

---

### Phase 2: Quality Assurance

**Rationale:** E2E tests validate existing app functionality and catch regressions as new features are added. Must come after Sentry (provides error visibility) and before user-facing features (tests validate them). This phase has moderate complexity due to Clerk auth and external service mocking requirements.

**Delivers:**
- Playwright test suite covering critical paths (enrollment, lesson completion, grading, coach review)
- Clerk testing setup with reusable auth state
- Mux video and n8n webhook mocking
- CI integration (tests run on every PR after lint/typecheck/build)

**Addresses:**
- E2E testing (table stakes)
- Mobile viewport tests (Playwright's cross-browser support)

**Avoids:**
- Pitfall #2: Flaky E2E tests (mock externals, separate test categories)
- Pitfall #14: Chinese text input edge cases (include IME composition tests)

**Stack elements:** @playwright/test, @clerk/testing (dev dependency)

**Architecture:** Testing layer — observes existing app from outside, zero production code changes

**Research flag:** Needs phase-level research for Clerk testing patterns and Mux/n8n mocking strategies

---

### Phase 3: User-Facing Features

**Rationale:** PWA, notifications, and search are the features students and coaches will directly interact with. They benefit from infrastructure monitoring (Sentry) and test coverage (E2E) being in place. These three features can be built in parallel by different developers or sequentially with minimal dependencies.

**Delivers:**
- **PWA:** Service worker, manifest, install prompt, offline fallback, app icons
- **In-app notifications:** Schema (notifications table), API routes (list, mark-read, SSE/polling), UI (bell icon, dropdown, toast)
- **Search:** Schema extensions (tsvector columns, GIN indexes), API route, SearchBar component, Cmd+K command palette

**Addresses:**
- PWA basics (table stakes)
- In-app notifications (table stakes)
- Basic search (table stakes)
- Notification preferences (differentiator)
- Fuzzy search with Chinese support (differentiator)

**Avoids:**
- Pitfall #3: Stale PWA cache (define caching strategy, never cache lesson content)
- Pitfall #8: Notification fatigue (notification tiers, frequency caps from start)
- Pitfall #9: Chinese search quality (pg_search with ICU tokenizer or Pinyin indexing)
- Pitfall #12: SSE on Vercel (use polling fallback, not pure SSE)

**Stack elements:** @serwist/next, serwist, posthog-js (for analytics event tracking)

**Architecture:** Client layer (PWA service worker), Data layer (notifications table, search indexes), new API routes

**Research flag:**
- **PWA sub-phase:** Needs phase-level research for Serwist caching strategies with Next.js 16 App Router dynamic routes
- **Notifications sub-phase:** Standard patterns (SSE/polling + Postgres), skip deeper research
- **Search sub-phase:** Needs phase-level research for pg_search Chinese tokenization on Neon (MEDIUM confidence from main research)

### Phase 4: Insights and Rewards

**Rationale:** Analytics and certificates are valuable but not blocking daily usage. Analytics queries existing tables (read-only, low risk). Certificates depend on completion tracking working correctly (already in v1.0). Both features benefit from full error monitoring and testing infrastructure being operational.

**Delivers:**
- **Analytics dashboard:** Admin overview page, coach dashboard with pending reviews, progress charts, engagement tables
- **PDF certificates:** Schema (certificates table), PDF template with Chinese font support, generation API route, download UI

**Addresses:**
- Analytics dashboard (differentiator)
- PDF certificates (table stakes for course completion)
- Public verification URL (differentiator, optional)

**Avoids:**
- Pitfall #7: PDF timeout (use @react-pdf/renderer, not Puppeteer, with proper font embedding)
- Pitfall #10: Vanity metrics (track learning metrics like interaction pass rate, lesson drop-off, coach response time)
- Pitfall #11: Neon cold starts (implement connection warming, cache analytics results with 15min TTL)

**Stack elements:** @react-pdf/renderer, posthog-node (server-side analytics)

**Architecture:** API/Server layer (analytics queries in RSC, PDF generation in route handler), Data layer (certificates table)

**Research flag:** Standard patterns, skip deeper research

### Phase Ordering Rationale

1. **Infrastructure first** because it makes everything safer. Sentry catches issues in all subsequent work. CI/CD prevents regressions. Rate limiting protects cost exposure on AI endpoints.

2. **Testing second** because it validates existing functionality and prevents regressions as new features are added. E2E tests running in CI provide confidence for parallel feature work.

3. **User-facing features third** because they're what students and coaches notice, but they depend on infrastructure stability. PWA, notifications, and search can be built in parallel with minimal interdependencies.

4. **Analytics and certificates last** because they're valuable but not blocking daily usage. Analytics is read-only (low risk). Certificates require completion tracking to be rock-solid.

**Dependency chain:**
```
Sentry (independent) ──┐
CI/CD (independent) ───┼─→ E2E Tests ──┐
Rate Limiting (indep) ─┘               ├─→ PWA ──────┐
                                       │              │
                                       ├─→ Notifs ───┼─→ Analytics ──┐
                                       │              │               ├─→ Done
                                       └─→ Search ───┘               │
                                                                      │
                                          Certificates ──────────────┘
```
