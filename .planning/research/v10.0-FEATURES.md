# Feature Landscape

**Domain:** Chinese Language Learning LMS -- SRS Flashcards, Smart Study Engine, Tone Training, HSK Assessment, Grammar Library, Auto-Exercise Generation, AI Prompt Testing Lab
**Researched:** 2026-02-16
**Milestone:** SRS + Study Intelligence (subsequent milestone, builds on v1.0 through v9.0)
**Confidence:** HIGH (features well-understood from established competitors; implementation patterns verified against open-source libraries and official documentation)

---

## Executive Summary

This research maps the feature landscape for seven interconnected capabilities that transform CantoMando from a content-delivery LMS into an adaptive learning platform. The features divide into two categories: **memorization infrastructure** (SRS flashcards, grammar library, tone training) and **intelligence layer** (smart study recommendations, assessment/placement, auto-exercise generation, prompt testing lab).

Research across Chinese learning apps (Anki, Pleco, Skritter, Duolingo, Busuu, Ka Tones, Dong Chinese, HanyuAce, SuperTest), open-source spaced repetition libraries (ts-fsrs, FSRS v6), grammar reference standards (AllSet Learning Grammar Wiki, ~450 patterns across HSK 1-6), and LLM exercise generation papers reveals clear patterns for what users expect and where CantoMando can differentiate.

The existing codebase provides strong foundations: `saved_vocabulary` with 145K dictionary entries, 7 exercise types with practice set builder, Azure TTS with Redis caching, HanziWriter for stroke animations, AI grading with versioned prompts, XP/streak gamification, and progress tracking. The new features extend these rather than replacing them.

**Key insight:** The single highest-impact feature is SRS flashcards because students already save vocabulary (the input data exists) and have no way to systematically review it (the output loop is missing). Everything else builds on or benefits from SRS being in place.

## Table Stakes

Features users expect from a premium Chinese learning platform with these capabilities. Missing any = product feels incomplete compared to Anki, Pleco, Skritter, Duolingo, Busuu, etc.

### SRS Flashcards

| Feature | Why Expected | Complexity | Dependencies on Existing | Notes |
|---------|--------------|------------|--------------------------|-------|
| **FSRS-based scheduling algorithm** | Industry standard since Anki adopted FSRS; SM-2 feels outdated. FSRS gives 20-30% fewer reviews for same retention. | Low | None -- `ts-fsrs` npm package handles algorithm | Use ts-fsrs v4+ (FSRS v6). Do NOT roll your own scheduler. |
| **Card states: New / Learning / Review / Relearning** | Users expect to see where cards are in the pipeline. FSRS natively tracks this. | Low | New `srs_cards` + `srs_reviews` tables | Maps directly to ts-fsrs Card state model |
| **Four rating buttons: Again / Hard / Good / Easy** | Standard SRS interaction. Anki, Mochi, every SRS app uses this. | Low | None | Show next review date for each rating choice |
| **Auto-card creation from saved vocabulary** | Users already save words in the reader (`saved_vocabulary` table). Cards should auto-generate. | Med | `saved_vocabulary`, `dictionary_entries` for enrichment | Each saved word creates card(s) automatically; user can also create manually |
| **Multiple card faces per word** | Chinese requires testing recognition + recall + audio. Minimum: Character->Meaning, Meaning->Character, Audio->Character. | Med | `dictionary_entries` (pinyin, jyutping, definitions), Azure TTS | Generate 2-3 sibling cards per vocabulary item |
| **Daily review queue with count badge** | "You have 42 cards due" -- core SRS loop motivator | Low | `srs_cards` table WHERE due <= now | Badge in sidebar nav, count on dashboard |
| **Review session player** | Full-screen card review with flip animation, rating buttons, progress bar | Med | SRS card data, TTS for audio cards | Card front -> tap to reveal back -> rate. Similar to Anki mobile. |

| **Vocabulary list to deck conversion** | Existing vocabulary lists should be reviewable as SRS decks | Low | `vocabulary_lists`, `vocabulary_list_items` tables | One-click "Start Reviewing" on any vocab list |

## Anti-Features

Features to explicitly NOT build. These are traps that waste development time, add unjustified complexity, or degrade the product experience.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Full CAT/IRT adaptive testing engine** | Item Response Theory requires thousands of calibrated test items and statistical analysis. Overkill for ~50-200 premium students. Building a proper item bank with calibrated difficulty parameters is a research project. | Simple adaptive logic: 3 correct at level N -> jump to N+1; 3 wrong -> drop to N-1. Good enough for placement. |
| **Real-time pitch detection with ML model** | Requires training data, browser ML inference (TensorFlow.js). Research papers show only moderate accuracy for Mandarin tone classification even in controlled environments. Noisy real-world conditions degrade further. | Record + playback comparison with reference audio. Static pitch contour diagrams for visual reference. Azure pronunciation assessment API if more precision needed later. |
| **Custom SRS algorithm** | FSRS is the result of years of academic research backed by the Anki community, outperforms SM-2 by 20-30% in benchmarks. Rolling your own is engineering waste with worse outcomes. | Use `ts-fsrs` package directly. Customize parameters (desired retention 0.85-0.95, maximum interval) but not the core scheduling algorithm. |
| **User-created card templates** | Anki's HTML/CSS template system is powerful but incredibly complex to build and support. Premium users on a curated platform don't need this when the platform controls the learning experience. | Provide 3-5 fixed card types: Character->Meaning, Meaning->Character, Audio->Character, Handwriting, Sentence Cloze. Covers all Chinese learning needs. |
| **Social/community grammar wiki editing** | Wikipedia-style community editing requires moderation, version control UI, spam prevention, edit conflict resolution. Small premium audience doesn't justify this infrastructure. | Coaches author/edit grammar content through admin panel. Students consume read-only reference material. |
| **Gamified assessment badges/rankings** | Placement tests should be diagnostic tools, not competitions. Gamifying them encourages "studying for the test" instead of honest self-assessment. Students optimize for scores rather than accurate placement. | Present results as "Your current level" with growth areas identified. No scores, rankings, or competitive elements on assessments. |
| **Offline-first SRS** | Service worker + IndexedDB sync for offline flashcard review is a massive engineering effort involving conflict resolution, sync queues, and data integrity across devices. PWA exists but full offline data sync is a fundamentally different problem. | Cards need network for TTS audio and AI features. Show "Connect to review" message offline. Browser cache handles brief disconnections only. |
| **Auto-generated TTS for every grammar example at seed time** | Pre-generating TTS for hundreds of example sentences during database seeding is expensive ($$$), slow, and wasteful since most sentences may never be viewed. | Generate TTS on-demand when student views a grammar pattern. Reuse existing Redis TTS caching (7-day TTL for characters, 24h for sentences -- already built). |
| **Character writing recognition via custom ML** | Building a handwriting recognition model from scratch is a research project, not a product feature. | Use HanziWriter's built-in quiz mode with configurable leniency parameter. Already handles stroke order, direction, and approximate positioning out of the box. |
| **Full knowledge graph with prerequisite tracking** | Academic research shows knowledge graphs with deep reinforcement learning improve adaptive learning. But building one requires extensive prerequisite mapping of hundreds of content items and is overkill at this scale. | Linear course progression with SRS-based review. Grammar patterns have HSK level ordering which provides implicit sequencing without explicit prerequisite modeling. |
| **Anki import/export** | Anki deck format (.apkg) is a SQLite database with media files. Parsing, converting, and handling media is significant complexity for a tiny user overlap with this premium platform. | CSV/JSON export of vocabulary data is sufficient. Students using both Anki and CantoMando can manually add words. |
| **Social/competitive SRS leaderboard** | SRS is personal-pace learning, not competition. Leaderboard pressure causes gaming behavior (clicking "Easy" to inflate review counts, avoiding "Again" to protect ranking). | Show personal stats only: cards reviewed, retention rate, streak. Existing XP system already provides gamification without SRS-specific competition. |
| **Tone contour pitch visualization (real-time)** | Web Audio API F0 extraction + canvas rendering is high complexity for v10.0. Academic tools like Praat do this well but are desktop-only for a reason. | Defer to future milestone. Azure per-character pronunciation scores are sufficient for initial tone training. Static tone diagrams for visual reference. |

### Critical Path

1. **SRS Cards** depend on `saved_vocabulary` (exists) + `ts-fsrs` (npm install). No blockers.
2. **Grammar Library** is independently buildable with seeded content. No blockers.
3. **Tone Training** is independent -- only needs Azure TTS (already integrated). No blockers.
4. **Assessment** depends on grammar patterns for item bank content (grammar library should come first).
5. **Smart Study** depends on SRS + practice data being available (SRS first, then time for data to accumulate).
6. **Auto-Exercise Generation** depends on grammar patterns being seeded (grammar library first).
7. **AI Prompt Lab** is independent -- only needs existing prompt/version tables (already built).

### Build Order Implication

SRS, Grammar Library, Tone Training, and AI Prompt Lab have zero cross-dependencies and could theoretically be built in parallel. Assessment and Auto-Exercises depend on Grammar Library. Smart Study depends on SRS data existing. This suggests three waves:

- **Wave 1 (parallel):** SRS Flashcards + Grammar Library + Tone Training
- **Wave 2 (depends on Wave 1):** HSK Assessment + Auto-Exercise Generation
- **Wave 3 (depends on Wave 1 data):** Smart Study Recommendations

AI Prompt Lab can slot into any wave since it's independent, but lower student impact suggests building it later.

## MVP Recommendation

### Build First (Wave 1 -- Core Memorization, no dependencies):

1. **SRS flashcard system with FSRS** -- Highest user impact. Students already save vocabulary (the input data exists) and have no way to systematically review it (the output loop is missing). Uses `ts-fsrs`, auto-generates cards from saved words. This is the single most-requested feature category in language learning.

2. **Grammar pattern library (seeded)** -- Foundation for assessment items and auto-exercises downstream. Seed ~220 patterns (HSK 1-4) with structure, explanation, 3-5 example sentences, and Cantonese notes where grammar diverges. Content-heavy but architecturally simple (one new table + seed script).

3. **Tone identification drills (listening)** -- Chinese-specific and high value. Reuses existing Azure TTS infrastructure. Start with Mandarin 4 tones and tone pairs, expand to Cantonese 6-tone system as differentiator.

### Build Second (Wave 2 -- Intelligence Layer, depends on Wave 1):

4. **HSK placement/assessment test** -- Builds on grammar patterns (for item bank) + existing exercise types (for test player). Critical for new student onboarding and placing students at the right level.

5. **Auto-exercise generation from vocabulary/grammar** -- High leverage for coaches (multiplies content creation speed). Needs grammar patterns to exist. Landing zone is existing practice set builder with draft status for coach review.

6. **Smart study recommendations** -- "What to study next" dashboard widget. Needs SRS data + practice data to be meaningful, so comes after SRS has been active for at least a few sessions.

### Build Third (Wave 3 -- Coach Power Tools):

7. **AI prompt testing lab** -- Valuable for coaches tuning grading prompts but lower student impact. Build after core student-facing features are stable.

### Defer to Future Milestones:

- **Voice conversation-driven SRS**: Complex NLP extraction from conversation transcripts. Needs SRS to be stable + richer conversation data.
- **Real-time pitch visualization**: High complexity, low incremental value over record-and-compare.
- **Assessment retake comparison**: Needs assessment to be actively used for months before comparison is meaningful.
- **Full adaptive difficulty in assessments**: Start with simple step-up/step-down; invest in proper adaptive engine only if student count and item bank justify it.
