# Domain Pitfalls

**Domain:** SRS Flashcards, Smart Study Engine, Tone Training, Assessment, Grammar Library, Auto-Exercise Generation, AI Prompt Testing
**Researched:** 2026-02-16

## Critical Pitfalls

Mistakes that cause rewrites or major issues.

### Pitfall 1: FSRS Implementation Gets the Math Wrong

**What goes wrong:** Custom FSRS-5 implementation produces intervals that are too short (overloading students with reviews) or too long (cards forgotten before review). The FSRS algorithm has several edge cases: first review after learning, relearning after a lapse, handling of "again" ratings on mature cards.
**Why it happens:** FSRS-5 looks simple (~100 lines) but the state transitions between New -> Learning -> Review -> Relearning have subtle rules. Off-by-one errors in elapsed_days calculation or incorrect stability/difficulty update formulas degrade the entire SRS experience.
**Consequences:** Students either drown in reviews (bad intervals too short) or forget everything (bad intervals too long). Both lead to abandonment. Unlike a UI bug, bad scheduling silently degrades learning over weeks before anyone notices.
**Prevention:**
- Port the [femto-fsrs](https://github.com/RickCarlino/femto-fsrs) reference implementation exactly, then write comprehensive unit tests
- Test against known FSRS-5 test vectors (available in the fsrs4anki wiki)
- Validate intervals against the [SRS benchmark](https://github.com/open-spaced-repetition/srs-benchmark) expected outputs
- Add logging: every review saves the computed interval to `srs_review_log` so drift can be detected post-launch
**Detection:** Monitor average retention rate after 2 weeks. FSRS-5 targets ~90% retention. If retention drops below 80% or rises above 95% (over-reviewing), the algorithm needs adjustment.

### Pitfall 2: SRS Card State Lost or Corrupted

**What goes wrong:** Concurrent reviews (two browser tabs), network failures mid-review, or race conditions in the UPDATE query cause card state to be overwritten with stale data. Student loses review progress or gets duplicate cards.
**Why it happens:** SRS card state is mutable (each review updates the row). Without optimistic locking, two concurrent updates can clobber each other.
**Consequences:** Lost review data. Student sees same card repeatedly or card disappears from queue. Trust in the system erodes.
**Prevention:**
- Add `updatedAt` check to review submission: `UPDATE srs_card_state SET ... WHERE id = $1 AND updated_at = $2` (optimistic lock)
- If lock fails, re-fetch card state and retry
- SRS scheduling runs server-side ONLY (no client-side FSRS calculations)
- API is idempotent: re-submitting the same review for the same card+timestamp is a no-op
**Detection:** Log UPDATE affected row counts. If UPDATE returns 0 rows (optimistic lock failure), alert.

### Pitfall 3: AI-Generated Exercises with Wrong Answers

**What goes wrong:** GPT-4o-mini generates a multiple-choice question where the "correct" answer is actually wrong, or generates fill-in-blank exercises with incorrect Chinese characters. Students learn incorrect information.
**Why it happens:** GPT makes mistakes in Chinese, especially with tones, traditional/simplified confusion, and Cantonese-specific vocabulary. Automated generation at scale means errors multiply.
**Consequences:** Students learn wrong answers. Trust in the platform is destroyed. Worse than no exercises at all.
**Prevention:**
- ALL generated exercises go through coach review before publishing (status: 'draft' until approved)
- NEVER auto-publish generated exercises
- Include a "Report Error" button on every exercise for students
- For vocabulary exercises: validate generated Chinese text against the `dictionary_entries` table
- For tone exercises: verify tone numbers match `pinyin-pro` output
**Detection:** Track error reports per exercise. Auto-generated exercises with >2 reports get auto-unpublished.

### Pitfall 4: Assessment Placement Too High or Too Low

**What goes wrong:** Placement test places a beginner at HSK 4, or an intermediate student at HSK 1. Student either struggles with impossible content or is bored with content they already know.
**Why it happens:** Question pool too small per level, question difficulty not calibrated, or adaptive algorithm converges on wrong level due to lucky/unlucky guessing.
**Consequences:** Student gets a bad first impression. If placed too high, they feel stupid. If placed too low, they feel the platform wastes their time. Both lead to churn.
**Prevention:**
- Minimum 5-8 questions per HSK level in the assessment pool
- Include both receptive (reading/listening) and productive (writing/speaking) questions
- Binary search adaptive algorithm should require at least 3 correct answers at a level before declaring mastery
- Allow students to retake placement test or manually adjust their level
- Show "This is our recommendation, but you can change it" -- not a locked decision
**Detection:** Track how many students change their level within 7 days of placement. High change rate = bad calibration.

## Moderate Pitfalls

### Pitfall 5: HSK Data Quality Issues in Seed Data

**What goes wrong:** The open-source HSK JSON datasets have encoding issues, missing pinyin for some entries, incorrect HSK level classifications, or entries that duplicate existing `dictionary_entries`.
**Prevention:**
- Run validation pass after seeding: check all entries have pinyin, definitions, and valid HSK levels
- Deduplicate against existing `dictionary_entries` by traditional+pinyin compound key
- Cross-reference with at least two HSK sources (drkameleon/complete-hsk-vocabulary AND ivankra/hsk30)
- Manual spot-check 10 random entries per level after seeding

### Pitfall 6: Grammar Content Bottleneck

**What goes wrong:** Building the grammar pattern CRUD and browse UI is easy, but the grammar library launches with 10 entries because content curation takes forever. Students see an empty library.
**Prevention:**
- Seed 50+ grammar patterns at launch (HSK 1-3 essentials) using AI draft generation via n8n
- Coach reviews and publishes in batches, not one at a time
- Prioritize the 54 HSK 1 grammar points first (from AllSet Learning structure)
- AI drafts need only 5-10 minutes of coach editing vs 30-60 minutes of writing from scratch

### Pitfall 7: Study Engine Recommendations Are Useless

**What goes wrong:** The smart study engine always recommends "Review SRS cards" because that always has the highest priority, making the feature feel like a glorified SRS reminder instead of intelligent guidance.
**Prevention:**
- Cap SRS review recommendation at position #1 only when >20 cards overdue
- Reserve slots 2-4 for different skill areas (grammar, tone, conversation)
- Include a "variety bonus" in the priority algorithm that penalizes repeated recommendation types
- A/B test recommendation ordering if data supports it later

### Pitfall 8: Tone Drill Audio Latency Ruins the Experience

**What goes wrong:** Each tone drill requires: (1) TTS API call for reference audio, (2) student recording, (3) Azure Speech API call for assessment. Three network round-trips per drill item makes the flow feel sluggish.
**Prevention:**
- Pre-fetch TTS audio for the next 5 drill items while student works on current one
- Cache all TTS audio in Upstash Redis (existing pattern, 7-day TTL for single characters)
- Pre-generate and cache TTS for all HSK 1-2 vocabulary (common tone drill targets)
- Show optimistic UI: display "Analyzing..." feedback immediately after recording stops

### Pitfall 9: XP Source Enum Migration Trap

**What goes wrong:** Adding new XP sources (like 'srs_review') to the `xp_source` Postgres enum requires an ALTER TYPE migration. If the migration fails or is missed, XP recording breaks for the new feature.
**Prevention:**
- Reuse existing XP source values: `practice_exercise` for SRS reviews, `practice_perfect` for perfect sessions
- The `entityType` text field in `xp_events` already disambiguates: `{ source: 'practice_exercise', entityType: 'srs_card' }`
- Only add new enum values if the reporting truly needs separate source categories (it doesn't for v10.0)

### Pitfall 10: n8n Webhook Timeout for Batch Exercise Generation

**What goes wrong:** Generating 10+ exercises in a single n8n webhook call exceeds the webhook timeout (default 60s). The request times out, coach sees an error, but exercises were partially generated server-side.
**Prevention:**
- Limit batch size to 5 exercises per webhook call
- For larger batches, split into multiple sequential calls with progress indicator
- Make the API endpoint idempotent: if called again with same input, return existing draft set
- Consider async pattern: webhook creates a "generation job" and returns immediately, poll for results

## Minor Pitfalls

### Pitfall 11: Flashcard UI Not Mobile-Friendly

**What goes wrong:** Card flip animations and swipe gestures don't work well on small screens. SRS review is the feature most likely to be used on phones.
**Prevention:** Design mobile-first. Test card flip with Framer Motion `rotateY` on iOS Safari (known quirks with 3D transforms). Use large tap targets for rating buttons.

### Pitfall 12: Grammar Pattern Slug Collisions

**What goes wrong:** Two grammar patterns get the same slug (e.g., both "using-le" and "using-了" normalize to similar slugs). Unique constraint violation on insert.
**Prevention:** Slugify with `nanoid` suffix: `using-le-abc123`. Or use the pattern's UUID in the URL instead of slug.

### Pitfall 13: SRS Deck Proliferation

**What goes wrong:** Students create too many decks and fragment their review across them, leading to uneven practice.
**Prevention:** Default to one deck. "Add to SRS" puts cards in default deck. Deck creation is optional and shown only in deck management page, not the main flow.

### Pitfall 14: Assessment Timer Drift

**What goes wrong:** Client-side timer counts down, but if the student's tab is backgrounded (phone notification, etc.), the timer pauses. Student gets more time than intended.
**Prevention:** Record `startedAt` server-side. On each answer submission, check `Date.now() - startedAt > timeLimitMs`. Server is the authority on time.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| FSRS-5 implementation | Getting the math wrong (Pitfall 1) | Port femto-fsrs exactly. Write 20+ unit tests against known test vectors. |
| SRS card schema | JSONB vs typed columns debate | Use typed columns for FSRS state (enables WHERE due <= now()). JSONB only for card front/back content. |
| HSK vocabulary seeding | Data quality (Pitfall 5) | Cross-reference two sources. Validate after seed. |
| Grammar library | Content bottleneck (Pitfall 6) | AI draft generation from day one. Seed 50+ HSK 1-3 patterns. |
| Tone training | Audio latency (Pitfall 8) | Pre-fetch TTS. Cache common words. Optimistic UI. |
| Assessment system | Bad calibration (Pitfall 4) | 5-8 questions per level. Allow manual override. Track level changes. |
| Auto-exercise generation | Wrong answers (Pitfall 3) | NEVER auto-publish. Coach review mandatory. Validate against dictionary. |
| Smart study engine | Useless recommendations (Pitfall 7) | Variety bonus. Cap SRS at #1 slot only when >20 overdue. |
| AI prompt testing lab | Testing against wrong webhook | Prompt lab MUST test against the ACTUAL production n8n webhook, not a mock. |
| XP integration | Enum migration (Pitfall 9) | Reuse existing enum values. Disambiguate via entityType. |
