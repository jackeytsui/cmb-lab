# Domain Pitfalls: Adding RBAC to an Existing LMS

**Domain:** Role-based access control migration -- adding Discord-style RBAC to an LMS with existing per-course access model, Clerk auth, and external webhook enrollment
**Researched:** 2026-02-14
**Confidence:** HIGH (codebase verified, Clerk docs verified, RBAC patterns verified with multiple sources)

---

## Critical Pitfalls

Mistakes that cause access loss, data corruption, or production lockouts.

### Pitfall 1: Clerk Session Token Staleness After Role Change (Up to 60-Second Blindspot)

**What goes wrong:**
External sales system fires a webhook that assigns a new role to a student (e.g., upgrading from "student" to a role with premium access). The webhook handler updates `users.role` in the database and calls `clerk.users.updateUserMetadata()` to set `publicMetadata.role`. The user's session token still has the OLD role cached. For up to 60 seconds, every `hasMinimumRole()` call in the middleware and API routes reads the stale role from `sessionClaims.metadata.role`. The user sees "Forbidden" errors or missing UI elements despite being correctly upgraded.

**Why it happens:**
Clerk's frontend SDK refreshes the session token on a **60-second interval**. When you update `publicMetadata` server-side (via webhook handler or admin action), the change is immediate in Clerk's backend but the JWT cookie the user's browser holds is stale until the next refresh cycle. The current `resolveRole()` function in `src/lib/auth.ts` checks `sessionClaims.metadata.role` FIRST and only falls back to the database if claims are empty -- meaning it will read the stale cached role and never hit the database fallback.

**This is uniquely dangerous in THIS codebase because:**
- `middleware.ts` reads `sessionClaims?.metadata?.role` directly (line 38) with NO database fallback
- `resolveRole()` in `src/lib/auth.ts` (line 16) returns the stale claims role immediately if it exists
- The database fallback only triggers when `claimsRole` is falsy, not when it's outdated
- 117 files with 286 occurrences of `hasMinimumRole`/`checkRole` are all affected

**Consequences:**
- User upgraded via webhook sees "Forbidden" for up to 60 seconds
- If the user navigates to a new page during the stale window, middleware redirects them away from their new role's pages
- Coach promoting a student to coach role during a live call: "I just gave you access, try refreshing" -- poor experience
- If the stale window overlaps with a time-sensitive action (e.g., live class starting), the student misses it entirely

**Prevention:**
```typescript
// APPROACH 1: Reverse the priority in resolveRole()
// When RBAC roles change more frequently (new milestone),
// make DB the source of truth, claims the fast-path optimization

async function resolveRole(): Promise<Roles> {
  const { sessionClaims, userId } = await auth();

  // For API routes (server-side): ALWAYS check DB for freshest role
  // Claims are an optimization, not the source of truth
  if (userId) {
    const user = await db.query.users.findFirst({
      where: eq(users.clerkId, userId),
      columns: { role: true },
    });
    if (user?.role) return user.role as Roles;
  }

  // Fallback to claims (for middleware where DB calls are expensive)
  const claimsRole = sessionClaims?.metadata?.role;
  if (claimsRole) return claimsRole;

  return "student";
}

// APPROACH 2: Keep claims-first but add DB verification for upgrades
// Use this if DB latency in every request is unacceptable

async function resolveRole(): Promise<Roles> {
  const { sessionClaims, userId } = await auth();
  const claimsRole = sessionClaims?.metadata?.role;

  // If claims say "student" but DB says "coach", DB wins (upgrade case)
  // If claims say "coach" but DB says "student", DB wins (downgrade case)
  // Only skip DB if we're confident claims are fresh
  if (userId) {
    const user = await db.query.users.findFirst({
      where: eq(users.clerkId, userId),
      columns: { role: true },
    });
    if (user?.role) return user.role as Roles;
  }

  if (claimsRole) return claimsRole;
  return "student";
}

// APPROACH 3: Hybrid with caching (best performance)
// Cache DB role in Redis with 30s TTL, invalidate on webhook

import { redis } from "@/lib/redis";

async function resolveRole(): Promise<Roles> {
  const { sessionClaims, userId } = await auth();
  if (!userId) return "student";

  // Check Redis cache first (invalidated by webhook handler)
  const cachedRole = await redis.get(`user:role:${userId}`);
  if (cachedRole) return cachedRole as Roles;

  // DB lookup
  const user = await db.query.users.findFirst({
    where: eq(users.clerkId, userId),
    columns: { role: true },
  });
  const role = (user?.role as Roles) || "student";

  // Cache for 30s (shorter than Clerk's 60s refresh)
  await redis.set(`user:role:${userId}`, role, { ex: 30 });
  return role;
}

// In webhook handler: invalidate cache immediately
async function handleRoleChange(clerkId: string, newRole: string) {
  await db.update(users).set({ role: newRole }).where(eq(users.clerkId, clerkId));
  await redis.del(`user:role:${clerkId}`);
  // Also update Clerk metadata for future session tokens
  const clerk = await clerkClient();
  await clerk.users.updateUserMetadata(clerkId, {
    publicMetadata: { role: newRole },
  });
}
```

**Detection (warning signs):**
- Users report "access denied" immediately after role change, resolves after ~1 minute
- Support tickets spike right after bulk role assignments
- Coach says "I changed their role but they can't see X yet"
- Logs show 403 responses for users who DO have the correct role in DB

**Which phase should address:** Schema Design Phase. The `resolveRole()` function must be refactored BEFORE any new role types are added. This is the foundation that everything else depends on.

### Pitfall 2: Dual Authority Problem -- Role vs courseAccess Creates Conflicting Access Decisions

**What goes wrong:**
After adding RBAC, the system has TWO independent access control mechanisms:
1. **Role-based:** `hasMinimumRole("coach")` checks the user's role enum
2. **Resource-based:** `courseAccess` table checks if a user has access to a specific course

These two systems can DISAGREE. A user with role "premium_student" (new RBAC role) might not have a `courseAccess` row for a specific course. Or a user with a `courseAccess` row might have an expired/downgraded role. The system now has to answer: "Does the role GRANT course access, or does the courseAccess table GRANT course access, or both?"

**Why it happens:**
The current system uses `courseAccess` as the SOLE authority for "can this user access this course." Role (`hasMinimumRole`) is used ONLY for "can this user access admin/coach features." These are orthogonal. When Discord-style roles are added that ALSO imply course access (e.g., "Tier 2 Student" role should grant access to Tier 2 courses), the two systems overlap and can contradict each other.

**This is uniquely dangerous in THIS codebase because:**
- Course access checks are scattered across 16 files with 141 occurrences of `courseAccess`
- The enrollment webhook (`/api/webhooks/enroll`) creates `courseAccess` rows -- if roles ALSO grant access, enrollment webhook must be updated
- `courses/page.tsx`, `lessons/[lessonId]/page.tsx`, `dashboard/page.tsx` all query `courseAccess` directly with JOINs -- not through a centralized access check function
- Admin bulk operations (`/api/admin/students/bulk`) manipulate `courseAccess` directly

**Consequences:**
- Student purchases Tier 2 via external sales system, gets "Tier 2" role via webhook, but no `courseAccess` row exists for the specific courses -- student sees empty course list
- Admin revokes a student's `courseAccess` row, but student's role still says "Tier 2" -- student can re-access via a different code path
- Two sources of truth = every permission bug is twice as hard to debug
- Coach grants `courseAccess` manually, but role says "free tier" -- which wins?

**Prevention:**
```typescript
// DECISION: Role is the AUTHORITY, courseAccess is the MECHANISM
//
// Roles define WHAT you can access (tier/level).
// courseAccess rows are DERIVED from roles (created/removed when role changes).
// courseAccess is NEVER the source of truth -- it's a denormalized cache
// that enables efficient SQL JOINs.
//
// This means: when a role changes, courseAccess rows are regenerated.
// Manual courseAccess grants are still allowed (admin override) but
// are flagged as "manual" grantedBy source.

// lib/access-resolver.ts -- SINGLE function for all access checks
export async function canAccessCourse(
  userId: string,
  courseId: string
): Promise<{ allowed: boolean; reason: string }> {
  // 1. Check explicit courseAccess grant (handles manual overrides)
  const explicit = await db.query.courseAccess.findFirst({
    where: and(
      eq(courseAccess.userId, userId),
      eq(courseAccess.courseId, courseId),
      or(isNull(courseAccess.expiresAt), gt(courseAccess.expiresAt, new Date()))
    ),
  });

  if (explicit) {
    return { allowed: true, reason: `explicit:${explicit.grantedBy}` };
  }

  // 2. Check role-based access (role implies access to a set of courses)
  const user = await db.query.users.findFirst({
    where: eq(users.id, userId),
    columns: { role: true },
  });

  if (user?.role) {
    const roleCourses = await getCoursesForRole(user.role);
    if (roleCourses.includes(courseId)) {
      return { allowed: true, reason: `role:${user.role}` };
    }
  }

  return { allowed: false, reason: "no_access" };
}

// CRITICAL: All 16 files that currently query courseAccess directly
// must be migrated to use canAccessCourse() or a wrapper.
// Do NOT leave raw courseAccess queries -- they bypass role checks.
```

**Detection (warning signs):**
- Student says "I bought the premium package but I can't see the courses"
- Admin revoked access but student can still view content
- Course list shows different results depending on which page the student navigates from
- `courseAccess` table has rows that contradict the user's role

**Which phase should address:** Architecture Phase (before schema design). The "who is the authority" decision must be made FIRST. Every subsequent design choice depends on whether roles derive from courseAccess, courseAccess derives from roles, or they are independent.

### Pitfall 3: Migration Destroys Existing Student Access (100+ Students at Risk)

**What goes wrong:**
The migration script converts the existing 3-role enum (`student`, `coach`, `admin`) to a new role system. During migration, the script either:
1. Drops the old `role` column before populating the new one -- students lose their role temporarily
2. Alters the enum to add new values but doesn't backfill existing users -- existing users have stale roles
3. Creates a new `roles` table but the old `hasMinimumRole()` function still reads the old `users.role` column -- broken during the cutover window
4. Maps existing `courseAccess` rows to roles but gets the mapping wrong -- students lose access to courses they paid for

**Why it happens:**
Postgres enum alterations are NOT transactional in the way you expect. `ALTER TYPE role ADD VALUE 'premium_student'` cannot be rolled back within a transaction. If the migration fails halfway through, you have a partially-modified enum that can't be undone without manual intervention. Additionally, the migration requires coordinating changes across:
- `users.role` column (Postgres enum)
- Clerk `publicMetadata.role` (external system, API calls)
- `courseAccess` table (if roles are being derived from it)
- `middleware.ts` (reads role from claims)
- `resolveRole()` (reads role from DB)
- All 117 files using `hasMinimumRole()`

**Consequences:**
- Students log in and see empty dashboards -- no courses, no progress
- Coaches can't access coach features during migration window
- Admin locked out of admin panel if their role mapping fails
- If Clerk metadata update fails for some users, those users have DB role but stale claims
- Rollback is nearly impossible if the Postgres enum was already modified

**Prevention:**
```sql
-- SAFE MIGRATION STRATEGY: Additive-only, never destructive

-- Step 1: Add new columns/tables alongside old ones (SAFE, rollback = drop column)
ALTER TABLE users ADD COLUMN role_v2 TEXT; -- NOT an enum yet, TEXT is flexible

-- Step 2: Backfill new column from old column (SAFE, old column still works)
UPDATE users SET role_v2 = role::TEXT;

-- Step 3: Deploy code that reads BOTH columns (SAFE, falls through to old)
-- resolveRole() checks role_v2 first, falls back to role
-- hasMinimumRole() unchanged, still works

-- Step 4: VERIFY all users have correct role_v2 values
-- Run reconciliation query:
SELECT id, email, role::TEXT, role_v2
FROM users
WHERE role::TEXT != role_v2 OR role_v2 IS NULL;
-- Must return 0 rows before proceeding

-- Step 5: Update code to write to BOTH columns on any role change
-- Dual-write ensures consistency during transition

-- Step 6: After 1 week of dual-write with no issues,
-- stop reading old column, switch fully to role_v2

-- Step 7: Only after full verification, drop old column
-- (This can be weeks or months later)
```

```typescript
// CLERK METADATA MIGRATION: Batch with verification
async function migrateClerkMetadata() {
  const allUsers = await db.select().from(users);
  const results = { success: 0, failed: 0, errors: [] as string[] };

  for (const user of allUsers) {
    try {
      const clerk = await clerkClient();
      await clerk.users.updateUserMetadata(user.clerkId, {
        publicMetadata: { role: user.role },
      });
      results.success++;
    } catch (error) {
      results.failed++;
      results.errors.push(`${user.email}: ${error.message}`);
    }
    // Rate limit: Clerk has API limits
    await new Promise(resolve => setTimeout(resolve, 100));
  }

  // CRITICAL: Log all failures for manual remediation
  if (results.failed > 0) {
    console.error(`Migration incomplete: ${results.failed} users failed`, results.errors);
    // Do NOT proceed to code cutover until all users are migrated
  }

  return results;
}

// ROLLBACK PLAN: Keep old column/code functional for 2 weeks
// If anything goes wrong, flip the feature flag back to old code path
const USE_NEW_ROLES = process.env.RBAC_V2_ENABLED === "true";

async function resolveRole(): Promise<string> {
  if (USE_NEW_ROLES) {
    return resolveRoleV2(); // New role resolution logic
  }
  return resolveRoleLegacy(); // Existing logic, unchanged

```

**Detection (warning signs):**
- Any student reports "I can't see my courses" after a deployment

**Which phase should address:** Migration Phase (dedicated phase, not combined with feature work). Migration must be a standalone deployment with its own verification checklist and rollback plan. NEVER combine schema migration with feature changes in the same deployment.

### Pitfall 4: Webhook Role Assignment Without Idempotency Creates Duplicate/Conflicting State

**What goes wrong:**
External sales system (GHL or other) fires a webhook to assign a role when a student purchases a tier. Due to network retry, webhook infrastructure retry, or user double-clicking "Purchase," the webhook fires twice within milliseconds. Both webhook handlers:
1. Look up the user (both find the same user)
2. Check current role (both see "student")
3. Update role to "tier_2" (both succeed, but now two audit log entries exist)
4. Create `courseAccess` rows (duplicate rows may be created if no unique constraint)
5. Update Clerk metadata (second call overwrites first, usually fine but wastes API calls)

Worse: what if the two webhooks have DIFFERENT role IDs due to a race in the sales system?

**Why it happens:**
The current enrollment webhook (`/api/webhooks/enroll`) does have a duplicate check for `courseAccess` (line 88-93: checks `existingAccess`), but it does NOT have a general-purpose idempotency mechanism. There is no `processed_webhooks` table. There is no idempotency key. If the external system sends the same event twice with slightly different payloads (e.g., one has `accessTier: "full"`, the other has `accessTier: "preview"`), both will process -- the second one DOWNGRADES the student.

**Consequences:**
- Student purchases premium, gets downgraded by retry webhook with stale data
- Duplicate `courseAccess` rows if unique constraint is missing
- Audit trail shows two role changes, impossible to determine which is "correct"
- If webhook handler is not idempotent, retry logic in the external system becomes a liability

**Prevention:**
```typescript
// 1. Processed webhooks table for idempotency
export const processedWebhooks = pgTable("processed_webhooks", {
  id: uuid("id").defaultRandom().primaryKey(),
  idempotencyKey: text("idempotency_key").notNull().unique(), // From external system
  source: text("source").notNull(), // "ghl", "enrollment", etc.
  eventType: text("event_type").notNull(),
  payload: jsonb("payload"),
  processedAt: timestamp("processed_at").notNull().defaultNow(),
  result: text("result"), // "success", "skipped", "error"
});

// 2. Idempotent webhook handler
export async function POST(req: NextRequest) {
  const body = await req.json();

  // Derive idempotency key from payload
  // Best: external system provides it. Fallback: hash of (email + courseId + timestamp rounded to minute)
  const idempotencyKey = body.idempotencyKey
    || `${body.email}:${body.roleId}:${Math.floor(Date.now() / 60000)}`;

  // Check if already processed
  const existing = await db.query.processedWebhooks.findFirst({
    where: eq(processedWebhooks.idempotencyKey, idempotencyKey),
  });

  if (existing) {
    return NextResponse.json({
      success: true,
      message: "Already processed",
      originalResult: existing.result
    });
  }

  // Process the webhook...
  try {
    await processRoleAssignment(body);

    // Record as processed AFTER success
    await db.insert(processedWebhooks).values({
      idempotencyKey,
      source: "sales_webhook",
      eventType: "role_assignment",
      payload: body,
      result: "success",
    });

    return NextResponse.json({ success: true });
  } catch (error) {
    // Record as error (so retries can try again)
    // Use onConflict to handle race condition on the insert itself
    await db.insert(processedWebhooks).values({
      idempotencyKey,
      source: "sales_webhook",
      eventType: "role_assignment",
      payload: body,
      result: `error:${error.message}`,
    }).onConflictDoNothing();

    throw error;
  }
}

// 3. Database-level unique constraint as last resort
// Prevents duplicate courseAccess even if application logic fails
ALTER TABLE course_access
  ADD CONSTRAINT course_access_user_course_unique
  UNIQUE (user_id, course_id);
```

**Detection (warning signs):**
- Multiple `courseAccess` rows for same user+course combination
- Audit log shows same role change applied twice within seconds
- Student reports access level flip-flopping (premium -> free -> premium)
- Webhook handler logs show same payload processed multiple times

**Which phase should address:** Webhook Infrastructure Phase (before connecting external sales system). Idempotency must be built into the webhook handler from day one. The existing enrollment webhook should also be retrofitted.

**Sources:**
- [Hookdeck: How to implement webhook idempotency](https://hookdeck.com/webhooks/guides/implement-webhook-idempotency) - HIGH confidence
- [Postmark: Why idempotency is important](https://postmarkapp.com/blog/why-idempotency-is-important) - MEDIUM confidence

### Pitfall 5: Unknown/Expired Role IDs from External System Silently Fail

**What goes wrong:**
External sales system sends a webhook with `roleId: "tier_3_vip"`. The LMS has never heard of this role. The webhook handler tries to look up the role, gets `null`, and either:
1. Silently ignores it (student paid but gets nothing)
2. Throws an unhandled error (webhook returns 500, external system may or may not retry)
3. Defaults to "student" role (student paid for premium, gets basic)

Later: sales team creates a new product tier but forgets to update the LMS role mapping. Or they retire an old tier but some students still have it. The LMS now has users with a role that no longer maps to any permissions.

**Why it happens:**
The current system has a hardcoded enum: `["student", "coach", "admin"]`. This works because there are only 3 roles and they never change. When roles are externally defined (sales system determines what tiers exist), the LMS role list can drift from the external system's list. There is no contract or validation layer between the two systems.

**Consequences:**
- Student pays for "Tier 3 VIP" but gets basic student access
- No error alert to admin -- the failure is silent
- Students with retired roles have undefined behavior (might have no access at all)
- Sales team adds new product tiers faster than engineering adds role mappings

**Prevention:**
```typescript
// 1. Role registry with strict validation
// roles-registry.ts

export const ROLE_REGISTRY = {
  // System roles (never from webhook)
  student: {
    label: "Student (Free)",
    courses: [],
    source: "system" as const,
  },
  coach: {
    label: "Coach",
    courses: "all" as const,
    source: "system" as const,
  },
  admin: {
    label: "Admin",
    courses: "all" as const,
    source: "system" as const,
  },
  // Purchasable roles (from external sales system)
  tier_1: {
    label: "Tier 1 Student",
    courses: ["course-id-1"],
    source: "webhook" as const,
    externalIds: ["ghl_tier1", "stripe_tier1"], // All external IDs that map here
  },
  tier_2: {
    label: "Tier 2 Student",
    courses: ["course-id-1", "course-id-2"],
    source: "webhook" as const,
    externalIds: ["ghl_tier2", "stripe_tier2"],
  },
} as const;

// 2. Webhook handler with strict validation and alerting
function resolveExternalRole(externalRoleId: string): string | null {
  for (const [roleKey, config] of Object.entries(ROLE_REGISTRY)) {
    if ('externalIds' in config && config.externalIds.includes(externalRoleId)) {
      return roleKey;
    }
  }
  return null; // Unknown role
}

async function handleRoleWebhook(payload: { email: string; roleId: string }) {
  const internalRole = resolveExternalRole(payload.roleId);

  if (!internalRole) {
    // CRITICAL: Alert admin, do NOT silently ignore
    console.error(`[RBAC] Unknown external role: "${payload.roleId}" for ${payload.email}`);

    await sendAdminAlert({
      type: "unknown_role",
      message: `External system sent unknown role "${payload.roleId}" for ${payload.email}. Student may not have received expected access.`,
      payload,
    });

    // Store the raw event for later processing
    await db.insert(pendingRoleAssignments).values({
      email: payload.email,
      externalRoleId: payload.roleId,
      status: "unknown_role",
    });

    // Return 200 to prevent retries (the data IS received, just unmapped)
    return { success: false, reason: "unknown_role" };
  }

  // Process the valid role assignment
  await assignRole(payload.email, internalRole);
  return { success: true };
}

// 3. Admin dashboard: show pending/unmapped role assignments
// So admin can manually resolve or add new mappings
```

**Detection (warning signs):**
- Students report "I paid but can't access anything"
- `pendingRoleAssignments` table has rows (means external IDs are unmapped)
- Admin alert emails for unknown roles
- Sales team launches new product tier without informing engineering

**Which phase should address:** Schema Design Phase. The role registry and external ID mapping must exist before the first webhook is connected. Include an admin UI for managing mappings in the Admin Dashboard Phase.

### Pitfall 6: Role Hierarchy Breaks When Adding Non-Hierarchical Roles

**What goes wrong:**
The current role system is strictly hierarchical: `student < coach < admin`. The `hasMinimumRole()` function works by comparing array indices: `roleHierarchy.indexOf(userRole) >= roleHierarchy.indexOf(minimumRole)`. When Discord-style roles are added (e.g., "tier_1", "tier_2", "content_creator"), they don't fit into a linear hierarchy. Is "tier_2" above or below "coach"? Is "content_creator" above "student" but below "coach"?

**Why it happens:**
The current `hasMinimumRole()` uses a simple array index comparison (line 44-48 of `auth.ts`). This pattern fundamentally assumes ALL roles are linearly ordered. Discord-style roles are a DAG (directed acyclic graph), not a line. A user can have "tier_2" access (premium courses) AND "content_creator" access (upload content) -- these are orthogonal, not hierarchical.

**Consequences:**
- `hasMinimumRole("coach")` returns `false` for "tier_2" users even if they should have some coach-like permissions
- Adding "tier_1" to the hierarchy array at position 1 breaks the existing `hasMinimumRole("coach")` check -- suddenly tier_1 students are considered "above" students but not coaches, and every `hasMinimumRole("student")` call now also matches tier_1
- The hierarchy becomes meaningless when roles are orthogonal

**Prevention:**
```typescript
// Replace hasMinimumRole with a capability-based system

// Define capabilities (what you can DO), not just roles (who you ARE)
type Capability =
  | "view_courses"        // All authenticated users
  | "access_tier_1"       // Tier 1 courses
  | "access_tier_2"       // Tier 2 courses (implies tier 1)
  | "submit_work"         // Submit assignments
  | "review_submissions"  // Coach capability
  | "manage_students"     // Coach capability
  | "manage_content"      // Admin capability
  | "manage_roles";       // Admin capability

// Map roles to capabilities (many-to-many)
const ROLE_CAPABILITIES: Record<string, Capability[]> = {
  student: ["view_courses"],
  tier_1: ["view_courses", "access_tier_1", "submit_work"],
  tier_2: ["view_courses", "access_tier_1", "access_tier_2", "submit_work"],
  coach: ["view_courses", "access_tier_1", "access_tier_2", "submit_work", "review_submissions", "manage_students"],
  admin: ["view_courses", "access_tier_1", "access_tier_2", "submit_work", "review_submissions", "manage_students", "manage_content", "manage_roles"],
};

// New check function: replaces hasMinimumRole
export async function hasCapability(capability: Capability): Promise<boolean> {
  const role = await resolveRole();
  const capabilities = ROLE_CAPABILITIES[role] || [];
  return capabilities.includes(capability);
}

// BACKWARD COMPATIBILITY: Keep hasMinimumRole working during migration
// Map old calls to new system
export async function hasMinimumRole(minimumRole: Roles): Promise<boolean> {
  const role = await resolveRole();
  // Legacy hierarchy still works for legacy roles
  const legacyHierarchy: Roles[] = ["student", "coach", "admin"];
  if (legacyHierarchy.includes(role as Roles)) {
    const userLevel = legacyHierarchy.indexOf(role as Roles);
    const requiredLevel = legacyHierarchy.indexOf(minimumRole);
    return userLevel >= requiredLevel;
  }
  // For new roles, map to capability
  const capabilityMap: Record<string, Capability> = {
    student: "view_courses",
    coach: "review_submissions",
    admin: "manage_content",
  };
  return hasCapability(capabilityMap[minimumRole] || "view_courses");
}
```

**Detection (warning signs):**
- Feature requests like "tier_2 students should be able to do X but not Y"
- `hasMinimumRole()` calls being added with comments like "// TODO: this doesn't work for tier_2"
- Developers creating workarounds like `if (role === "tier_2" || hasMinimumRole("coach"))`
- New roles being squeezed into the hierarchy array at awkward positions

**Which phase should address:** Architecture Phase. Decide whether to keep hierarchy (only works for simple role sets) or move to capabilities (required for Discord-style roles). This decision affects every access check in the codebase.

### Pitfall 7: Middleware Role Check Diverges from API Route Role Check

**What goes wrong:**
`middleware.ts` reads the role from `sessionClaims?.metadata?.role` (line 38) and makes routing decisions. API route handlers call `hasMinimumRole()` which falls through to the database. After adding RBAC, the middleware might allow access (stale claims say "coach") while the API denies it (DB says role was revoked), or vice versa. The user sees a coach dashboard with all the UI elements but every API call returns 403.

**Why it happens:**
Middleware runs on the Edge Runtime and cannot access the database (no DB driver for Edge). API routes run on Node.js and CAN access the database. This creates two fundamentally different role resolution paths:
- Middleware: claims-only, up to 60 seconds stale, no DB fallback
- API routes: DB-verified via `resolveRole()`, always fresh

When these two paths disagree, the user experiences a "ghost" UI -- they can see the page but can't interact with anything.

**Consequences:**
- User sees coach page but every button click fails with 403
- User is redirected away from a page they DO have access to (middleware rejects, API would accept)
- Debugging is confusing because middleware and API checks return different results
- "It works when I refresh" (refresh triggers new session token fetch)

**Prevention:**
```typescript
// STRATEGY 1: Make middleware permissive, let API routes be the authority
// Middleware does authentication (is the user logged in?),
// not authorization (does the user have the right role?)

// middleware.ts
export default clerkMiddleware(async (auth, req) => {
  if (isPublicRoute(req)) return NextResponse.next();

  const { userId } = await auth();
  if (!userId && isProtectedRoute(req)) {
    return NextResponse.redirect(new URL("/sign-in", req.url));
  }

  // DO NOT check roles in middleware
  // Let the page/API route handle authorization with fresh DB data
  return NextResponse.next();
});

// Individual pages handle their own authorization
// app/(dashboard)/admin/page.tsx
export default async function AdminPage() {
  const hasAccess = await hasMinimumRole("admin");
  if (!hasAccess) {
    redirect("/dashboard"); // Server-side redirect, fresh DB check
  }
  // ... render admin page
}

// STRATEGY 2: If middleware MUST check roles (for performance),
// use a "soft check" that allows borderline cases through
// and lets the page do the hard check

export default clerkMiddleware(async (auth, req) => {
  // ...auth check...

  const role = sessionClaims?.metadata?.role || "student";

  // Only block OBVIOUS mismatches (student accessing /admin)
  // Let borderline cases through (new roles, role transitions)
  if (isAdminRoute(req) && role === "student") {
    // Claims say "student" -- probably correct, redirect
    return NextResponse.redirect(new URL("/dashboard", req.url));
  }

  // For all other cases, let the page decide
  // This means coach pages might render for a moment for non-coaches
  // but the server component will redirect before paint
  return NextResponse.next();
});
```

**Detection (warning signs):**
- Users report seeing a page briefly before being redirected
- API calls return 403 but the page itself loaded fine
- Role changes take effect for API calls immediately but UI navigation stays stale
- Different behavior between hard refresh and client-side navigation

**Which phase should address:** Architecture Phase. Decide the middleware strategy (permissive vs. authoritative) BEFORE adding new roles. This is a one-time architectural decision.

### Pitfall 8: Permission Explosion When Mapping Roles to Courses

**What goes wrong:**
Initial design: 3 roles, 2 courses. Clean and simple. After RBAC: 6 roles, 5 courses, 2 access tiers = 60 potential role-course-tier combinations. Each combination needs explicit mapping. When a new course is added, someone must update the mapping for all 6 roles. When a new role is added, someone must decide its access to all 5 courses. The mapping table grows quadratically.

**Why it happens:**
Flat role-to-course mappings don't scale. The number of mapping entries is `roles * courses * tiers`. With Discord-style roles where users can have MULTIPLE roles, the resolution logic becomes: "user has roles [tier_1, early_bird_bonus] -- what courses can they access?" This requires union/intersection logic that is easy to get wrong.

**Consequences:**
- Admin forgets to add new course to tier_2 role mapping -- tier_2 students can't see new content
- Role-course mapping table becomes a maintenance burden
- Conflicting mappings (role A says "full access", role B says "preview only") with no resolution rule
- Every new course launch requires a role mapping review

**Prevention:**
```typescript
// USE COURSE TAGS, NOT EXPLICIT MAPPINGS
// Courses have a "tier" tag. Roles grant access to tiers, not specific courses.
// Adding a new course to tier 2 = setting its tier tag. Done.

// Schema
export const courses = pgTable("courses", {
  // ... existing fields
  accessTier: integer("access_tier").notNull().default(0), // 0=free, 1=tier1, 2=tier2
});

// Role definition: what TIER does this role unlock?
const ROLE_MAX_TIER: Record<string, number> = {
  student: 0,      // Free courses only
  tier_1: 1,       // Tier 1 and below
  tier_2: 2,       // Tier 2 and below (inherits tier 1)
  coach: 999,      // All courses
  admin: 999,      // All courses
};

// Access check: simple numeric comparison
function canAccessCourse(userRole: string, courseTier: number): boolean {
  const maxTier = ROLE_MAX_TIER[userRole] ?? 0;
  return courseTier <= maxTier;
}

// Result: adding a new course = set its tier number
// Adding a new role = set its max tier number
// No combinatorial explosion
// New tier? Just increase the number. Existing roles automatically include lower tiers.
```

**Detection (warning signs):**
- Role-course mapping table has more rows than `roles * courses` (indicates manual overrides accumulating)
- Admin asks "which roles have access to this new course?" before every launch
- Bug reports about missing access after new course added

**Which phase should address:** Schema Design Phase. The tier-based model must be designed into the schema from the start. Retrofitting a flat mapping table into a tiered model later requires data migration.

### Pitfall 9: Clerk Cookie Size Limit (4KB) Exceeded by Role Metadata

**What goes wrong:**
As RBAC grows, you store more data in Clerk's `publicMetadata`: role name, permissions array, tier level, accessible course IDs, feature flags. The session token (stored as a JWT in a cookie) exceeds the 4KB browser cookie limit. Clerk silently truncates or fails to set the cookie. The user's session breaks -- they appear logged out or have missing metadata.

**Why it happens:**
Clerk stores session tokens in cookies. Most browsers cap cookies at 4KB. After Clerk's default claims, only ~1.2KB is available for custom data. If you store `publicMetadata: { role: "tier_2", permissions: ["access_tier_1", "access_tier_2", ...], courses: ["uuid1", "uuid2", ..."] }`, you can easily exceed this.

**Consequences:**
- Session cookie fails to set -- user appears logged out
- Metadata silently truncated -- role data missing, user defaults to "student"
- Works in development (small metadata) but breaks in production (more roles, more data)
- Extremely hard to debug because the cookie is opaque

**Prevention:**
```typescript
// RULE: Session token metadata must be MINIMAL
// Only store: { role: "tier_2" }
// NEVER store: permissions array, course IDs, feature flags
// Derive everything else from the role name at runtime

// Clerk session token customization:
// { "metadata": "{{user.public_metadata}}" }
// public_metadata should ONLY contain:
// { "role": "tier_2" }
// That's it. ~20 bytes. Well within limits.

// Everything else lives in the database or role registry:
// - Permissions: derived from ROLE_CAPABILITIES[role]
// - Courses: derived from ROLE_MAX_TIER[role] + course.accessTier
// - Feature flags: derived from role or separate feature flag system
```

**Detection (warning signs):**
- Users randomly appear logged out after metadata update
- `document.cookie` in browser devtools shows truncated JWT
- Session token size warnings in Clerk dashboard
- Works for some users but not others (different metadata sizes)

**Which phase should address:** Schema Design Phase. Establish the "minimal metadata" rule before any role data is added to Clerk. This is a design constraint, not a bug to fix later.

**Sources:**
- [Clerk: Session token size limit ~1.2KB custom data](https://clerk.com/docs/backend-requests/resources/session-tokens) - HIGH confidence

### Pitfall 10: TypeScript Type Safety Lost When Extending Role Enum

**What goes wrong:**
The current `Roles` type is `"student" | "coach" | "admin"` (from `globals.d.ts`). Adding new roles means updating this type everywhere it's referenced. If you miss one location, TypeScript won't catch it at compile time because the old `switch` statements have `default` cases that silently handle unknown roles.

**Prevention:**
```typescript
// Use a const assertion with exhaustive checking
const ROLES = ["student", "tier_1", "tier_2", "coach", "admin"] as const;
type Role = typeof ROLES[number];

// Exhaustive switch helper (compile error if a role is missing)
function assertNever(x: never): never {
  throw new Error(`Unexpected role: ${x}`);
}

function getRoleLabel(role: Role): string {
  switch (role) {
    case "student": return "Student";
    case "tier_1": return "Tier 1";
    case "tier_2": return "Tier 2";
    case "coach": return "Coach";
    case "admin": return "Admin";
    default: return assertNever(role); // Compile error if missing case
  }
}
```

### Pitfall 11: Admin Accidentally Removes Their Own Admin Role

**What goes wrong:**
Admin panel for managing roles allows an admin to change any user's role. Admin selects themselves and accidentally changes their role from "admin" to "student." They are immediately locked out of the admin panel. No other admin exists (single-admin system). Recovery requires direct database access.

**Prevention:**
```typescript
// Self-demotion guard
async function updateUserRole(targetUserId: string, newRole: string, actorUserId: string) {
  if (targetUserId === actorUserId && newRole !== "admin") {
    throw new Error("Cannot remove your own admin role. Ask another admin to make this change.");
  }

  // Additional guard: ensure at least one admin remains
  const adminCount = await db
    .select({ count: sql<number>`COUNT(*)` })
    .from(users)
    .where(eq(users.role, "admin"));

  if (adminCount[0].count <= 1 && newRole !== "admin") {
    const targetUser = await db.query.users.findFirst({
      where: eq(users.id, targetUserId),
    });
    if (targetUser?.role === "admin") {
      throw new Error("Cannot remove the last admin. Promote another user to admin first.");
    }
  }

  // Safe to proceed
  await db.update(users).set({ role: newRole }).where(eq(users.id, targetUserId));
}
```

### Pitfall 12: Feature Flags and Roles Conflated

**What goes wrong:**
Beta features are gated behind roles: "only tier_2 can see the new reader." When the feature launches to everyone, the role check must be removed from code. But it was hardcoded in 5 different components. Some get updated, some don't. Tier_1 users report the feature is "broken" (it's actually just hidden behind a stale role check).

**Prevention:**
- Use a separate feature flag system (even a simple env var or DB flag)
- NEVER use roles to gate beta features
- Roles = permanent access levels. Feature flags = temporary rollout controls.

## Performance Traps

Patterns that work during development but fail at scale.

| Trap | Symptoms | Prevention | When It Breaks |
|---|---|---|---|
| DB lookup in resolveRole() on every request | Slow page loads, DB connection exhaustion | Redis cache with 30s TTL, invalidate on role change | >100 concurrent users |
| Role capability check with no caching | Multiple hasCapability() calls per request each hit DB | Resolve role ONCE per request, cache in request context | >5 capability checks per page |
| courseAccess regeneration on every role change | Slow webhook response, timeout | Queue role-change events, process async | >50 students changing roles simultaneously |
| Listing all users with role-course JOIN | Full table scan on users + courseAccess | Add composite index on (user_id, course_id), paginate | >10,000 users |
| Clerk metadata update in tight loop | Clerk API rate limit (20 req/10s for Backend API) | Batch updates with 100ms delay between calls | >50 users in migration batch |
