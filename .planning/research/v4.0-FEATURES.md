# Feature Landscape: v4.0 Practice & Homework Platform

**Domain:** Language learning LMS with standalone practice, homework builder, pronunciation scoring, and enhanced AI chat
**Researched:** 2026-02-06
**Milestone:** v4.0 Practice & Homework Platform (subsequent milestone, builds on v1.0-v3.1)
**Confidence:** HIGH (features) / MEDIUM (pronunciation scoring APIs)

---

## Executive Summary

This research maps the feature landscape for transforming CantoMando Blueprint from a video-only LMS into a full practice platform. The five target capabilities -- visual homework builder, standalone practice sets, pronunciation scoring, iMessage-style AI practice chat, and custom phonetic fonts -- span different maturity levels in the language learning ecosystem.

**Homework/exercise builders** are well-established territory. Competitors like H5P, Articulate Rise, Typeform, and SurveyJS all provide drag-and-drop question builders with JSON schema output. The standard question types (multiple choice, fill-in-blank, matching, free text, audio recording) are table stakes. The differentiator is making these exercise types specifically useful for Chinese language learning -- tone pair matching, pinyin/jyutping annotation in prompts, character writing, and audio recording with immediate AI feedback.

**Pronunciation scoring for Chinese** is the most technically complex area. Three viable API options exist: SpeechSuper (purpose-built for Mandarin with per-character tone scoring at $0.004-$0.008/assessment), Azure Speech Pronunciation Assessment (supports zh-CN with phoneme-level accuracy but unclear tone-specific scoring), and a custom approach using OpenAI Whisper for transcription + GPT for analysis. SpeechSuper is the strongest candidate for per-character tone accuracy, but Cantonese pronunciation scoring has NO reliable third-party API -- this will need to be AI-prompt-based using existing OpenAI infrastructure.

**iMessage-style chat for practice** builds directly on the existing ChatPanel/ChatWidget components. The transformation from a "help assistant" to a "practice partner" is primarily a prompt engineering and UI enhancement task, not a fundamental architecture change. Competitors like HelloChinese and Duolingo use structured exercise flows within chat, but for adult learners, free-form conversation practice with lesson-specific scaffolding is more appropriate.

**Custom phonetic fonts** (Hanzi-Pinyin-Font for Mandarin, Cantonese Visual font for Cantonese) are a rendering approach where the font file itself contains pinyin/jyutping annotations above each hanzi glyph. This is simpler than HTML ruby markup for bulk text but limits annotation customization. The existing `ChineseAnnotation` component uses ruby markup for chatbot output; the font approach would complement this for body text and practice content.

## Anti-Features

Features to deliberately NOT build. Common mistakes in this domain that would waste effort or hurt the product.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Gamification (points, streaks, badges, leaderboards)** | Already in PROJECT.md "Out of Scope." Adult learners find gamification condescending. Duolingo does this well for casual learners; CantoMando targets serious paid adult learners who want depth, not dopamine | Track progress meaningfully: completion rates, pronunciation improvement trends, skill coverage. Motivation comes from visible progress and coach feedback, not artificial rewards |
| **Spaced repetition vocabulary system** | Already in PROJECT.md "Out of Scope" for v4.0. SRS is a deep product (Anki has decades of refinement). Building a mediocre SRS wastes effort and competes with tools students already use (Anki, Pleco). Defer to future milestone | Provide vocabulary lists per lesson that students can export to Anki/Pleco. Focus on contextual practice (exercises, chat) not isolated flashcards |
| **Character stroke order / handwriting input** | HelloChinese and Pleco do this excellently with years of refinement. Building a stroke order system requires custom canvas drawing, stroke detection, and a complete stroke database for thousands of characters | Support typed Chinese input (IME, already built). Recommend Pleco/Skritter for stroke practice. Handwriting is a different product |
| **Full SCORM/xAPI export** | H5P and Articulate exist for standards-based content. CantoMando is a self-contained LMS, not a content authoring tool. SCORM compatibility adds complexity with zero user value | Keep exercise data in own JSON schema. If external LMS integration ever needed, build an API, not SCORM export |
| **Automated speech-to-text transcription for practice** | Whisper can transcribe Chinese but doesn't score pronunciation quality. Transcribing what the student said is different from scoring how well they said it. Don't confuse STT with pronunciation assessment | Use purpose-built pronunciation APIs (SpeechSuper) for scoring. Use Whisper only where transcription is needed (voice conversation turns, which already exist) |
| **AI-generated exercises** | Tempting to have AI auto-generate practice questions from lesson content. But quality control for Chinese exercises is critical -- wrong tones, incorrect pinyin, or culturally inappropriate examples are worse than no exercises | Let coaches author exercises manually using the builder. AI can suggest improvements or validate, but coaches must approve all content. Quality over quantity |
| **Real-time multiplayer / competitive practice** | Some language apps add multiplayer quiz battles. Requires WebSocket infrastructure, matchmaking, latency optimization -- massive complexity for a small user base of paid adult learners | Focus on single-player mastery. Coach-student interaction is the social element, not student-student competition |
| **Offline practice mode** | PWA already provides basic caching. Full offline exercise mode requires local storage for exercise data, offline grading logic, and sync-on-reconnect -- significant complexity | Practice requires AI grading (online). Keep exercises online-only. PWA handles the "install to home screen" use case |
| **Voice-based exercise navigation** | "Say the answer instead of clicking" -- sounds cool but voice command recognition in Chinese is unreliable for navigation, and it confuses "I'm answering the question" with "I'm navigating the app" | Keep voice for pronunciation scoring and conversation practice. UI navigation stays touch/click |
| **Auto-grading character handwriting** | OCR + scoring for handwritten Chinese characters. Extremely complex, requires training data, and competitors (Skritter) have spent years on this | Stick to typed input + audio recording. Handwriting recognition is a standalone product |

### JSON Schema Design for Exercises

Every exercise should be stored as a JSON schema with a discriminated union pattern:

```typescript
// Base exercise structure
interface BaseExercise {
  id: string;
  type: ExerciseType;
  prompt: string;           // Can contain [char|pinyin|jyutping] annotations
  language: 'cantonese' | 'mandarin' | 'both';
  points: number;
  explanation?: string;     // Shown after grading
}

// Multiple Choice
interface MultipleChoiceExercise extends BaseExercise {
  type: 'multiple_choice';
  options: { id: string; text: string; isCorrect: boolean }[];
  shuffleOptions: boolean;
}

// Fill in the Blank
interface FillInBlankExercise extends BaseExercise {
  type: 'fill_in_blank';
  template: string;          // "I want to go to ___"
  acceptedAnswers: string[]; // Multiple correct answers
  gradingMode: 'exact' | 'ai';
}

// Matching Pairs
interface MatchingExercise extends BaseExercise {
  type: 'matching';
  pairs: { left: string; right: string }[];
  shuffleSides: boolean;
}

// Audio Recording
interface AudioRecordingExercise extends BaseExercise {
  type: 'audio_recording';
  targetText: string;        // Text to read aloud
  scoringMode: 'ai_grading' | 'pronunciation_api';
}

// Free Text
interface FreeTextExercise extends BaseExercise {
  type: 'free_text';
  expectedAnswer?: string;   // Hint for AI grading
  minLength?: number;
  maxLength?: number;
}

type Exercise = MultipleChoiceExercise | FillInBlankExercise | MatchingExercise | AudioRecordingExercise | FreeTextExercise;
```

This schema is stored in JSONB columns in Postgres. Drizzle ORM supports JSONB well. This approach matches SurveyJS and H5P patterns where exercises are data, not code.

## Pronunciation Scoring Deep Dive

### API Options Comparison

| Provider | Mandarin Support | Tone Scoring | Per-Character | Cantonese | Pricing | Integration |
|----------|-----------------|--------------|---------------|-----------|---------|-------------|
| **SpeechSuper** | YES (primary focus) | YES (initial, final, tone per syllable) | YES | NO | $0.004-$0.008/assessment, $20/mo minimum | REST API, most audio formats |
| **Azure Speech** | YES (zh-CN) | UNCLEAR (prosody scoring exists, tone-specific unclear) | YES (phoneme-level accuracy) | NO | Pay-per-use, ~$1/1000 assessments | SDK + REST, well-documented |
| **Speechace** | UNCLEAR (primarily English focus) | UNCLEAR | YES for supported languages | NO | Contact sales (not public) | REST API |
| **Custom (Whisper + GPT)** | YES (via transcription + analysis) | LIMITED (can compare transcribed tones, not acoustic) | YES (can analyze per-character) | YES (Whisper v3 supports yue) | OpenAI API costs (~$0.01/assessment) | Already using OpenAI, n8n compatible |

### Recommendation

**For Mandarin:** Use **SpeechSuper** as primary pronunciation scoring API. It provides exactly what's needed: per-character accuracy, tone score, initial/final consonant scores. The pricing ($0.004/word assessment) is reasonable for a paid learner product.

**For Cantonese:** Use **custom AI approach** -- send audio to Whisper for transcription, then use GPT to compare transcribed jyutping with expected pronunciation. This won't give acoustic-level tone scoring like SpeechSuper does for Mandarin, but Cantonese pronunciation scoring APIs simply don't exist from any major provider. Flag this as lower-quality scoring than Mandarin and be transparent with users.

**Fallback/MVP:** If SpeechSuper integration is too complex for early phases, start with **Azure Speech pronunciation assessment** for Mandarin (simpler SDK, Microsoft-maintained, known quantity) and upgrade to SpeechSuper later for tone-specific features.

### What the Scoring UI Should Look Like

Based on HelloChinese and ELSA Speak patterns:

1. **Target text display** -- large, centered Chinese text with pinyin above
2. **Record button** -- single tap to record, tap again to stop
3. **Overall score** -- large number (0-100) with color indicator (red/yellow/green)
4. **Per-character breakdown** -- each character color-coded:
   - Green: pronunciation correct, tone correct
   - Yellow: pronunciation close, or tone slightly off
   - Red: pronunciation incorrect or wrong tone
5. **Tone detail (expandable)** -- tap a character to see: "Expected: 3rd tone, Detected: 2nd tone"
6. **Retry button** -- "Try again" with previous score shown for comparison
7. **Coach review flag** -- option to send attempt to coach for detailed feedback

## iMessage-Style Practice Chat Deep Dive

### Transformation from Help Assistant to Practice Partner

The existing chatbot (`ChatWidget` + `ChatPanel`) is a help assistant -- floating widget, generic questions. The v4.0 enhanced chatbot should be a **practice partner** -- full-screen or large panel, lesson-specific, with structured practice flows.

**Current state:**
- Floating bottom-right widget (380px wide, 520px tall)
- Generic "Ask me anything" prompt
- No conversation persistence (resets on close)
- RAG search over knowledge base
- Chinese annotation rendering in AI responses

**Target state (iMessage-style practice chat):**
- Dedicated practice chat page (full width, not a floating widget)
- Lesson-specific context and suggested conversation starters
- Conversation persistence (save and resume)
- Student text also annotated (not just AI responses)
- Practice-oriented system prompt: "Let's practice ordering food using vocabulary from Lesson 3"
- Canto-to-Mando pedagogical awareness: "In Cantonese you'd say X, in Mandarin it's Y"
- Suggested practice topics derived from lesson content

**UI Patterns from Competitors:**

| App | Chat Approach | What Works |
|-----|--------------|------------|
| HelloChinese | Structured chat with multiple-choice in-line | Good for beginners, too constrained for adults |
| Duolingo | AI conversation with correction pop-ups | Natural flow with gentle correction |
| ChatGPT | Free-form with excellent Chinese | Too generic, no pedagogical structure |
| Speak (app) | Scenario-based roleplay chat | Excellent for conversation practice; specific situations |

**Recommendation:** Free-form conversation (like ChatGPT) with lesson scaffolding (like Speak app). Not structured multiple-choice in chat (too constrained for adult learners).

## Custom Font Deep Dive

### Two Font Types Needed

1. **Hanzi-Pinyin Font** (Mandarin) -- TTF/OTF font that renders pinyin above each hanzi character automatically. Available from `parlr/hanzi-pinyin-font` on GitHub. When applied to Chinese text via CSS `font-family`, each character displays its pinyin pronunciation above it without any HTML ruby markup.

2. **Cantonese Visual Font** -- Equivalent font for jyutping above Cantonese characters. User will provide the specific font files.

### Implementation Approach

```css
/* Global font-face declarations */
@font-face {
  font-family: 'HanziPinyin';
  src: url('/fonts/Hanzi-Pinyin-Font.woff2') format('woff2');
  font-display: swap;
}

@font-face {
  font-family: 'CantoneseVisual';
  src: url('/fonts/Cantonese-Visual-Font.woff2') format('woff2');
  font-display: swap;
}

/* Apply based on language preference */
.chinese-text-mandarin { font-family: 'HanziPinyin', sans-serif; }
.chinese-text-cantonese { font-family: 'CantoneseVisual', sans-serif; }
```

### Font vs Ruby: When to Use Each

| Context | Approach | Reason |
|---------|----------|--------|
| **Body text, lesson content** | Custom font | Automatic, no markup needed, works on any text |
| **Chat messages (AI output)** | Existing `ChineseAnnotation` ruby markup | AI already generates `[char|pinyin|jyutping]` format; more control |
| **Exercise prompts** | Custom font | Exercise text is coach-authored plain Chinese text |
| **Exercise answers/feedback** | Ruby markup | Need precise annotation per character for grading display |
| **Pronunciation scoring display** | Ruby markup | Need per-character color coding that fonts can't provide |

**Key limitation of font approach:** The font always shows pinyin for its built-in character set. You cannot selectively annotate some characters but not others. You cannot color-code individual annotations (for scoring). For any context requiring per-character control, use the existing ruby markup approach.

---

## MVP Recommendation

For MVP (first 2-3 phases of v4.0), prioritize:

### Must Ship First
1. **Custom fonts loaded site-wide** -- prerequisite for all Chinese text rendering improvements
2. **Voice AI language preference bug fix** -- quick fix, high impact
3. **Exercise schema and question types (MC, fill-blank, matching, audio, free text)** -- data model must be solid before builder UI
4. **Homework builder MVP (add/edit/reorder questions, preview)** -- coaches create content first
5. **Practice set player (sequential flow, immediate feedback, score summary)** -- students consume content

### Ship Second
6. **iMessage-style practice chat with conversation persistence** -- parallel to homework builder
7. **Practice set assignment (link to lesson/module/course)** -- connect practice to curriculum

### Ship Third
8. **Pronunciation scoring integration (SpeechSuper for Mandarin, AI-based for Cantonese)** -- most complex API integration
9. **Per-character tone highlights in voice tutor** -- depends on pronunciation scoring data
10. **Suggested practice topics from conversation analysis** -- connects chat + practice

### Defer to Post-MVP
- Word order exercise type
- Tone selection exercise type
- Listening comprehension exercise type
- Per-character tone heatmap over time
- AI-suggested practice based on conversation weaknesses
- Coach review of practice attempts (coach can see scores, but detailed attempt review is post-MVP)
- Reusable question bank (start with questions embedded in practice sets, extract to bank later)
