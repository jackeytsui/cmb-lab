# Project Research Summary

**Project:** CantoMando Blueprint LMS - v3.0 CRM-LMS Integration
**Domain:** GoHighLevel CRM integration with existing LMS
**Researched:** 2026-01-31
**Confidence:** HIGH

## Executive Summary

The v3.0 milestone integrates GoHighLevel (GHL) as the CRM backbone for student management in an existing Next.js LMS. Research across GHL API documentation, CRM-LMS integration patterns, and bidirectional sync architecture reveals a clear path: event-driven sync with n8n as middleware, storing only CRM data essential to coaching workflows, and treating GHL as the source of truth for contact data while the LMS owns learning data.

The recommended approach uses GHL's official SDK with Private Integration Tokens (not OAuth), event-sourced sync logging for conflict resolution, and a "window into CRM, not a mirror" philosophy. This prevents the critical pitfalls identified in research: infinite webhook loops from tag sync, lost events from webhook delivery failures, OAuth token refresh race conditions, and scope creep that transforms the LMS into a shadow CRM. The integration adds exactly one new dependency (`@gohighlevel/api-client`) and leverages existing infrastructure (Upstash Redis for rate limiting, Neon Postgres for sync state, n8n for automation).

The key architectural decision is bidirectional tag sync with explicit ownership rules: LMS owns milestone tags (`module_1_complete`, `inactive_7d`), GHL owns marketing tags (`vip_student`, `coaching_tier_premium`), and both systems can write coaching tags (`needs_coaching`) with last-write-wins conflict resolution using echo detection. All other CRM data (timezone, goals, pipeline stages) flows one-way from GHL to LMS as read-only cached display. This prevents 90% of potential conflict scenarios while giving coaches the context they need without rebuilding GHL inside the LMS.

## Key Findings

### Recommended Stack

**Official GHL SDK with Private Integration Token approach** — The official `@gohighlevel/api-client` v2.2.2 provides typed TypeScript interfaces for GHL V2 API with built-in token management and webhook signature verification. Private Integration Tokens simplify authentication for single-tenant integrations (one GHL sub-account to one LMS instance) compared to full OAuth marketplace app registration. Rate limiting managed through existing Upstash Redis infrastructure (already in stack for webhook limiting) with 100 requests per 10 seconds burst limit and 200,000 daily limit. Expected usage: 50-200 API calls per day for 500-1000 students.

**Core technologies:**
- **@gohighlevel/api-client v2.2.2**: Official TypeScript SDK for GHL V2 API — only new dependency, handles auth and signature verification
- **Upstash Redis (existing)**: Rate-limit outbound GHL API calls and echo detection for webhook loops — reuses infrastructure already deployed for webhook rate limiting
- **n8n Cloud (existing)**: Middleware for complex automations — LMS fires events to n8n webhooks, n8n orchestrates GHL workflows without adding GHL credentials to LMS
- **Drizzle ORM (existing)**: Schema additions for sync state — three new tables: `ghl_contacts` (contact link cache), `sync_events` (audit log and retry queue), `ghl_tags` (local tag cache)
- **Event-sourced sync log pattern**: PostgreSQL table as both audit trail and processing queue — eliminates need for Redis queue or BullMQ at this scale

**Critical version requirements:**
- GHL API V2 only (V1 end-of-support January 2026)
- Private Integration Token (NOT OAuth) for single-tenant simplicity
- `@gohighlevel/api-client` compatible with Node.js 18+ (already satisfied by Next.js 16)

### Expected Features

**Must have (table stakes for student management):**
- **Searchable, sortable, paginated student table** — coaches expect to find students quickly by name/email, sort by progress/last active, not scroll through flat list
- **Filter by course enrollment and progress status** — "show me students in Course X who haven't started" is basic cohort management
- **Bulk course assignment and removal** — coaches onboard cohorts of 20-50 students, not one-by-one
- **Student detail view with progress data** — click student to see enrolled courses, progress per course, submissions, coach notes (extends existing admin student profile)
- **Last active indicator** — coaches need to spot disengaged students ("no activity in 7+ days")
- **GHL contact linking** — map LMS user to GHL contact via persistent ID, not email (email changes break lookup)
- **CRM context display (read-only)** — show timezone, goals, native language from GHL on student profile without API call on every load (cached with 5-min TTL)

**Should have (competitive differentiation):**
- **Student tagging system** — coaches tag students as "beginner-cohort-jan", "needs-extra-help", "vip" for cohort filtering and re-engagement
- **Bidirectional GHL tag sync** — tags created in LMS sync to GHL (enables CRM automations), tags from GHL sync to LMS (coach sees CRM segmentation)
- **Bulk tagging** — select 15 students, add tag "needs-review" in one action (batch insert, not 15 API calls)
- **Activity timeline on student profile** — chronological view of enrolled, completed lesson, submitted audio, received feedback (aggregates multiple tables)
- **Engagement status badge** — auto-calculated "Active" (<7 days), "Inactive" (7-30 days), "At Risk" (>30 days) based on `lastAccessedAt`
- **"View Full Profile in GHL" deep link** — one-click to student's GHL contact page for features LMS doesn't replicate (pipeline, emails, appointments)

**Defer (scope boundary, NOT building a shadow CRM):**
- **Pipeline stage editor** — GHL owns sales pipeline, LMS should display stage name only with link to edit in GHL
- **Email/SMS sending from LMS** — communication channels are CRM territory, keep in GHL
- **Payment/invoice management** — revenue tracking belongs in GHL/Stripe, not LMS
- **Calendar/scheduling** — booking coaching calls stays in GHL (Calendly integration)
- **Full contact history (call logs, email history)** — show LMS activity timeline only, link to GHL for full CRM history
- **Workflow automation builder** — automations belong in n8n or GHL workflows, not in LMS UI

### Architecture Approach

**Event-sourced sync with explicit ownership rules** — All sync operations (inbound and outbound) are logged in a `sync_events` PostgreSQL table which serves as both audit trail and processing queue. Every webhook event, API call, and tag change creates a row. This enables conflict resolution (check for recent outbound before applying inbound), retry logic (filter by `status = 'failed' AND retryCount < 5`), and debugging ("why did this tag disappear?"). The pattern eliminates need for separate message queue infrastructure (Redis queue, BullMQ) at expected scale (<1000 students, 50-200 API calls/day).

**Major components:**

1. **GhlClient** (`lib/ghl/client.ts`) — Typed wrapper around GHL REST API with rate limiting, token management, and retry logic; all GHL API calls go through this, never direct `fetch()`
2. **GhlSyncService** (`lib/ghl/sync.ts`) — Orchestrates bidirectional sync with conflict resolution; owns tag ownership rules (LMS tags vs GHL tags vs shared tags)
3. **GhlWebhookHandler** (`app/api/webhooks/ghl/route.ts`) — Receives GHL webhook events (`ContactTagUpdate`, `ContactUpdate`), verifies RSA signature, creates `sync_events` row, returns 200 immediately (never blocks webhook response)
4. **MilestoneDetector** (`lib/ghl/milestones.ts`) — Detects learning events that trigger GHL actions (module complete → add tag, 7 days inactive → add tag); called after progress updates, fire-and-forget pattern
5. **Student Management Dashboard** (extends existing `/coach/students`) — Data table with search, filters, bulk operations, tag management; integrates with existing `StudentAccessManager` component

**Data ownership rules (critical for conflict avoidance):**
- **LMS owns:** Learning data (progress, milestones, submissions), milestone tags (`module_1_complete`, `course_complete`, `inactive_7d`)
- **GHL owns:** CRM data (contact fields, pipeline stages, marketing tags), marketing tags (`vip_student`, `coaching_tier_*`)
- **Bidirectional (last-write-wins):** Coaching tags (`needs_coaching`, `received_feedback`) with echo detection to prevent infinite loops

**Integration with existing architecture:**
- Webhook pattern: Follows existing `webhooks/clerk` and `webhooks/enroll` pattern (signature verification, rate limiting, return 200 quickly)
- Rate limiting: Extends existing `src/lib/rate-limit.ts` webhookLimiter with GHL-specific limiter (100/min vs 10/min for general webhooks)
- Progress tracking: Hooks into existing progress API route (`api/progress/[lessonId]`) to call MilestoneDetector after updates
- n8n integration: LMS fires events to n8n webhook URLs (existing pattern from coach feedback notifications), n8n routes to GHL

### Critical Pitfalls

1. **Infinite webhook loop from tag sync** — LMS adds tag → GHL fires `ContactTagUpdate` webhook → LMS processes as "new" tag → syncs back to GHL → loop. GHL fires webhooks for ALL tag changes including API-initiated changes with no `changeSource` field. Prevention: Redis-based echo detection (mark outbound changes with 60s TTL, ignore webhooks matching recent outbound changes). Detection: rapid spike in GHL API calls, same tag appearing/disappearing in logs, 429 responses within seconds of deploying tag sync. Recovery: deploy echo detection, clear Redis, events self-heal. **Must address: Sync Infrastructure Phase (before any bidirectional sync enabled).**

2. **GHL webhook delivery failures silently lose events** — LMS endpoint returns 500 error (DB timeout, deployment), GHL does NOT retry on 5xx errors (only retries 429 with 6 attempts over 70 minutes). Lost enrollment upgrades, lost tag changes, no error notification. Prevention: ALWAYS return 200 immediately after storing raw webhook in `ghl_webhook_events` table, process async via cron or background job; implement daily reconciliation as safety net (fetch all contacts, compare with LMS state, fix drift). Detection: GHL Webhook Logs Dashboard shows "failed" deliveries, students report "paid but can't access", tag counts don't match. Recovery: run reconciliation job to detect and fill gaps. **Must address: Webhook Infrastructure Phase (before connecting any GHL webhook events).**

3. **OAuth token refresh race condition kills all GHL API access** — Two simultaneous operations detect expired token, both call `/token` refresh endpoint, GHL uses single-use refresh tokens (rotation), first succeeds and gets new token, second fails with `invalid_grant` using old token, stored token corrupted or overwritten with garbage. Prevention: Redis mutex lock for token refresh (only one process refreshes, others wait and read new token), proactive refresh (refresh 5 minutes before expiration, not on-demand). Detection: `invalid_grant` errors in logs (CRITICAL), multiple simultaneous 401s from GHL, GHL features stop working for all users. Recovery: manual re-authorization via OAuth flow, update stored tokens. **Must address: Sync Infrastructure Phase (before ANY GHL API calls).**

4. **Rate limit exhaustion cascades across all GHL features** — Bulk operation (sync tags for 50 students) burns through 100-requests-per-10-seconds burst limit, ALL subsequent GHL API calls fail with 429 (shared limit across endpoints), coach dashboard shows "CRM unavailable", new enrollment processing delayed. Prevention: Shared queue for ALL GHL calls with concurrency limit (5 concurrent, 80 per 10s window for headroom), priority queue for urgent vs batch, use GHL bulk tag endpoint (up to 500 contacts in single call instead of 50 individual calls). Detection: `X-RateLimit-Remaining` headers dropping to 0, 429 responses during batch operations. Recovery: pause batch operations, wait for limit reset (10s burst, 24h daily). **Must address: Sync Infrastructure Phase (rate-limited client wrapper before ANY calls).**

5. **Tag schema mismatch creates silent sync failures** — Coach creates tag "module-1-complete" in LMS, GHL already has "Module 1 Complete" (spaces, capitals) from manual workflow, GHL now has TWO tags meaning same thing, GHL automations trigger on "Module 1 Complete" but LMS only sets "module-1-complete", automations never fire. GHL tags are case-sensitive and space-sensitive. Prevention: Canonical tag registry (`lib/ghl/tag-registry.ts`) with both internal key and GHL name mapping, all tags defined in one place, bidirectional lookup (LMS key ↔ GHL name), validate direction before syncing. Detection: GHL dashboard shows duplicate-looking tags, automations not firing despite tags set, tag count grows unexpectedly. Recovery: audit tags in both systems, deduplicate in GHL manually, establish registry. **Must address: Tag Schema Phase (before any tag sync implemented).**

6. **Contact ID mapping breaks on email changes** — LMS maps users to GHL contacts by email, student changes email in Clerk, Clerk webhook updates LMS email, GHL contact still has old email, LMS can't find GHL contact (email lookup returns nothing), all sync silently fails. Prevention: Persistent `ghl_contact_mapping` table linking `userId` to `ghlContactId` (never rely on email for lookups), when Clerk email changes, also update GHL contact email via API. Detection: "Contact not found" errors in sync logs, students with CRM data that suddenly goes blank, duplicate contacts in GHL. Recovery: manual contact merge in GHL, update mapping table. **Must address: Contact Sync Phase (mapping table must be FIRST schema created).**

7. **Scope creep turns LMS into shadow CRM** — Coaches want to see pipeline stages, then move contacts between stages, then send emails, then view conversation history; soon 40% of GHL rebuilt inside LMS. Each feature doubles sync complexity and bug surface area. Prevention: Tier boundary document (Tier 1: display only, Tier 2: LMS-owned writes, Tier 3: bidirectional carefully, Tier 4: DO NOT BUILD - use GHL directly), "Open in GoHighLevel" deep links for Tier 4 features (10 minutes vs 10 weeks). Detection: feature requests starting with "Can we also show...", sync logic getting more complex each sprint. Recovery: rip out excess features, restore scope boundary (significant code deletion). **Must address: Planning Phase (write boundary document, get sign-off before development).**

## Implications for Roadmap

Based on research, suggested phase structure emphasizes infrastructure before features to avoid critical pitfalls:

### Phase 1: Sync Infrastructure & Contact Linking

**Rationale:** Everything depends on having a working GHL API client with bulletproof token management and rate limiting. All critical pitfalls (infinite loops, token refresh races, rate limit cascades) are prevented in this phase. Building features first and "adding sync later" guarantees hitting these pitfalls in production.

**Delivers:**
- `@gohighlevel/api-client` integration with rate-limited wrapper
- OAuth token management with Redis mutex lock (or PIT token management if using Private Integration)
- `ghl_contacts` table linking LMS users to GHL contacts (by persistent ID, not email)
- `sync_events` table (audit log and processing queue)
- Contact linking logic (match by email, store mapping)
- Echo detection infrastructure (Redis markers for outbound changes)

**Addresses:**
- Stack: GHL SDK, Upstash Redis, Drizzle schema additions
- Pitfalls: Token refresh race condition, rate limit exhaustion, contact mapping breaks

**Avoids:**
- Infinite webhook loops (echo detection built from day one)
- Lost events (event log pattern established)
- Silent sync failures (contact ID mapping, not email)

**Research flag:** Standard integration pattern, well-documented GHL API. Skip `/gsd:research-phase` — official SDK docs and OAuth guides are comprehensive.

---

### Phase 2: Outbound Events (LMS → GHL)

**Rationale:** Outbound is simpler than inbound (no conflict resolution needed). Provides immediate value — GHL automations can trigger on LMS milestones (course complete, student inactive). Proves sync infrastructure works before adding bidirectional complexity.

**Delivers:**
- `MilestoneDetector` service (module complete, course complete, inactive detection)
- `GhlOutboundService` (process pending sync events, retry failed)
- Integration points: progress API route, coach feedback route, enrollment webhook
- Admin sync log page (read-only view of `sync_events` for debugging)
- Webhook events fired to n8n (optional, if GHL automations need LMS events)

**Addresses:**
- Features: GHL tag sync (one direction), milestone-based automations
- Architecture: Event-sourced sync pattern, MilestoneDetector component
- Pitfalls: Rate limiting (uses GhlClient queue), sync failures logged in events table

**Avoids:**
- Sync every progress update (only milestones trigger GHL calls)
- Blocking user requests (fire-and-forget milestone detection)
- Silent failures (all events logged, retry logic built-in)

**Research flag:** Skip phase research — milestone detection and tag API calls are straightforward, covered in project research.

---

### Phase 3: Inbound Webhooks & Tag Sync (GHL → LMS)

**Rationale:** Requires conflict resolution logic (tags are bidirectional) which builds on sync_events infrastructure from Phase 2. Webhook handling follows existing Clerk/enrollment webhook patterns. Tag ownership rules prevent most conflicts.

**Delivers:**
- GHL webhook endpoint (`/api/webhooks/ghl`)
- Webhook signature verification (RSA SHA256)
- Tag sync with conflict resolution (ownership rules + echo detection)
- Contact field sync (timezone, custom fields — read-only from GHL)
- `ghl_tags` table (local cache with source tracking)
- Admin resync trigger (manual reconciliation button)
- Separate webhook rate limiter (100/min vs 10/min generic)

**Addresses:**
- Features: Bidirectional tag sync, CRM context display
- Architecture: GhlWebhookHandler, GhlSyncService with conflict resolution
- Pitfalls: Infinite webhook loops (echo detection), lost events (receive-and-queue pattern), webhook rate limiting conflicts

**Avoids:**
- Tag schema mismatch (canonical registry from Phase 1)
- Stale CRM data (webhook-driven cache invalidation)
- 5xx errors losing events (return 200 immediately, process async)

**Research flag:** **Needs `/gsd:research-phase`** — conflict resolution patterns for bidirectional sync need deeper validation. Research areas: tag ownership strategies, echo detection alternatives, webhook delivery guarantees.

---

### Phase 4: Student Management Dashboard

**Rationale:** UI layer that consumes data from Phases 1-3. Extends existing coach student list with filters, search, bulk operations. Can be built in parallel with Phase 3 if sync infrastructure is stable.

**Delivers:**
- Enhanced `/coach/students` page: searchable data table, sort by columns, pagination
- Filters: by course enrollment, by progress status, by tag, by engagement status
- Bulk operations: select students (checkboxes, shift-click), bulk assign/remove course, bulk add/remove tag
- Student detail slide-out: progress per course, activity timeline, coach notes inline, CRM context card
- Engagement status badge (Active/Inactive/At Risk computed from lastAccessedAt)
- Tag management UI: create tags, rename, delete, color coding

**Addresses:**
- Features: All table stakes (search, filter, bulk assign), differentiators (tagging, activity timeline, CRM context)
- Pitfalls: Bulk operation error handling (per-item results, retry failed, undo capability)

**Avoids:**
- Scope creep (CRM context is display-only, "View in GHL" link for editing)
- Stale data display (5-min cache TTL, freshness indicator, manual refresh button)
- Poor bulk UX (detailed error feedback, undo capability)

**Research flag:** Skip phase research — data table patterns and bulk operations are well-established UI patterns.

---

### Phase 5: Reconciliation & Hardening

**Rationale:** Safety net for webhook failures and drift detection. Runs after core sync is proven stable. Optimization and reliability layer.

**Delivers:**
- Daily reconciliation cron job (fetch all GHL contacts, compare tags/fields with LMS, fix drift)
- Inactive student detection cron (no activity in 7/14/30 days, queue tag events)
- Sync health dashboard (success rate, failed events chart, drift detection alerts)
- Retry failed events job (process `sync_events` where `status = 'failed'` and `retryCount < 5`)
- Alert on daily rate limit approaching (email admin if <10K remaining)
- Bulk tag operation batching (use GHL bulk endpoint for 50+ students)

**Addresses:**
- Pitfalls: Lost webhook events (reconciliation catches gaps), rate limit exhaustion (bulk endpoint usage), stale data (drift detection)
- Features: Inactive student tagging (automated re-engagement)

**Avoids:**
- Relying solely on webhooks (reconciliation as safety net)
- Rate limit waste (batch operations instead of individual calls)
- Silent drift accumulation (proactive monitoring)

**Research flag:** Skip phase research — reconciliation patterns covered in project research.

### Phase Ordering Rationale

**Infrastructure-first approach (Phases 1-3 before UI):**
- All critical pitfalls relate to sync infrastructure (infinite loops, lost events, token races, rate limits)
- Building UI first creates pressure to ship before infrastructure is hardened
- Outbound-before-inbound reduces complexity (one-way sync proves infrastructure before adding conflict resolution)
- Each phase validates previous phase's stability (Phase 2 proves Phase 1 client works, Phase 3 proves Phase 2 event log works)

**Dependencies discovered in research:**
- Phase 3 requires Phase 2's `sync_events` table for conflict resolution (check for recent outbound before applying inbound)
- Phase 4 requires Phase 3's contact linking and tag cache (dashboard displays GHL data)
- Phase 5 requires stable Phases 1-3 before adding reconciliation (can't reconcile until baseline sync works)

**Pitfall avoidance mapping:**
- Infinite loops: Prevented in Phase 1 (echo detection infrastructure)
- Lost events: Prevented in Phase 2 (event log pattern) and Phase 3 (webhook receive-and-queue)
- Token races: Prevented in Phase 1 (mutex lock)
- Rate limits: Prevented in Phase 1 (rate-limited client) and Phase 5 (bulk operations)
- Tag mismatch: Prevented in Phase 1 (tag registry) and Phase 3 (validation)
- Scope creep: Prevented in Phase 4 (boundary document, display-only CRM context)

### Research Flags

**Phases needing `/gsd:research-phase` during planning:**

- **Phase 4 (Student Dashboard):** If adding Chinese language search to student table (pg_trgm doesn't support CJK) — research pg_search ICU tokenizer effectiveness or Meilisearch alternative

**Phases with standard patterns (skip research-phase):**

- **Phase 2 (Outbound Events):** Milestone detection, tag API calls — straightforward, covered in project research

### Gaps to Address

**GHL custom field mapping configuration:**
- Research shows goals/native language stored as custom fields (not standard fields)
- Field IDs vary per GHL sub-account (must be discovered via API)
- Gap: Admin UI for mapping GHL custom field IDs to LMS concepts needs design during Phase 1 planning
- Mitigation: Create one-time setup script to fetch field definitions, store mapping in database, admin can update without redeployment

**Chinese text search in student table:**
- Standard `pg_trgm` doesn't support CJK characters (documented limitation)
- Neon supports `pg_search` with ICU tokenizer (Unicode-aware segmentation for Chinese)
- Gap: pg_search effectiveness for Chinese needs validation during implementation
- Mitigation: Start with standard `tsvector` for English fields (student names, course titles), add pg_search for Chinese vocabulary search only if needed, defer Meilisearch integration to post-MVP if pg_search insufficient

**GHL webhook signature verification complexity:**
- GHL uses RSA SHA256 public key verification (not simple HMAC)
- Public key available from GHL marketplace app settings
- Gap: RSA verification implementation needs validation with actual GHL webhook payloads
- Mitigation: Fallback option is webhook secret header (same pattern as enrollment webhook `x-webhook-secret`), less secure but simpler; test RSA verification in Phase 3 planning, fall back to secret if RSA proves problematic

**Reconciliation job frequency:**
- Daily reconciliation catches drift but 24h lag for critical data
- Gap: Determine which data needs hourly vs daily reconciliation during Phase 5 planning
- Mitigation: Start with daily for all data types, add hourly reconciliation only for critical fields (contact linking, access-related tags) based on monitoring

**Scale thresholds for infrastructure changes:**
- Postgres-as-queue works up to ~10,000 events/day (research validated)
- Current expected scale: 50-200 API calls/day for 500-1000 students
- Gap: Monitoring thresholds for when to migrate to dedicated queue (Redis Stream, BullMQ) not defined
- Mitigation: Set alerts at 5,000 events/day (50% threshold), 8,000 events/day (80% threshold) to trigger architecture review before hitting limits
