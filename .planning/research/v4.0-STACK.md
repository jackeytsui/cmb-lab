## New Stack Additions

### 1. Drag-and-Drop: @dnd-kit/react (Canvas Homework Builder)

| Package | Version | Purpose |
|---------|---------|---------|
| @dnd-kit/react | ^0.2.1 | Drag-and-drop primitives for homework builder |
| @dnd-kit/helpers | ^0.2.1 | Array manipulation utilities for sortable lists |

**Why @dnd-kit/react over alternatives:**

| Library | Verdict | Reason |
|---------|---------|--------|
| **@dnd-kit/react** | RECOMMENDED | Grid + list + canvas support, React-native, fine-grained control, active rewrite with simplified API |
| @hello-pangea/dnd | NO | List-only (no grid/canvas), heavier bundle, community fork (Atlassian abandoned original) |
| pragmatic-drag-and-drop | MAYBE | Lightweight + native DnD API, but documentation tied to Atlassian's Atlaskit, steeper learning curve, less community support |
| react-dnd | NO | Older, lower-level, more boilerplate, less active maintenance |
| framer-motion (Reorder) | SUPPLEMENT ONLY | Already installed; good for simple list reorder animations but lacks the canvas/grid primitives needed for a homework builder |

**Rationale for @dnd-kit/react:**
- The homework builder requires **multiple drag-and-drop patterns**: sortable question list, drag question types from palette onto canvas, reorder answer options within questions, and potentially nested containers (question groups). @dnd-kit is the only library that supports all of these in a single package.
- The new `@dnd-kit/react` rewrite (replacing `@dnd-kit/core` + `@dnd-kit/sortable`) ships a simplified API: `DragDropProvider` replaces `DndContext`, hooks return a single `ref` object instead of spreading `attributes`/`listeners`/`setNodeRef` separately.
- React 19 explicitly supported in peer dependencies (fixed in 0.2.x).
- 0.2.x has StrictMode fixes and React lifecycle regression fixes, making it production-viable.

**Version note:** @dnd-kit/react 0.2.1 is the new rewrite, NOT the legacy @dnd-kit/core 5.x. The legacy packages (@dnd-kit/core, @dnd-kit/sortable, @dnd-kit/utilities) are being replaced. Use the NEW packages.

**Integration with existing stack:**
- framer-motion (already installed) provides the animation layer. Use `motion.div` for drag preview animations and drop transitions. @dnd-kit handles the drag logic, framer-motion handles the visual polish.
- shadcn/ui components wrap @dnd-kit `useSortable` for consistent styling of draggable question blocks.
- XState (already installed) manages builder state machine: idle -> dragging -> dropped -> editing -> saving.

**What NOT to use:**
- Do NOT use `framer-motion` `Reorder` component as the primary DnD solution. It handles simple vertical/horizontal reorder but cannot do: drag from palette to canvas, multi-container sorting, or grid layouts.
- Do NOT install the legacy `@dnd-kit/core` + `@dnd-kit/sortable`. Use the new `@dnd-kit/react` + `@dnd-kit/helpers`.
- Do NOT install `react-beautiful-dnd` (deprecated by Atlassian) or `@hello-pangea/dnd` (fork, list-only).

**Confidence:** HIGH -- npm registry confirms 0.2.1 published Jan 2026, React 19 peer dependency supported, migration guide available at next.dndkit.com.

### 2. Pronunciation Scoring: Azure Speech SDK (Mandarin + Cantonese)

| Package | Version | Purpose |
|---------|---------|---------|
| microsoft-cognitiveservices-speech-sdk | ^1.47.0 | Pronunciation assessment for zh-CN and zh-HK |

**Why Azure Speech over alternatives:**

| Service | Mandarin Tones | Cantonese | Per-Character | Browser SDK | Verdict |
|---------|---------------|-----------|---------------|-------------|---------|
| **Azure Speech** | YES (phoneme accuracy) | YES (zh-HK supported) | YES (word/phoneme level) | YES (JS SDK) | RECOMMENDED |
| SpeechSuper | YES (tone prediction, initial/final/tone scores) | NO (8 languages, Cantonese not listed) | YES (single character SKU) | REST API only (no browser SDK, iOS/Android SDKs only) | NO -- missing Cantonese |
| OpenAI Whisper | NO (transcription only, no scoring) | Partial (transcription) | NO | Via API only | NO -- not a scoring engine |
| ELSA Speak API | English-focused | NO | YES | REST API | NO -- wrong language focus |
| Speechace | Limited Chinese | NO | YES | REST API | NO -- primarily English |

**Rationale for Azure Speech:**
- **Both Mandarin AND Cantonese supported** for pronunciation assessment. Confirmed: zh-CN (Mandarin, Simplified) and zh-HK (Chinese, Cantonese, Traditional) are both in the 33 supported locales. This is the critical differentiator -- SpeechSuper has better Mandarin tone scores but does NOT support Cantonese at all.
- Per-word and per-phoneme accuracy scores returned in JSON response for Chinese.
- JavaScript SDK (`microsoft-cognitiveservices-speech-sdk`) works in both browser and Node.js. Uses WebSocket internally, auto-selects appropriate implementation per environment.
- Phoneme names available for zh-CN with SAPI format (not IPA).
- Pricing: same as Speech-to-Text standard tier (~$1.32/hour, prorated per second). An 8-second pronunciation check costs ~$0.003.

**CRITICAL LIMITATION -- Tone-Specific Scoring:**
Azure Speech provides **phoneme-level accuracy** for zh-CN and zh-HK, but **prosody assessment (which includes tone/intonation scoring) is currently English-only (en-US)**. This means:
- You GET: per-word accuracy score, per-phoneme accuracy score, fluency score, completeness score
- You DO NOT GET: explicit "tone 1 vs tone 2 vs tone 3 vs tone 4" classification per character

**Workaround for tone scoring:**
1. Use Azure Speech phoneme accuracy as the primary score (a mispronounced tone will still reduce phoneme accuracy).
2. Supplement with a **post-processing step**: compare the recognized text (from Azure STT) against the expected text character-by-character. If a character is recognized differently (e.g., student says "ma3" but expected "ma1"), flag it as a tone error. This uses pinyin-pro (already installed) to map characters to expected tones.
3. For Cantonese, use to-jyutping (already installed) for the same comparison.
4. Display results as: overall accuracy score (from Azure) + per-character tone match/mismatch (from post-processing).

**Alternative for richer Mandarin tone scoring (future consideration):**
SpeechSuper provides explicit tone prediction (actual tone spoken vs expected tone) for Mandarin single characters, with initial sound, final sound, and tone scores. If Mandarin-only tone accuracy becomes insufficient with Azure, consider adding SpeechSuper as a supplementary API for Mandarin-specific exercises only. But do NOT use it as the primary service because it lacks Cantonese entirely.

**Integration with existing stack:**
- The voice tutor already uses OpenAI Realtime API via WebRTC (`useRealtimeConversation.ts`, `realtime-utils.ts`). Pronunciation scoring is a SEPARATE flow -- not real-time conversation, but discrete "record -> assess -> show results" interactions.
- Create a new API route: `POST /api/pronunciation/assess` that receives audio blob + reference text + locale (zh-CN or zh-HK), sends to Azure Speech, returns scored JSON.
- Rate limit via Upstash (already configured) to prevent abuse.
- Store assessment results in Neon Postgres for progress tracking.

**Azure service setup required:**
- Create Azure Speech resource in Azure portal (Free tier: 5 hours/month of speech-to-text, sufficient for development and early users).
- Environment variables: `AZURE_SPEECH_KEY`, `AZURE_SPEECH_REGION`.

**Confidence:** MEDIUM -- Azure Speech zh-CN and zh-HK pronunciation assessment confirmed via official language support docs. Tone-specific scoring limitation confirmed (prosody is en-US only). The post-processing workaround for tone identification is an engineering approach, not a verified product feature.

### 3. Custom Font Loading: next/font/local (NO new package needed)

| Package | Version | Purpose |
|---------|---------|---------|
| *No new package* | -- | Use built-in `next/font/local` from Next.js 16 |

**Why no new package:**
Next.js has built-in support for local font files via `next/font/local`. This handles:
- Automatic font subsetting and optimization
- Self-hosting (no external font CDN requests)
- CSS variable injection for Tailwind integration
- Zero layout shift (font-display: swap with size-adjust)

**Implementation pattern for the two custom Chinese fonts:**

The project needs two custom font files:
1. **Hanzi Pinyin Font** (.ttf/.otf) -- renders Pinyin annotations above Chinese characters for Mandarin
2. **Cantonese Visual Font** (.ttf/.otf) -- renders Jyutping annotations above Chinese characters for Cantonese

```typescript
// src/app/layout.tsx
import localFont from "next/font/local";
import { Inter } from "next/font/google";

const inter = Inter({
  subsets: ["latin"],
  variable: "--font-inter",
});

const hanziPinyin = localFont({
  src: "../fonts/HanziPinyin.ttf",  // path relative to layout.tsx
  variable: "--font-hanzi-pinyin",
  display: "swap",
});

const cantoneseVisual = localFont({
  src: "../fonts/CantoneseVisual.ttf",
  variable: "--font-cantonese-visual",
  display: "swap",
});

// In the body tag:
<body className={`${inter.variable} ${hanziPinyin.variable} ${cantoneseVisual.variable} font-sans ...`}>
```

**Tailwind CSS 4 integration:**
```css
/* In globals.css or tailwind config */
@theme {
  --font-hanzi-pinyin: var(--font-hanzi-pinyin);
  --font-cantonese-visual: var(--font-cantonese-visual);
}
```

Then use in components: `className="font-hanzi-pinyin"` or `className="font-cantonese-visual"`.

**Font file placement:**
- Place font files in `src/fonts/` (co-located with app directory, recommended by Next.js docs)
- OR in `public/fonts/` and reference with absolute path `/fonts/HanziPinyin.ttf`
- The `src/fonts/` approach is preferred because `next/font/local` optimizes fonts at build time (subsetting, format conversion).

**IMPORTANT CAVEAT:** These are specialty annotation fonts -- they render phonetic annotations above characters as part of the glyph. They should ONLY be applied to specific components (exercise displays, reading practice) where phonetic annotations are desired, NOT globally to all Chinese text. Use the CSS variable approach to scope font application.

**Format recommendation:** If the provided files are .ttf, convert to .woff2 for 30-50% smaller file size. `next/font/local` supports both, but .woff2 loads faster. Use a tool like `woff2_compress` or an online converter.

**Confidence:** HIGH -- next/font/local is a core Next.js feature, documented extensively. The CSS variable approach for multiple fonts in App Router is the standard pattern.

### 4. Exercise Rendering: No dedicated library needed (build with existing stack)

| Package | Version | Purpose |
|---------|---------|---------|
| *No new package* | -- | Build exercise types with React + shadcn/ui + framer-motion |

**Why build custom instead of using a library:**

There is no off-the-shelf React library for "language learning exercise rendering" that handles MCQ, matching, fill-in-blank, ordering, audio recording, and free text in a unified way. The closest options are:
- **h5p-react** -- H5P content player for React. H5P is an open-source interactive content framework used in Moodle/WordPress. However, it is extremely opinionated, hard to style (its own CSS framework), and the React wrapper is poorly maintained.
- **react-quiz-component** -- Simple MCQ quiz component. Too basic, no matching/ordering/audio support, not customizable enough.
- **Survey.js** -- Powerful form builder with many question types. But it is a full survey engine with its own styling system, not designed for language learning exercises. Massive bundle size.

**Recommended approach -- build exercise renderers as React components using existing stack:**

| Exercise Type | Implementation | Existing Stack Used |
|---------------|---------------|---------------------|
| Multiple Choice (MCQ) | Radio/checkbox group with feedback states | shadcn/ui RadioGroup, framer-motion for reveal animation |
| Matching | Two-column drag-to-match or tap-to-pair | @dnd-kit/react (new addition) for drag, or tap-based pairing with state |
| Fill-in-the-Blank | Inline input fields within text | shadcn/ui Input, react-hook-form for validation |
| Ordering | Sortable list of items | @dnd-kit/react sortable (new addition) |
| Audio Recording | Record and submit audio for assessment | MediaRecorder API (browser native), Azure Speech SDK (new addition) |
| Free Text | Text area with AI grading | shadcn/ui Textarea, AI SDK (existing) via n8n webhook |

**Architecture pattern -- Exercise Component Interface:**
```typescript
interface ExerciseProps {
  type: "mcq" | "matching" | "fill-blank" | "ordering" | "audio" | "free-text";
  config: ExerciseConfig;  // Question data from DB
  onSubmit: (answer: ExerciseAnswer) => Promise<ExerciseResult>;
  locale: "zh-CN" | "zh-HK";  // For font and phonetic display
}
```

Each exercise type gets its own component (`McqExercise.tsx`, `MatchingExercise.tsx`, etc.) with a shared `ExerciseShell` wrapper that handles timing, attempts, feedback display, and progress tracking.

**Why this approach is better than a library:**
- Full control over styling (dark mode, cinematic aesthetic matches existing app)
- Chinese character rendering with custom fonts requires tight integration
- AI grading via n8n webhook is custom to this app's architecture
- Exercise types are relatively simple UI patterns -- the complexity is in the grading logic, not the rendering

**Confidence:** HIGH -- standard React component patterns, no novel technology.

## Integration Architecture Summary

```
Homework Builder (Coach):
  @dnd-kit/react (drag blocks from palette to canvas)
    + framer-motion (animations, already installed)
    + XState (builder state machine, already installed)
    + react-hook-form + zod (question configuration forms, already installed)
    -> Saves to Neon Postgres via Drizzle

Exercise Player (Student):
  Custom React components (MCQ, matching, ordering, fill-blank, audio, free-text)
    + @dnd-kit/react (matching + ordering exercises only)
    + framer-motion (feedback animations, already installed)
    + Custom fonts via next/font/local (phonetic annotations)
    -> Grades via n8n webhook or client-side check
    -> Pronunciation via Azure Speech SDK
    -> Results stored in Neon Postgres

Enhanced Chatbot (Student):
  Existing AI SDK useChat hook
    + Restyled ChatWidget (iMessage aesthetic)
    + Custom fonts for Chinese annotation display
    + Lesson-specific prompt injection (existing prompt system)

Pronunciation Scoring (Student):
  Browser MediaRecorder API (capture audio)
    -> POST /api/pronunciation/assess
    -> Azure Speech SDK (server-side assessment)
    -> Post-process with pinyin-pro / to-jyutping (tone matching)
    -> Return scored JSON to client
```
