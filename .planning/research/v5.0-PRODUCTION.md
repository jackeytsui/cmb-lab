## 1. Next.js 16 Build Configuration & Clerk Auth

**Confidence:** HIGH (verified via Next.js docs + Clerk docs)

### The Problem

Pages that call `auth()` or `currentUser()` from `@clerk/nextjs/server` fail during `next build` because Next.js attempts static generation at build time, but auth data requires request-time cookies/headers. This triggers the "app-static-to-dynamic" error.

Currently only 2 cron API routes have `export const dynamic = 'force-dynamic'`. **Zero page routes** have it, meaning any page calling Clerk auth will break the build.

### Solution: Route Segment Config

Add `export const dynamic = 'force-dynamic'` to every page that calls Clerk auth functions.

```typescript
// src/app/(dashboard)/dashboard/page.tsx
import { auth, currentUser } from "@clerk/nextjs/server";

export const dynamic = 'force-dynamic';

export default async function DashboardPage() {
  const { userId } = await auth();
  // ...
}
```

### Pages That Need `force-dynamic`

Every server component page that calls `auth()`, `currentUser()`, or reads `cookies()`/`headers()` needs this. Based on codebase analysis:

**Student pages:**
- `src/app/(dashboard)/dashboard/page.tsx` -- calls `auth()`, `currentUser()`
- `src/app/(dashboard)/courses/[courseId]/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/lessons/[lessonId]/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/my-feedback/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/my-conversations/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/dashboard/practice/page.tsx` -- calls `auth()`

**Coach pages:**
- `src/app/(dashboard)/coach/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/coach/submissions/[submissionId]/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/coach/conversations/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/coach/students/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/coach/pronunciation/page.tsx` -- calls `auth()`

**Admin pages:**
- `src/app/(dashboard)/admin/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/students/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/courses/*/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/knowledge/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/ai-logs/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/prompts/*/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/analytics/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/exercises/*/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/ghl/page.tsx` -- calls `auth()`
- `src/app/(dashboard)/admin/uploads/page.tsx` -- calls `auth()`

**API routes** that already work: API route handlers are always dynamic by default in Next.js App Router, so they do not need this export.

### Alternative: Layout-Level Dynamic

Instead of adding `force-dynamic` to every page, you could add it to a shared layout:

```typescript
// src/app/(dashboard)/layout.tsx (if it existed -- currently no dashboard layout)
export const dynamic = 'force-dynamic';

export default function DashboardLayout({ children }: { children: React.ReactNode }) {
  return children;
}
```

**Recommendation:** Create a `(dashboard)/layout.tsx` that exports `force-dynamic`. This is cleaner than adding it to 20+ individual pages. Any page under `(dashboard)/` will inherit the dynamic behavior. Pages outside this group (like `/verify/[certificateId]`) remain statically optimizable.

### Future: Partial Prerendering (PPR)

Next.js PPR (stable in Next.js 15+) allows pages to have a static shell with dynamic holes. With PPR, you would:

1. Enable PPR in `next.config.ts`: `experimental: { ppr: true }`
2. Wrap auth-dependent content in `<Suspense>` boundaries
3. The static shell (header, layout chrome) gets prerendered; auth content streams in

**For v5.0:** Use `force-dynamic` (simple, proven). PPR is a future optimization, not needed now.

## 2. Database Indexing Strategy for Drizzle ORM

**Confidence:** HIGH (verified via Drizzle ORM official docs)

### The Problem

Foreign key columns across 12+ tables have no indexes. PostgreSQL creates indexes for PRIMARY KEY and UNIQUE constraints automatically, but **does NOT auto-index foreign key columns**. This means every JOIN on `userId`, `courseId`, `lessonId`, or `moduleId` does a sequential scan.

### How to Add Indexes in Drizzle

Use the `index()` function from `drizzle-orm/pg-core` in the table's third parameter (array-based callback):

```typescript
import { pgTable, uuid, text, index } from "drizzle-orm/pg-core";

export const courseAccess = pgTable("course_access", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  courseId: uuid("course_id").notNull().references(() => courses.id, { onDelete: "cascade" }),
  // ... other columns
}, (table) => [
  index("course_access_user_id_idx").on(table.userId),
  index("course_access_course_id_idx").on(table.courseId),
]);
```

### Migration Workflow

1. Update schema files with `index()` definitions
2. Run `npm run db:generate` (drizzle-kit generate)
3. Drizzle Kit detects the new indexes and generates a migration with `CREATE INDEX` statements
4. Run `npm run db:migrate` (drizzle-kit migrate) to apply
5. The migration is additive-only (no data changes, no table locks beyond momentary index build)

## 3. N+1 Query Elimination with Drizzle

**Confidence:** HIGH (verified via Drizzle ORM docs)

### The Problem

Several pages make multiple sequential queries where a single query with joins or relational includes would suffice. Classic N+1 patterns:

1. **Course detail page:** Fetches course, then fetches modules, then fetches lessons per module, then fetches progress per lesson
2. **Coach dashboard:** Fetches submissions list, then fetches user details per submission
3. **Practice results:** Fetches attempt, then fetches exercises individually

### Drizzle Relational Query API (with clause)

Drizzle's relational query API generates a **single SQL statement** regardless of nesting depth. This is the primary N+1 elimination tool.

```typescript
// BEFORE (N+1): 1 query for course + N queries for modules + M queries for lessons
const course = await db.query.courses.findFirst({ where: eq(courses.id, courseId) });
const courseModules = await db.query.modules.findMany({ where: eq(modules.courseId, courseId) });
for (const mod of courseModules) {
  const modLessons = await db.query.lessons.findMany({ where: eq(lessons.moduleId, mod.id) });
}

// AFTER (single query): Drizzle generates one SQL statement
const course = await db.query.courses.findFirst({
  where: eq(courses.id, courseId),
  with: {
    modules: {
      where: isNull(modules.deletedAt),
      orderBy: { sortOrder: 'asc' },
      with: {
        lessons: {
          where: isNull(lessons.deletedAt),
          orderBy: { sortOrder: 'asc' },
        },
      },
    },
  },
});
```

### Pattern: Course Detail with Progress

The dashboard page already uses a good SQL join pattern. The course detail page should follow a similar approach:

```typescript
// Course detail: get course with modules, lessons, and user's progress
const courseData = await db.query.courses.findFirst({
  where: and(eq(courses.id, courseId), isNull(courses.deletedAt)),
  with: {
    modules: {
      where: isNull(modules.deletedAt),
      orderBy: { sortOrder: 'asc' },
      with: {
        lessons: {
          where: isNull(lessons.deletedAt),
          orderBy: { sortOrder: 'asc' },
        },
      },
    },
  },
});

// Then get progress in a single query (not per-lesson)
const progress = await db
  .select()
  .from(lessonProgress)
  .where(and(
    eq(lessonProgress.userId, dbUserId),
    // Get all progress for lessons in this course at once
  ));

// Map progress by lessonId for O(1) lookup
const progressMap = new Map(progress.map(p => [p.lessonId, p]));
```

### Pattern: Coach Submission Queue

```typescript
// BEFORE (N+1): fetch submissions, then fetch user for each
const subs = await db.query.submissions.findMany({ where: eq(submissions.status, 'pending_review') });
for (const sub of subs) {
  const user = await db.query.users.findFirst({ where: eq(users.id, sub.userId) });
}

// AFTER (single query with relations)
const subs = await db.query.submissions.findMany({
  where: eq(submissions.status, 'pending_review'),
  orderBy: { createdAt: 'desc' },
  with: {
    user: {
      columns: { id: true, name: true, email: true },
    },
    lesson: {
      columns: { id: true, title: true },
    },
    interaction: {
      columns: { id: true, prompt: true, type: true },
    },
    feedback: true,
  },
});
```

### Pattern: Practice Results with Exercises

```typescript
// Single query: attempt + practice set + all exercises
const attempt = await db.query.practiceAttempts.findFirst({
  where: eq(practiceAttempts.id, attemptId),
  with: {
    practiceSet: {
      with: {
        exercises: {

      },

```

### When to Use `with` vs Manual Joins

| Use Case | Approach | Why |
|----------|----------|-----|
| Loading related entities (course -> modules -> lessons) | `with` clause | Clean, single query, nested result |
| Aggregation (COUNT, SUM) across joins | Manual `.select().from().leftJoin()` with `sql` | `with` doesn't support aggregation |
| Complex filtering across joined tables | Manual joins with `.where()` | More control over join conditions |
| Simple FK traversal | `with` clause | Less code, same performance |

### Key N+1 Pages to Fix

| Page | Current Pattern | Fix |
|------|----------------|-----|
| Course detail | Sequential module/lesson queries | `with: { modules: { with: { lessons: true } } }` |
| Coach submissions | Loop to fetch user per submission | `with: { user: true, lesson: true }` |
| Coach conversations | Loop to fetch user per conversation | `with: { user: true, lesson: true }` |
| Practice results | Separate exercise fetch | `with: { practiceSet: { with: { exercises: true } } }` |
| Admin student list | Multiple queries for progress stats | Single aggregation query with joins |

## 4. Security Headers

**Confidence:** HIGH (verified via Next.js docs, Clerk docs, Mux docs)

### Current State

The `next.config.ts` already has:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: strict-origin-when-cross-origin`

**Missing:**
- Content-Security-Policy (CSP)
- Permissions-Policy
- Strict-Transport-Security (HSTS)
- X-DNS-Prefetch-Control

### Approach: next.config.ts Headers vs Middleware

| Approach | Pros | Cons |
|----------|------|------|
| `next.config.ts` headers | Simple, declarative, no runtime cost | Cannot generate per-request nonces |
| Middleware | Per-request nonces for CSP, dynamic values | Adds latency to every request |

**Recommendation:** Use `next.config.ts` for all headers EXCEPT CSP. For CSP, use middleware with per-request nonces for maximum security, OR use `next.config.ts` with `'unsafe-inline'` for simplicity.

Given this is a small LMS (not a banking app), the simpler `next.config.ts` approach with `'unsafe-inline'` for styles is acceptable and recommended for v5.0.

### Content-Security-Policy

This CSP must allow: Clerk auth, Mux video, OpenAI Realtime API, Upstash Redis, n8n webhooks, and Neon database connections (server-side only, not relevant to CSP).

```typescript
// next.config.ts
const ContentSecurityPolicy = [
  "default-src 'self'",
  // Scripts: self + Clerk + Mux analytics
  "script-src 'self' 'unsafe-inline' 'unsafe-eval' https://*.clerk.accounts.dev https://challenges.cloudflare.com https://src.litix.io",
  // Styles: unsafe-inline required by Clerk components
  "style-src 'self' 'unsafe-inline'",
  // Connect: Clerk API + Mux streaming + OpenAI Realtime + Litix analytics
  "connect-src 'self' https://*.clerk.accounts.dev https://*.mux.com https://*.litix.io wss://*.openai.com https://api.openai.com https://storage.googleapis.com",
  // Media: Mux video streaming + blob for audio recording
  "media-src 'self' blob: https://*.mux.com",
  // Images: Clerk avatars + Mux thumbnails
  "img-src 'self' data: https://img.clerk.com https://image.mux.com https://*.litix.io",
  // Frames: Clerk Cloudflare challenges
  "frame-src https://challenges.cloudflare.com",
  // Workers: Mux + audio recording + service worker
  "worker-src 'self' blob:",
  // Fonts: self (local font files)
  "font-src 'self' data:",
  // Form action
  "form-action 'self'",
].join("; ");
```

**Important notes:**
- `'unsafe-inline'` for `script-src` is needed because Next.js injects inline scripts for hydration. For stricter CSP, use middleware with nonces.
- `'unsafe-eval'` may be needed for some Next.js dev features -- test in production build without it first.
- `wss://*.openai.com` is for the WebRTC realtime voice conversation feature.
- Replace `*.clerk.accounts.dev` with your production Clerk FAPI domain (e.g., `clerk.cantomando.com`).

### Clerk Automatic CSP (Alternative)

Clerk v6.14.0+ supports automatic CSP injection through middleware:

```typescript
// middleware.ts
import { clerkMiddleware } from "@clerk/nextjs/server";

export default clerkMiddleware({
  contentSecurityPolicy: {
    mode: "standard", // or "strict" for nonce-based
  },
});
```

However, this only handles Clerk's own CSP needs. You still need to manually add Mux and OpenAI domains. The manual approach in `next.config.ts` gives you full control.

### Role Gating

The sidebar needs to know the user's role to show/hide sections. Options:

1. **Server component approach:** Fetch role in the layout, pass as prop
2. **Client component approach:** Use Clerk's `useUser()` to read `publicMetadata.role`

**Recommendation:** Use `useUser()` in the client sidebar component. The role is already stored in Clerk session claims (per the middleware), so no extra DB query is needed.

```typescript
"use client";
import { useUser } from "@clerk/nextjs";

export function AppSidebar() {
  const { user } = useUser();
  const role = (user?.publicMetadata?.role as string) || "student";
  const isCoach = role === "coach" || role === "admin";
  const isAdmin = role === "admin";

  return (
    <Sidebar>
      <SidebarContent>
        <SidebarGroup>
          <SidebarGroupLabel>Learning</SidebarGroupLabel>
          {studentNavItems.map(item => <NavItem key={item.url} item={item} />)}
        </SidebarGroup>

        {isCoach && (
          <SidebarGroup>
            <SidebarGroupLabel>Coach</SidebarGroupLabel>
            {coachNavItems.map(item => <NavItem key={item.url} item={item} />)}
          </SidebarGroup>
        )}

        {isAdmin && (
          <SidebarGroup>
            <SidebarGroupLabel>Admin</SidebarGroupLabel>
            {adminNavItems.map(item => <NavItem key={item.url} item={item} />)}
          </SidebarGroup>
        )}
      </SidebarContent>
    </Sidebar>
  );
}
```

### Pattern: loading.tsx

```typescript
// Template for consistent loading skeletons
import { Skeleton } from "@/components/ui/skeleton";

export default function PageLoading() {
  return (
    <div className="container mx-auto px-4 py-8">
      {/* Page title skeleton */}
      <Skeleton className="h-8 w-48 bg-zinc-800 mb-6" />

      {/* Content skeleton matching the page layout */}
      <div className="space-y-4">
        <Skeleton className="h-12 w-full bg-zinc-800" />
        <Skeleton className="h-12 w-full bg-zinc-800" />
        <Skeleton className="h-12 w-full bg-zinc-800" />
      </div>
    </div>
  );
}
```

### When loading.tsx is Enough vs Suspense Boundaries

| Scenario | Approach |
|----------|----------|
| Entire page loads from one data source | `loading.tsx` is sufficient |
| Page has independent data sections | Use `<Suspense>` per section |
| Heavy component inside a page (e.g., analytics chart) | Wrap in `<Suspense>` with inline skeleton |
| Streaming from server component | `loading.tsx` shows until first chunk |

### Recommendation for v5.0

With the new sidebar layout, `loading.tsx` files will NOT include the header/sidebar chrome (the layout handles that). They only need to show skeleton content for the page area:

```typescript
// BEFORE (current pattern -- loading.tsx includes AppHeader)
export default function DashboardLoading() {
  return (
    <div className="min-h-screen bg-zinc-900 text-white">
      <AppHeader title="Dashboard" />  {/* Redundant once layout exists */}
      <div className="container mx-auto px-4 py-8">
        <Skeleton ... />
      </div>
    </div>
  );
}

// AFTER (with sidebar layout)
export default function DashboardLoading() {
  return (
    <div className="container mx-auto px-4 py-8">
      <Skeleton className="h-9 w-72 bg-zinc-800 mb-8" />
      <div className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-6">
        {[1, 2, 3].map(i => (
          <Skeleton key={i} className="h-48 bg-zinc-800 rounded-lg" />
        ))}
      </div>
    </div>
  );
}
```

**This means all existing 11 loading.tsx files need to be updated** to remove the `AppHeader` and outer `min-h-screen` wrapper once the sidebar layout is in place.

## 9. Roadmap Implications

### Suggested Phase Structure for Production Hardening

**Phase A: App Shell & Build Fix** (do first -- enables all other work)
1. Install shadcn/ui sidebar
2. Create `(dashboard)/layout.tsx` with `SidebarProvider` + `export const dynamic = 'force-dynamic'`
3. Build `AppSidebar` with role-gated navigation
4. Update all pages to remove redundant `AppHeader` wrapper
5. Update all `loading.tsx` files for new layout structure
6. Verify `npm run build` passes

**Phase B: Database Performance**
1. Add all ~24 indexes to schema files
2. Run `drizzle-kit generate` for migration
3. Apply migration
4. Refactor N+1 queries on course detail, coach dashboard, practice results
5. Verify query performance improvement

**Phase C: Security & Environment**
1. Add CSP header to `next.config.ts`
2. Add remaining security headers (HSTS, Permissions-Policy)
3. Complete `.env.example` with all 12 missing variables
4. Fix lint errors (45 pre-existing)

**Phase D: Loading States & Polish**
1. Add missing `loading.tsx` files (~8-10 pages)
2. Create `not-found.tsx` and `error.tsx` at root level
3. Add 404 page

### Why This Order

1. **App shell first** because it creates the `(dashboard)/layout.tsx` which also fixes the build issue (one layout with `force-dynamic` vs 20+ pages). Every subsequent phase benefits from being able to build and test.
2. **Database second** because indexes are additive and safe, and N+1 fixes are isolated per-page.
3. **Security third** because headers don't affect functionality and can be tested independently.
4. **Loading states last** because they depend on the new layout structure from Phase A.
