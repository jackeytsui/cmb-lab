**Domain:** SRS flashcards, smart study engine, tone training, assessment tests, grammar library, auto-exercise generation, AI prompt testing lab
**Researched:** 2026-02-16
**Confidence:** HIGH (all recommendations grounded in existing codebase patterns)

## Recommended Architecture

v10.0 adds seven feature domains to an existing monolith with 29 schema files, 30+ API route groups, and 25+ component directories. The architecture must integrate seamlessly with existing patterns rather than introducing new ones.

### Guiding Principle: Extend, Don't Reinvent

Every new feature domain follows the established pattern:
```
src/db/schema/{domain}.ts          -- Drizzle table + relations + types
src/app/api/{domain}/route.ts      -- REST endpoints (auth + rate limit)
src/components/{domain}/           -- UI components
src/lib/{domain}.ts or {domain}/   -- Business logic
src/hooks/use{Domain}.ts           -- Client-side data hooks
src/types/{domain}.ts              -- Shared TypeScript types (if complex)
```

## Component Boundaries

### 1. SRS Flashcard System

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/db/schema/srs.ts` | SRS card + review log tables | Drizzle ORM, savedVocabulary |
| `src/lib/srs-scheduler.ts` | FSRS scheduling algorithm (pure functions) | Nothing external (pure math) |
| `src/app/api/srs/` | CRUD cards, submit reviews, get due cards | srs-scheduler, DB, auth |
| `src/components/srs/` | Flashcard player, deck manager, review stats | API routes via hooks |
| `src/hooks/useSRS.ts` | Client state for review session | API routes |
| `src/app/(dashboard)/srs/` | Page routes: /srs (review), /srs/decks | Components |

**Key Design Decision: Extend savedVocabulary, don't duplicate**

The existing `saved_vocabulary` table has 6 columns (traditional, simplified, pinyin, jyutping, definitions, notes). Rather than creating a separate `srs_cards` table that duplicates vocabulary data, create an `srs_card_state` table that references `saved_vocabulary.id` for vocabulary cards, while also supporting standalone cards (grammar points, sentences) via a self-contained content JSONB column.

```
saved_vocabulary (existing)
    |
    v
srs_card_state (NEW -- one per user per card)
    |-- id: uuid PK
    |-- userId: uuid FK -> users
    |-- savedVocabularyId: uuid FK -> saved_vocabulary (nullable)
    |-- deckId: uuid FK -> srs_decks (nullable, for organization)
    |-- cardType: enum('vocabulary', 'grammar', 'sentence', 'character')
    |-- front: jsonb  -- { text, pinyin?, audio?, image? }
    |-- back: jsonb   -- { text, definition, examples?, notes? }
    |-- state: enum('new', 'learning', 'review', 'relearning')
    |-- difficulty: real (0.0-1.0, FSRS parameter)
    |-- stability: real (FSRS parameter)
    |-- due: timestamp
    |-- lastReview: timestamp
    |-- reps: integer
    |-- lapses: integer
    |-- scheduledDays: integer
    |-- elapsedDays: integer
    |-- createdAt, updatedAt
```

```
srs_review_log (NEW -- append-only for analytics)
    |-- id: uuid PK
    |-- cardId: uuid FK -> srs_card_state
    |-- userId: uuid FK -> users
    |-- rating: enum('again', 'hard', 'good', 'easy')
    |-- state: enum (card state at time of review)
    |-- scheduledDays: integer
    |-- elapsedDays: integer
    |-- reviewedAt: timestamp
```

```
srs_decks (NEW -- user-created groupings)
    |-- id: uuid PK
    |-- userId: uuid FK -> users
    |-- name: varchar(100)
    |-- description: text
    |-- isDefault: boolean (one default deck per user)
    |-- cardCount: integer (denormalized for display)
    |-- newCardsPerDay: integer (default 20)
    |-- reviewsPerDay: integer (default 200)
    |-- createdAt, updatedAt
```

**Why this structure:**
- `savedVocabularyId` lets vocabulary cards pull fresh data (definitions may update)
- `front`/`back` JSONB handles non-vocabulary card types (grammar, sentences) without schema changes
- `srs_review_log` is append-only like `xp_events` -- enables future FSRS parameter optimization
- `srs_decks` mirrors `vocabulary_lists` concept but with SRS-specific settings

### 2. Smart Study Engine

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/lib/study-engine.ts` | Weak area identification, plan generation | DB queries across all domains |
| `src/app/api/study/` | Get recommendations, daily plan | study-engine, auth |
| `src/components/study/` | Study plan cards, weak area highlights | API routes |
| `src/app/(dashboard)/dashboard/` | Extend existing dashboard with study plan section | study components |

**No new tables needed.** The study engine queries across existing tables:

```typescript
// Study engine reads from:
// - srs_card_state: due cards count, lapse patterns
// - practice_attempts: score distributions per exercise type
// - interaction_attempts: lesson interaction scores
// - daily_activity: recent activity patterns
// - lesson_progress: unfinished lessons
// - saved_vocabulary: unseen words needing flashcards
// - conversations: speaking practice frequency

interface StudyRecommendation {
  type: 'srs_review' | 'weak_grammar' | 'tone_drill' | 'lesson_continue' | 'practice_retry' | 'speaking_practice';
  priority: number; // 1-10
  title: string;
  description: string;
  actionUrl: string;
  estimatedMinutes: number;
  data: Record<string, unknown>; // type-specific payload
}
```

**Design: Compute on-demand, cache in Redis**

The study engine computes recommendations on API call, caches result in Upstash Redis for 30 minutes (keyed by `study:${userId}`), and invalidates on any learning activity. This avoids a new table while keeping response times fast.

### 3. AI Prompt Testing Lab

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/db/schema/prompt-lab.ts` | Test cases, test runs, comparison results | Drizzle ORM, aiPrompts |
| `src/app/api/prompt-lab/` | Run tests, save cases, compare, promote | n8n webhooks, DB, aiPrompts |
| `src/components/prompt-lab/` | Test runner UI, comparison view, batch results | API routes |
| `src/app/(dashboard)/coach/prompt-lab/` | Page route (coach only) | Components |

```
prompt_test_cases (NEW)
    |-- id: uuid PK
    |-- promptId: uuid FK -> ai_prompts
    |-- name: varchar(200)
    |-- input: jsonb  -- { studentResponse, expectedAnswer, language, ... }
    |-- expectedOutput: jsonb  -- { minScore?, shouldBeCorrect?, containsFeedback? }
    |-- createdBy: uuid FK -> users
    |-- createdAt, updatedAt
```

```
prompt_test_runs (NEW)
    |-- id: uuid PK
    |-- promptId: uuid FK -> ai_prompts
    |-- promptVersion: integer  -- which version was tested
    |-- compareVersion: integer (nullable)  -- for A/B comparison
    |-- status: enum('running', 'completed', 'failed')
    |-- results: jsonb  -- array of { testCaseId, output, passed, latencyMs }
    |-- summary: jsonb  -- { totalTests, passed, failed, avgLatency, avgScore }
    |-- createdBy: uuid FK -> users
    |-- startedAt, completedAt
```

**Integration with existing `aiPrompts` system:** The prompt lab reads from `ai_prompts` and `ai_prompt_versions` tables. "Promote to production" writes to `ai_prompts.currentContent` and creates a new `ai_prompt_versions` entry -- exactly the existing update flow.

**AI calls go through n8n:** Test execution sends the prompt + test case input to the existing n8n grading webhooks (text or audio). The response is compared against `expectedOutput` assertions. This maintains the "all AI through n8n" constraint.

### 4. Tone Training Module

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/lib/tone-training.ts` | Tone pair generation, drill logic, scoring | tone-sandhi.ts, pronunciation.ts |
| `src/app/api/tone/` | Get drills, submit answers, get tone stats | tone-training, Azure Speech, DB |
| `src/components/tone/` | Tone identification UI, production recorder, minimal pairs | API routes, TTS |
| `src/app/(dashboard)/tone/` | Page route: /tone (drill modes) | Components |
| `src/types/tone.ts` | Drill definitions, result types | Shared |

```
tone_drill_results (NEW)
    |-- id: uuid PK
    |-- userId: uuid FK -> users
    |-- drillType: enum('identification', 'production', 'minimal_pair', 'sandhi')
    |-- language: enum('cantonese', 'mandarin')
    |-- targetPhrase: varchar(200)
    |-- targetTones: integer[]  -- e.g., [1, 4] for first + fourth tone
    |-- userAnswer: text  -- what user selected/said
    |-- isCorrect: boolean
    |-- score: integer  -- 0-100 for production
    |-- pronunciationData: jsonb  -- Azure pronunciation details (nullable)
    |-- createdAt: timestamp
```

**Leverages existing infrastructure:**
- `src/lib/pronunciation.ts` (Azure Speech) already scores per-character pronunciation with tone accuracy
- `src/lib/tone-sandhi.ts` already implements Mandarin third-tone sandhi rules
- `src/lib/tts.ts` already generates Azure TTS audio for Mandarin and Cantonese
- `pinyin-pro` (already installed) extracts tone numbers from characters
- `to-jyutping` (already installed) provides Cantonese tone numbers

**Drill types built from existing data:**

1. **Identification drills**: Play TTS audio, student identifies tone number. Uses `pinyin-pro` to extract correct tone, Azure TTS for audio.
2. **Production drills**: Show character + target tone, student records audio. Uses Azure pronunciation assessment (already working in practice grading).
3. **Minimal pairs**: Generated from `dictionary_entries` where entries differ by one tone. Query: find pairs like ma1/ma3 from existing dictionary data.
4. **Sandhi drills**: Use `tone-sandhi.ts` to generate before/after sandhi examples. Student predicts sandhi output.

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|

| `src/components/assessment/` | Assessment player, results page | API routes |

```
assessments (NEW)
    |-- id: uuid PK
    |-- title: text
    |-- type: enum('placement', 'hsk_mock', 'custom')
    |-- hskLevel: integer (nullable, 1-6 for HSK mocks)
    |-- language: enum('cantonese', 'mandarin', 'both')
    |-- description: text
    |-- timeLimit: integer (nullable, minutes)
    |-- questionCount: integer
    |-- passingScore: integer (0-100)
    |-- questions: jsonb  -- array of ExerciseDefinition (reuse existing type!)
    |-- isAdaptive: boolean (for placement tests)
    |-- status: enum('draft', 'published', 'archived')
    |-- createdBy: uuid FK -> users
    |-- createdAt, updatedAt
```

```
assessment_attempts (NEW)
    |-- id: uuid PK
    |-- assessmentId: uuid FK -> assessments
    |-- userId: uuid FK -> users
    |-- answers: jsonb  -- per-question answers
    |-- results: jsonb  -- per-question scores
    |-- totalScore: integer (0-100)
    |-- placementLevel: varchar(20) (nullable -- "HSK3", "Beginner", etc.)
    |-- timeSpentSeconds: integer
    |-- startedAt, completedAt
```

**Key Design: Reuse ExerciseDefinition**

The existing `ExerciseDefinition` union type (from `src/types/exercises.ts`) already supports 7 exercise types with Zod validation. Assessment questions use the exact same type. The `assessments.questions` JSONB column stores an array of `ExerciseDefinition` objects. Grading reuses `src/lib/practice-grading.ts` functions (`gradeMultipleChoice`, `gradeFillInBlank`, etc.).

**Adaptive placement algorithm:**
```typescript
// Simple adaptive: start at medium difficulty, binary search
// Each question tagged with difficulty 1-6 (HSK level equivalent)
// Correct answer -> harder question, wrong -> easier
// After 15-20 questions, converge on level estimate
interface AdaptiveState {
  currentDifficulty: number;
  correctByLevel: Record<number, number>;
  totalByLevel: Record<number, number>;
  questionsRemaining: number;
}
```

### 6. Grammar Pattern Library

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/db/schema/grammar.ts` | Grammar patterns, examples, tags | Drizzle ORM |
| `src/app/api/grammar/` | CRUD patterns, search, AI draft generation | DB, n8n webhooks |
| `src/components/grammar/` | Pattern browser, detail view, editor | API routes |
| `src/app/(dashboard)/grammar/` | Page routes: /grammar (browse), /grammar/[slug] | Components |
| `src/app/(dashboard)/coach/grammar/` | Coach editor page | Components |

```
grammar_patterns (NEW)
    |-- id: uuid PK
    |-- slug: text UNIQUE
    |-- title: text  -- e.g., "Using 了 (le) for completed actions"
    |-- structure: text  -- e.g., "Subject + Verb + 了"
    |-- level: integer (1-6, maps to HSK levels)
    |-- language: enum('cantonese', 'mandarin', 'both')
    |-- explanation: text  -- rich text
    |-- cantoneseDiff: text (nullable)  -- how Cantonese differs
    |-- examples: jsonb  -- array of { chinese, pinyin?, jyutping?, english, notes? }
    |-- relatedPatterns: uuid[] (nullable)  -- FK references to other patterns
    |-- tags: text[]  -- searchable tags like "aspect", "comparison", "particles"
    |-- status: enum('draft', 'published', 'archived')
    |-- aiGenerated: boolean  -- flag for AI drafts vs coach-authored
    |-- createdBy: uuid FK -> users
    |-- createdAt, updatedAt
```

**Design: Coach-first, AI-assisted**

Grammar patterns are created by coaches with an optional "Generate AI Draft" button. AI draft generation goes through n8n webhook:
1. Coach enters a grammar point title (e.g., "Using 把 (ba) construction")
2. Frontend sends to `/api/grammar/generate` which calls n8n webhook
3. n8n sends to OpenAI with a grammar-generation prompt (stored in `ai_prompts`)
4. Response is inserted as a `draft` status pattern with `aiGenerated: true`
5. Coach reviews, edits, publishes

**Search uses existing text search pattern:**
```sql
-- Same approach as dictionary search (ILIKE on key columns)
WHERE title ILIKE '%pattern%'
   OR structure ILIKE '%pattern%'
   OR explanation ILIKE '%pattern%'
   OR '%pattern%' = ANY(tags)
```

### 7. Auto-Exercise Generation

| Component | Responsibility | Communicates With |
|-----------|---------------|-------------------|
| `src/lib/exercise-generator.ts` | Template-based exercise generation from content | n8n webhooks, DB |
| `src/app/api/exercises/generate/` | Generate exercises from lesson/reader content | exercise-generator, n8n |
| `src/components/admin/exercise-approval/` | Coach approval queue for generated exercises | API routes |

**No new tables.** Generated exercises are inserted into the existing `practice_exercises` table (within a new practice set) with `status: 'draft'` on the parent `practice_sets` row. Coaches approve by changing status to `published`.

**Generation pipeline:**

```
Input Source                    Exercise Types Generated
-----------                    -------------------------
Lesson content (text)    -->   Cloze (fill_in_blank), Reorder (ordering)
Saved vocabulary list    -->   Matching, Multiple choice
Reader text segment      -->   Fill-in-blank, Free text translation
Grammar pattern          -->   Multiple choice, Ordering (sentence build)
Dictionary entries       -->   Matching (char -> meaning), MCQ (tone select)
```

**Generation flow:**
1. Coach selects source content (lesson, vocab list, reader text, grammar pattern)
2. Frontend sends content to `/api/exercises/generate`
3. API route calls n8n webhook with content + generation prompt (from `ai_prompts`)
4. n8n returns array of `ExerciseDefinition` objects (already validated by Zod schema)
5. API creates a new `practice_set` (status: draft) and inserts generated exercises
6. Coach reviews in approval UI, edits if needed, publishes

## Data Flow

### SRS Review Session Flow

```
1. Student opens /srs
2. GET /api/srs/due -> queries srs_card_state WHERE due <= now()
3. Card rendered (front JSONB or joined savedVocabulary data)
4. Student rates (again/hard/good/easy)
5. POST /api/srs/review { cardId, rating }
   a. srs-scheduler.ts computes new state (pure function)
   b. UPDATE srs_card_state with new due/stability/difficulty
   c. INSERT srs_review_log (append-only)
   d. awardXP({ source: 'practice_exercise', ... })  -- reuse existing XP source
6. Next card loaded (client-side from prefetched batch)
```

### Smart Study Plan Flow

```
1. Student opens dashboard (existing page)
2. GET /api/study/recommendations
   a. Check Redis cache (study:${userId})
   b. If miss: query across all learning domains
      - SRS: count due cards
      - Practice: find low-score exercises
      - Lessons: find incomplete lessons
      - Tone: find weak tone patterns from tone_drill_results
      - Conversations: check speaking frequency
   c. Rank by priority (due reviews > weak areas > new content)
   d. Cache in Redis for 30 min
3. Render StudyPlanCards component in dashboard
4. Each card links to the relevant section (/srs, /tone, /practice, etc.)
```

### Tone Drill Flow

```
1. Student selects drill type (identification/production/minimal_pair/sandhi)
2. GET /api/tone/drill?type=identification&language=mandarin
   a. Generate drill from dictionary_entries + pinyin-pro
   b. For identification: select random character, extract tone, generate TTS
   c. For minimal pairs: find dictionary entries differing by one tone
3. Student answers
4. POST /api/tone/submit { drillType, targetPhrase, userAnswer, audioBlob? }
   a. For identification: instant client-side grading (tone number match)
   b. For production: send audio to Azure Speech (existing pronunciation.ts)
   c. INSERT tone_drill_results
   d. awardXP (reuse practice_exercise source)
```

### Assessment Flow

```
1. Student selects assessment (placement quiz or HSK mock)
2. POST /api/assessment/start { assessmentId }
   a. Create assessment_attempt row
   b. For adaptive: return first question at mid-level difficulty
   c. For HSK mock: return all questions (timed)
3. Student answers each question
4. POST /api/assessment/answer { attemptId, questionIndex, answer }
   a. Grade using existing practice-grading.ts functions
   b. For adaptive: compute next difficulty, return next question
5. POST /api/assessment/complete { attemptId }
   a. Calculate final score and placement level
   b. UPDATE assessment_attempts
```

### Auto-Exercise Generation Flow

```
1. Coach selects source content in admin panel
2. POST /api/exercises/generate { sourceType, sourceId, exerciseTypes[] }
3. API constructs prompt with source content + exercise generation template
4. Sends to n8n webhook (N8N_EXERCISE_GEN_WEBHOOK_URL)
5. n8n calls OpenAI, returns ExerciseDefinition[] (validated by Zod)
6. API creates practice_set (draft) + practice_exercises rows
7. Coach opens approval queue, reviews/edits/publishes
```

### Pattern 3: Redis-Cached Aggregations

**What:** Compute-heavy cross-domain queries cached in Upstash Redis
**When:** Study recommendations, tone accuracy stats, assessment readiness

```typescript
// Existing pattern from the codebase (rate limiting uses Upstash Redis)
import { Redis } from "@upstash/redis";

const CACHE_TTL = 1800; // 30 minutes

export async function getStudyRecommendations(userId: string) {
  const cacheKey = `study:${userId}`;
  const cached = await redis.get<StudyRecommendation[]>(cacheKey);
  if (cached) return cached;

  const recommendations = await computeRecommendations(userId);
  await redis.set(cacheKey, recommendations, { ex: CACHE_TTL });
  return recommendations;
}

// Invalidate on any learning activity
export async function invalidateStudyCache(userId: string) {
  await redis.del(`study:${userId}`);
}
```

## Anti-Patterns to Avoid

### Anti-Pattern 1: Separate SRS Content Table

**What:** Creating a `srs_cards` table that stores Chinese text, pinyin, definitions alongside scheduling state.
**Why bad:** Duplicates `saved_vocabulary` data. When a dictionary definition updates, SRS cards have stale data. Two sources of truth for vocabulary.
**Instead:** Reference `saved_vocabulary.id` for vocabulary cards. Store content in JSONB only for non-vocabulary card types (grammar, sentences).

### Anti-Pattern 2: Real-Time Study Plan Computation

**What:** Computing study recommendations on every dashboard render without caching.
**Why bad:** The study engine queries 5+ tables across different domains. On a dashboard that loads frequently, this creates unnecessary DB pressure.
**Instead:** Cache in Redis with 30-minute TTL. Invalidate on learning activity. The recommendation is not so time-sensitive that it needs live computation.

### Anti-Pattern 3: Direct OpenAI Calls for Exercise Generation

**What:** Calling OpenAI directly from API routes for grammar generation or exercise generation.
**Why bad:** Breaks the "all AI through n8n" constraint. Removes workflow flexibility (can't add logging, fallback models, rate limiting at the n8n level).
**Instead:** All AI calls go through n8n webhooks. Create new webhook URLs for grammar generation (`N8N_GRAMMAR_GEN_WEBHOOK_URL`) and exercise generation (`N8N_EXERCISE_GEN_WEBHOOK_URL`).

**Exception:** The existing `/api/grade` route calls OpenAI directly as a fallback. This is a legacy pattern -- new features should use n8n exclusively.

### Anti-Pattern 4: Monolithic Assessment Table

**What:** Storing all assessment questions as separate rows in a relational table with complex joins.
**Why bad:** Assessment questions have complex, type-varied structure (MCQ vs fill-blank vs ordering). Normalizing this into rows creates schema complexity.
**Instead:** Store questions as a JSONB array of `ExerciseDefinition` objects in `assessments.questions`. Assessments are read-heavy, write-rarely -- JSONB is perfect for this access pattern.

### Anti-Pattern 5: Overly Complex Adaptive Algorithm

**What:** Implementing IRT (Item Response Theory) with theta estimation for placement tests.
**Why bad:** Requires large question pools with calibrated difficulty parameters. CantoMando has a small content team. The complexity is disproportionate to the user base size.
**Instead:** Simple binary-search adaptive: start at mid-difficulty, adjust based on correct/incorrect. 15-20 questions converge on a level estimate. Good enough for placement; HSK mocks are fixed question sets anyway.

## Migration Strategy

All new tables are additive (no ALTER TABLE on existing tables). Migration sequence:

1. Schema files created (srs.ts, tone.ts, assessment.ts, grammar.ts, prompt-lab.ts)
2. `drizzle-kit generate` creates migration SQL
3. `drizzle-kit migrate` applies to Neon Postgres
4. No existing data is modified -- all new tables start empty
5. Existing `saved_vocabulary` rows can be bulk-converted to SRS cards via a one-time script

**Zero-downtime:** Since all changes are additive (new tables, new routes, new components), they can be deployed incrementally without affecting existing functionality.

## Scalability Considerations

| Concern | At 100 users | At 1K users | At 10K users |
|---------|-------------|-------------|--------------|
| SRS due card queries | Index on (userId, due) sufficient | Same | Add partial index WHERE state != 'new' |
| Review log size | ~50K rows total | ~500K rows | Partition by month or archive old logs |
| Study engine computation | Inline queries OK | Redis cache essential | Pre-compute nightly via cron |
| Tone drill generation | On-demand fine | Same | Cache drill pools in Redis |
| Grammar search | ILIKE fine | Same | Add pg_trgm GIN index on title + tags |
| Assessment attempts | Trivial volume | Same | Same |

Current user base is small (~100 active students per MEMORY.md CSV export). All recommendations are designed for this scale with clear upgrade paths.
