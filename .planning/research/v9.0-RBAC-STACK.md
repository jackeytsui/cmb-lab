# Technology Stack: RBAC Additions for CantoMando LMS

**Project:** CantoMando Blueprint - Discord-Style RBAC (v9.0)
**Researched:** 2026-02-14
**Overall confidence:** HIGH (all recommendations use existing stack or zero-dependency patterns)

---

## Core Thesis: Zero New Dependencies

The RBAC system requires **no new npm packages**. Every capability needed -- schema definition, permission resolution, form validation, UI components, webhook handling -- is already covered by the existing stack. The research below explains what to use from the existing stack, what to add as custom code (with rationale for why a library is unnecessary), and the two shadcn/ui components worth installing via CLI.

## Recommended Stack Additions

### Database Layer (No new packages)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Drizzle ORM (existing) | 0.45+ | New RBAC tables: `roles`, `role_courses`, `role_features`, `user_roles` | Already used for all 27 schema files. Drizzle supports `pgTable`, `pgEnum`, `jsonb`, `uuid`, `timestamp` -- everything needed. No reason to add another ORM or query builder. |
| `pg-core` types (existing) | via drizzle-orm | `text`, `boolean`, `timestamp`, `uuid`, `jsonb` columns | The new tables use only column types already in use across the codebase. `jsonb` is used in 7 existing schema files. `pgEnum` is used for `roleEnum`, `accessTierEnum`, `grantedByEnum`. |
| Neon Serverless (existing) | 1.0+ | Database connection | No change. Same connection pool, same `db` import from `@/db`. |

**Schema approach:** Use `text` columns (not `pgEnum`) for the new role-related fields. Rationale from PITFALLS research: Postgres enum `ALTER TYPE ADD VALUE` cannot be rolled back within a transaction. The existing `roleEnum` on the `users` table stays as-is (it controls coach/admin dashboard routing, an orthogonal concern). New role names in the `roles` table use `text` for flexibility.

**Feature keys approach:** Use a TypeScript `const` array as the source of truth for valid feature keys, stored as `text` in the database. Zod validates at the API boundary. This avoids the enum rigidity problem while maintaining type safety.

```typescript
// src/lib/rbac/features.ts -- source of truth
export const FEATURE_KEYS = [
  "ai_conversation",
  "practice_sets",
  "dictionary_reader",
  "listening_lab",
  "video_threads",
  "certificates",
  "ai_chat",
] as const;

export type FeatureKey = typeof FEATURE_KEYS[number];
```

### Permission Resolver (No new packages)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Pure TypeScript (custom) | -- | `resolvePermissions()` function | A custom 50-100 line function. No RBAC library needed because: (1) the permission model is additive-only (no deny, no inheritance hierarchy), (2) feature keys are a fixed enum, (3) course access is a simple set union. CASL, Permit.io, and similar libraries solve complex ABAC/PBAC problems this system does not have. |
| Drizzle queries (existing) | 0.45+ | Fetch user roles, role courses, role features | Two-three SELECT queries with JOINs. The resolver gathers all active (non-expired) roles for a user, unions their courses and features, merges with legacy `course_access` rows. |

**Why NOT use CASL:** CASL (`@casl/ability`) is designed for attribute-based access control with subject/action/conditions patterns (`can('read', 'Post', { author: userId })`). The CantoMando RBAC is simpler: "does this student's role set include course X or feature Y?" This is a Set.has() check, not a CASL ability check. Adding CASL would introduce 15KB of bundle, a learning curve for the `AbilityBuilder` API, and React Context boilerplate (`@casl/react`) -- all for a problem solvable with a single function.

**Why NOT use Permit.io / OpenFGA / Oso:** These are authorization-as-a-service platforms designed for multi-tenant B2B SaaS with thousands of resources and complex policy engines. CantoMando has ~10 courses, ~7 features, and <1000 students. An external authorization service adds latency (network call per permission check), cost, and operational complexity for a problem that fits in 100 lines of application code.

**Resolver shape:**

```typescript
// src/lib/rbac/resolver.ts
export interface ResolvedPermissions {
  courses: Map<string, { roleIds: string[]; accessTier: "preview" | "full" }>;
  features: Map<FeatureKey, string[]>;  // feature -> roleIds that grant it
  hasAllCourses: boolean;               // any role has allCourses flag
  roleNames: string[];                  // for display
}

export async function resolvePermissions(userId: string): Promise<ResolvedPermissions> {
  // 1. Fetch user_roles WHERE (expiresAt IS NULL OR expiresAt > NOW())
  // 2. Fetch role definitions with role_courses and role_features
  // 3. Union courses (highest access tier wins per course)
  // 4. Union features
  // 5. Merge legacy course_access rows
  // 6. Return typed result
}
```

**Zod schemas needed:**

```typescript
// src/lib/rbac/schemas.ts
import { z } from "zod";
import { FEATURE_KEYS } from "./features";

export const createRoleSchema = z.object({
  name: z.string().min(1).max(50),
  description: z.string().max(500).optional(),
  color: z.string().regex(/^#[0-9a-fA-F]{6}$/).optional(),
  allCourses: z.boolean().default(false),
  courseIds: z.array(z.string().uuid()).default([]),
  featureKeys: z.array(z.enum(FEATURE_KEYS)).default([]),
});

export const assignRoleSchema = z.object({
  userId: z.string().uuid(),
  roleId: z.string().uuid(),
  expiresAt: z.string().datetime().optional().nullable(),
});

export const webhookRoleSchema = z.object({
  email: z.string().email(),
  roleId: z.string().uuid().optional(),
  roleName: z.string().optional(),
  courseId: z.string().uuid().optional(),  // backward compat
  accessTier: z.enum(["preview", "full"]).default("full"),
  expiresAt: z.string().datetime().optional(),
}).refine(
  (data) => data.roleId || data.roleName || data.courseId,
  { message: "At least one of roleId, roleName, or courseId is required" }
);
```

### Authentication / Session (No new packages)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Clerk (existing) | @clerk/nextjs 6.36+ | Session claims for `users.role` (coach/admin routing), user metadata | No change to Clerk configuration. The RBAC system stores roles in the DATABASE, not in Clerk metadata. Clerk metadata continues to hold only `{ role: "student" | "coach" | "admin" }` for the existing dashboard routing (middleware). RBAC permissions are resolved server-side via DB queries in the resolver, not via session claims. This avoids the 1.2KB cookie size limit pitfall (see PITFALLS research). |

**Critical decision: Clerk metadata stays minimal.**

The existing `sessionClaims.metadata.role` continues to carry the user's `users.role` value (student/coach/admin) for middleware routing. The new RBAC role assignments (`user_roles` table) are NOT synced to Clerk metadata. Rationale:

1. A user can have multiple RBAC roles -- Clerk metadata is a single value, not a list
2. Clerk session tokens refresh every 60 seconds -- stale RBAC data would cause access flicker
3. Cookie size limit (1.2KB custom data) would be exceeded with multiple role IDs + feature sets
4. The resolver already hits the DB -- no performance gain from duplicating in claims

The `hasMinimumRole()` function in `src/lib/auth.ts` continues to work unchanged for coach/admin gating. RBAC permission checks use the new `resolvePermissions()` function which queries the database directly.

### Caching (No new packages)

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Per-request memoization (custom) | -- | Cache `resolvePermissions()` result within a single request | A page may check permissions multiple times (course access in header, feature access in sidebar, course list in main content). The resolver should run its DB queries once per request, not once per check. |

**Implementation: React `cache()` from `react` (already available in React 19).**

```typescript
import { cache } from "react";

// Deduplicated within a single RSC render pass
export const getPermissions = cache(async (userId: string) => {
  return resolvePermissions(userId);
});
```

This is the standard Next.js 16 pattern for request-scoped deduplication. No Redis, no external cache. The resolver runs 2-3 SQL queries per request -- with Neon serverless connection pooling, this adds ~5-15ms. Acceptable for <1000 concurrent users.

**When to add Redis caching:** If permission resolution latency becomes a bottleneck (>50ms measured in production), add `@upstash/redis` (already installed, version 1.36+) with a 30-second TTL cache, invalidated on role changes. But do NOT add this preemptively -- the per-request `cache()` is sufficient until proven otherwise.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Permission library | Custom resolver (~100 LOC) | CASL (`@casl/ability`) | Overkill. CASL's subject/action/conditions model is designed for ABAC. This system has additive-only set unions. Adding CASL introduces 15KB bundle, `AbilityBuilder` boilerplate, and React Context wiring for a problem solved by `Set.has()`. |
| Permission library | Custom resolver | Permit.io SDK | External service dependency. Adds network latency per permission check, requires API key management, costs money. Justified for multi-tenant SaaS with complex policies, not for a single-tenant LMS with 7 features. |
| Permission library | Custom resolver | Oso / OpenFGA | Same as Permit.io -- external policy engine for enterprise RBAC. The CantoMando permission model (union of role grants) is too simple to justify a policy engine. |
| Permission storage | Separate tables (`role_courses`, `role_features`) | Bitwise flags (Discord-style integer) | Bitwise flags are efficient but opaque. Discord uses them because they serve millions of concurrent users. This LMS has <1000 users. Relational join tables are readable, queryable, and debuggable with standard SQL. The performance difference is negligible at this scale. |
| Permission storage | Relational tables | JSONB column on `roles` table | JSONB works but loses foreign key integrity on course IDs. A `role_courses` join table with a FK to `courses.id` ensures courses can't be deleted while roles reference them (or cascade properly). JSONB also makes "which roles grant access to course X?" a harder query (requires `jsonb_array_elements` vs simple `WHERE courseId = X`). |
| Session integration | DB-only (no Clerk metadata for RBAC) | Sync RBAC roles to Clerk metadata | Cookie size limit (1.2KB), 60-second stale window, multi-role doesn't fit single metadata value. See PITFALLS research for full analysis. |
| Caching | React `cache()` per-request | Redis via @upstash/redis | Premature optimization. Per-request deduplication is sufficient. Redis adds complexity (cache invalidation on role changes, TTL management, another failure mode). Add Redis only if measured latency exceeds 50ms. |
| UI calendar | Native `<input type="date">` | react-day-picker / shadcn Calendar | The existing codebase uses native date inputs for `expiresAt` and `dueDate` fields. Adding a calendar component library is unnecessary unless the product explicitly requires a more polished date picker. Defer until feedback requests it. |
| Admin table | @tanstack/react-table (existing) | Build custom table | Already used in `StudentDataTable` and `ActiveStudentDataTable`. Role listing and user-role assignment tables follow the exact same pattern. |

## Integration Points with Existing Stack

### 1. Drizzle Schema Integration

The new `src/db/schema/rbac.ts` exports tables that reference existing `users` and `courses` tables via foreign keys. It gets added to `src/db/schema/index.ts` as `export * from "./rbac"`.

```typescript
// New tables reference existing tables
import { users } from "./users";
import { courses } from "./courses";

export const roles = pgTable("roles", { ... });
export const roleCourses = pgTable("role_courses", {
  roleId: uuid("role_id").references(() => roles.id, { onDelete: "cascade" }),
  courseId: uuid("course_id").references(() => courses.id, { onDelete: "cascade" }),
  // ...
});
export const userRoles = pgTable("user_roles", {
  userId: uuid("user_id").references(() => users.id, { onDelete: "cascade" }),
  roleId: uuid("role_id").references(() => roles.id, { onDelete: "cascade" }),
  // ...
});
```

### 2. Webhook Integration

The existing `/api/webhooks/enroll/route.ts` is EXTENDED (not replaced). The handler accepts the existing payload plus optional `roleId`/`roleName`. If a role identifier is present, it assigns the role via `user_roles` insert. If only `courseId` is present, it falls back to the existing `course_access` insert (backward compatible).

### 3. Access Check Migration

~50 files currently query `courseAccess` directly. These migrate to call `canAccessCourse(userId, courseId)` from `src/lib/rbac/helpers.ts`. The helper checks both RBAC roles and legacy `course_access` rows (union model). Migration is file-by-file, not big-bang.

### 4. Feature Gating

~8 feature entry points add a `canAccessFeature(userId, featureKey)` check. This is a new check (features are currently ungated). The check returns `{ allowed: boolean; upgradeRole?: string }` so the UI can show "Unlock with Premium" instead of a blank redirect.

## Performance Budget

| Operation | Expected Latency | Method |
|-----------|-----------------|--------|
| `resolvePermissions()` cold (first call in request) | 5-15ms | 2-3 SQL queries via Neon serverless |
| `resolvePermissions()` warm (subsequent calls in same request) | <1ms | React `cache()` returns memoized result |
| `canAccessCourse()` | <1ms after resolver | `Set.has(courseId)` on resolved permissions |
| `canAccessFeature()` | <1ms after resolver | `Map.has(featureKey)` on resolved permissions |
| Role CRUD (admin action) | 10-30ms | Standard Drizzle insert/update |
| Webhook role assignment | 20-50ms | User lookup + role assignment + optional course_access sync |

At <1000 concurrent users with Neon serverless, these latencies are well within acceptable bounds. No Redis caching needed at launch.
