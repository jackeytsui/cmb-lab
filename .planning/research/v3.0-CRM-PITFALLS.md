# Domain Pitfalls: GoHighLevel CRM Integration

**Domain:** CRM-LMS integration -- GoHighLevel contact sync, tag sync, webhook automations, bidirectional data flow
**Researched:** 2026-01-31
**Confidence:** MEDIUM-HIGH (GHL API docs verified, bidirectional sync patterns verified with multiple sources, codebase patterns reviewed)

## Critical Pitfalls

Mistakes that cause data corruption, infinite loops, or production outages.

### Pitfall 1: Bidirectional Tag Sync Creates Infinite Webhook Loop

**What goes wrong:**
Coach adds tag "Module-1-Complete" to a student in the LMS. LMS syncs this tag to GoHighLevel via API. GHL fires a `ContactTagUpdate` webhook back to the LMS. LMS sees a "new" tag and tries to sync it back to GHL. Infinite loop. Each cycle fires more webhooks, burning through the 100-requests-per-10-seconds rate limit in seconds, then the 200,000 daily limit within minutes.

**Why it happens:**
GHL fires `ContactTagUpdate` webhooks for ALL tag changes, including those made via API. There is no built-in "change source" field in GHL webhook payloads (unlike HubSpot's `changeSource` field). GHL's same-app credential suppression only works for OAuth marketplace apps, not for webhook event subscriptions.

**Consequences:**
- Infinite webhook loop consuming all API quota
- GHL rate limiting (429 errors) blocking ALL API calls, not just tag calls
- Duplicate tag entries if the loop creates slightly different tag names
- Potential GHL account suspension for API abuse
- LMS webhook endpoint overwhelmed, affecting Clerk and enrollment webhooks too (they share the same Vercel deployment)

**Prevention:**
```typescript
// 1. Echo detection: Mark outbound changes and ignore their echoes
// Store a short-lived record of changes YOUR system initiated

// lib/ghl/sync-tracker.ts
import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv();
const ECHO_TTL = 60; // seconds - GHL webhooks typically arrive within 5-30s

export async function markOutboundChange(
  contactId: string,
  changeType: string,
  changeValue: string
): Promise<void> {
  const key = `ghl:outbound:${contactId}:${changeType}:${changeValue}`;
  await redis.set(key, "1", { ex: ECHO_TTL });
}

export async function isEchoWebhook(
  contactId: string,
  changeType: string,
  changeValue: string
): Promise<boolean> {
  const key = `ghl:outbound:${contactId}:${changeType}:${changeValue}`;
  const exists = await redis.get(key);
  if (exists) {
    await redis.del(key); // Consume the echo marker
    return true;
  }
  return false;
}

// 2. Use in outbound sync (LMS -> GHL)
async function syncTagToGHL(contactId: string, tag: string) {
  await markOutboundChange(contactId, "tag_add", tag);
  await ghlClient.addTag(contactId, tag);
}

// 3. Use in inbound webhook handler (GHL -> LMS)
async function handleGHLTagWebhook(payload: GHLWebhookPayload) {
  const { contactId, tag, action } = parseTagPayload(payload);

  if (await isEchoWebhook(contactId, `tag_${action}`, tag)) {
    console.log(`Ignoring echo: ${action} tag "${tag}" for ${contactId}`);
    return; // This was our own change, skip it
  }

  // Genuine GHL-originated change, process it
  await applyTagChangeToLMS(contactId, tag, action);
}
```

**Detection (warning signs):**
- Rapid spike in GHL API calls (monitor `X-RateLimit-Remaining` header)
- Same tag appearing and disappearing in logs repeatedly
- 429 responses from GHL within seconds of deploying tag sync
- Redis memory spike from echo markers accumulating (means echoes are not being matched)
- Webhook endpoint response times degrading

**Which phase should address:** Sync Infrastructure Phase (before any bidirectional sync is enabled). This must be the FIRST thing built, not bolted on after tag sync works.

### Pitfall 2: GHL Webhook Delivery Failures Silently Lose Events

**What goes wrong:**
Your LMS endpoint returns a 500 error (database connection timeout, unhandled exception, deployment in progress). GHL does NOT retry on 5xx errors. The webhook event is permanently lost. A student completes a milestone in GHL (e.g., paid for next tier), but the LMS never receives the notification. Student's access is never upgraded.

**Why it happens:**
GoHighLevel's retry policy is unusually restrictive: they ONLY retry on HTTP 429 (rate limit), with 6 attempts over ~70 minutes. All other error codes (including 5xx server errors) are treated as final, permanent failures. This is unlike Clerk (which retries), Stripe (which retries with exponential backoff), or most webhook providers.

This interacts badly with:
- Vercel cold starts (first request after idle may timeout)
- Database connection pool exhaustion under load
- Deployment windows (brief 502/503 during deploy)

**Consequences:**
- Lost enrollment upgrades (student paid but access not granted)
- Lost tag changes (coach updated CRM but LMS shows stale data)
- No error notification -- GHL marks delivery as failed but does not alert you
- Difficult to diagnose because the failure is silent

**Prevention:**
```typescript
// 1. ALWAYS return 200 immediately, process async
// app/api/webhooks/ghl/route.ts

export async function POST(req: NextRequest) {
  const rawBody = await req.text();

  // Verify signature FIRST (fast, no DB)
  if (!verifyGHLSignature(rawBody, req.headers)) {
    return NextResponse.json({ error: "Invalid signature" }, { status: 401 });
  }

  const payload = JSON.parse(rawBody);

  // Store raw event in database IMMEDIATELY (fast INSERT, no processing)
  await db.insert(ghlWebhookEvents).values({
    eventType: payload.type,
    payload: payload,
    status: "pending",
    receivedAt: new Date(),
  });

  // Return 200 BEFORE any business logic
  return NextResponse.json({ received: true }, { status: 200 });
}

// 2. Process events via separate mechanism (cron or queue)
// lib/ghl/event-processor.ts
export async function processGHLEvents() {
  const pendingEvents = await db
    .select()
    .from(ghlWebhookEvents)
    .where(eq(ghlWebhookEvents.status, "pending"))
    .orderBy(asc(ghlWebhookEvents.receivedAt))
    .limit(50);

  for (const event of pendingEvents) {
    try {
      await processEvent(event);
      await db.update(ghlWebhookEvents)
        .set({ status: "processed", processedAt: new Date() })
        .where(eq(ghlWebhookEvents.id, event.id));
    } catch (error) {
      await db.update(ghlWebhookEvents)
        .set({
          status: "failed",
          errorMessage: error.message,
          retryCount: event.retryCount + 1
        })
        .where(eq(ghlWebhookEvents.id, event.id));
    }
  }
}

// 3. Implement periodic reconciliation as safety net
// Run daily: fetch all contacts from GHL API, compare with LMS state
export async function reconcileGHLContacts() {
  const ghlContacts = await fetchAllGHLContacts();
  const lmsUsers = await db.select().from(users);

  for (const contact of ghlContacts) {
    const lmsUser = lmsUsers.find(u => u.email === contact.email);
    if (!lmsUser) continue;

    // Compare tags, access levels, etc.
    const drift = detectDrift(contact, lmsUser);
    if (drift.length > 0) {
      console.warn(`Drift detected for ${contact.email}:`, drift);
      // Auto-fix or flag for manual review
    }
  }
}
```

- Students report "I paid but still can't access Module 2"
- Tag counts don't match between GHL and LMS
- Webhook endpoint has 5xx errors in Vercel logs around the same time as GHL failures

**Which phase should address:** Webhook Infrastructure Phase. The "receive-and-queue" pattern must be built BEFORE connecting any GHL webhook events. The reconciliation job should be added in a later hardening phase.

### Pitfall 3: OAuth Token Refresh Race Condition Kills All GHL API Access

**What goes wrong:**
Two server-side operations happen simultaneously: a tag sync and a contact update. Both detect the access token is expired. Both call the GHL `/token` refresh endpoint at roughly the same time. The first call succeeds and gets a new refresh token. The second call uses the OLD refresh token (which is now invalidated). The second call gets `invalid_grant`. Now your stored refresh token is from the first call, but the second call's error handler might overwrite it with garbage. Result: all GHL API access is broken until manual re-authorization.

**Why it happens:**
GHL uses **single-use refresh tokens** (refresh token rotation). Each refresh token can only be used once. GHL added a 30-second grace window for concurrent calls with the same credentials, but:
- You must wait at least 8 seconds between refresh calls or get `invalid_grant`
- If your retry logic hits the endpoint a third time with the original token (outside the 30s window), it fails permanently
- The 30-second window only helps if requests use the EXACT same `refresh_token` -- if one request already rotated it, subsequent requests using the old token fail

**Consequences:**
- Complete loss of GHL API access
- All sync operations fail silently (or noisily, flooding error logs)
- Requires manual re-authorization via OAuth flow
- Coach-facing features show stale data with no way to update
- Potential data drift accumulating during the outage

**Prevention:**
```typescript
// lib/ghl/token-manager.ts
// SINGLETON token refresh with mutex lock

import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv();
const LOCK_KEY = "ghl:token:refresh:lock";
const TOKEN_KEY = "ghl:token:current";
const LOCK_TTL = 15; // seconds

interface GHLTokens {
  accessToken: string;
  refreshToken: string;
  expiresAt: number; // Unix timestamp
}

export async function getValidAccessToken(): Promise<string> {
  const tokens = await getStoredTokens();

  // Proactive refresh: refresh if expiring within 5 minutes
  if (tokens.expiresAt > Date.now() + 5 * 60 * 1000) {
    return tokens.accessToken;
  }

  return await refreshWithLock(tokens.refreshToken);
}

async function refreshWithLock(currentRefreshToken: string): Promise<string> {
  // Try to acquire lock (prevents concurrent refresh)
  const acquired = await redis.set(LOCK_KEY, "1", { nx: true, ex: LOCK_TTL });

  if (!acquired) {
    // Another process is refreshing -- wait and read the new token
    await new Promise(resolve => setTimeout(resolve, 3000));
    const tokens = await getStoredTokens();
    return tokens.accessToken;
  }

  try {
    const response = await fetch("https://services.leadconnectorhq.com/oauth/token", {
      method: "POST",
      headers: { "Content-Type": "application/x-www-form-urlencoded" },
      body: new URLSearchParams({
        client_id: process.env.GHL_CLIENT_ID!,
        client_secret: process.env.GHL_CLIENT_SECRET!,
        grant_type: "refresh_token",
        refresh_token: currentRefreshToken,
      }),
    });

    if (!response.ok) {
      throw new Error(`Token refresh failed: ${response.status}`);
    }

    const data = await response.json();

    // IMMEDIATELY persist new tokens
    const newTokens: GHLTokens = {
      accessToken: data.access_token,
      refreshToken: data.refresh_token,
      expiresAt: Date.now() + (data.expires_in * 1000),
    };

    await storeTokens(newTokens);
    return newTokens.accessToken;
  } finally {
    await redis.del(LOCK_KEY); // Always release lock
  }
}

// Store in both Redis (fast reads) and database (durability)
async function storeTokens(tokens: GHLTokens): Promise<void> {
  await Promise.all([
    redis.set(TOKEN_KEY, JSON.stringify(tokens)),
    db.update(ghlIntegrationSettings)
      .set({
        accessToken: encrypt(tokens.accessToken),
        refreshToken: encrypt(tokens.refreshToken),
        tokenExpiresAt: new Date(tokens.expiresAt),
      })
      .where(eq(ghlIntegrationSettings.id, "default")),
  ]);
}
```

**Detection (warning signs):**
- `invalid_grant` errors in logs (CRITICAL -- means refresh token was consumed)
- Multiple simultaneous 401 responses from GHL
- Token refresh happens more than once within 30 seconds
- GHL features suddenly stop working for all users simultaneously

**Which phase should address:** Sync Infrastructure Phase. Token management must be rock-solid before ANY GHL API calls are made. Consider using GHL's official Node.js SDK which handles some of this automatically.

**Sources:**
- [GoHighLevel OAuth 2.0 Docs](https://marketplace.gohighlevel.com/docs/Authorization/OAuth2.0/index.html) - HIGH confidence

### Pitfall 4: Rate Limit Exhaustion Cascades Across All GHL Features

**What goes wrong:**
A bulk operation (e.g., syncing tags for 50 students after course completion) burns through the 100-requests-per-10-seconds burst limit. Now ALL GHL API calls fail with 429 for the remainder of the 10-second window -- including real-time tag lookups for the coach dashboard, contact creation for new enrollments, and webhook delivery acknowledgments. One batch job takes down all CRM features.

**Why it happens:**
GHL rate limits are per-app per-location (sub-account), not per-endpoint. The burst limit of 100 requests per 10 seconds and daily limit of 200,000 requests are SHARED across all API operations. A naive `Promise.all()` on 50 tag updates fires 50 requests simultaneously, immediately hitting the burst limit.

**Consequences:**
- Coach sees "CRM unavailable" errors while viewing student profiles
- New enrollment webhook processing delayed (GHL returns 429)
- Daily limit (200K) can be exhausted by runaway batch operations
- GHL retries webhook deliveries on 429 with 10-minute delays, causing stale data

**Prevention:**
```typescript
// lib/ghl/api-client.ts
// Rate-limited API client that respects GHL's limits

import PQueue from "p-queue";

// Shared queue for ALL GHL API calls
const ghlQueue = new PQueue({
  concurrency: 5,          // Max 5 concurrent requests
  interval: 10_000,        // Per 10-second window
  intervalCap: 80,         // Leave 20% headroom for webhooks/urgent calls
});

// Priority queue for urgent vs batch operations
export async function ghlApiCall(
  endpoint: string,
  options: RequestInit,
  priority: "urgent" | "batch" = "batch"
): Promise<Response> {
  return ghlQueue.add(async () => {
    const token = await getValidAccessToken();

    const response = await fetch(
      `https://services.leadconnectorhq.com${endpoint}`,
      {
        ...options,
        headers: {
          ...options.headers,
          Authorization: `Bearer ${token}`,
          "Content-Type": "application/json",
          Version: "2021-07-28",
        },
      }
    );

    // Track rate limit headers
    const remaining = parseInt(response.headers.get("X-RateLimit-Remaining") || "100");
    const dailyRemaining = parseInt(
      response.headers.get("X-RateLimit-Daily-Remaining") || "200000"
    );

    // Alert if approaching limits
    if (remaining < 20) {
      console.warn(`GHL burst rate limit low: ${remaining} remaining`);
    }
    if (dailyRemaining < 10000) {
      console.error(`GHL daily rate limit critically low: ${dailyRemaining}`);
      // Consider pausing non-essential sync operations
    }

    // Handle 429 with backoff
    if (response.status === 429) {
      const retryAfter = parseInt(response.headers.get("Retry-After") || "10");
      await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
      // Re-queue the request
      return ghlApiCall(endpoint, options, priority);
    }

    return response;
  }, { priority: priority === "urgent" ? 1 : 0 });
}

// For batch operations, use explicit throttling
export async function batchTagSync(
  operations: Array<{ contactId: string; tag: string; action: "add" | "remove" }>
) {
  // Process in chunks of 10, with delays between chunks
  const chunks = chunkArray(operations, 10);

  for (const chunk of chunks) {
    await Promise.all(
      chunk.map(op => ghlApiCall(
        `/contacts/${op.contactId}/tags`,
        { method: "POST", body: JSON.stringify({ tags: [op.tag] }) },
        "batch"
      ))
    );
    // Explicit pause between chunks
    await new Promise(resolve => setTimeout(resolve, 2000));
  }
}
```

**Detection (warning signs):**
- `X-RateLimit-Remaining` headers dropping to 0
- 429 responses in logs, especially during batch operations
- Coach dashboard shows loading spinners for CRM data
- `X-RateLimit-Daily-Remaining` declining faster than expected

**Which phase should address:** Sync Infrastructure Phase. The rate-limited API client wrapper must exist before any GHL API calls are made. All subsequent phases MUST use this client, never direct `fetch()`.

### Pitfall 5: Tag Schema Mismatch Causes Silent Sync Failures

**What goes wrong:**
Coach creates tag "module-1-complete" in the LMS. LMS syncs it to GHL. But GHL already has a tag "Module 1 Complete" (with spaces and capitals) from a previous manual workflow. GHL now has TWO tags that mean the same thing. GHL automations trigger on "Module 1 Complete" but the LMS only sets "module-1-complete". Automations never fire. Coach blames the integration.

Worse: GHL tags are case-sensitive and space-sensitive. "module-1" and "Module-1" and "module 1" are three different tags.

**Why it happens:**
- No shared tag registry or naming convention between systems
- GHL allows free-form tag creation via API (no validation against existing tags)
- Tags created in GHL UI use human-friendly names ("Module 1 Complete")
- Tags created programmatically tend to use slug-style names ("module-1-complete")
- No duplicate detection or fuzzy matching when creating tags

**Consequences:**
- GHL automations don't trigger (wrong tag name)
- Coach confusion: "I set the tag but nothing happened"
- Tag proliferation in GHL (dozens of near-duplicate tags)
- Reports based on GHL tags show incorrect counts
- Difficult to clean up once tag drift has accumulated

**Prevention:**
```typescript
// lib/ghl/tag-registry.ts
// Canonical tag registry with GHL name mapping

// Define ALL tags in one place with both internal and GHL representations
export const TAG_REGISTRY = {
  // Progress tags
  "module_1_complete": {
    ghlName: "Module 1 Complete",    // MUST match exactly what GHL expects
    description: "Student completed all lessons in Module 1",
    direction: "lms_to_ghl" as const, // Only set from LMS
  },
  "module_2_complete": {
    ghlName: "Module 2 Complete",
    description: "Student completed all lessons in Module 2",
    direction: "lms_to_ghl" as const,
  },
  // Coaching tags
  "needs_coaching": {
    ghlName: "Needs Coaching",
    description: "Student flagged for coach outreach",
    direction: "bidirectional" as const,
  },
  "inactive_7d": {
    ghlName: "Inactive 7 Days",
    description: "No activity in 7 days",
    direction: "lms_to_ghl" as const,
  },
  // CRM-owned tags (coach sets in GHL, syncs to LMS)
  "vip_student": {
    ghlName: "VIP Student",
    description: "High-value student flagged by coach",
    direction: "ghl_to_lms" as const,
  },
} as const;

type TagKey = keyof typeof TAG_REGISTRY;

// Reverse lookup: GHL name -> internal key
const GHL_NAME_TO_KEY = Object.fromEntries(
  Object.entries(TAG_REGISTRY).map(([key, val]) => [val.ghlName, key])
);

export function ghlNameToTagKey(ghlName: string): TagKey | null {
  return (GHL_NAME_TO_KEY[ghlName] as TagKey) ?? null;
}

export function tagKeyToGHLName(key: TagKey): string {
  return TAG_REGISTRY[key].ghlName;
}

// Validate direction before syncing
export function canSyncDirection(
  key: TagKey,
  direction: "lms_to_ghl" | "ghl_to_lms"
): boolean {
  const tag = TAG_REGISTRY[key];
  return tag.direction === "bidirectional" || tag.direction === direction;
}

// In webhook handler: reject unknown tags
function handleInboundTag(ghlTagName: string) {
  const key = ghlNameToTagKey(ghlTagName);
  if (!key) {
    console.warn(`Unknown GHL tag received: "${ghlTagName}" - ignoring`);
    return; // Don't create random tags in LMS
  }
  if (!canSyncDirection(key, "ghl_to_lms")) {
    console.warn(`Tag "${ghlTagName}" is LMS-owned, ignoring GHL change`);
    return; // This tag should only flow LMS -> GHL
  }
  // Process the tag change
}
```

**Detection (warning signs):**
- GHL dashboard shows duplicate-looking tags ("Module 1 Complete" AND "module-1-complete")
- GHL automations not firing despite tags being set
- Coach reports "tags aren't syncing" (they are, just to the wrong tag name)
- Tag count in GHL grows unexpectedly over time

**Which phase should address:** Tag Schema Phase (before any tag sync is implemented). The registry MUST be agreed upon with the coach/admin before writing any sync code. Store the registry in the database (not just code) so it can be updated without deployments.

### Pitfall 6: Stale CRM Data Displayed to Coaches Due to Aggressive Caching

**What goes wrong:**
Coach opens student profile in LMS, sees CRM context (last GHL activity, tags, pipeline stage). Coach calls the student based on this info. But the data is 2 hours stale -- the student already responded to an email, moved pipeline stages, and was contacted by another coach. Coach duplicates outreach, annoying the student.

**Why it happens:**
To avoid hitting GHL rate limits (Pitfall 4), developers cache CRM data aggressively. But CRM data changes frequently from multiple sources (other coaches, automations, the student themselves). The caching TTL is too long, or cache invalidation doesn't account for webhook updates.

**Consequences:**
- Coaches make decisions on stale data
- Duplicate outreach to students
- Trust erosion: "The CRM data in LMS is never accurate"
- Coaches revert to using GHL directly, defeating the integration purpose

**Prevention:**
```typescript
// Use a layered approach: webhook-driven cache invalidation + short TTL fallback

// 1. When GHL webhook arrives, invalidate the cache for that contact
async function handleGHLWebhook(contactId: string) {
  await redis.del(`ghl:contact:${contactId}`);
  // Process the event...
}

// 2. When reading, use cache with SHORT TTL (5 min max)
async function getGHLContact(contactId: string) {
  const cached = await redis.get(`ghl:contact:${contactId}`);
  if (cached) return JSON.parse(cached);

  const contact = await ghlApiCall(`/contacts/${contactId}`, { method: "GET" }, "urgent");
  const data = await contact.json();

  await redis.set(`ghl:contact:${contactId}`, JSON.stringify(data), { ex: 300 }); // 5 min
  return data;
}

// 3. Show data freshness indicator to coach
function CRMContextPanel({ contact, lastSyncedAt }) {
  const staleMinutes = Math.floor((Date.now() - lastSyncedAt) / 60000);
  return (
    <div>
      <p className="text-xs text-zinc-500">
        Last synced: {staleMinutes < 1 ? "just now" : `${staleMinutes}m ago`}
        {staleMinutes > 10 && (
          <button onClick={refreshContact}>Refresh</button>
        )}
      </p>
      {/* CRM data display */}
    </div>
  );
}
```

**Detection (warning signs):**
- Coaches complain about "wrong" CRM data
- Cache hit rate is suspiciously high (near 100% means TTL is too long)
- Webhook-driven invalidations not happening (check GHL webhook delivery logs)

**Which phase should address:** UI Phase (coach dashboard). But caching strategy should be decided in Sync Infrastructure Phase.

### Pitfall 7: Existing Webhook Endpoint Conflicts with GHL Webhooks

**What goes wrong:**
The LMS already has `/api/webhooks/clerk` and `/api/webhooks/enroll` endpoints. Developer adds `/api/webhooks/ghl`. The existing `webhookLimiter` (10 requests/minute per IP) rate-limits GHL webhook deliveries because GHL sends bursts of webhooks from the same IP range. Legitimate GHL events get 429'd. Since GHL retries on 429 with 10-minute delays and only 6 attempts, events can be significantly delayed or lost if the rate limit is persistently hit.

**Why it happens:**
The existing `webhookLimiter` in `src/lib/rate-limit.ts` applies 10 requests/minute per IP. This is appropriate for the enrollment webhook (called infrequently by n8n) but too restrictive for GHL, which can send dozens of webhook events in rapid succession (e.g., when a coach bulk-updates tags for 20 students).

**Consequences:**
- GHL webhook events delayed by 10+ minutes per retry cycle
- After 6 retries (~70 minutes), events are permanently lost
- Intermittent data sync failures that are hard to reproduce
- Existing Clerk/enrollment webhooks unaffected (they use different IPs)

**Prevention:**
```typescript
// Create separate rate limiters for different webhook sources
// lib/rate-limit.ts -- additions

// Existing: restrictive for unknown sources
export const webhookLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(10, "1 m"),
  prefix: "ratelimit:webhook",
});

// NEW: More permissive for GHL (they send bursts)
export const ghlWebhookLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(100, "1 m"), // GHL can send many events
  prefix: "ratelimit:webhook:ghl",
});

// In /api/webhooks/ghl/route.ts -- use GHL-specific limiter
export async function POST(req: NextRequest) {
  const ip = getClientIp(req);
  const rl = await ghlWebhookLimiter.limit(ip);
  if (!rl.success) {
    // Return 429 -- GHL will retry with backoff
    return rateLimitResponse(rl);
  }
  // ... signature verification and processing
}
```

**Detection (warning signs):**
- GHL Webhook Logs Dashboard shows 429 responses from your endpoint
- Webhook events arrive with significant delays
- Rate limit Redis keys for "webhook" prefix show high usage

**Which phase should address:** Webhook Infrastructure Phase. Must be addressed when creating the GHL webhook endpoint, before connecting any events.

**Sources:**
- [GoHighLevel Automated Webhook Retries](https://help.gohighlevel.com/support/solutions/articles/155000007071-automated-webhook-retries) - HIGH confidence
- Existing codebase: `src/lib/rate-limit.ts` webhookLimiter at 10/min - verified

### Pitfall 8: Contact ID Mapping Breaks on Email Changes

**What goes wrong:**
LMS maps users to GHL contacts by email address. Student changes their email in Clerk. Clerk webhook updates the LMS email. But the GHL contact still has the old email. Now the LMS can't find the GHL contact for that student. All sync operations silently fail because the email lookup returns nothing.

**Why it happens:**
- Email is used as the shared identifier between systems
- Clerk allows email changes (and fires `user.updated` webhook)
- GHL contact emails don't auto-update when Clerk emails change
- No persistent mapping table linking LMS user ID to GHL contact ID

**Consequences:**
- CRM data disappears from coach dashboard for that student
- Tag syncs silently fail (contact not found)
- Duplicate contacts created in GHL if system falls back to creating new ones
- Progress data no longer syncs to the student's CRM record

**Prevention:**
```typescript
// db/schema/ghl-mapping.ts
// Persistent mapping table -- NEVER rely on email for cross-system lookups

export const ghlContactMapping = pgTable("ghl_contact_mapping", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id")
    .notNull()
    .unique()
    .references(() => users.id, { onDelete: "cascade" }),
  ghlContactId: text("ghl_contact_id").notNull().unique(),
  ghlLocationId: text("ghl_location_id").notNull(),
  // Track sync state
  lastSyncedAt: timestamp("last_synced_at"),
  syncStatus: text("sync_status").notNull().default("active"), // active | error | disconnected
  createdAt: timestamp("created_at").notNull().defaultNow(),
});

// On initial sync: create mapping and STORE the GHL contact ID
async function linkContactToUser(userId: string, ghlContactId: string, locationId: string) {
  await db.insert(ghlContactMapping).values({
    userId,
    ghlContactId,
    ghlLocationId: locationId,
  }).onConflictDoUpdate({
    target: ghlContactMapping.userId,
    set: { ghlContactId, ghlLocationId: locationId, syncStatus: "active" },
  });
}

// For ALL sync operations: lookup by mapping, not email
async function getGHLContactId(userId: string): Promise<string | null> {
  const mapping = await db.query.ghlContactMapping.findFirst({
    where: eq(ghlContactMapping.userId, userId),
  });
  return mapping?.ghlContactId ?? null;
}

// When Clerk email changes, also update GHL contact
async function handleClerkEmailUpdate(clerkId: string, newEmail: string) {
  const user = await db.query.users.findFirst({
    where: eq(users.clerkId, clerkId),
  });
  if (!user) return;

  const ghlContactId = await getGHLContactId(user.id);
  if (ghlContactId) {
    await ghlApiCall(`/contacts/${ghlContactId}`, {
      method: "PUT",
      body: JSON.stringify({ email: newEmail }),
    });
  }
}
```

**Detection (warning signs):**
- "Contact not found" errors in sync logs
- Students with CRM data that suddenly goes blank
- Duplicate contacts appearing in GHL
- Sync error count correlates with email change frequency

**Which phase should address:** Contact Sync Phase. The mapping table must be the FIRST schema created for CRM integration. Update the existing Clerk webhook handler to propagate email changes.

### Pitfall 9: Scope Creep -- Building a CRM Inside the LMS

**What goes wrong:**
Coaches want to see pipeline stages in the LMS. Then they want to move contacts between stages. Then they want to send emails from the LMS. Then they want to view conversation history. Soon you've rebuilt 40% of GoHighLevel inside the LMS. The user explicitly said "I don't want it to fully replace GoHighLevel."

**Why it happens:**
Each feature request is individually reasonable ("just show me the pipeline stage"). But cumulatively, they create a shadow CRM that is always inferior to the real GHL. Every new feature doubles the sync complexity and the surface area for bugs.

**Consequences:**
- Months of development on CRM features that GHL already does better
- Massive sync complexity for every new data type
- Coaches use a half-baked LMS CRM view instead of going to GHL directly
- Every GHL UI change potentially breaks the LMS mirror

**Prevention:**
```
SCOPE BOUNDARY RULES (enforce in planning, not just development):

TIER 1 - DISPLAY ONLY (safe to build):
  - Show GHL tags on student profile (read-only from cache)
  - Show "last contacted" timestamp
  - Show pipeline stage name (NOT a stage editor)
  - Show contact notes count (link to GHL for full view)

TIER 2 - LMS-OWNED WRITES (safe, data flows one direction):
  - Set progress tags (Module X Complete, Inactive 7d)
  - Update custom fields with LMS data (last_lesson, completion_%)
  - Fire webhook events for GHL automations

TIER 3 - BIDIRECTIONAL (build carefully, needs echo detection):
  - Coach-editable tags (e.g., "Needs Coaching", "VIP")
  - Student notes that sync both ways

TIER 4 - DO NOT BUILD (use GHL directly):
  - Pipeline stage editor
  - Email/SMS sending
  - Appointment scheduling
  - Conversation history viewer
  - Invoice/payment management
  - Workflow builder

For Tier 4 features: Add a "Open in GoHighLevel" button that deep-links to
the contact's GHL profile. This is 10 minutes of work vs 10 weeks.
```

**Detection (warning signs):**
- Feature requests that start with "Can we also show..."
- Coach saying "It would be nice if I didn't have to switch to GHL for..."
- New GHL API endpoints being added that aren't in the original scope
- Sync logic getting more complex with each sprint

**Which phase should address:** Planning Phase (before development starts). Write the scope boundary document and get explicit sign-off. Reference it in every feature discussion.

## Minor Pitfalls

Mistakes that cause annoyance but are relatively fixable.

---

### Pitfall 10: GHL API V1 vs V2 Confusion

**What goes wrong:**
Developer finds code examples or Stack Overflow answers using V1 API endpoints. Implements them. They work for now. But V1 is officially end-of-support as of January 2026. New API key generation has been disabled. When the V1 key eventually stops working, all integrations break simultaneously.

**Prevention:**
- Use ONLY V2 API (OAuth-based) from day one
- All GHL API calls must go through `https://services.leadconnectorhq.com` (V2 base URL)
- Include `Version: 2021-07-28` header in all requests
- Use Private Integration tokens as a simpler alternative to full OAuth if single-tenant

---

### Pitfall 11: Webhook Signature Verification Skipped During Development

**What goes wrong:**
Developer skips GHL webhook signature verification during local development ("I'll add it later"). Code ships to production without it. Anyone who discovers the webhook URL can inject fake events -- creating contacts, granting access, modifying tags.

**Prevention:**
- GHL uses RSA SHA256 signatures via `x-wh-signature` header
- Implement verification from the FIRST webhook handler
- Use environment variable for the public key
- For local development, use the same verification with GHL's test tools

---

### Pitfall 12: Timezone Mismatches in "Last Active" Timestamps

**What goes wrong:**
LMS stores `lastAccessedAt` in UTC. GHL displays times in the coach's local timezone. Coach sees "Last active: 2 hours ago" in GHL but "Last active: today" in LMS. Discrepancy erodes trust in the data.

**Prevention:**
- Always send timestamps to GHL in ISO 8601 with explicit timezone
- Store timezone preference for coaches
- Display relative times ("2 hours ago") instead of absolute times in the LMS UI
- Document that GHL and LMS may show slightly different "last active" due to different data sources

## Integration Gotchas

Specific to adding GHL integration to this existing LMS codebase.

| Integration Point | Common Mistake | Correct Approach |
|---|---|---|
| Clerk webhook + GHL contact create | Creating GHL contact before LMS user exists | Wait for Clerk `user.created` -> LMS user insert -> THEN create GHL contact |
| Enrollment webhook + GHL tags | Setting "enrolled" tag before course access is confirmed | Set tag AFTER `courseAccess` insert succeeds, not before |
| Progress tracking + tag sync | Syncing every progress update to GHL | Only sync MILESTONE tags (module complete, course complete), not every 1% progress update |
| Rate limiting + GHL calls | Using existing `webhookLimiter` for GHL endpoints | Create separate `ghlWebhookLimiter` with higher limits |
| n8n grading webhook + GHL | Trying to route GHL webhooks through n8n | Keep GHL webhooks direct to LMS; n8n is for AI grading only |
| Redis (Upstash) + token storage | Storing GHL tokens only in Redis (volatile) | Store in BOTH Redis (fast) and Postgres (durable) |
| Vercel deployments + webhook URLs | GHL webhook URL breaks on preview deployments | Use production URL only for GHL webhooks; never preview URLs |

## Performance Traps

Patterns that work during development but fail at scale.

| Trap | Symptoms | Prevention | When It Breaks |
|---|---|---|---|
| Fetching GHL contact on every page load | Slow page loads, rate limit exhaustion | Cache with webhook-driven invalidation | >50 concurrent coaches |
| Full contact list fetch for reconciliation | Timeout, memory exhaustion | Paginated fetch with cursor, run off-peak | >500 contacts |
| Sync every progress update to GHL | 429 errors, daily limit exhaustion | Debounce: sync only on milestone completion | >100 active students |
| Storing full GHL contact payload in cache | Redis memory bloat | Store only needed fields (tags, stage, lastActive) | >1000 contacts |
| No index on `ghl_contact_mapping.ghl_contact_id` | Slow webhook processing | Add unique index on `ghl_contact_id` column | >1000 mappings |

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---|---|---|
| Infinite webhook loop | LOW | Deploy echo detection, clear Redis, events self-heal |
| Lost webhook events | MEDIUM | Run reconciliation job to detect and fill gaps |
| Token refresh corruption | MEDIUM | Manual re-authorization via OAuth flow, update stored tokens |
| Rate limit exhaustion | LOW | Pause batch operations, wait for limit reset (10s burst, 24h daily) |
| Tag schema mismatch | HIGH | Audit all tags in both systems, deduplicate in GHL manually, establish registry |
| Duplicate GHL contacts | HIGH | Manual merge in GHL (no bulk merge API), update mapping table |
| Scope creep | VERY HIGH | Rip out excess features, restore scope boundary -- significant code deletion |
| Stale data display | LOW | Reduce cache TTL, add manual refresh button |
