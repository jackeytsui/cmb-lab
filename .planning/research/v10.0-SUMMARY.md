# Research Summary: v10.0 Mastery & Intelligence

**Domain:** SRS Flashcards, Smart Study Engine, Tone Training, Assessment/Placement, Grammar Library, Auto-Exercise Generation, AI Prompt Testing Lab
**Researched:** 2026-02-16
**Overall confidence:** HIGH

## Executive Summary

v10.0 adds seven interconnected feature domains to the CantoMando Blueprint LMS, transforming it from a lesson-delivery platform into a comprehensive study system with spaced repetition, intelligent study guidance, tone-specific training, proficiency assessment, structured grammar reference, AI-powered content generation, and coach tooling for prompt quality assurance.

The critical finding is that the existing stack covers every technical requirement. Zero new npm packages are needed. The SRS scheduling algorithm (FSRS-5) is small enough (~100 lines of pure TypeScript) to implement in-house, following the same pattern as existing utilities like `tone-sandhi.ts` and `practice-grading.ts`. Tone training reuses the existing Azure Speech pronunciation pipeline (per-character accuracy scores work as tone scoring for zh-CN since each character = one syllable). Auto-exercise generation routes through n8n webhooks using the existing OpenAI integration. Assessment tests are thin wrappers over the existing practice set system.

The primary risks are not technical but content-related: grammar library launches empty without AI-assisted seeding, assessment calibration requires real student testing, and AI-generated exercises will produce wrong answers that need coach review. The architecture is designed so that all AI output goes through human approval before reaching students.

The smart study recommendation engine is deliberately heuristic-based (not ML), which is appropriate for the current user scale (<100 students). It queries across SRS card states, practice scores, daily activity, and grammar progress to produce a prioritized study plan cached in Redis.

## Key Findings

**Stack:** Zero new npm packages. Custom FSRS-5 implementation (~100 lines). All AI through n8n webhooks. 9 new Postgres tables via Drizzle migrations.
**Architecture:** SRS cards reference existing `saved_vocabulary` (no content duplication). Assessments reuse `ExerciseDefinition` types and grading functions. Study engine caches in Redis.
**Critical pitfall:** FSRS implementation math errors silently degrade learning over weeks. Must port femto-fsrs exactly and write comprehensive unit tests against known test vectors.

## Implications for Roadmap

Based on research, suggested phase structure:

1. **SRS Foundation** - Schema, FSRS-5 algorithm, basic review UI, deck management
   - Addresses: SRS flashcard reviews (table stakes), card state management
   - Avoids: FSRS math errors (Pitfall 1) by porting tested reference implementation
   - Rationale: Everything else depends on SRS schema existing (vocabulary seeding, smart study, auto-card creation)

2. **HSK Vocabulary + Grammar Data** - Seed HSK vocab, create grammar schema, AI draft generation
   - Addresses: HSK vocabulary lists (table stakes), grammar pattern reference (table stakes)
   - Avoids: Content bottleneck (Pitfall 6) by using AI draft generation from day one
   - Rationale: Provides content for SRS cards, assessment questions, and grammar exercises

3. **Tone Training** - 4 drill types (identification, production, minimal pairs, sandhi)
   - Addresses: Tone pair drills (table stakes), dual Cantonese/Mandarin support (differentiator)
   - Avoids: Audio latency (Pitfall 8) with TTS pre-fetching and Redis caching
   - Rationale: Builds on existing Azure Speech pipeline. Independent of SRS/grammar.

4. **Assessment System** - Placement test, HSK mocks, adaptive question selection
   - Addresses: Placement test (table stakes), adaptive assessment (differentiator)
   - Avoids: Bad calibration (Pitfall 4) by requiring 5-8 questions per level and allowing manual override
   - Rationale: Needs HSK vocab data from phase 2 for question pools

5. **Auto-Exercise Generation + Prompt Lab** - AI exercise creation, coach approval queue, prompt A/B testing
   - Addresses: AI-generated exercises (differentiator), prompt testing lab (differentiator)
   - Avoids: Wrong answers (Pitfall 3) by requiring coach review before publish
   - Rationale: Content generation tools that multiply coach productivity. Prompt lab ensures AI quality.

6. **Smart Study Engine** - Recommendation algorithm, dashboard integration, Redis caching
   - Addresses: Study recommendations (table stakes), cross-skill balance (differentiator)
   - Avoids: Useless recommendations (Pitfall 7) with variety bonus and SRS capping
   - Rationale: Build last because it queries data from all other subsystems

**Phase ordering rationale:**
- Phase 1 first because SRS schema is a dependency for phases 2, 4, and 6
- Phase 2 before phases 4 and 5 because assessment questions and exercise generation need HSK/grammar content
- Phase 3 is independent and can run in parallel with phase 2
- Phase 5 before phase 6 because the study engine needs the full feature surface to make good recommendations
- Phases 1-2 could be one milestone, phases 3-4 another, phases 5-6 another, or all six in one large milestone

**Research flags for phases:**
- Phase 1: Needs careful implementation validation (FSRS-5 math). Write tests first. HIGH priority to get right.
- Phase 2: Content seeding is the risk, not tech. Validate HSK data quality post-seed.
- Phase 3: Standard patterns from existing pronunciation pipeline. LOW risk.
- Phase 4: Question calibration needs real student testing. Mark as "beta" initially.
- Phase 5: n8n webhook design for exercise generation needs workflow creation. MEDIUM complexity.
- Phase 6: Straightforward aggregation queries. Mostly a UX design challenge, not tech.

## Gaps to Address

- **Grammar content volume:** How many grammar patterns can be AI-drafted and coach-reviewed per day? Need to estimate throughput before committing to "50+ at launch."
- **FSRS-5 test vectors:** Need to find or create definitive test vectors for the custom implementation. The femto-fsrs reference exists but may not have comprehensive tests.
- **Cantonese tone training specifics:** Azure zh-HK pronunciation assessment is confirmed, but Cantonese has 6-9 tones (depending on analysis). The tone drill UI for Cantonese needs more design research.
- **n8n workflow design:** Two new n8n workflows needed (grammar generation, exercise generation). Their exact prompt structure needs to be designed during the implementation phase.
- **Assessment question pool size:** How many questions are needed per HSK level for reliable placement? Literature suggests 15-20 total, but the per-level minimum needs validation.
