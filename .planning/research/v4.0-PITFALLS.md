# Domain Pitfalls: v4.0 Practice & Homework Platform

**Domain:** Language learning exercise platform with pronunciation scoring
**Researched:** 2026-02-06

## Critical Pitfalls

Mistakes that cause rewrites or major issues.

### Pitfall 1: Azure Speech Prosody/Tone Scoring is English-Only

**What goes wrong:** Developer assumes Azure Speech pronunciation assessment provides per-character tone classification (tone 1/2/3/4/neutral) for Chinese because it supports zh-CN. Builds the entire tone highlighting UI expecting `toneScore` or `toneClass` in the API response. Discovers at integration time that prosody assessment (which covers intonation/tone) is only available for en-US.

**Why it happens:** Azure's documentation says "pronunciation assessment supports zh-CN" which is true for phoneme accuracy, but the prosody dimension that would capture tones is gated to English.

**Consequences:** Per-character tone accuracy highlighting feature is blocked. Significant rework to implement tone detection via alternative means.

**Prevention:**
- Design the pronunciation scoring API to return two types of scores: (1) Azure's native phoneme accuracy per word/character, and (2) a post-processed tone match using pinyin-pro/to-jyutping comparison against the recognized text.
- Build the UI to display both: an accuracy heatmap AND a tone match indicator, as separate visual layers.
- Plan the post-processing step from day one, not as a fallback.

**Detection:** Test Azure Speech zh-CN response JSON early in development. If `ProsodyScore` is absent or null, the workaround is needed.

### Pitfall 2: Building the Homework Builder Before Exercise Renderers

**What goes wrong:** Team starts with the visual drag-and-drop builder (exciting, shiny) before building the exercise rendering components. Builder has no way to show live previews of questions. Coach creates questions blind, with no visual feedback. Rework needed to retrofit preview rendering into the builder.

**Why it happens:** The builder feels like the "main" feature, so it gets prioritized. But a builder without preview is like a WYSIWYG editor without the WYS part.

**Consequences:** Builder ships without preview capability. Coaches create broken exercises because they cannot see what students will see. Must retrofit preview, causing component API redesign.

**Prevention:**
- Build exercise renderer components FIRST (McqExercise, MatchingExercise, etc.)
- Builder uses the same renderer components in a read-only preview mode
- Shared component interface: `<ExerciseRenderer type={type} config={config} mode="preview" | "interactive" />`

**Detection:** If builder work starts and there is no `<ExerciseRenderer>` component to import, the ordering is wrong.

### Pitfall 3: Applying Annotation Fonts Globally

**What goes wrong:** Custom Hanzi Pinyin font or Cantonese Visual font is applied via `font-family` on a high-level container (body, main content area). ALL Chinese text site-wide now shows phonetic annotations above characters -- including chat messages, admin panels, course descriptions, and navigation. This is visually overwhelming and breaks layouts (annotation text adds vertical height).

**Why it happens:** Developer adds the font class to the body tag for convenience, not realizing these are specialty fonts where every glyph includes annotation text rendered above the base character.

**Consequences:** Entire site looks broken. Chinese text everywhere has unexpected vertical annotations. Layout shifts. Must hunt down every usage and scope correctly.

**Prevention:**
- Load fonts via CSS variables (`--font-hanzi-pinyin`, `--font-cantonese-visual`) in layout.tsx but do NOT apply them to body.
- Create dedicated wrapper components: `<PinyinText>` and `<JyutpingText>` that apply the font class.
- Only these wrapper components should use `font-hanzi-pinyin` or `font-cantonese-visual` classes.
- Default Chinese text rendering uses the system font stack (no annotations).

**Detection:** If Chinese text in the navigation bar or admin panel shows phonetic annotations, the font scoping is wrong.

## Moderate Pitfalls

Mistakes that cause delays or technical debt.

### Pitfall 4: Using Legacy @dnd-kit/core Instead of @dnd-kit/react

**What goes wrong:** Developer installs `@dnd-kit/core` + `@dnd-kit/sortable` (the legacy packages with more documentation/tutorials) instead of the new `@dnd-kit/react` + `@dnd-kit/helpers`. Legacy packages have React 19 compatibility issues, different API patterns, and are being deprecated.

**Prevention:** npm install command in STACK.md specifies `@dnd-kit/react @dnd-kit/helpers`. Documentation is at next.dndkit.com (not docs.dndkit.com which is legacy).

---

### Pitfall 5: Recording Audio Without Proper Format for Azure Speech

**What goes wrong:** Browser MediaRecorder captures audio in webm/opus format (Chrome default). Azure Speech SDK expects WAV (PCM 16-bit, 16kHz or higher). Assessment fails with cryptic errors or returns garbage scores.

**Prevention:**
- Configure MediaRecorder with `mimeType: 'audio/webm;codecs=opus'` and convert to WAV before sending to Azure.
- OR use Azure Speech SDK's built-in microphone input (`AudioConfig.fromDefaultMicrophoneInput()`) which handles format natively.
- The SDK approach is simpler for the pronunciation assessment use case (record -> assess in one step).

---

### Pitfall 6: Storing Exercise Answers as Unstructured JSON

**What goes wrong:** All exercise types store their answers as a generic `jsonb` column with no schema validation. Over time, MCQ answers, matching pairs, ordering sequences, and free text all have inconsistent shapes. Querying for analytics becomes impossible. Data migration nightmares.

**Prevention:**
- Define Zod schemas for each exercise type's answer format:
  - MCQ: `{ selectedOptionId: string }`
  - Matching: `{ pairs: Array<{ left: string, right: string }> }`
  - Ordering: `{ orderedItemIds: string[] }`
  - Fill-blank: `{ answers: Record<string, string> }` (keyed by blank ID)
  - Audio: `{ audioUrl: string, assessmentResult: PronunciationResult }`
  - Free-text: `{ text: string }`
- Validate with Zod before storage. Store the exercise type alongside the answer.
- Use a discriminated union in TypeScript for the answer type.

---

### Pitfall 7: Overcomplicating the Homework Builder State Machine

**What goes wrong:** Builder state management becomes a tangled mess of useState hooks, context providers, and reducer patterns. Adding/removing/reordering questions creates race conditions. Undo/redo is impossible to add later.

**Prevention:**
- Use XState (already installed) for the builder state machine from day one.
- Model states: idle -> adding_question -> configuring -> previewing -> saving -> saved.
- Model the question list as immutable state transitions, not in-place mutations.
- XState's built-in history states enable undo/redo if needed later.

---

### Pitfall 8: Not Rate-Limiting Pronunciation Assessment Calls

**What goes wrong:** Students spam the "Record" button. Each attempt calls Azure Speech API. Monthly bill spikes. Free tier exhausted in days.

**Prevention:**
- Apply Upstash rate limiting (already configured) to the pronunciation assessment API route.
- Suggested limit: 30 pronunciation assessments per student per hour, 200 per day.
- Show a cooldown timer in the UI after each attempt (3-5 seconds minimum).
- Track daily usage per student in the database for admin visibility.

## Minor Pitfalls

Mistakes that cause annoyance but are fixable.

### Pitfall 9: Font Files Too Large for Initial Page Load

**What goes wrong:** Custom Chinese annotation fonts are 5-20MB each (CJK font files are large). First page load takes seconds for font download, during which Chinese text either shows fallback font (FOUT) or invisible text (FOIT).

**Prevention:**
- Convert .ttf to .woff2 (30-50% size reduction).
- next/font/local automatically applies `font-display: swap` (shows fallback first, swaps when loaded).
- Since annotation fonts are only used in specific exercise/practice contexts, lazy-load them: do NOT include the font variable in the root layout className. Instead, apply it only on pages/components that need it.
- Consider font subsetting if the font file supports it (remove unused Unicode blocks).

---

### Pitfall 10: Drag Preview Flicker on Touch Devices

**What goes wrong:** @dnd-kit drag preview looks correct on desktop but flickers or has wrong positioning on mobile touch devices. This is a known issue with the HTML Drag and Drop API on mobile.

**Prevention:**
- @dnd-kit uses PointerSensor by default, which handles both mouse and touch. Ensure PointerSensor is configured, not MouseSensor.
- Test on real mobile devices early (not just responsive mode in DevTools).
- For touch devices, consider using a "long press to drag" pattern to avoid conflicts with scrolling.

## Phase-Specific Warnings

| Phase Topic | Likely Pitfall | Mitigation |
|-------------|---------------|------------|
| Custom fonts | Applying annotation fonts globally (Pitfall 3) | Scope to wrapper components only |
| Exercise renderers | Inconsistent answer schemas (Pitfall 6) | Define Zod schemas per exercise type upfront |
| Homework builder | Legacy @dnd-kit packages (Pitfall 4) | Use @dnd-kit/react, NOT @dnd-kit/core |
| Homework builder | No live preview (Pitfall 2) | Build renderers before builder |
| Homework builder | State management mess (Pitfall 7) | Use XState from day one |
| Pronunciation scoring | Expecting tone scores from Azure (Pitfall 1) | Plan post-processing workaround |
| Pronunciation scoring | Wrong audio format (Pitfall 5) | Use Azure SDK microphone input or convert to WAV |
| Pronunciation scoring | API cost overrun (Pitfall 8) | Rate limit with Upstash |
