# Project Research Summary

**Project:** CantoMando Blueprint - Discord-Style RBAC (v9.0)
**Domain:** Role-based access control migration for existing language learning LMS
**Researched:** 2026-02-14
**Confidence:** HIGH

## Executive Summary

This research evaluates how to add Discord-style RBAC to an existing LMS that currently uses per-course access grants and a simple three-role hierarchy (student/coach/admin). The recommended approach is **additive and zero-dependency**: build the entire RBAC system using the existing stack (Drizzle ORM, Zod, react-hook-form, Clerk) with only two shadcn/ui components added (Badge, Checkbox). No RBAC libraries like CASL or Permit.io are needed because the permission model is simpler than what those tools solve — this is a pure additive union model with no deny mechanics or complex hierarchies.

The core architecture adds four new database tables (roles, role_courses, role_features, user_roles) alongside the existing course_access table. A centralized Permission Resolver replaces scattered courseAccess queries with a single function that unions role-based grants with direct course_access overrides. The critical design decision is keeping RBAC orthogonal to the existing hasMinimumRole() system: the legacy student/coach/admin role continues controlling dashboard routing via Clerk claims, while RBAC controls student content access via database-backed role resolution. This separation eliminates the need to refactor 117 files that use hasMinimumRole().

The primary risk is **session token staleness**: Clerk refreshes session tokens every 60 seconds, creating a window where students upgraded via webhook see access denied errors. The research identifies five critical pitfalls (token staleness, dual authority conflicts, migration data loss, webhook idempotency gaps, unknown role IDs) and provides concrete prevention strategies for each. The recommended migration path is additive-only with dual-write and feature flag cutover to eliminate rollback risk.

## Key Findings

### Recommended Stack

**Zero new npm packages.** The entire RBAC system is built with existing infrastructure plus ~1,200 lines of application code across 15 new files.

**Core technologies:**
- Drizzle ORM (existing) — four new tables with standard pgTable/uuid/jsonb patterns already proven in 27 schema files
- Pure TypeScript resolver (~100 LOC) — no RBAC library needed; additive union logic is simpler than CASL's ABAC or Permit.io's policy engine
- Zod (existing) — API validation for role CRUD and webhook payloads using established react-hook-form patterns
- React cache() for per-request memoization — no Redis needed; 3 parallel queries resolve in ~50ms with Neon HTTP driver
- Clerk (unchanged) — RBAC roles stay in database, NOT in session claims (avoids 1.2KB cookie limit and 60s stale window for role changes)

**Two shadcn/ui additions:**
- Badge component — role name display on student profiles (no existing badge component in the 24 UI components)
- Checkbox component — multi-select for courses/features in role builder (existing UI only has Select for single-choice and Switch for binary toggles)

**Critical architecture decision from STACK research:** Use text columns for role names (not pgEnum) to avoid Postgres enum rollback limitations. Feature keys use a fixed TypeScript const array as source of truth, validated by Zod at API boundaries. This avoids the "cannot roll back ALTER TYPE ADD VALUE" pitfall while maintaining type safety.

### Expected Features

**Must have (table stakes):**
- Named roles (TS-1) — coaches create "Bronze", "Premium", "Trial" packages with description and color
- Role-to-course mapping (TS-2) — each role grants access to specific courses or wildcard "all courses"
- Role assignment to students (TS-3) — students can hold multiple roles that stack additively like Discord
- Permission resolver (TS-4) — centralized function computing effective permissions from all roles plus direct course_access overrides
- Feature-level permissions (TS-5) — roles grant access to 7 platform features: ai_conversation, practice_sets, dictionary_reader, listening_lab, video_threads, certificates, ai_chat
- Role expiration (TS-6) — optional expiresAt on role assignments for trial/seasonal access
- Admin/coach UI (TS-7) — role builder and student assignment pages
- Webhook enrollment (TS-8) — webhook accepts roleId/roleName for automated tier assignment via GHL

**Should have (competitive):**
- Wildcard course access (D-1) — "all courses" flag eliminates maintenance for premium tiers when new courses are added
- Role stacking visualization (D-2) — student profile shows which role grants which permission (critical for debugging multi-role scenarios)
- Bulk role assignment (D-3) — extends existing bulk operations to assign/remove roles for cohort onboarding
- Role-based access tier (D-4) — Trial role grants preview access, Premium grants full (tier override per role-course pair)
- Student-facing tier display (D-5) — locked features show "Unlock with Premium" instead of invisibly hidden (drives upsells)
- Role templates (D-6) — preset "Free Trial", "Standard", "Premium" roles with clone-and-customize
- Migration tool (D-7) — analyzes existing course_access patterns, suggests roles, migrates students
- GHL role name lookup (D-8) — webhook accepts human-readable "Premium" instead of UUIDs for coach-configured automations

**Explicitly rejected (anti-features):**
- Arbitrary permission strings — fixed feature enum prevents permission sprawl
- Per-lesson/module overrides — course-level gating is sufficient; per-lesson breaks linear unlock flow
- Explicit deny permissions — pure additive model only; three-state (allow/deny/inherit) is the #1 RBAC confusion source
- Role hierarchy with inheritance — flat roles with explicit permissions; inheritance chains create invisible dependencies
- Dynamic/conditional permissions — keep RBAC static; progress-gating stays in checkLessonUnlock
- Per-student feature overrides — create small roles like "AI Add-on" instead of shadow permission system
- Real-time permission push — resolve on each page load; stale session after role change is acceptable until next navigation
- Role-based UI theming — show role badge with role color, not entire dashboard theme changes

### Architecture Approach

The architecture is **additive and non-destructive**. Four new tables coexist with the existing course_access table. The Permission Resolver unions both sources, making migration incremental and rollback-safe. The existing hasMinimumRole() system for coach/admin dashboard gating remains unchanged — RBAC is orthogonal, controlling student content access.

**Major components:**
1. **Permission Resolver** (src/lib/permissions.ts) — executes 3 parallel queries (user_roles JOIN roles, role_courses, role_features, course_access) and returns PermissionSet with canAccessCourse(), canUseFeature(), getAccessTier() methods. Query cost: ~50ms with Neon HTTP driver. Per-request caching via React cache() eliminates duplicate queries.
2. **RBAC Schema** (src/db/schema/roles.ts) — roles table with allCourses wildcard flag, role_courses join table with accessTier per mapping, role_features join table with fixed feature_key enum, user_roles join table with expiresAt. All use CASCADE on delete for automatic cleanup.
3. **Feature Gate Component** (src/components/auth/FeatureGate.tsx) — server component wrapper that checks permissions.canUseFeature(key) and renders children or fallback. Enforced at 8 entry points (ai_conversation, practice_sets, dictionary_reader, listening_lab, video_threads, certificates, ai_chat pages/routes).
4. **Role Admin API** (src/app/api/admin/roles/) — CRUD endpoints for roles, role-course mappings, role-feature mappings. Uses existing hasMinimumRole("admin") guards. Zod validation with existing react-hook-form patterns.
5. **Enrollment Webhook Extension** (src/app/api/webhooks/enroll/route.ts) — accepts roleId/roleName in addition to courseId. Backward compatible: courseId-only payloads continue working. Idempotent via processed_webhooks table with unique constraint on idempotency key.

**Critical pattern from ARCHITECTURE research:** Resolve permissions once at the top of a server component, pass the PermissionSet down. Never call resolvePermissions() multiple times in the same request. Existing ~50 files that query courseAccess directly migrate incrementally — the resolver includes course_access in its union, so switching files from direct query to resolver does not change behavior.

**Migration strategy:** Phase 1 adds schema with no behavior change. Phase 2 uses dual-path resolution (new code uses resolver, existing code unchanged). Phase 3 creates roles and assigns students (access now comes from BOTH roles and course_access). Phase 4 migrates existing queries file-by-file. No destructive changes. course_access table becomes "direct override" layer permanently.

### Critical Pitfalls

1. **Clerk Session Token Staleness (60-Second Blindspot)** — Webhook upgrades student role in DB but Clerk session token caches old role for up to 60 seconds. The existing resolveRole() in src/lib/auth.ts reads sessionClaims FIRST with DB fallback only when claims are empty. Result: student sees "Forbidden" for up to 60s after upgrade. **Prevention:** Reverse resolveRole() priority to DB-first (claims become fast-path optimization, not authority) OR use Redis cache with 30s TTL invalidated on role change webhook. CRITICAL: This affects 117 files with 286 hasMinimumRole() calls.

2. **Dual Authority Conflict (Role vs courseAccess)** — After RBAC, TWO systems grant access: role-based (new) and resource-based (existing course_access). These can disagree. Student purchases Tier 2, gets role via webhook but no course_access row — sees empty course list. Or admin revokes course_access but role still says Tier 2 — student re-accesses via different code path. **Prevention:** Decide single source of truth BEFORE schema design. Recommended: Role is authority, course_access is derived/override. Centralize all access checks in canAccessCourse() resolver function — eliminates scattered courseAccess queries in 16 files with 141 occurrences.

3. **Migration Data Loss (100+ Students at Risk)** — Postgres enum ALTER TYPE ADD VALUE cannot be rolled back within transaction. If migration fails halfway, partially-modified enum can't be undone. Coordinating changes across users.role column, Clerk publicMetadata, courseAccess table, middleware, resolveRole(), and 117 hasMinimumRole() files creates rollback nightmare. **Prevention:** Additive-only migration with dual-write. Add role_v2 TEXT column (not enum), backfill, deploy code reading BOTH columns with feature flag cutover, verify reconciliation query returns 0 rows before dropping old column. Clerk metadata batch update with rate limiting (20 req/10s) and failure logging.

4. **Webhook Idempotency Gaps (Duplicate/Conflicting State)** — External sales system fires webhook twice due to network retry. Both handlers create duplicate course_access rows or second webhook downgrades student with stale data. Existing enrollment webhook has per-record duplicate check but NO general idempotency mechanism (no processed_webhooks table, no idempotency key). **Prevention:** Add processed_webhooks table with unique constraint on idempotencyKey. Webhook handler checks existing before processing, records result AFTER success. Unknown roleId/roleName returns 404 (not 200) to alert external system to misconfiguration.

5. **Unknown External Role IDs (Silent Failures)** — External sales system sends roleId: "tier_3_vip" that LMS has never heard of. Webhook handler either silently ignores (student paid but gets nothing), throws unhandled error (500 may not retry), or defaults to "student" (paid for premium, gets basic). Sales team creates new product tiers faster than engineering adds role mappings. **Prevention:** Role registry with external ID mapping. Webhook resolves external ID to internal role, alerts admin on unknown roles, stores in pending_role_assignments table for manual resolution. Admin dashboard shows unmapped roles.

**Additional moderate pitfalls from research:**
- Role hierarchy breaks with non-hierarchical roles (need capability-based system)
- Middleware role check diverges from API route check (make middleware permissive, API authoritative)
- Permission explosion with role-course mappings (use tier-based model: role unlocks tier number, courses have tier)
- Clerk cookie size limit exceeded (keep publicMetadata minimal: only role name, never permissions array or course IDs)

## Implications for Roadmap

Based on research, suggested phase structure addresses dependencies discovered in FEATURES.md and avoids critical pitfalls from PITFALLS.md:

### Phase 1: Schema + Resolver Foundation
**Rationale:** Build the entire data model and permission resolver in one phase with zero UI. This is the critical path foundation that everything else depends on. Include wildcard "all courses" flag (D-1) because it's a single boolean column with no extra complexity.

**Delivers:**
- Drizzle schema: roles, role_courses, role_features, user_roles tables
- Permission resolver: resolvePermissions() with PermissionSet interface
- Feature key constants: FEATURE_KEYS array with 7 platform features
- Zod schemas: createRoleSchema, assignRoleSchema, webhookRoleSchema
- Helper functions: canAccessCourse(), canAccessFeature() wrappers
- Unit tests: resolver with mock data verifying union logic

**Addresses features:** TS-1, TS-2, TS-3, TS-5, TS-6, TS-4, D-1

**Avoids pitfalls:**
- Uses TEXT columns for role names (not pgEnum) — avoids enum rollback limitation (Pitfall 3)
- Resolver includes both user_roles AND course_access in union — establishes single source of truth pattern (Pitfall 2)
- Per-request caching via React cache() — no premature Redis optimization

**Research flag:** Standard patterns only. Drizzle table creation and union queries are well-documented. No /gsd:research-phase needed.

### Phase 2: Admin UI + Role Management
**Rationale:** Coaches need to create and manage roles before anything is enforced. Build the admin tooling that makes RBAC configurable without code changes. Include role templates (D-6) as seed data. Include bulk role assignment (D-3) because infrastructure already exists in bulk operations system.

**Delivers:**
- Role admin pages: /admin/roles (list), /admin/roles/[roleId] (builder)
- Role CRUD API: POST/GET/PATCH/DELETE /api/admin/roles
- Role builder form: course toggles (Checkbox), feature toggles (Checkbox), "all courses" switch (Switch), name/description/color inputs
- Student role assignment: section on /admin/students/[studentId] with role badges, add/remove buttons
- Bulk operations extension: assign_role/remove_role operations in bulkSchema
- Preset roles seed data: "Free Trial" (preview all courses), "Standard" (selected courses + core features), "Premium" (all courses + all features)

**Addresses features:** TS-7, D-3, D-6

**Avoids pitfalls:**
- Admin self-protection guard: cannot remove own admin role (Pitfall 11)
- Last admin guard: cannot remove last admin in system (Pitfall 11)
- Role templates reduce setup friction — avoids "permission explosion" from manual per-course mapping (Pitfall 8)

**Uses from STACK:** Badge component (new shadcn/ui), Checkbox component (new shadcn/ui), existing @tanstack/react-table for role listing, existing react-hook-form + Zod validation patterns

### Phase 3: Webhook + Feature Enforcement + Student Experience
**Rationale:** Wire up the webhook for automated enrollment via roles. Enforce feature permissions across all entry points (the actual "gates"). Add student-facing tier display so locked features show upgrade prompts instead of invisible absence.

**Delivers:**

- Student tier display: role badges on dashboard, locked features show "Unlock with [role name]" prompts
- Upgrade CTA components: feature-specific upgrade prompts with role comparison

**Addresses features:** TS-8, D-8, D-5, feature gating enforcement

**Avoids pitfalls:**
- Idempotency via processed_webhooks — prevents duplicate/conflicting state from webhook retries (Pitfall 4)
- Unknown role alerts — prevents silent failures when external system sends unmapped role IDs (Pitfall 5)
- Role registry with external ID mapping — supports GHL "Premium" → internal "tier_2" translation (Pitfall 5)

**Uses from STACK:** @upstash/ratelimit (existing), Zod webhookRoleSchema, existing enrollment webhook patterns

**Research flag:** Webhook idempotency patterns are well-documented (Hookdeck guide). Feature gating is standard pattern (show/hide based on permission check). No /gsd:research-phase needed.

### Phase 4: Migration + Attribution + Access Tier Overrides
**Rationale:** Migrate existing students from course_access to roles using automated analysis tool. Add role stacking visualization for debugging multi-role scenarios. Add access tier overrides per role-course if trial funnels are needed.

**Delivers:**

**Addresses features:** D-7, D-2, D-4

### Phase 5: Role Resolution Caching + Performance
**Rationale:** Address session token staleness (Pitfall 1) and optimize permission resolution for production scale. This phase can be deferred until after launch if initial user count is low (<100 concurrent).

**Delivers:**
- resolveRole() refactor: DB-first priority (not claims-first) OR Redis-cached with 30s TTL
- Redis integration: @upstash/redis (already installed) with cache invalidation on role change webhooks
- Permission resolution benchmark: measure p95 latency under load (target <50ms)
- Conditional Redis: only add if measured latency exceeds 50ms in production
- Cache invalidation hooks: webhook handler calls redis.del() on role assignment/revocation
- Monitoring: track permission resolution latency, cache hit rate

**Addresses pitfalls:**
- Session token staleness — DB-first or cached resolution eliminates 60s blindspot (Pitfall 1)
- Middleware/API divergence — making middleware permissive and API authoritative prevents stale claims blocking fresh DB access (Pitfall 7)

**Uses from STACK:** @upstash/redis (existing), React cache() for per-request memoization

**Research flag:** Redis caching patterns and Clerk session token refresh mechanics are well-documented. If Pitfall 1 is critical for launch (students upgrade via live webhook during onboarding calls), promote this to Phase 3. Otherwise defer until measured latency justifies it.

### Phase Ordering Rationale

**Why schema + resolver first:**
- Everything depends on the permission model existing. No UI or enforcement can be built without resolvePermissions().
- Schema is pure infrastructure with no user-facing changes — safe to deploy and verify before adding UI.
- Testable via seed data and API tests without requiring admin UI or student experience changes.

**Why admin UI before webhook:**
- Coaches must create roles before webhook can assign them. Webhook handler fails on unknown roleId if no roles exist.
- Role builder establishes the "role registry" that webhook depends on for external ID mapping.
- Allows manual testing and role configuration before connecting external sales automation.

**Why webhook + enforcement together:**
- Webhook assigns roles but roles grant nothing until feature enforcement exists — these must launch together for end-to-end flow.
- Student-facing tier display (locked features showing upgrade prompts) justifies adding roles in the first place — delivers immediate value.

**Why migration last:**
- Existing students continue working via course_access throughout Phases 1-3. No migration urgency.
- Migration tool benefits from battle-tested role management UI (Phase 2) and webhook patterns (Phase 3).
- Allows production validation with NEW students before touching existing student access.

**Why caching/performance is optional final phase:**
- Per-request React cache() is sufficient for <1000 concurrent users (research shows ~50ms query time).
- Redis adds operational complexity (another service to monitor, cache invalidation bugs) without proven need.
- Can be added later if measured latency justifies it — does not block launch.
