## 1. XP Point System Design

### 1.1 XP Value Table (Recommended for CantoMando)

The point values below are calibrated for a premium language learning LMS where lessons are longer (10-30 min video) and exercises are substantive. Unlike Duolingo (5-min micro-lessons at 10 XP each), this system values depth over frequency.

| Action | XP | Rationale |
|--------|----|-----------|
| **Complete a lesson** (video + all interactions passed) | 50 | Core learning activity; video lessons are substantial (10-30 min). This is the anchor value. |
| **Pass a practice exercise** (MCQ, fill-blank, matching, ordering) | 5 | Quick exercises, graded client-side. Low individual value, accumulates across a practice set. |
| **Pass an AI-graded exercise** (free text, audio recording) | 10 | Harder than deterministic exercises; requires AI grading round-trip. |
| **Complete a practice set** (all exercises done) | 20 | Completion bonus on top of per-exercise XP. Rewards finishing the whole set. |
| **Practice set perfect score** (100%) | +10 bonus | Bonus for perfection. Added on top of completion XP. |
| **Voice AI conversation** (5+ min) | 15 | Conversation practice is high-effort, high-value for language learning. |
| **Voice AI conversation** (10+ min) | 25 | Extended conversation demonstrates deeper engagement. |
| **Chatbot interaction** (5+ messages exchanged) | 5 | Lightweight engagement with the AI chatbot. |
| **Pronunciation score 80+** | +5 bonus | Rewards good pronunciation on audio exercises. |
| **Pronunciation score 95+** | +10 bonus | Rewards excellent pronunciation. |
| **Daily goal met** | 10 | Rewards consistency. Applied once per day when daily XP target is reached. |
| **Streak milestone** (7, 14, 30, 60, 90, 180, 365 days) | 25-100 | One-time bonus. See milestone table below. |

**Design principle:** XP should correlate with genuine learning effort. A student who completes a lesson (50 XP) earns more than one who does 5 quick chatbot exchanges (25 XP). This prevents XP farming through low-effort actions.

### 1.2 Streak Milestone Bonuses

| Milestone | Bonus XP | Badge |
|-----------|----------|-------|
| 7 days | 25 | "First Week" |
| 14 days | 35 | "Two Weeks Strong" |
| 30 days | 50 | "Monthly Milestone" |
| 60 days | 75 | "Two Month Journey" |
| 90 days | 100 | "Quarter Master" |
| 180 days | 150 | "Half Year Dedication" |
| 365 days | 300 | "Annual Scholar" |

### 1.3 XP Inflation Prevention

**Problem:** Over time, students accumulate massive XP totals that make levels meaningless. Duolingo has this problem -- users reach level 25 and XP becomes decorative.

**Prevention strategies (recommended combination):**

1. **Capped daily XP from repeatable actions:** Limit XP from chatbot interactions to 25/day and voice AI to 50/day. Lesson and practice set XP is uncapped (there is a natural limit -- you can only complete so many lessons in a day).

2. **No XP for re-completing already-passed content:** Replaying a lesson you already completed gives 0 XP. Retrying a practice set gives 50% XP (to encourage review without enabling farming).

3. **Linear progression curve with increasing intervals:** Use a "linear increase in increases" formula rather than pure exponential. See section 1.4.

4. **XP is a measure, not a currency:** XP is NOT spent on anything. It purely tracks cumulative effort. This removes the inflation/deflation problem entirely -- there is nothing to "buy" so the absolute number doesn't matter, only progress toward the next level.

### 1.4 Level Progression Curve

**Recommended: Linear Progression Curve (increase in increases)**

This formula from game design literature provides accessible early levels that scale proportionally without exponential blowup:

```
Level 1:  100 XP   (cumulative: 100)
Level 2:  150 XP   (cumulative: 250)
Level 3:  200 XP   (cumulative: 450)
Level 4:  250 XP   (cumulative: 700)
Level 5:  300 XP   (cumulative: 1000)
...
Level N:  100 + (N-1) * 50 XP
```

**Formula:** `xpForLevel(n) = 100 + (n - 1) * 50`

**Cumulative:** `totalXpForLevel(n) = n * 100 + (n * (n-1) / 2) * 50`

This means:
- Level 1-5: Achievable in the first week (encouraging early engagement)
- Level 10: Requires ~3,250 cumulative XP (roughly 2-3 weeks of daily use)
- Level 20: Requires ~11,500 cumulative XP (roughly 2-3 months)
- Level 50: Requires ~63,750 cumulative XP (roughly 1 year of consistent use)

**Why not exponential?** An exponential curve (e.g., `100 * 1.4^n`) makes early levels trivial and late levels unreachable. With a 1.4x multiplier, level 20 would require 2.5M XP -- absurd. With a lower multiplier like 1.1x, early levels don't feel distinct enough. The linear increase approach avoids these extremes.

**Level cap:** Set at 50. After level 50, XP continues to accumulate but levels stop incrementing. This prevents the awkwardness of "Level 347" and preserves the premium feel.

### 1.5 Level Names (Optional, Premium Feel)

Instead of generic "Level 5", use language-learning themed names:

| Range | Theme | Examples |
|-------|-------|---------|
| 1-5 | Beginner | Listener, Observer, Initiate, Learner, Student |
| 6-10 | Elementary | Speaker, Reader, Writer, Conversant, Explorer |
| 11-20 | Intermediate | Practitioner, Communicator, Bridge Builder, etc. |
| 21-35 | Advanced | Scholar, Interpreter, Linguist, etc. |
| 36-50 | Mastery | Master, Ambassador, Polyglot, etc. |

## 2. Streak Mechanics & Psychology

### 2.1 Key Research Findings

**Duolingo data (HIGH confidence, from official Duolingo blog and third-party analysis):**

- Users are **3x more likely** to return daily when streaks are active
- Streak features showed **+14% improvement** in Day-7 retention (the greatest improvement among all engagement features)
- DAU/MAU ratio reached **37.3%** in Q3 2025 (meaning more than 1 in 3 monthly users practice daily)
- **Day 7 is the critical threshold** -- loss aversion kicks in around this point, making users much less likely to churn
- **Streak freezes increased engagement** -- counterintuitively, being more lenient with streaks boosts daily active users

### 2.2 Streak Design for Premium Adult Learners

**Key difference from Duolingo:** CantoMando serves high-ticket adult learners, not free-tier casual users. Adults have unpredictable schedules (work travel, family obligations, illness). The streak system must accommodate real life without inducing anxiety or shame.

**Recommended streak system:**

| Feature | Implementation | Rationale |
|---------|---------------|-----------|
| **Streak definition** | Complete your daily XP goal | Simple, unified with the goal system |
| **Streak freeze** | 2 free freezes per month, auto-replenish on the 1st | Adults need flexibility without feeling punished |
| **Grace period** | 4-hour grace past midnight in user's timezone | Late-night learners get buffer |
| **Streak repair** | After a break, show "Welcome back! Day 1 of your new streak" with positive framing | No shame; celebrate the return |
| **Milestone badges** | Permanent badges at 7, 14, 30, 60, 90, 180, 365 days | Badges persist even if streak breaks later |
| **Notification** | Single reminder 3 hours before streak expiration | No guilt trips, no escalating notifications |
| **Longest streak** | Track and display "Personal best" alongside current streak | A broken streak doesn't erase the achievement |

### 2.3 Anti-Anxiety Design Patterns

**Do:**
- "Keep your 30-day streak alive!" (positive, forward-looking)
- "Your streak freeze protected your 42-day streak yesterday" (acknowledge life happens)
- "Welcome back! You've practiced 847 minutes total" (focus on cumulative progress)
- "Daily practice helped you complete 23 lessons" (tie streak to real learning outcomes)

**Do NOT:**
- "You lost your streak!" (shame-inducing)
- "Don't lose your streak!" (anxiety-inducing)
- Multiple escalating notifications ("You haven't practiced yet today!", "Your streak is about to end!", "LAST CHANCE!")
- Flame/fire imagery (Duolingo's aggressive owl memes are fun for kids, inappropriate for premium adult product)

### 2.4 Streak Reset Behavior

When a streak breaks (no freeze available, grace period expired):

1. Current streak resets to 0
2. Longest streak (personal best) is preserved forever
3. Show encouraging message on next login: "Starting a new streak! Your personal best was 42 days."
4. Previously earned milestone badges remain displayed
5. XP earned during the streak is never lost

## 3. Daily Goal System

### 3.1 Goal Tiers

Let users choose their daily XP target. Map targets to time estimates for clarity:

| Tier | Daily XP Target | Estimated Time | Label |
|------|----------------|----------------|-------|
| Light | 20 XP | ~5 minutes | "Casual" |
| Regular | 50 XP | ~15 minutes | "Regular" |
| Serious | 100 XP | ~30 minutes | "Committed" |
| Intense | 150 XP | ~45 minutes | "Intensive" |

**Why these values?**
- **20 XP (Light):** Achievable by completing 2-4 practice exercises or a chatbot interaction. Keeps the streak alive on busy days.
- **50 XP (Regular):** Requires completing a lesson OR a full practice set + a few exercises. This is the recommended default.
- **100 XP (Serious):** Requires a lesson + practice set, or multiple practice activities. For dedicated learners.
- **150 XP (Intense):** Requires significant daily study. For learners in an intensive phase (exam prep, pre-trip).

### 3.2 Goal Selection UX

- Present during onboarding as a simple 4-option selector
- Default to "Regular" (50 XP)
- Changeable at any time from Settings page
- Show time estimate alongside XP target: "Regular: ~15 min/day"
- When user changes goal mid-day, apply the new goal to the current day (don't retroactively break the streak)

### 3.3 Adaptive Goals (Future Enhancement, Not MVP)

For post-v5.0: Track the user's actual daily XP over the past 14 days. If they consistently exceed their goal by 2x+, suggest upgrading. If they consistently miss it, suggest downgrading. Show this as a gentle suggestion, never automatically change the goal.

## 4. Activity Ring Design Patterns

### 4.1 Three-Ring Model (Recommended)

Apple Watch uses three rings: Move, Exercise, Stand. For CantoMando, use three learning-specific metrics:

| Ring | Metric | Color | Why |
|------|--------|-------|-----|
| **Outer ring** | Daily XP progress (toward daily goal) | Green (#34C759) | Primary engagement metric. Maps to Apple's "Move" ring. |
| **Middle ring** | Lessons/practice completed today | Blue (#007AFF) | Tracks structured learning activities. Like Apple's "Exercise" ring. |
| **Inner ring** | Streak days (current streak / next milestone) | Orange (#FF9500) | Long-term consistency. Like Apple's "Stand" ring. Shows progress toward next streak milestone. |

**Alternative 3-ring model considered and rejected:**
- XP / Time Spent / Conversations -- "Time Spent" is too easy to game (leave app open) and doesn't correlate with learning.
- Lessons / Exercises / Pronunciation -- Too granular; users won't track three separate learning modes.

### 4.2 SVG Implementation Technique

**Use framer-motion (already in project) with SVG circles.** Do NOT add `react-activity-rings` -- that library has limited customization and hasn't been updated since 2023. A custom implementation with framer-motion gives full control.

**Core SVG pattern:**

```tsx
// Pseudocode -- actual implementation in phase plans
const ActivityRing = ({ progress, radius, strokeWidth, color }) => {
  const circumference = 2 * Math.PI * radius;

  return (
    <svg viewBox="0 0 200 200">
      {/* Background track */}
      <circle
        cx={100} cy={100} r={radius}
        fill="none"
        stroke={color}
        strokeWidth={strokeWidth}
        opacity={0.2}
      />
      {/* Progress arc */}
      <motion.circle
        cx={100} cy={100} r={radius}
        fill="none"
        stroke={color}
        strokeWidth={strokeWidth}
        strokeLinecap="round"
        strokeDasharray={circumference}
        initial={{ strokeDashoffset: circumference }}
        animate={{ strokeDashoffset: circumference * (1 - progress) }}
        transition={{ duration: 1.5, ease: "easeOut" }}
        transform="rotate(-90 100 100)"
      />
    </svg>
  );
};
```

**Key details:**
- `transform="rotate(-90 100 100)"` -- start fill from 12 o'clock position (SVG circles start at 3 o'clock by default)
- `strokeLinecap="round"` -- rounded endpoints like Apple Watch
- Two circles per ring: background (20% opacity) + progress (full opacity, animated)
- `framer-motion`'s `motion.circle` handles smooth animation via `strokeDashoffset`
- For completion (100%), add a subtle glow/pulse animation

### 4.3 Ring Completion Celebrations

| Condition | Animation |
|-----------|-----------|
| Single ring reaches 100% | Ring pulses once, subtle glow effect |
| All 3 rings complete | Confetti burst + "Day Complete!" toast |
| Ring exceeds 100% (XP > goal) | Ring continues past full circle with overlap indicator |

### 4.4 Accessibility

- Add `aria-label` to SVG: "Daily XP progress: 75% of goal"
- Use `aria-valuemin`, `aria-valuemax`, `aria-valuenow` on the ring wrapper
- Provide a text-based summary below rings for screen readers
- Ensure colors pass WCAG contrast against dark background

## 5. Database Schema Design

### 5.1 Approach: Event-Sourced XP Ledger + Materialized Summaries

**Recommendation: Hybrid approach.** Use an append-only XP events table (source of truth) with materialized daily summaries for fast queries.

**Why not just a running total?**
- A single `xp_total` column on the users table is fragile: if a bug awards wrong XP, there's no audit trail to fix it.
- Event sourcing lets you recalculate totals from scratch if needed.
- The events table also powers the XP timeline chart on the progress dashboard.

**Why not pure event sourcing?**
- Querying "what is this user's total XP?" by aggregating all events is expensive as the events table grows.
- Daily summaries provide O(1) lookups for current totals and streak calculations.

### 5.2 Schema Definitions (Drizzle ORM)

#### Table 1: `xp_events` (Append-Only Ledger)

```typescript
import { pgTable, uuid, text, timestamp, integer, pgEnum } from "drizzle-orm/pg-core";
import { users } from "./users";

export const xpSourceEnum = pgEnum("xp_source", [
  "lesson_complete",
  "practice_exercise",
  "practice_set_complete",
  "practice_set_perfect",
  "voice_conversation",
  "chatbot_interaction",
  "pronunciation_bonus",
  "daily_goal_met",
  "streak_milestone",
]);

export const xpEvents = pgTable("xp_events", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  source: xpSourceEnum("source").notNull(),
  xpAmount: integer("xp_amount").notNull(),
  // Reference to the entity that triggered the XP (lesson ID, practice set ID, etc.)
  sourceEntityId: uuid("source_entity_id"),
  // Human-readable description for the XP timeline
  description: text("description"),
  createdAt: timestamp("created_at", { withTimezone: true }).notNull().defaultNow(),
});
```

**Indexes:**
- `(userId, createdAt)` -- for querying a user's XP timeline
- `(userId, source, sourceEntityId)` -- for dedup (prevent double-awarding)

### 5.3 Timezone Handling Strategy

**Critical design decision.** Streaks and daily goals must reset at midnight in the USER's timezone, not server time.

**Implementation:**

1. **Store user timezone** on the `users` table (add `timezone TEXT DEFAULT 'UTC'`). Set during onboarding or settings page. Use IANA timezone names (e.g., `America/New_York`, `Asia/Hong_Kong`).

2. **Determine "today" at write time:** When awarding XP, compute the user's local date:
   ```typescript
   import { formatInTimeZone } from 'date-fns-tz';
   const userToday = formatInTimeZone(new Date(), userTimezone, 'yyyy-MM-dd');
   ```
   Note: `date-fns` is already in the project. Add `date-fns-tz` for timezone support (or use the built-in `Intl.DateTimeFormat` API which requires no additional dependency).

3. **Grace period:** Add 4 hours to the midnight cutoff. If it's 2 AM in the user's timezone, treat it as "still yesterday" for streak purposes.
   ```typescript
   function getUserActivityDate(userTimezone: string): string {
     const now = new Date();
     // Subtract 4 hours for grace period
     const adjusted = new Date(now.getTime() - 4 * 60 * 60 * 1000);
     return formatInTimeZone(adjusted, userTimezone, 'yyyy-MM-dd');
   }
   ```

4. **No timezone conversion in SQL:** Store `activityDate` as a plain date string (`YYYY-MM-DD`). All timezone logic lives in the application layer. This avoids PostgreSQL `AT TIME ZONE` complexity and is simpler to reason about.

### 5.4 Streak Calculation Algorithm

```typescript
async function updateStreak(userId: string, userTimezone: string) {
  const today = getUserActivityDate(userTimezone);
  const streak = await db.query.userStreaks.findFirst({
    where: eq(userStreaks.userId, userId),
  });

  if (!streak) {
    // First time - create streak record
    await db.insert(userStreaks).values({
      userId,
      currentStreak: 1,
      longestStreak: 1,
      lastActiveDate: today,
      streakFreezes: 2,
    });
    return;
  }

  const yesterday = getYesterday(today); // subtract 1 day from date string

  if (streak.lastActiveDate === today) {
    // Already active today, no change
    return;
  }

  if (streak.lastActiveDate === yesterday) {
    // Consecutive day - extend streak
    const newStreak = streak.currentStreak + 1;
    await db.update(userStreaks)
      .set({
        currentStreak: newStreak,
        longestStreak: Math.max(newStreak, streak.longestStreak),
        lastActiveDate: today,
      })
      .where(eq(userStreaks.userId, userId));
  } else {
    // Missed day(s) - check for streak freeze
    const daysMissed = dateDiff(streak.lastActiveDate, today) - 1;

    if (daysMissed <= streak.streakFreezes) {
      // Use freeze(s)
      const newStreak = streak.currentStreak + 1; // +1 for today (frozen days don't add)
      await db.update(userStreaks)
        .set({
          currentStreak: newStreak,
          longestStreak: Math.max(newStreak, streak.longestStreak),
          lastActiveDate: today,
          streakFreezes: streak.streakFreezes - daysMissed,
          freezesUsedThisMonth: streak.freezesUsedThisMonth + daysMissed,
        })
        .where(eq(userStreaks.userId, userId));
    } else {
      // Streak broken
      await db.update(userStreaks)
        .set({
          currentStreak: 1, // Restart at 1 (today counts)
          lastActiveDate: today,
        })
        .where(eq(userStreaks.userId, userId));
    }
  }
}
```

### 5.5 XP Award Flow

```
User completes action (lesson, exercise, conversation, etc.)
  |
  v
1. Insert into xp_events (append-only ledger)
  |
  v
2. Upsert daily_activity for user's today
   - Increment xp_earned
   - Increment relevant counters (lessons, exercises, etc.)
   - Check if xp_earned >= dailyGoalXp, set goalMet = true
  |
  v
3. Update user_xp_summary
   - Add XP to total
   - Recalculate level if threshold crossed
  |
  v
4. If goalMet just flipped to true:
   - Award "daily_goal_met" XP (10 XP) -- insert another xp_event
   - Call updateStreak()
   - Check for streak milestone badges
  |
  v
5. Return updated XP state to client for immediate UI update
```

### 7.3 Timezone Handling

**Option A: `date-fns-tz` (recommended)**
- Small addition to existing `date-fns` ecosystem
- `formatInTimeZone(date, timezone, format)` is the exact function needed
- 3KB gzipped

**Option B: Native `Intl.DateTimeFormat`**
- Zero dependency, uses browser/Node built-in API
- Slightly more verbose but avoids any new package
- Available in Node 16+ (the project uses Next.js 16 with Node 20+)

**Recommendation:** Use native `Intl.DateTimeFormat` to avoid adding a dependency. The timezone operations needed are simple:

```typescript
function getUserLocalDate(timezone: string): string {
  return new Intl.DateTimeFormat('en-CA', {
    timeZone: timezone,
    year: 'numeric',
    month: '2-digit',
    day: '2-digit'
  }).format(new Date());
  // Returns 'YYYY-MM-DD' format
}
```

### 7.4 Custom Activity Rings vs Library

**Recommendation: Build custom.** The `react-activity-rings` library is:
- Last updated 2023
- Limited customization (can't control animation timing, glow effects, completion celebrations)
- 120 weekly downloads (low adoption)

Building custom with `framer-motion` + SVG is straightforward (~100 lines for the core component) and gives full control over the premium look and feel.

## 8. Roadmap Implications

### 8.1 Suggested Phase Structure

Based on this research, the XP/streak/goal system should be built in this order:

1. **Schema & XP Engine** (data layer first)
   - Create all 5 database tables (xp_events, daily_activity, user_streaks, user_xp_summary, user_badges)
   - Add `timezone` and `dailyGoalXp` columns to users table
   - Build core XP award service (the flow from section 5.5)
   - Build streak calculation algorithm
   - Avoids: Building UI before the data model is solid

2. **Daily Goal & Streak Tracking** (settings + tracking)
   - Daily goal selection in student settings page
   - Streak freeze management
   - Goal checking (did user meet today's goal?)
   - Wire XP awards into existing lesson completion, practice completion, conversation endpoints

3. **Activity Rings & XP Display** (visual layer)
   - Custom ActivityRings component with framer-motion
   - XP float animation component
   - Integration into student dashboard
   - Daily/weekly XP summary display

4. **Celebrations & Badges** (reward layer)
   - Confetti integration for milestone moments
   - Badge system with unlock animations
   - Level-up notifications
   - Streak milestone celebrations

5. **Progress Dashboard** (comprehensive view)
   - XP timeline chart (weekly/monthly)
   - Badge collection display
   - Streak history
   - Level progression visualization

### 8.2 Dependencies on Other v5.0 Work

- **App shell/sidebar** should come BEFORE XP display (rings need a consistent location in the layout)
- **Student settings page** should come BEFORE or WITH daily goal selection
- **Celebrations** should come BEFORE or WITH XP celebrations (shared confetti infrastructure)

### 8.3 Research Flags for Phase Planning

| Topic | Flag | Reason |
|-------|------|--------|
| Timezone edge cases | MEDIUM risk | Daylight saving transitions can cause a "day" to be 23 or 25 hours |
| XP retroactive award | MEDIUM risk | Should existing lesson completions be retroactively awarded XP? Needs decision. |
| Activity ring performance | LOW risk | SVG animations with framer-motion are well-established |
| Streak freeze UX | LOW risk | Simple toggle with clear state display |

### 8.4 Out of Scope for v5.0

- **Leaderboards:** Explicitly excluded per project requirements. Premium adult learners prefer personal progress.
- **XP marketplace / currency:** XP is a measure of effort, not something to spend. No shop, no cosmetics.
- **Adaptive daily goals:** Defer to v5.1+. MVP ships with manual goal selection only.
- **Spaced repetition tied to XP:** Deferred per project requirements.
