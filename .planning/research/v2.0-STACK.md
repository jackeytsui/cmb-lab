# Technology Stack: v2.0 GoHighLevel CRM Integration & Student Management

**Project:** CantoMando Blueprint LMS
**Milestone:** v2.0 CRM-LMS Integration
**Researched:** 2026-01-31
**Confidence:** HIGH (verified via npm registry, official GHL docs, and SDK repository)

---

## Executive Summary

The v2.0 milestone requires integrating GoHighLevel (GHL) as the CRM backbone for student management. The LMS already has the enrollment webhook pattern (`/api/webhooks/enroll`) that receives data from external systems -- the v2.0 work extends this to bidirectional sync with GHL's V2 API.

**Key decisions:**

- **GHL SDK:** Use official `@gohighlevel/api-client` v2.2.2 with Private Integration Token (NOT OAuth app, NOT API v1)
- **Sync strategy:** Event-driven with n8n as middleware (NOT polling, NOT direct bidirectional real-time sync)
- **Data ownership:** GHL owns contact/marketing data, LMS owns learning data. Sync only the intersection.
- **Queue:** Upstash Redis (already in stack) for rate-limited GHL API calls (NOT BullMQ, NOT new infrastructure)
- **No new infrastructure:** Everything integrates with existing Next.js 16 + Neon Postgres + Drizzle + Upstash stack

## Recommended Stack Additions

### 1. GoHighLevel API Client

| Technology | Version | Purpose | Confidence |
|------------|---------|---------|------------|
| `@gohighlevel/api-client` | 2.2.2 | Official TypeScript SDK for GHL V2 API | HIGH |

**Why the official SDK:**
1. **Maintained by GoHighLevel** -- not a community wrapper that can go stale
2. **TypeScript-first** -- typed endpoints for contacts, tags, custom fields, webhooks
3. **Built-in token management** -- handles Private Integration Token auth natively
4. **Webhook signature verification** -- validates incoming GHL webhook payloads
5. **All V2 endpoints covered** -- contacts CRUD, tags, custom fields, search, bulk operations

**Why NOT alternatives:**

| Alternative | Why Not |
|-------------|---------|
| `gohl` (M2KDevelopments) | Community-maintained, no guarantee of V2 parity |
| `ghlSDK` (ModernTyrTech) | Unofficial, smaller community, update lag |
| Raw `fetch` calls | Loses type safety, must handle auth/retry/rate-limits manually |
| API v1 | End-of-life, no support, missing V2 features |

**Authentication: Private Integration Token (PIT)**

Use a PIT instead of OAuth because:
- This is a **single-tenant integration** (one GHL sub-account to one LMS instance)
- No need for OAuth marketplace app registration
- Static token, simpler setup -- just set in env vars
- Supports all V2 API scopes needed (contacts, tags, custom fields, webhooks)
- Token rotation recommended every 90 days (manual process)

```typescript
// lib/ghl/client.ts
import HighLevel from '@gohighlevel/api-client';

export const ghl = new HighLevel({
  privateIntegrationToken: process.env.GHL_PRIVATE_INTEGRATION_TOKEN!,
});

// Usage: contacts
const contact = await ghl.contacts.getContact({
  contactId: 'abc-123',
}, { headers: { locationId: process.env.GHL_LOCATION_ID! } });
```

### 2. GHL API Rate Limit Management

| Technology | Version | Purpose | Confidence |
|------------|---------|---------|------------|
| `@upstash/ratelimit` | 2.0.8 (already installed) | Rate-limit outbound GHL API calls | HIGH |
| `@upstash/redis` | 1.36.1 (already installed) | Redis backend for rate limiter | HIGH |

**GHL Rate Limits:**
- **Burst:** 100 requests per 10 seconds per location
- **Daily:** 200,000 requests per day per location
- **Effectively:** ~600 requests/minute max

**Why reuse Upstash (already in stack):**
The project already uses `@upstash/ratelimit` for webhook and AI chat rate limiting (see `src/lib/rate-limit.ts`). Adding a GHL-specific limiter is a configuration change, not a new dependency.

**Why NOT BullMQ or separate queue:**
- BullMQ requires a separate Redis instance and a worker process -- operational overhead
- GHL API calls from the LMS are low-volume (contact syncs, tag updates, not bulk ETL)
- Expected volume: ~50-200 API calls/day (enrollment events, tag changes, milestone completions)
- At this volume, simple rate-limited `fetch` with retry is sufficient

**When to upgrade to BullMQ:** If daily GHL API calls exceed 10,000 or you need guaranteed delivery with retry persistence across server restarts. This is unlikely for a language tutoring LMS.

```typescript
// lib/ghl/rate-limit.ts
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";

const redis = Redis.fromEnv();

// GHL allows 100 requests per 10 seconds per location
// Stay under at 80 requests per 10 seconds for safety margin
export const ghlLimiter = new Ratelimit({
  redis,
  limiter: Ratelimit.slidingWindow(80, "10 s"),
  prefix: "ratelimit:ghl",
  analytics: true,
});

// Wrapper for rate-limited GHL API calls
export async function ghlApiCall<T>(
  fn: () => Promise<T>,
  retries = 3
): Promise<T> {
  const locationId = process.env.GHL_LOCATION_ID!;

  for (let attempt = 0; attempt < retries; attempt++) {
    const { success, reset } = await ghlLimiter.limit(locationId);

    if (!success) {
      const waitMs = Math.max(1000, reset - Date.now());
      await new Promise(resolve => setTimeout(resolve, waitMs));
      continue;
    }

    try {
      return await fn();
    } catch (error: any) {
      if (error?.status === 429 && attempt < retries - 1) {
        // GHL returned 429 despite our limiter -- back off harder
        await new Promise(resolve => setTimeout(resolve, (attempt + 1) * 5000));
        continue;
      }
      throw error;
    }
  }

  throw new Error("GHL API call failed after retries");
}
```

**Source:** [GHL Rate Limits](https://marketplace.gohighlevel.com/docs/oauth/Faqs/index.html), existing `src/lib/rate-limit.ts`

### 3. Webhook Event System (LMS to GHL)

| Technology | Version | Purpose | Confidence |
|------------|---------|---------|------------|
| n8n webhooks (existing) | -- | Middleware between LMS events and GHL | HIGH |
| Native `fetch` | -- | Fire-and-forget webhook calls | HIGH |

**Architecture: LMS fires webhooks to n8n, n8n routes to GHL**

This follows the existing pattern in the codebase (see `src/app/api/notify/coach-feedback/route.ts`) where the LMS sends events to n8n webhook URLs, and n8n handles the downstream integration.

**Why n8n as middleware (NOT direct LMS-to-GHL):**

1. **Already in the stack** -- n8n Cloud is configured and running workflows for this project
2. **Visual workflow editing** -- coaches/admins can adjust GHL automation logic without code deploys
3. **Error handling** -- n8n has built-in retry, error paths, and execution history
4. **Decoupling** -- LMS doesn't need GHL credentials for outbound events; n8n manages the connection
5. **Flexibility** -- same event can trigger GHL update AND email AND Slack notification

**Event types the LMS should fire:**

| Event | Trigger | GHL Action |
|-------|---------|------------|
| `student.milestone_completed` | Course/module completion | Add tag, update custom field |
| `student.inactive` | No activity for X days | Add tag, trigger re-engagement workflow |
| `student.feedback_received` | Coach sends feedback | Update contact note |
| `student.tag_added` | Coach adds tag in LMS | Sync tag to GHL contact |
| `student.tag_removed` | Coach removes tag in LMS | Remove tag from GHL contact |
| `student.enrolled` | New enrollment | Already exists via enroll webhook |

**Webhook payload pattern:**
```typescript
// lib/ghl/events.ts
interface LmsEvent {
  type: string;
  timestamp: string;
  data: Record<string, unknown>;
}

export async function fireLmsEvent(event: LmsEvent): Promise<void> {
  const webhookUrl = process.env.N8N_GHL_SYNC_WEBHOOK_URL;
  if (!webhookUrl) {
    console.warn(`GHL sync webhook not configured. Event skipped: ${event.type}`);
    return;
  }

  try {
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), 10000);

    await fetch(webhookUrl, {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        ...(process.env.N8N_WEBHOOK_AUTH_HEADER && {
          Authorization: process.env.N8N_WEBHOOK_AUTH_HEADER,
        }),
      },
      body: JSON.stringify(event),
      signal: controller.signal,
    });

    clearTimeout(timeoutId);
  } catch (error) {
    // Fire-and-forget: log but don't block the user action
    console.error(`Failed to fire LMS event ${event.type}:`, error);
  }
}
```

**Source:** Existing pattern in `src/app/api/notify/coach-feedback/route.ts`, [GHL Inbound Webhooks](https://help.gohighlevel.com/support/solutions/articles/155000003147-workflow-trigger-inbound-webhook)

**Signature verification uses RSA (SHA256):**
```typescript
// app/api/webhooks/ghl/route.ts
import { createVerify } from "crypto";

const GHL_WEBHOOK_PUBLIC_KEY = process.env.GHL_WEBHOOK_PUBLIC_KEY!;

function verifyGhlWebhook(payload: string, signature: string): boolean {
  const verifier = createVerify("SHA256");
  verifier.update(payload);
  return verifier.verify(GHL_WEBHOOK_PUBLIC_KEY, signature, "base64");
}
```

### 5. Database Schema Additions

| Technology | Version | Purpose | Confidence |
|------------|---------|---------|------------|
| Drizzle ORM | 0.45.1 (already installed) | Schema for GHL sync state | HIGH |
| Neon Postgres | -- (already running) | Storage | HIGH |

**New tables needed:**

```typescript
// db/schema/ghl-sync.ts
import { pgTable, uuid, text, timestamp, jsonb, boolean, pgEnum } from "drizzle-orm/pg-core";
import { users } from "./users";

export const syncDirectionEnum = pgEnum("sync_direction", ["ghl_to_lms", "lms_to_ghl"]);

// Link LMS users to GHL contacts
export const ghlContactLinks = pgTable("ghl_contact_links", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }).unique(),
  ghlContactId: text("ghl_contact_id").notNull().unique(),
  ghlLocationId: text("ghl_location_id").notNull(),
  lastSyncedAt: timestamp("last_synced_at"),
  syncEnabled: boolean("sync_enabled").notNull().default(true),
  createdAt: timestamp("created_at").notNull().defaultNow(),
});

// Cache GHL tags locally (avoid API calls for display)
export const ghlTags = pgTable("ghl_tags", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  tagName: text("tag_name").notNull(),
  source: syncDirectionEnum("source").notNull(), // Who added: GHL or LMS
  syncedAt: timestamp("synced_at").notNull().defaultNow(),
});

// Cache GHL custom fields locally
export const ghlCustomFields = pgTable("ghl_custom_fields", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull().references(() => users.id, { onDelete: "cascade" }),
  fieldKey: text("field_key").notNull(),     // e.g., "timezone", "learning_goals"
  fieldValue: text("field_value"),
  lastSyncedAt: timestamp("last_synced_at").notNull().defaultNow(),
});

// Audit log for sync operations
export const ghlSyncLog = pgTable("ghl_sync_log", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").references(() => users.id),
  direction: syncDirectionEnum("direction").notNull(),
  eventType: text("event_type").notNull(),   // e.g., "tag_added", "contact_updated"
  payload: jsonb("payload"),
  success: boolean("success").notNull(),
  errorMessage: text("error_message"),
  createdAt: timestamp("created_at").notNull().defaultNow(),
});
```

**Schema rationale:**
- `ghl_contact_links`: Maps LMS user IDs to GHL contact IDs. This is the core link table. One-to-one relationship.
- `ghl_tags`: Local cache of GHL tags per user. Avoids API call every time a coach views a student profile. The `source` field tracks who added the tag (for conflict resolution).
- `ghl_custom_fields`: Local cache of GHL custom fields. Coaches see timezone, goals, etc. without API calls.
- `ghl_sync_log`: Audit trail for debugging sync issues. Essential for "why did this tag disappear?" investigations.

**What NOT to store locally:**
- GHL pipeline stages (display via deep-link, don't replicate)
- GHL email campaign data (not LMS's concern)
- GHL conversation history (view in GHL directly)
- GHL appointment data (not relevant to LMS)

### 6. User Schema Extension

The existing `users` table needs additional fields for GHL-synced data that coaches reference frequently:

```typescript
// Additions to existing users table
export const users = pgTable("users", {
  // ... existing fields ...

  // GHL-synced fields (cached for coach convenience)
  timezone: text("timezone"),                // From GHL custom field
  learningGoals: text("learning_goals"),     // From GHL custom field
  ghlProfileUrl: text("ghl_profile_url"),    // Deep-link to GHL contact page
  lastGhlSyncAt: timestamp("last_ghl_sync_at"),
});
```

**Why add to users table (not just ghl_custom_fields):**
These fields are accessed on every student profile view. Joining through `ghl_custom_fields` for every page load adds complexity. Keep frequently-accessed CRM data denormalized on the user record, with the `ghl_custom_fields` table for less-used fields.

## Sync Strategy: Event-Driven with n8n Middleware

### Architecture Overview

```
                    LMS (Next.js)                    n8n Cloud                   GoHighLevel

[Coach adds tag] -----> POST /n8n-webhook ---------> [GHL Sync Workflow] -------> GHL API: Add Tag
                                                          |
                                                          v
                                                     [Error? Retry]

[GHL tag change] <----- POST /api/webhooks/ghl <--- [n8n optional] <----------- GHL Webhook Event
       |
       v
[Update ghl_tags table]
[Update sync log]
```

### Why Event-Driven Over Polling

| Approach | Pros | Cons | Verdict |
|----------|------|------|---------|
| **Event-driven (webhooks)** | Real-time, low API usage, simple | Requires webhook endpoints | USE THIS |
| **Polling** | Simple to implement | Wastes API quota, delayed data, scales badly | AVOID |
| **Real-time bidirectional** | Instant consistency | Complex conflict resolution, race conditions | OVERKILL |

### Conflict Resolution Strategy

**Rule: Last-write-wins with source tracking.**

For tags (the most common bidirectional sync):
1. When a coach adds a tag in the LMS, it syncs to GHL (source: `lms_to_ghl`)
2. When a tag is added in GHL, it syncs to LMS (source: `ghl_to_lms`)
3. If the same tag exists from both sources, no conflict -- it's additive
4. Tag removal syncs bidirectionally, logged in `ghl_sync_log`

For custom fields (timezone, goals):
1. GHL is the source of truth for contact profile data
2. LMS caches but does NOT push custom field changes back to GHL
3. One-directional: GHL -> LMS only

**Why GHL owns contact data:**
GHL is the CRM. Marketing team manages contact profiles there. LMS should read and display, not compete for ownership. This eliminates 90% of conflict scenarios.

## What NOT to Add

| Technology | Why Not |
|------------|---------|
| BullMQ + Redis worker | Over-engineered for expected volume (~200 API calls/day). Upstash rate limiter is sufficient. |
| Zapier / Make.com | n8n Cloud is already in the stack and handles this exact use case. |
| GHL OAuth app | Single-tenant integration. PIT is simpler, no marketplace registration needed. |
| Separate sync service | Adds deployment complexity. Next.js API routes + n8n handle this cleanly. |
| Full GHL data replication | LMS should NOT mirror pipelines, campaigns, emails. Display via deep-link instead. |
| WebSocket for real-time sync | Unnecessary latency requirement. Event-driven with 1-5 second delay is fine for CRM data. |
| `p-queue` for API throttling | Adds dependency. Upstash rate limiter already provides this pattern. |

### Data Flow: Enrollment (Enhanced)

Current flow:
```
GHL Workflow -> n8n -> POST /api/webhooks/enroll -> Create Clerk user + LMS user + course access
```

Enhanced flow:
```
GHL Workflow -> n8n -> POST /api/webhooks/enroll (with ghlContactId in payload)
  -> Create Clerk user + LMS user + course access
  -> Create ghl_contact_links record
  -> Fetch GHL contact data (tags, custom fields, timezone)
  -> Cache in ghl_tags + ghl_custom_fields + users.timezone
```

The enrollment webhook already receives data from GHL (via n8n). The enhancement is to include the GHL contact ID in the payload and immediately cache CRM data.
