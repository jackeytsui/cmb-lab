<task type="auto">
  <name>Task 1: Add rate limiting to authenticated AI routes (chat, grade, grade-audio)</name>
  <files>
    src/app/api/chat/route.ts
    src/app/api/grade/route.ts
    src/app/api/grade-audio/route.ts
  </files>
  <action>
    For each of the three authenticated routes, add rate limiting AFTER the auth check but BEFORE any processing:

    **1. src/app/api/chat/route.ts (POST handler)**
    - Add imports: `import { aiChatLimiter, aiChatLimiterElevated, rateLimitResponse, selectLimiter } from "@/lib/rate-limit";`
    - After the existing `const { userId } = await auth()` and 401 check, add:
      ```
      // Rate limiting
      const { sessionClaims } = await auth();  // OR destructure sessionClaims alongside userId if not already
      const role = (sessionClaims?.metadata?.role as string) || "student";
      const limiter = selectLimiter(role, aiChatLimiter, aiChatLimiterElevated);
      const rl = await limiter.limit(userId);
      if (!rl.success) {
        return rateLimitResponse(rl);
      }
      ```
    - IMPORTANT: The chat route uses `auth()` already -- destructure `sessionClaims` alongside `userId` in the same call. Do NOT call `auth()` twice.

    **2. src/app/api/grade/route.ts (POST handler)**
    - Add imports: `import { gradingLimiter, gradingLimiterElevated, rateLimitResponse, selectLimiter } from "@/lib/rate-limit";`
    - After auth check, add rate limit check using `gradingLimiter`/`gradingLimiterElevated`
    - Same pattern: get role from sessionClaims, selectLimiter, limiter.limit(userId), check success

    **3. src/app/api/grade-audio/route.ts (POST handler)**
    - Same pattern as grade/route.ts -- uses `gradingLimiter`/`gradingLimiterElevated`
    - Both grading endpoints share the same limiter (same prefix "ratelimit:grade") so they share the rate limit budget

    For all three routes:
    - The rate limit check must be BEFORE any request body parsing (don't waste compute on rate-limited requests)
    - Do NOT move the existing auth check -- rate limiting goes after auth (we need userId as the identifier)
    - Ensure `sessionClaims` is destructured from the same `auth()` call as `userId` (avoid double auth calls)
  </action>
  <verify>
    Run `npm run build` -- all three routes compile without errors. Grep for `rateLimitResponse` in all three files to confirm integration.
  </verify>
  <done>All three authenticated routes import from rate-limit.ts and check rate limits before processing requests. Chat uses aiChatLimiter (20/min), grade and grade-audio both use gradingLimiter (10/min). All select elevated limiter for coach/admin roles.</done>
</task>

<task type="auto">
  <name>Task 2: Add IP-based rate limiting to webhook enrollment route</name>
  <files>src/app/api/webhooks/enroll/route.ts</files>
  <action>
    The enrollment webhook is unauthenticated (uses secret header, not Clerk). Rate limit by IP address.

    - Add imports: `import { webhookLimiter, rateLimitResponse, getClientIp } from "@/lib/rate-limit";`
    - At the VERY TOP of the POST handler (BEFORE secret verification), add:
      ```
      // Rate limit by IP (before any processing)
      const ip = getClientIp(req);
      const rl = await webhookLimiter.limit(ip);
      if (!rl.success) {
        return rateLimitResponse(rl);
      }
      ```
    - Rate limiting must be BEFORE the webhook secret check -- a brute-force attacker trying to guess the secret should be rate limited too.
    - The request parameter is `req: NextRequest` -- pass it to `getClientIp()` (the function accepts `Request`, and `NextRequest` extends it).
  </action>
  <verify>
    Run `npm run build` -- webhook route compiles. Grep for `webhookLimiter` and `getClientIp` in the file.
  </verify>
  <done>Enrollment webhook rate limits by IP at 10 requests/min. Rate check happens before secret verification. Returns 429 with Retry-After on limit exceeded.</done>
