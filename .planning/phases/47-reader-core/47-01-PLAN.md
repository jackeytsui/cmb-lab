**src/lib/segmenter.ts** — Intl.Segmenter wrapper:
- Export `WordSegment` interface: `{ text: string; index: number; isWordLike: boolean }`
- Create a module-level `Intl.Segmenter('zh', { granularity: 'word' })` instance (create once, reuse)

**src/lib/tone-sandhi.ts** — Third-tone sandhi for Mandarin pinyin:
- Import `pinyin` and `convert` from `pinyin-pro`
- Export `applyThirdToneSandhi(text: string): string[]` that:
  1. Gets per-syllable pinyin with `pinyin(text, { toneType: 'num', type: 'array' })`
  2. Gets tone numbers with `pinyin(text, { pattern: 'num', type: 'array' }).map(Number)`
  3. Walks right-to-left: when tone 3 precedes tone 3, change first to tone 2 by replacing trailing `3` with `2` in the numbered pinyin string
  4. Converts each numbered pinyin back to tone marks via `convert(s)` from pinyin-pro
  5. Returns array of tone-mark pinyin syllables
- Also export `getPinyinWithSandhi(text: string): string` convenience function that joins the array with spaces
- IMPORTANT: pinyin-pro's built-in `toneSandhi` option only handles 一 and 不. The third-tone 3+3 rule must be implemented manually. Verified: `pinyin('你好')` returns `nǐ hǎo` not `ní hǎo`.

**src/lib/chinese-convert.ts** — opencc-js wrapper with lazy initialization:
- Export `ScriptMode` type: `'original' | 'simplified' | 'traditional'`
- Use `dynamic import('opencc-js')` to avoid bundling 2MB dictionary data eagerly
- Memoize converters in module-level variables (`toSimplifiedFn`, `toTraditionalFn`)
- Export `async function convertScript(text: string, from: ScriptMode, to: ScriptMode): Promise<string>`:
  - If `from === to` or `to === 'original'`, return text unchanged
  - For `to === 'simplified'`: lazy-load opencc-js, create `Converter({ from: 'hk', to: 'cn' })`, memoize, convert
  - For `to === 'traditional'`: lazy-load opencc-js, create `Converter({ from: 'cn', to: 'hk' })`, memoize, convert
  - Use HK variant (`hk`) not generic traditional (`t`) — per research, this project focuses on Cantonese/Hong Kong Chinese

  - Return spread `...preferences` plus the three setters

**src/app/api/reader/import/route.ts** — POST endpoint for file import:
- Import `auth` from `@clerk/nextjs/server` for authentication
- Import `NextRequest`, `NextResponse` from `next/server`
- Import `extractTextFromPdf` from `@/lib/chunking`
- Export `async function POST(request: NextRequest)`:
  1. Auth check: `const { userId } = await auth(); if (!userId) return 401`
  2. Parse multipart: `const formData = await request.formData()`
  3. Get file: `const file = formData.get('file') as File | null; if (!file) return 400`
  4. Get file type from formData or infer from file.name extension
  5. For PDF files (`.pdf`):
     - `const buffer = Buffer.from(await file.arrayBuffer())`
     - `const text = await extractTextFromPdf(buffer)`
     - Validate extraction: check `text.trim()` is non-empty and contains CJK chars via `/[\u4e00-\u9fff]/`
     - If empty or no CJK: return 422 with message "Could not extract Chinese text from this PDF. The PDF may contain scanned images instead of text."
  6. For text files (`.txt` or other):
     - `const arrayBuffer = await file.arrayBuffer()`
     - First try UTF-8: `new TextDecoder('utf-8').decode(new Uint8Array(arrayBuffer))`
     - Check if decoded text contains CJK characters
     - If no CJK found, try encoding detection:
       - `import jschardet from 'jschardet'`
       - `jschardet.detect(Buffer.from(arrayBuffer))`
       - Map encoding names: GB2312→gbk, GBK→gbk, GB18030→gb18030, Big5→big5, others→utf-8
       - If confidence < 0.5, fall back to utf-8
       - Decode with detected encoding
     - If still no CJK after re-decode: return 422 with message "No Chinese text detected in file."
  7. Enforce 20,000 character soft limit: if text.length > 20000, truncate and add a `truncated: true` flag in response
  8. Return `NextResponse.json({ text, encoding, truncated })`
  - Wrap entire handler in try/catch, return 500 on unexpected errors
  </action>
