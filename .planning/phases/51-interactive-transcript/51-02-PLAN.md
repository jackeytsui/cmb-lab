  <action>
Create `src/hooks/useVideoSync.ts` with the following implementation. This is the core sync engine -- it polls the YouTube player's current time and determines which caption is active.

**Interface:**
```typescript
interface CaptionLine {
  startMs: number;
  endMs: number;
  text: string;
  sequence: number;
}
```

**Binary search function** (`findActiveCaptionIndex`):
- Takes `captions: CaptionLine[]` and `currentMs: number`
- Guard: if `captions.length === 0` return -1
- Binary search for the last caption where `startMs <= currentMs`
- Verify the found caption's `endMs > currentMs` (not in a gap)
- Return the index, or -1 if in a gap or before first caption
- This is O(log n) -- critical for 200+ caption videos (per research anti-patterns)

**Hook** (`useVideoSync`):
- Input: `captions: CaptionLine[]`
- Internal refs: `playerRef` (YT.Player | null), `currentTimeRef` (number), `intervalRef` (ReturnType<typeof setInterval> | null), `isPlayingRef` (boolean)
- State: `activeCaptionIndex` (number, default -1)

**Polling logic:**
- `startPolling` (useCallback): If already polling (intervalRef.current exists), return early. Create a `setInterval` at 250ms that:
  1. Gets `playerRef.current` -- if null, skip
  2. Calls `player.getCurrentTime()` (synchronous on raw YT.Player, NOT the promisified wrapper)
  3. Converts to ms: `Math.round(timeS * 1000)`
  4. Stores in `currentTimeRef.current`
  5. Calls `findActiveCaptionIndex(captions, timeMs)`
  6. Updates state ONLY if changed: `setActiveCaptionIndex(prev => prev === newIndex ? prev : newIndex)` -- the functional updater avoids unnecessary re-renders per research Pitfall 1
- `stopPolling` (useCallback): If `intervalRef.current` exists, `clearInterval` and set to null

**Cleanup:** `useEffect(() => stopPolling, [stopPolling])` -- clears interval on unmount

**Event handlers:**
- `handlePlayerReady(player: YT.Player)`: Store in `playerRef.current`. This must capture `event.target` which is the raw synchronous YT.Player (NOT `getInternalPlayer()` which returns promisified methods -- see research Pitfall 3).
- `handlePlay()`: Set `isPlayingRef.current = true`, call `startPolling()`
- `handlePause()`: Set `isPlayingRef.current = false`, call `stopPolling()`
- `handleEnd()`: Set `isPlayingRef.current = false`, call `stopPolling()`, set `activeCaptionIndex` to -1

**Seek function:**
- `seekToCaption(index: number)`: Guard against null player, invalid index. Call `player.seekTo(captions[index].startMs / 1000, true)`. Immediately set `activeCaptionIndex(index)` for instant UI feedback (don't wait for next poll tick). If video is not playing (`!isPlayingRef.current`), call `player.playVideo()` to auto-play on seek.

**Return object:**
```typescript
return {
  activeCaptionIndex,
  handlePlayerReady,
  handlePlay,
  handlePause,
  handleEnd,
  seekToCaption,
};
```

**IMPORTANT anti-patterns to avoid (from research):**
- Do NOT store currentTime in React state (causes 4+ re-renders/sec)
- Do NOT use requestAnimationFrame (fires at 60fps, overkill for sentence-level sync)
- Do NOT use linear scan for active caption (O(n) vs O(log n) binary search)
- Do NOT poll when video is not playing (wastes CPU)
- Do NOT use `getInternalPlayer()` -- it wraps methods in Promises

Export: `export function useVideoSync(captions: CaptionLine[])`

For the YT.Player type, use `import type { YouTubePlayer as YTPlayer } from "react-youtube"` to get the correct type (this is what YouTubePlayer.tsx already uses).
