**Create `src/lib/tags.ts`** — Tag CRUD service with these functions:

- `createTag(data: { name, color, type?, description?, createdBy })` — inserts into tags table, returns created tag
- `getTags(filters?: { type?: "coach" | "system" })` — returns all tags, optionally filtered by type
- `updateTag(tagId: string, data: { name?, color?, description? })` — updates tag, returns updated
- `deleteTag(tagId: string)` — deletes tag (cascades to studentTags and autoTagRules)
- `assignTag(userId: string, tagId: string, assignedBy?: string, options?: { source?: "api" | "webhook" | "system" })` — inserts into studentTags with ON CONFLICT DO NOTHING (idempotent). The `source` parameter controls sync behavior: `"api"` (default) triggers outbound GHL sync, `"webhook"` skips outbound sync (prevents echo loops), `"system"` triggers outbound sync. Returns `{ assigned: boolean, tag: Tag }` so callers know if a new assignment was made.
- `removeTag(userId: string, tagId: string, options?: { source?: "api" | "webhook" | "system" })` — deletes from studentTags. Same `source` semantics as assignTag.
- `getStudentTags(userId: string)` — returns all tags for a student (join studentTags with tags)
- `getStudentsWithTag(tagId: string)` — returns all user IDs that have a given tag

Use Drizzle `db.insert().onConflictDoNothing()` for assignTag to handle the unique constraint gracefully.

**Create `src/app/api/admin/tags/route.ts`**:
- `GET` — calls `getTags()`, returns JSON array. Accepts optional `?type=coach|system` query param.
- `POST` — validates body with Zod (`{ name: string, color: string, type?: "coach" | "system", description?: string }`), calls `createTag()`. Requires admin/coach role (check auth via Clerk `auth()`).

**Create `src/app/api/admin/tags/[tagId]/route.ts`**:
- `PATCH` — validates body, calls `updateTag()`. Requires admin/coach.
- `DELETE` — calls `deleteTag()`. Requires admin/coach.

**Create `src/app/api/students/[studentId]/tags/route.ts`**:
- `GET` — calls `getStudentTags(studentId)`, returns tags array.
- `POST` — body `{ tagId: string }`, calls `assignTag(studentId, tagId, currentUserId, { source: "api" })`. The `source: "api"` flag signals that this assignment originated from the LMS UI and should trigger outbound GHL sync (Plan 02 will wire this). Requires admin/coach.
- `DELETE` — body `{ tagId: string }`, calls `removeTag(studentId, tagId, { source: "api" })`. Same `source: "api"` flag. Requires admin/coach.

**Important:** The `assignTag` and `removeTag` functions return the tag data so that Plan 02's sync layer can determine the tag name for GHL sync. The API routes call sync fire-and-forget after successful assignment. This wiring is completed in Plan 02 (which modifies this route to import and call `syncTagToGhl`).

**Create `src/app/api/admin/auto-tag-rules/route.ts`**:
- `GET` — returns all auto-tag rules with their associated tag info (join with tags table).
- `POST` — validates body with Zod (`{ tagId, conditionType, conditionValue }`), inserts new rule.
- `PATCH` — body `{ ruleId, isActive: boolean }`, toggles rule active state.
- `DELETE` — body `{ ruleId }`, deletes rule.

All routes follow existing project patterns: Clerk `auth()` for authentication, role check for admin/coach, Zod validation, try/catch with proper error responses.
