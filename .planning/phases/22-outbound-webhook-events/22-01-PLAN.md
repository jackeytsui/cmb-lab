  <name>Task 1: Create WebhookDispatcher service</name>
  <files>src/lib/ghl/webhooks.ts</files>
  <action>
Create `src/lib/ghl/webhooks.ts` with the following components:

**Types:**
- `WebhookEventType` union: `"module.completed" | "course.completed" | "lesson.milestone" | "student.inactive" | "feedback.sent"`
- `WebhookPayload` interface with `eventType`, `timestamp` (ISO 8601), `student` object (`email`, `name`, `userId`, `ghlContactId`), and `context` (Record<string, unknown>)
- Context-specific interfaces: `ModuleCompletedContext`, `CourseCompletedContext`, `StudentInactiveContext`, `FeedbackSentContext`, `MilestoneLessonContext`

**Core function - `dispatchWebhook(params)`:**
- Accepts: `{ userId, userEmail?, eventType, entityType, entityId, context }` (context is the event-specific data; `userEmail` is an optional string -- if provided, skip the users table lookup for email and use this value directly; this allows callers like the feedback route that already have the email loaded to avoid a redundant DB query)
- Step 1: Check for duplicate -- query `sync_events` for a recent (within 1 hour) event matching `(entityId, eventType, direction='outbound', status IN ('completed', 'pending'))`. If found, return early (skip).
- Step 2: Resolve GHL contact -- call `getGhlContactId(userId)`. If null, attempt `findOrLinkContact(userId, email)` (look up user email from users table). If still no contact, log a sync_event with status "completed" and payload `{ skipped: true, reason: "no_ghl_contact" }`, then return.
- Step 3: Build the payload -- look up user email/name from users table, construct `WebhookPayload` object.
- Step 4: Log sync_event with status "pending" via `logSyncEvent()`.
- Step 5: Attempt delivery via `deliverWebhook(payload, ghlContactId)`.
- Step 6: If delivery succeeds, call `markEventCompleted(eventId)`. If fails, call `markEventFailed(eventId, errorMessage)`.

**Delivery function - `deliverWebhook(payload, ghlContactId)`:**
- Uses `markOutboundChange()` for echo detection BEFORE calling GHL API.
- Adds a tag to the GHL contact via `ghlClient.post(\`/contacts/\${ghlContactId}/tags\`, { tags: [tagName] })`.
  - Tag naming convention: `lms:${eventType}` (e.g., `lms:module.completed`, `lms:course.completed`).
- If `GHL_WEBHOOK_URL` env var is set, also POST the full payload to that URL via `ghlClient.post()` or plain `fetch()` (this is a GHL incoming webhook URL, not the GHL contacts API).
- Throws on failure so the caller can catch and mark as failed.

**Re-delivery function - `deliverWebhookFromEvent(event: SyncEvent)`:**
- Reconstructs the delivery from a stored sync_event record (used by cron retry).
- Reads `payload` and `ghlContactId` from the event, calls `deliverWebhook()`.
- Export this for use by the cron route.

Important: Do NOT block on errors from echo detection or tag addition -- these are best-effort. The sync_event logging is the source of truth for retry.

<task type="auto">
  <name>Task 2: Create MilestoneDetector service</name>
  <files>src/lib/ghl/milestones.ts</files>
  <action>
Create `src/lib/ghl/milestones.ts` with the following components:

**`checkModuleCompletion(userId, moduleId)` -> boolean:**
- Model after `checkCourseCompletion()` in `src/lib/certificates.ts`.
- Query 1: Get all non-deleted lessons in the module (`lessons` where `moduleId` matches and `deletedAt` is null).
- Query 2: Get completed lessons for this user in this module (`lessonProgress` inner join `lessons`, where `userId` matches, `moduleId` matches, `deletedAt` is null, `completedAt` is not null).
- Return `completedLessons.length >= allLessons.length` (same pattern as certificates.ts).
- Return false if allLessons.length === 0.

**`getMilestoneLessonIds()` -> string[]:**
- Query `ghl_field_mappings` table for entries where `lmsConcept` starts with `'milestone_lesson:'` and `isActive = true`.
- Extract the lesson ID from the `lmsConcept` value (split on ':' and take the second part).
- Return array of lesson IDs that are configured as milestones.
- Cache the result in a module-level variable with a 5-minute TTL (simple in-memory cache) to avoid querying on every progress update.

**`detectAndDispatchMilestones(userId, lessonId)` -> void:**
- This is the main entry point called from the progress route after a lesson is newly completed.
- Step 1: Look up the lesson with its module and course (`lessons` join `modules` join `courses`).
- Step 2: Check if `lessonId` is in the milestone lesson list (`getMilestoneLessonIds()`). If yes, dispatch a `lesson.milestone` webhook:
  ```typescript
  await dispatchWebhook({
    userId,
    eventType: "lesson.milestone",
    entityType: "lesson",
    entityId: lessonId,
    context: { lessonTitle: lesson.title, moduleTitle: module.title, courseTitle: course.title },
  });
  ```
- Step 3: Call `checkModuleCompletion(userId, lesson.moduleId)`. If true, dispatch a `module.completed` webhook with module title, course title, total lessons in module.
- Step 4: Call `checkCourseCompletion(userId, lesson.module.courseId)` (import from certificates.ts). If true, dispatch a `course.completed` webhook with course title, total modules, total lessons, completion date.
- Each dispatch is fire-and-forget (catch errors and log them, don't throw).
- Import `dispatchWebhook` from `./webhooks.ts`.

Important: Use the existing `checkCourseCompletion` from `src/lib/certificates.ts` -- do NOT rewrite it. Only `checkModuleCompletion` is new.
  </action>
