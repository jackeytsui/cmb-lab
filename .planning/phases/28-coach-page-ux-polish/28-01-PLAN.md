<objective>
Add loading skeletons and server-side error handling to coach dashboard, submission detail, students page, and conversations list.

Purpose: Coach server-rendered pages currently show blank screens while loading and either show generic 404/error.tsx boundaries or silently fail when DB queries error. This plan creates loading.tsx skeletons and wraps DB queries in try/catch with inline ErrorAlert rendering, following the same patterns established in Phase 27 for student pages.

Note on pre-existing coverage: The SubmissionQueue client component (coach/page.tsx) already handles its own empty states (filter-aware "No submissions to review" / "All caught up!" / "No reviewed submissions yet") and error states (ErrorAlert with onRetry). This plan does NOT touch SubmissionQueue -- those requirements are already satisfied. This plan focuses on the server-rendered pages that lack loading skeletons and the server-component pages (submission detail, students) that lack try/catch around their DB queries.

Output: 3 new loading.tsx files and 2 modified page.tsx files with try/catch error handling.
</objective>

**Submission detail page (`src/app/(dashboard)/coach/submissions/[submissionId]/page.tsx`):**
- Add import: `import { ErrorAlert } from "@/components/ui/error-alert";`
- The current `getSubmission` function catches errors and returns null, then the page calls `notFound()`. The problem is that DB errors and genuinely missing submissions both result in the same 404 page.
- Refactor `getSubmission` to return a discriminated result: change the return type to `Promise<{ data: SubmissionDetail | null; error: boolean }>`.
  - Success path: `return { data: result, error: false };`
  - Empty result (no rows found): `return { data: null, error: false };`
  - Catch block: `console.error("Error fetching submission:", error); return { data: null, error: true };`
- Update the page component:
  - `const result = await getSubmission(submissionId, currentUser?.id ?? "");`
  - If `result.error` is true: return the page shell (min-h-screen, container, back link to /coach) with `<ErrorAlert variant="block" message="Unable to load submission details. Please try again or go back to the dashboard." />`
  - If `result.data` is null (and error is false): call `notFound();` as before (genuinely missing submission)
  - Otherwise destructure and render as before: `const { submission, student, lesson, interaction, feedback } = result.data;`
- Handle missing audio gracefully: the current code at line 309 uses `submission.type === "audio" && submission.audioData` to conditionally render the audio player, which means if audioData is null/empty for an audio-type submission, it falls through to the text response branch (which would show an empty or irrelevant text). Add an explicit third branch: when `submission.type === "audio" && !submission.audioData`, render: `<div className="p-4 bg-zinc-800/50 rounded-lg text-center"><p className="text-zinc-400">Audio recording is unavailable</p></div>`. Note: transcription is already conditionally rendered (line 318: `{submission.transcription && ...}`) so no changes needed for transcription edge cases -- it simply won't render when transcription is null.
