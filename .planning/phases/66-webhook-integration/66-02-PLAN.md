**2. Replace the `EnrollmentPayload` interface with a Zod schema:**
```typescript
import { z } from "zod";

const enrollmentSchema = z.object({
  email: z.string().email(),
  name: z.string().optional(),
  // Legacy course enrollment (WEBHOOK-05)
  courseId: z.string().uuid().optional(),
  accessTier: z.enum(["preview", "full"]).default("full"),
  expiresAt: z.string().optional(),
  // RBAC role assignment (WEBHOOK-01, WEBHOOK-02)
  roleId: z.string().uuid().optional(),
  roleName: z.string().optional(),
  roleExpiresAt: z.string().optional(), // WEBHOOK-03
  // Idempotency (WEBHOOK-04)
  idempotencyKey: z.string().optional(),
}).refine(
  (d) => d.courseId || d.roleId || d.roleName,
  { message: "At least one of courseId, roleId, or roleName is required" }
);
```

**3. POST handler flow (in order):**

a. Rate limiting (keep existing -- unchanged)
b. Secret verification (keep existing -- unchanged)
c. Parse body with `enrollmentSchema.safeParse(body)`. Return 400 if invalid with `{ error: "Validation failed", details: result.error.flatten() }`.

d. **Idempotency check (WEBHOOK-04):**
   - Derive key: `data.idempotencyKey || \`enroll:${data.email}:${data.roleId || data.roleName || ''}:${data.courseId || ''}\``
   - Query `processedWebhooks` by idempotencyKey using `eq()`
   - If found: return `NextResponse.json({ success: true, message: "Already processed", ...existing.resultData }, { status: 200 })`

e. **Role resolution (WEBHOOK-01, WEBHOOK-02, WEBHOOK-06):**
   If `data.roleId` or `data.roleName` is present:
   - If `roleId`: query `roles` table by id with `isNull(roles.deletedAt)` filter
   - If `roleName` (and no roleId): query `roles` table using `ilike(roles.name, data.roleName)` AND `isNull(roles.deletedAt)`
   - If role not found (WEBHOOK-06):
     * Query all admin users: `db.select({ id: users.id }).from(users).where(eq(users.role, "admin"))`
     * For each admin, call `createNotification({ userId: admin.id, type: "system", category: "system", title: "Unknown role in webhook", body: \`Enrollment webhook referenced unknown role: "${data.roleId || data.roleName}". Student ${data.email} was NOT assigned a role.\`, linkUrl: "/admin/roles" })`
     * Return `NextResponse.json({ error: "Role not found", identifier: data.roleId || data.roleName }, { status: 404 })`
   - Store resolved roleId in a variable `resolvedRoleId`

f. **User find-or-create (keep existing logic -- unchanged):**
   Clerk lookup by email, create if not found, sync to DB users table. This block is identical to the current implementation.

g. **Course access grant (WEBHOOK-05 -- backward compatible):**
   If `data.courseId` is present:
   - Keep the ENTIRE existing courseId logic (verify course exists, check existing access, update or insert courseAccess). This is lines 42-113 of the current file, moved into a conditional block.
   - If course not found, return 404 (existing behavior).

h. **Role assignment (WEBHOOK-01, WEBHOOK-02, WEBHOOK-03):**
   If `resolvedRoleId` exists (role was found in step e):
   - Parse roleExpiresAt: `data.roleExpiresAt ? new Date(data.roleExpiresAt) : undefined`
   - Call `assignRole(dbUser.id, resolvedRoleId, null, roleExpiration)` -- null for assignedBy (Pitfall 1 fix from Plan 01)
   - Store the result

i. **Record processed webhook (WEBHOOK-04):**
   - PLACEMENT: This insert MUST be inside the try block, AFTER all mutations (course access grant in step g + role assignment in step h) have completed, and BEFORE the final return statement in step j. If an error occurs during mutations, the catch block fires and NO idempotency record is written -- allowing the webhook to be retried on the next call.
   - Insert into processedWebhooks: `{ idempotencyKey, source: "enrollment", eventType: resolvedRoleId ? "role_assignment" : "course_access", payload: data, result: "success", resultData: { userId: dbUser.id, roleAssignment: roleResult, courseId: data.courseId } }`
   - Use `.onConflictDoNothing()` for race condition safety (Research Pattern 2)
   - IMPORTANT: Record AFTER processing, not before (Anti-pattern from research). If recorded before mutations and the mutation fails, the webhook becomes permanently "processed" despite never completing -- defeating idempotency.

j. **Response:**
   Return 200 with `{ success: true, userId: dbUser.id, courseId: data.courseId, roleAssigned: resolvedRoleId ? true : undefined, roleAction: roleResult?.action }`

**4. Error handling:**
Keep existing try/catch with console.error and 500 response. The idempotency record (step 3.i) is inside the try block after mutations, so if any mutation throws, the catch block fires, no idempotency record is written, and the webhook can be safely retried.

**Key pitfalls to avoid (from research):**
- Do NOT require courseId (Pitfall 2) -- it is now optional
- Do NOT use `%` wildcards with ilike (Pitfall 3) -- bare ilike IS exact case-insensitive match
- DO include all discriminating fields in idempotency key (Pitfall 4)
- DO return 404 for unknown roles (Pitfall 5), not 200
- DO record processedWebhook AFTER success (Anti-pattern from research)
