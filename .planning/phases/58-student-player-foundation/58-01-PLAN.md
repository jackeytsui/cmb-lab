<objective>
Create the response storage schema (sessions + responses tables) and student-facing API routes for fetching thread data and submitting responses with persistent session tracking.

Purpose: Without DB tables for sessions and responses, the player has no persistence layer. Without a student-accessible GET route, only coaches can load thread data. This plan builds the data foundation that Plan 02's player UI will consume.

Output: Two new DB tables (video_thread_sessions, video_thread_responses), a student GET endpoint for thread data, and a refactored respond POST that creates sessions and stores responses.
</objective>

**video_thread_sessions table:**
- `id`: uuid, defaultRandom, primaryKey
- `threadId`: uuid, FK to video_threads.id (cascade delete), notNull
- `studentId`: uuid, FK to users.id (cascade delete), notNull
- `status`: text (use pgEnum "session_status" with values: "in_progress", "completed", "abandoned"), notNull, default "in_progress"

- `completedAt`: timestamp, nullable
- `lastStepId`: uuid, nullable (tracks where student is for future resume)
- Index on (threadId, studentId)

**video_thread_responses table:**
- `id`: uuid, defaultRandom, primaryKey
- `sessionId`: uuid, FK to video_thread_sessions.id (cascade delete), notNull
- `stepId`: uuid, FK to video_thread_steps.id (cascade delete), notNull
- `responseType`: responseTypeEnum (reuse existing), notNull
- `content`: text, nullable (text answer or selected option value)
- `metadata`: jsonb, nullable (for future expansion: timing data, etc.)
- `createdAt`: timestamp, notNull, defaultNow
- Index on (sessionId)
- Index on (stepId)

- videoThreadSessions: many-to-one with videoThreads, many-to-one with users, one-to-many with videoThreadResponses
- videoThreadResponses: many-to-one with videoThreadSessions, many-to-one with videoThreadSteps
- Update existing videoThreadsRelations to add `sessions: many(videoThreadSessions)`
- Update existing videoThreadStepsRelations to add `responses: many(videoThreadResponses)`

**Types:**
- Export VideoThreadSession, NewVideoThreadSession, VideoThreadResponse, NewVideoThreadResponse

**1. Create student-facing GET route at `src/app/api/video-threads/[threadId]/route.ts`:**

This is a NEW file (the existing GET is at `/api/admin/video-threads/[threadId]/route.ts` and requires coach role).

- Use Clerk auth to get the current user (import from `@/lib/auth` -- use `getCurrentUser` or `auth()` from `@clerk/nextjs/server`)
- Require authenticated user (return 401 if not logged in)
- Fetch thread with steps ordered by sortOrder ASC, include upload relation (for muxPlaybackId)
- Return `{ thread, steps }` (flatten steps out of the thread object for easier client consumption)
- Do NOT require coach/admin role -- any authenticated user can view published threads

**2. Refactor `src/app/api/video-threads/[threadId]/respond/route.ts`:**

The existing route does logic traversal but has NO session or response storage. Refactor it to:

- Import Clerk auth to get current user
- Require authenticated user (return 401 if not)
- Accept body: `{ stepId, sessionId?, response: { type: ResponseType, content: string } }`
- **Session handling:**
  - If `sessionId` is provided, validate it exists and belongs to this user + thread
  - If `sessionId` is NOT provided, create a new session: insert into videoThreadSessions with threadId, studentId, status "in_progress"
  - Return the sessionId in the response (client stores it for subsequent calls)
- **Response storage:**
  - Insert into videoThreadResponses: sessionId, stepId, responseType (from response.type), content (from response.content)
- **Next step resolution:**
  - Keep the existing logic traversal code (legacy logic, fallbackStepId, sortOrder fallback, recursive logic node resolution with visited set)
  - After resolving nextStepId, update session's lastStepId to the next step
  - If nextStepId is null (end of thread), update session status to "completed" and set completedAt
- Return: `{ nextStepId, sessionId, completed: boolean }`

Import the new tables from `@/db/schema`:
```typescript
import { videoThreadSteps, videoThreadSessions, videoThreadResponses } from "@/db/schema";
```

Use `auth()` from `@clerk/nextjs/server` to get userId, then look up internal user ID from users table.
