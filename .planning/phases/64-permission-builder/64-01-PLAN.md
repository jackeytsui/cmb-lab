**Schema changes in `src/db/schema/roles.ts`:**

1. Import `modules` and `lessons` from `./courses` (add to existing imports).

2. Add two nullable FK columns to the `roleCourses` table definition:
   - `moduleId: uuid("module_id").references(() => modules.id, { onDelete: "cascade" })` (nullable, no `.notNull()`)
   - `lessonId: uuid("lesson_id").references(() => lessons.id, { onDelete: "cascade" })` (nullable, no `.notNull()`)

3. Replace the existing unique index `role_courses_role_course_unique` on `(roleId, courseId)` with a new expression-based approach. Since Drizzle ORM does not natively support COALESCE in unique indexes, use `sql` to create a raw unique index. Add this import: `import { sql } from "drizzle-orm";`. Then replace the uniqueIndex line with:
   ```
   uniqueIndex("role_courses_unique").on(table.roleId, table.courseId, table.moduleId, table.lessonId)
   ```
   IMPORTANT: PostgreSQL NULL handling -- this standard unique index will NOT prevent duplicate NULL rows. To handle this, the API layer (Task 2) will use delete-then-insert pattern instead of relying on the DB unique constraint for NULL columns. Keep the index for non-null uniqueness and query performance.

4. Update the `roleCoursesRelations` to add two new relations:
   ```
   module: one(modules, { fields: [roleCourses.moduleId], references: [modules.id] }),
   lesson: one(lessons, { fields: [roleCourses.lessonId], references: [lessons.id] }),
   ```

5. Run the schema migration: `npm run db:push` (this project uses db:push, not drizzle-kit migrate). If db:push fails due to the existing unique index, use the Neon MCP tool to run direct SQL:
   - `DROP INDEX IF EXISTS role_courses_role_course_unique;`
   - Then retry `npm run db:push`.

**Permission resolver changes in `src/lib/permissions.ts`:**

6. Update the `PermissionSet` interface to add two new methods:
   ```typescript
   canAccessModule(moduleId: string): boolean;
   canAccessLesson(lessonId: string): boolean;
   ```

7. In `_resolvePermissions`, update the role-based course grants query (the first of the three parallel queries) to also select `moduleId` and `lessonId`:
   ```typescript
   .select({
     courseId: roleCourses.courseId,
     moduleId: roleCourses.moduleId,
     lessonId: roleCourses.lessonId,
     accessTier: roleCourses.accessTier,
   })
   ```

8. After building courseIdSet/featureSet/accessTierMap, add two new data structures for granular grants:
   ```typescript
   const moduleGrants = new Set<string>(); // moduleIds with module-level grants
   const lessonGrants = new Set<string>(); // lessonIds with lesson-level grants
   ```

9. Update the role-based course grants loop to categorize each grant:
   - Row with `moduleId === null && lessonId === null` -> course-level grant (existing behavior: add to courseIdSet)
   - Row with `moduleId !== null && lessonId === null` -> module-level grant (add courseId to courseIdSet, add moduleId to moduleGrants)
   - Row with `lessonId !== null` -> lesson-level grant (add courseId to courseIdSet, add moduleId to moduleGrants if present, add lessonId to lessonGrants)
   - Always call `mergeTier()` with the courseId for access tier tracking.

10. Update the returned PermissionSet object:
    - `canAccessCourse(courseId)` stays the same: `hasWildcardAccess || courseIdSet.has(courseId)` (returns true if ANY grant exists for that course, at any granularity level -- this is correct because courseId is always added to courseIdSet regardless of grant level)
    - Add `canAccessModule(moduleId)`: `hasWildcardAccess || moduleGrants.has(moduleId)`. Note: This checks if a module-level or lesson-level grant exists. For course-level wildcard grants, the caller should check `canAccessCourse()` first.
    - Add `canAccessLesson(lessonId)`: `hasWildcardAccess || lessonGrants.has(lessonId)`. Same note applies.

11. Update the existing standalone `canAccessModule()` and `canAccessLesson()` async helper functions at the bottom of the file. They currently do a DB lookup to find the parent course and check `canAccessCourse()`. Update them to ALSO check the new sync methods:
    - `canAccessModule`: Return true if `permissions.canAccessModule(moduleId)` OR if the parent course is accessible via `permissions.canAccessCourse(mod.courseId)`.
    - `canAccessLesson`: Return true if `permissions.canAccessLesson(lessonId)` OR if the parent module is accessible via `permissions.canAccessModule(lesson.module.id)` (need to also select moduleId in the query) OR if the parent course is accessible via `permissions.canAccessCourse(lesson.module.courseId)`.

12. Also expose `moduleGrants` and `lessonGrants` on the PermissionSet interface so callers can check set membership. Add to the interface:
    ```typescript
    moduleGrants: Set<string>;
    lessonGrants: Set<string>;
    ```

**Extend GET /api/admin/roles/[roleId] in `src/app/api/admin/roles/[roleId]/route.ts`:**

1. Add a `tree=true` query parameter check. When `?tree=true` is passed, the GET handler returns the full course tree + permission grants in addition to the role. This keeps the existing GET behavior unchanged for other callers (like the edit dialog).

2. When `tree=true`, after fetching the role, run three parallel queries:
   - All non-deleted courses ordered by sortOrder: `db.select().from(courses).where(isNull(courses.deletedAt)).orderBy(asc(courses.sortOrder))`
   - All non-deleted modules ordered by sortOrder (filter by courseIds using `inArray`)
   - All non-deleted lessons ordered by sortOrder (filter by moduleIds from above -- but since modules query runs in parallel, do a second round: fetch all lessons where moduleId is in the set of fetched module IDs. Alternatively, fetch ALL non-deleted lessons in one query, then filter in JS -- with <200 lessons this is fine.)

3. Also fetch in parallel:
   - `roleCourses` rows where `roleId` matches: `db.select().from(roleCourses).where(eq(roleCourses.roleId, roleId))`
   - `roleFeatures` rows where `roleId` matches: `db.select({ featureKey: roleFeatures.featureKey }).from(roleFeatures).where(eq(roleFeatures.roleId, roleId))`

4. Build the nested course tree in JS: for each course, attach its modules (filtered from module list), and for each module, attach its lessons (filtered from lesson list). Return shape:
   ```json
   {
     "role": { ... },
     "courseTree": [
       { "id": "...", "title": "...", "modules": [
         { "id": "...", "title": "...", "lessons": [
           { "id": "...", "title": "..." }
         ]}
       ]}
     ],
     "courseGrants": [
       { "courseId": "...", "moduleId": null, "lessonId": null }
     ],
     "featureGrants": ["ai_conversation", "practice_sets"]
   }
   ```

5. Import the needed schemas: `courses, modules, lessons` from `@/db/schema` and `asc, isNull, inArray` from `drizzle-orm`. Import `roleCourses, roleFeatures` from `@/db/schema`.

**Create PUT /api/admin/roles/[roleId]/courses in `src/app/api/admin/roles/[roleId]/courses/route.ts`:**

6. Create the file with a PUT handler. Auth: `hasMinimumRole("coach")`. Validate body with Zod:
   ```typescript
   const schema = z.object({
     courseId: z.string().uuid(),
     moduleId: z.string().uuid().nullable(),
     lessonId: z.string().uuid().nullable(),
     granted: z.boolean(),
   });
   ```

7. Extract `roleId` from params. Logic:
   - If `granted === false`:
     - If `moduleId === null && lessonId === null` (course-level revoke): Delete ALL roleCourses rows matching this `roleId + courseId` (any granularity). This handles "uncheck entire course."
     - If `moduleId !== null && lessonId === null` (module-level revoke): Delete ALL roleCourses rows matching `roleId + courseId + moduleId` (module-level and any lesson-level within that module).
     - If `lessonId !== null` (lesson-level revoke): Delete the specific row matching `roleId + courseId + moduleId + lessonId`.
   - If `granted === true`:
     - If `moduleId === null && lessonId === null` (course-level grant / "Select All"): First delete ALL existing rows for this `roleId + courseId`. Then insert ONE row with `moduleId: null, lessonId: null`. This collapses granular grants into a course-level wildcard.
     - If `moduleId !== null && lessonId === null` (module-level grant): First delete any existing course-level row for this `roleId + courseId` where `moduleId IS NULL` (because a course-level grant would make this redundant -- actually, do NOT delete course-level grants; they are broader). Instead: delete any existing row matching `roleId + courseId + moduleId` where `lessonId IS NULL`, then insert a new row. Also delete any lesson-level rows for this module (they are now redundant since the module-level grant covers all lessons).
     - If `lessonId !== null` (lesson-level grant): Delete any existing row matching `roleId + courseId + moduleId + lessonId` (dedup), then insert.

8. Use delete-then-insert pattern for all mutations to handle the NULL uniqueness issue. Return `{ success: true }` on 200.
