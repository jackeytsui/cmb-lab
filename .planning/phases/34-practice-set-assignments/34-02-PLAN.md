Create a GET endpoint that returns available assignment targets for the dialog's cascading selects.

**GET /api/admin/assignments/targets**

- `hasMinimumRole("coach")` guard
- Query params: `type` (required: 'course' | 'module' | 'lesson' | 'student' | 'tag'), `parentId` (optional, for cascading)
- Based on `type`:
  - `course`: Return all courses (where deletedAt IS NULL), ordered by title. Fields: `{ id, title }`
  - `module`: Require `parentId` (courseId). Return modules for that course (where deletedAt IS NULL), ordered by sortOrder. Fields: `{ id, title, courseId }`
  - `lesson`: Require `parentId` (moduleId). Return lessons for that module (where deletedAt IS NULL), ordered by sortOrder. Fields: `{ id, title, moduleId }`
  - `student`: Return all users with role 'student' (or all users), ordered by name/email. Fields: `{ id, name, email }`. Use: `db.select({ id: users.id, name: users.name, email: users.email }).from(users).orderBy(asc(users.name))`
  - `tag`: Return all tags, ordered by name. Fields: `{ id, name, color }`
- Return 200 with `{ targets: [...] }`
- If `type` is missing or invalid, return 400.
- If `parentId` required but missing, return 400.
  </action>

**AssignmentDialog.tsx** — "use client" component using AlertDialog (Radix) pattern:

Props: `{ practiceSetId: string; practiceSetTitle: string; open: boolean; onOpenChange: (open: boolean) => void }`

State:
- `targetType`: 'course' | 'module' | 'lesson' | 'student' | 'tag' | '' (initially '')
- `selectedCourseId`, `selectedModuleId`, `selectedTargetId`: string states for cascading
- `dueDate`: string (datetime-local input value, initially '')
- `targets`: array of fetched targets for current dropdown
- `assignments`: array of existing assignments for this set
- `isLoading`, `isCreating`, `error`: loading/error states

Behavior:
1. On open, fetch existing assignments: `GET /api/admin/assignments?practiceSetId=...`
2. When targetType changes, fetch targets: `GET /api/admin/assignments/targets?type=...`
3. For curriculum types, implement cascading:
   - 'course': Show course select. selectedTargetId = selected courseId.
   - 'module': Show course select, then module select (parentId=courseId). selectedTargetId = selected moduleId.
   - 'lesson': Show course select, then module select, then lesson select. selectedTargetId = selected lessonId.
   - 'student': Show student select (search by name/email). selectedTargetId = selected userId.
   - 'tag': Show tag select with color indicators. selectedTargetId = selected tagId.
4. "Assign" button: POST /api/admin/assignments with `{ practiceSetId, targetType, targetId: selectedTargetId, dueDate: dueDate || undefined }`
5. On success: refresh assignments list, clear form, show success toast or clear error.
6. On 409 (duplicate): show "Already assigned" message.
7. Due date: native `<input type="datetime-local" />` with label "Due date (optional)".

UI Layout (inside AlertDialog):
- Title: "Assign: {practiceSetTitle}"
- Section 1: Target Type radio/select (5 options: Course, Module, Lesson, Student, Tag)
- Section 2: Dynamic cascading selects based on type
- Section 3: Due date input
- Section 4: "Assign" button
- Divider
- Section 5: Existing Assignments (render AssignmentList)
- Footer: Close button

Use `<Select>` from shadcn for dropdowns. Use `<Dialog>` from shadcn (NOT AlertDialog — AlertDialog is for confirmations, Dialog is for forms). Import from `@/components/ui/dialog`.
