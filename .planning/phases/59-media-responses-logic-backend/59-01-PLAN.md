Create a POST endpoint at `/api/video-threads/[threadId]/upload-response` that handles two operations based on the request body `action` field:

**Action: "get-upload-url"**
- Authenticate via `getCurrentUser()` from `@/lib/auth` (same as the respond route -- student-level, no role gate)
- Verify the threadId exists in the database
- Call `mux.video.uploads.create()` with `cors_origin` from `NEXT_PUBLIC_APP_URL` and `playback_policy: ["public"]`, `encoding_tier: "baseline"`
- Insert a record into `videoUploads` table with `category: "response"`, `uploadedBy: user.id`, `status: "pending"`, filename from request body
- Return `{ uploadUrl, uploadId, dbUploadId }`

**Action: "check-status"**
- Same auth check (getCurrentUser)
- Accept `uploadId` (Mux upload ID) in the body
- Replicate the same polling logic from `/api/admin/mux/check-status/route.ts`: retrieve upload from Mux, check asset status, update `videoUploads` table, return `{ status, muxPlaybackId?, muxAssetId? }`
- Verify the upload record belongs to the requesting user (uploadedBy === user.id) to prevent students from polling other users' uploads

Use a single POST route with `action` discriminator to keep the API surface small (one route, two operations). Do NOT reuse the admin routes -- they require coach role and are under `/api/admin/`.

  <action>
Create a client component `StudentMediaRecorder` that supports both audio-only and video (webcam+audio) recording modes. This component will render INSIDE the player overlay (not full-screen), so it must be compact.

**Props interface:**
```typescript
interface StudentMediaRecorderProps {
  mode: "audio" | "video";  // Determines getUserMedia constraints
  threadId: string;          // For the upload API route
  onUploadComplete: (result: { muxPlaybackId: string; uploadId: string }) => void;
  onCancel: () => void;
  disabled?: boolean;
}
```

**Recording flow:**
1. On mount, call `navigator.mediaDevices.getUserMedia()`:
   - `mode === "audio"`: `{ audio: true, video: false }`
   - `mode === "video"`: `{ audio: true, video: true }`
2. Show live preview: for video mode, render `<video>` with srcObject; for audio mode, show an animated waveform indicator (simple CSS pulsing circle is fine)
3. Record button starts `new MediaRecorder(stream)` collecting chunks via `ondataavailable`
4. Stop button stops recording, creates Blob (`video/webm` for video, `audio/webm` for audio)
5. Show playback preview: `<video>` for video, `<audio>` for audio, with Retake and Upload buttons

**Upload flow (mirrors builder's VideoRecorder pattern):**
1. POST to `/api/video-threads/${threadId}/upload-response` with `{ action: "get-upload-url", filename: "response-{timestamp}.webm" }`
2. PUT blob to the returned `uploadUrl`
3. Poll `/api/video-threads/${threadId}/upload-response` with `{ action: "check-status", uploadId }` every 3s, max 20 polls
4. On `status === "ready"`, call `onUploadComplete({ muxPlaybackId, uploadId })`
5. On error or timeout, show error message with retry option

**UI states:** idle (showing preview/permissions), recording (with red indicator + timer), recorded (playback + retake/upload), uploading (spinner + progress text), error (message + retry)

**Styling:** Use Tailwind classes consistent with the player overlay aesthetic -- semi-transparent backgrounds, white text, rounded corners. Keep compact: the recorder sits inside the gradient overlay at the bottom of the video player.

Do NOT import from sonner (toast) -- use inline error display matching the player's error pattern. The builder's `VideoRecorder.tsx` uses toast but the player should be self-contained.
