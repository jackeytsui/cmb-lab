## Summary

Phase 44 is a housekeeping and foundation phase with five distinct work streams: (1) apply 4 pending database migrations plus practice tables to Neon, (2) commit untracked files, (3) install 4 new npm packages for the v6.0 reader feature, (4) add the shadcn popover component, and (5) define Drizzle schema for dictionary_entries, character_data, and saved_vocabulary tables with proper indexes.

The technical risk is low. All migrations are already generated as SQL files. The packages to install are well-established. The schema design is well-researched in prior v6.0 research docs (`.planning/research/v6-dictionary-data.md`). The main decision points are schema column choices for the dictionary tables and whether to use `serial` vs `uuid` primary keys (the codebase convention is `uuid` for all tables).

**Primary recommendation:** Execute in order -- migrations first (validates DB connection), then git commit, then npm install, then shadcn popover, then schema definition + migration generation. This order minimizes risk because each step validates the previous.

### Pattern 1: Schema File Convention (from existing codebase)

**What:** Each domain gets its own schema file with tables, relations, enums, and type exports. The barrel export in `index.ts` re-exports everything.

**When to use:** Always -- this is the established pattern across 21 existing schema files.

**Example:**
```typescript
// Source: Existing pattern in src/db/schema/practice.ts, xp.ts, etc.
import { pgTable, uuid, text, timestamp, integer, index } from "drizzle-orm/pg-core";
import { relations } from "drizzle-orm";
import { users } from "./users";

export const savedVocabulary = pgTable("saved_vocabulary", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  // ... columns
  createdAt: timestamp("created_at").notNull().defaultNow(),
}, (table) => [
  index("saved_vocabulary_user_id_idx").on(table.userId),
]);

// Relations
export const savedVocabularyRelations = relations(savedVocabulary, ({ one }) => ({
  user: one(users, {
    fields: [savedVocabulary.userId],
    references: [users.id],
  }),
}));

// Type inference
export type SavedVocabulary = typeof savedVocabulary.$inferSelect;
export type NewSavedVocabulary = typeof savedVocabulary.$inferInsert;
```

### Pattern 2: Index Convention (from existing codebase)

**What:** Every FK column gets a B-tree index. Composite indexes for common query patterns. Index names follow `tablename_columnname_idx` convention.

**When to use:** Always for FKs and commonly queried columns.

**Example:**
```typescript
// Source: Existing pattern from Phase 38 (42 FK indexes in migration 0008)
(table) => [
  index("dictionary_entries_traditional_idx").on(table.traditional),
  index("dictionary_entries_simplified_idx").on(table.simplified),
]
```

### Pattern 3: UUID Primary Keys (codebase convention)

**What:** All tables use `uuid("id").defaultRandom().primaryKey()` -- NOT `serial`.

**When to use:** Always. The entire codebase (21 tables) uses UUIDs. The dictionary tables should follow suit despite the v6.0 research doc suggesting `serial` -- consistency is more important than marginal performance.

**Exception:** The `character_data` table can use the character itself as a natural primary key (`varchar('character', { length: 1 }).primaryKey()`) since each character is unique and lookups are always by character. This is valid because the character IS the identity.

### Pattern 4: Text Array Columns

**What:** Drizzle supports Postgres text arrays via `.array()` notation.

**When to use:** For the dictionary `definitions` column (stores multiple English definitions per entry).

**Example:**
```typescript
// Source: Context7 /drizzle-team/drizzle-orm-docs
import { sql } from 'drizzle-orm';
import { text } from 'drizzle-orm/pg-core';

definitions: text('definitions').array().notNull().default(sql`'{}'::text[]`),
```

### Anti-Patterns to Avoid

- **Serial IDs for dictionary tables:** The research doc recommends `serial` but the codebase uses `uuid` everywhere. Using serial would create an inconsistency in query patterns across the app. Use UUID for `dictionary_entries` and `saved_vocabulary`. Exception: `character_data` uses the character itself as PK.
- **Splitting dictionary schema across multiple files:** Keep `dictionary_entries` and `character_data` in one file (`dictionary.ts`) since they are tightly coupled domain concepts. Keep `saved_vocabulary` in its own file since it's a user-facing feature.
- **Generating columns in Drizzle that Neon may not support:** The research doc suggests a `GENERATED ALWAYS AS` column for `is_single_char`. Drizzle ORM supports `generatedAlwaysAs()` but verify the generated SQL works on Neon before relying on it. A simpler approach: compute `isSingleChar` in application code or as a regular boolean column populated during seeding.
- **Running `db:push` instead of `db:migrate`:** The project uses migration files. Do NOT use `db:push` for schema changes. Always generate a migration with `db:generate` and apply with `db:migrate`. Exception: The practice tables note in STATE.md says "Run `npm run db:push` or apply migration" -- for consistency, use `db:migrate` with the existing generated migrations.

## Common Pitfalls

### Pitfall 1: Migration Order Matters

**What goes wrong:** Running migration 0009 (xp_events, daily_activity) before 0007 (dailyGoalXp column on users) would fail because 0009's daily_activity table references columns that 0007 adds.

**Why it happens:** Migrations are designed to run sequentially. Skipping or reordering causes FK constraint failures.

**How to avoid:** Run `npm run db:migrate` once. Drizzle Kit applies all pending migrations in order automatically. Do NOT cherry-pick migrations.

**Warning signs:** Error messages mentioning `column "X" does not exist` or `relation "X" does not exist`.

### Pitfall 2: Migration 0005 is Untracked in Git

**What goes wrong:** Migration 0005 (tags, bulk_operations, practice tables) exists on disk and in the journal but is untracked by git. If it gets lost, the migration history becomes inconsistent.

**Why it happens:** The file was generated during Phase 23/31 development but never committed.

**How to avoid:** Commit it as part of CLEAN-02 before running any new migrations. The Drizzle journal references it and expects it to exist.

**Warning signs:** `git status` showing `?? src/db/migrations/0005_careful_green_goblin.sql`.

### Pitfall 3: Practice Tables May Already Exist

**What goes wrong:** STATE.md says "Run `npm run db:push` or apply migration to create practice tables." Migration 0005 already contains the CREATE TABLE statements for practice tables. If someone ran `db:push` during development, the tables already exist in Neon. Running the migration would then fail with `relation already exists`.

**Why it happens:** Ambiguity between `db:push` (direct schema push, no migration tracking) and `db:migrate` (tracked migrations).

**How to avoid:** Check if migrations 0005-0010 have already been applied by querying `SELECT * FROM __drizzle_migrations ORDER BY created_at;` in Neon. If some are missing, `db:migrate` will apply only the unapplied ones. If tables exist but migration records don't, you may need to manually insert migration records or use `db:push` to sync.

**Warning signs:** Error `relation "practice_sets" already exists` during migration.

### Pitfall 4: opencc-js Import Style Changed

**What goes wrong:** The Context7 docs show TWO import patterns for opencc-js -- the simple `OpenCC.Converter()` API and the newer `ConverterFactory` with separate `Locale` imports. Using the wrong pattern causes runtime errors.

**Why it happens:** opencc-js 1.0.5 supports both patterns, but the simple `Converter()` API is the recommended one for most use cases.

**How to avoid:** Use the simple API: `import * as OpenCC from 'opencc-js'; const converter = OpenCC.Converter({ from: 'hk', to: 'cn' });`. Only use `ConverterFactory` if you need tree-shaking optimization.

**Warning signs:** `OpenCC.Converter is not a function` or `Cannot read properties of undefined`.

### Pitfall 5: Dictionary Schema Generated Column Compatibility

**What goes wrong:** The v6.0 research doc suggests `is_single_char BOOLEAN GENERATED ALWAYS AS (char_length(traditional) = 1) STORED` -- a Postgres generated column. Drizzle ORM supports `.generatedAlwaysAs()` but the generated SQL may need testing on Neon specifically.

**Why it happens:** Generated columns are Postgres 12+ feature. Neon supports them, but Drizzle's `generatedAlwaysAs()` output should be verified.

**How to avoid:** For Phase 44, skip the generated column. Use a regular boolean column populated during the Phase 45 seeding step. This is simpler and avoids a potential migration compatibility issue. The generated column is an optimization that can be added later.

**Warning signs:** Migration fails with `syntax error at or near "GENERATED"`.

### Pitfall 6: shadcn CLI May Overwrite Tailwind Config

**What goes wrong:** Running `npx shadcn@latest add popover` may attempt to modify `tailwind.config.ts` or `globals.css` if it detects a missing configuration.

**Why it happens:** The shadcn CLI checks for required configuration and may auto-add it.

**How to avoid:** The project already has a valid `components.json` with all aliases configured. The popover component should install cleanly as just a `.tsx` file in `src/components/ui/`. Review the diff after installation to ensure no unexpected changes.

**Warning signs:** Unexpected changes in `globals.css` or `tailwind.config.ts` after running the CLI.

### Dictionary Entries Schema

```typescript
// Source: Codebase pattern from src/db/schema/practice.ts + v6.0 research
import {
  pgTable,
  uuid,
  varchar,
  text,
  timestamp,
  integer,
  boolean,
  smallint,
  jsonb,
  pgEnum,
  index,
} from "drizzle-orm/pg-core";
import { sql } from "drizzle-orm";

// Source enum for dictionary entries
export const dictionarySourceEnum = pgEnum("dictionary_source", [
  "cedict",
  "canto",
  "both",
]);

export const dictionaryEntries = pgTable("dictionary_entries", {
  id: uuid("id").defaultRandom().primaryKey(),
  traditional: varchar("traditional", { length: 50 }).notNull(),
  simplified: varchar("simplified", { length: 50 }).notNull(),
  pinyin: varchar("pinyin", { length: 200 }).notNull(),        // "Zhong1 guo2" (numbered tones)
  pinyinDisplay: varchar("pinyin_display", { length: 200 }),    // "Zhong guo" (with tone marks)
  jyutping: varchar("jyutping", { length: 200 }),               // "zung1 gwok3" (nullable)
  definitions: text("definitions").array().notNull(),            // PostgreSQL text[]
  source: dictionarySourceEnum("source").notNull().default("cedict"),
  isSingleChar: boolean("is_single_char").notNull().default(false), // computed during seeding
  frequencyRank: integer("frequency_rank"),
  createdAt: timestamp("created_at").notNull().defaultNow(),
}, (table) => [
  index("dictionary_entries_traditional_idx").on(table.traditional),
  index("dictionary_entries_simplified_idx").on(table.simplified),
  index("dictionary_entries_pinyin_idx").on(table.pinyin),
]);
```
